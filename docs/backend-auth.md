# Backend Authentication Mechanics

This doc covers the procedural detail of the auth surface: password hashing, the account/login/select/switch orchestration, session rehydration, and the two abuse-mitigation layers (per-IP rate limiting and per-account login backoff). It is split from [backend.md](./backend.md) so the main doc keeps only the high-level auth decisions (JWT-not-sessions, token TTLs, the selected-player claim, token-as-sole-authority, one-active-session, backoff-not-lockout). Start there for the architecture; this file is the mechanics.

## Password hashing

Passwords use **PBKDF2-HMAC-SHA256** behind an `IPasswordHasher` abstraction. A hash is stored in a self-contained, tunable format carrying the iteration count and a per-hash salt (default 600k iterations, the OWASP floor); an application-wide pepper is folded in via HMAC and validated at startup so a missing pepper fails fast. Verification is constant-time, and a valid credential stored with an outdated work factor is transparently re-hashed on next login. An **unknown username** still runs equivalent derivation work via `IPasswordHasher.VerifyDummy` (against a fixed dummy hash), so the login response time does not reveal whether an account exists — a constant-time hash compare alone cannot cover the present-vs-absent-account branch.

## Account/login orchestration

Account creation and the login/select/refresh/logout flows are orchestrated by `AccountService` in the application layer; the controller is a thin HTTP adapter that maps the result and wires the request-scoped session (a presentation concern). The new-player defaults are a domain concern built by a `NewPlayerFactory` in `Game.Core`, so the application layer neither encodes the defaults nor constructs entity graphs. **Signup creates the account only — no character** (#1256): `CreateAccount` takes just credentials, so the class-selection surface is a single one (the create-character screen) rather than being duplicated into the auth form. **Class drives creation:** every character is created through `CreatePlayer` — the account's first on the select screen right after signup, and any additional ones — which takes a `classId`, resolves it against the class catalogue (rejecting an unknown or retired class), and hands the resolved `Class` to the factory, which produces the character's starter kit and starting attribute spread from class data. The starter kit also includes **equipped** starting items (a class's `StarterEquipment`, the one new-player mechanism with no prior precedent — players carried no inventory at creation before): the blueprint carries them as raw `(ItemId, EEquipmentSlot)` pairs and the data tier persists each as an unlocked-and-equipped row, so the weapon's innate `GrantedSkillId` comes online through the normal equip path at battle assembly. The slot↔category match is an authoring-time guard (`AdminClasses.SetStarterEquipment`), so creation trusts the kit as well-formed, exactly as it trusts the starter skills' `Player`-acquirable flag. The chosen class persists as the non-nullable `Player.ClassId` seam — permanent (a different archetype is a new character) and the anchor the durable class effects (locked-base fingerprint, signature passive) build on. Access-token issuance sits behind an `IAccessTokenService` abstraction so the application layer stays free of the JWT libraries, while the concrete implementation lives at the presentation edge alongside the bearer-validation pipeline.

**Login is split around player selection.** `Login` verifies credentials and returns the issued (pre-selection) token pair plus a summary of each character on the account — it binds no player. A second authenticated `SelectPlayer(playerId, refreshToken)` call then validates that the player belongs to the account (anti-cheat), loads it, binds the session, and **rotates the token pair** to carry the chosen player as a `player_id` claim (consuming the caller's refresh token only after the validations pass, so a rejected selection leaves it intact). The single-active-session takeover check (`ActiveSession`) is a per-player presence check, so it runs *after* selection. This is the seam for multiple characters per account (the in-game character switcher and lossless switching land on top of it).

**Switching characters in-game is lossless.** A `SwitchPlayer(playerId, refreshToken)` flow lets a logged-in account change its live character without re-logging in. Before binding the new character (via the same `SelectPlayer` path), it **credits the departed character** — the one the caller's token currently binds — for its elapsed idle time through the offline-rewards simulator, resolving its in-flight battle and **dropping the ≥5-minute floor** (that floor is a login-time concern; a deliberate switch credits any elapsed time). The credit runs over HTTP off the departed character's battle loop, so the client must tear its game socket down first; the target's ownership is validated up front so a rejected switch never mutates the departed character. That socket-teardown expectation is also **enforced server-side** via an atomic claim on the departed character's presence key (`SocketManagerService.TryClaimForSwitchCredit`/`ReleaseSwitchCreditClaim`): the credit is skipped (and logged) when the key is already held — a live socket, since its battle loop is already serializing its own saves under the per-socket command lock and running the off-lock HTTP credit then would reintroduce the lost-update race the lock exists to prevent — or another switch-away credit already in flight. A socket that registers while the claim is held defers behind it (bounded by the claim's short TTL) rather than immediately taking over, so a socket racing the credit's own start can no longer slip in between a presence check and the credit that follows it. The switch itself still proceeds. The entered character's own welcome-back catch-up then runs through the normal offline-rewards path on reconnect. This makes one-live-at-a-time outcome-equivalent to parallel progression — the parked character loses nothing while another is played. The switcher lists the account's characters to pick from via an authenticated `Players` endpoint (the login-time summaries are returned only by `Login`, and that handoff is one-shot, so an in-session switch re-fetches the current list).

**Characters are created post-signup with their own names, bounded by a cap.** Signup creates no character, so an authenticated account creates *every* character — its first and any additional ones — via `CreatePlayer(name, classId)` (HTTP, part of the pre-game select flow — it binds no session). Each character carries its own user-supplied name (validated by the `PlayerName` domain rule: trimmed, non-blank, ≤ the 20-char `Player.Name` limit, no control characters; names need not be globally unique). A **config-bound per-account cap** (`PlayerCreation:MaxPlayersPerAccount`, default 6) is enforced server-side as anti-cheat — the count check and insert run under a lock on the owning user row and commit in-tier, so concurrent creations can't slip past the cap.

**The class picker's data is delivered over HTTP, not the socket.** Choosing a class happens on the create-character screen, *before* a player is selected — but reference data (including the class catalogue) is socket-delivered and the socket handshake requires a selected player. So a bespoke `CharacterCreationData` endpoint serves the creatable classes (active only) as a self-contained view model with the starter skill/item **names resolved server-side**, deliberately distinct from the socket `GetClasses` reference catalogue: it requires authentication but no selected player, and pre-resolving the names means the picker needs no other reference set. (This is one of the few reads still on HTTP, like the `Players` list — see [HTTP vs WebSocket](./backend.md#http-vs-websocket-communication).)

At most one active account may hold a given username, enforced by a **partial unique index** (`Username WHERE ArchivedAt IS NULL`) so two concurrent creations can't both slip past the up-front availability check and insert duplicate active rows. The filter excludes archived users, preserving username reuse after archival. Account creation therefore commits its own insert in the data tier (rather than deferring to the per-request unit of work) so the index's unique-violation surfaces as a clean "username taken" result instead of a 500 raised after the action returns.

## Session rehydration & the session-cache boundary

The gameplay session cache (`Session_{userId}` → the in-flight `PlayerState`) is a volatile presentation convenience; it can be evicted (Redis flush, sliding-TTL lapse) or simply never established on a given instance, and it is **never** an authority on authentication (see [backend.md → Authentication](./backend.md#authentication) for the principle). A valid token with no cached session is *authenticated-but-uncached*: the session is rehydrated in memory by binding the player named in the token's selected-player claim instead of reporting "not logged in" (a pre-selection token with no such claim is authenticated but has no player, surfaced as a graceful error). The claim is authoritative — a cached session bound to a different character than the claim names is re-bound to the claim's player. `SessionService.Authenticated` is gated only on the token-derived `UserId`; a cache hit/miss governs `HasPlayerSession`, never authentication. The rehydration is **in-memory only** — it re-binds the request's `PlayerState` but does **not** write the session cache, since that write would land on the concurrent HTTP thread and player-state writes belong on the socket. The cache is (re)populated only by login (which establishes the binding before any socket exists) and by the socket's write-behind path.

`SessionLoaderMiddleware` records only the token-derived identity (the `UserId` and the selected-player claim) on every authenticated request; the `GetSession` read (and the rehydration on a miss) is deferred to the consumers that actually need player state — the socket handshake, the `Status`/`ActiveSession` auth endpoints, and `SwitchPlayer`'s departed-character credit — via `SessionInitializer`. So the rest of the authenticated HTTP surface (admin tooling, refresh, device-info) pays no per-request session-cache read.

## Auth endpoint rate limiting

The anonymous auth endpoints (login, refresh, account creation, logout) are throttled by a **per-client-IP fixed-window** rate limiter (ASP.NET Core's built-in `AddRateLimiter`), blunting credential stuffing, refresh-token brute force, and the PBKDF2 resource-exhaustion vector. The four endpoints share one `"auth"` policy, so the limit is a **combined per-IP budget** an attacker can't multiply across endpoints. The partition key is the client IP resolved the same trusted-proxy way as login tracking (`ClientIp.Resolve`), so a spoofed `X-Forwarded-For` can't shard a caller across partitions. A throttled request returns the standard `{ errorMessage }` envelope (429) with a `Retry-After` header. Unlike the CORS origins, the limits carry **safe positive defaults** so an unconfigured deployment is still protected — config only tunes them — and are validated as positive on start so an accidental zero fails fast rather than silently disabling the guard.

## Per-account login backoff

Layered on top of the per-IP limiter as defence-in-depth against a **slow, distributed** credential guess on one account that stays under any single IP's rate limit, `Login` applies a **per-account exponential backoff**. Consecutive failures are tracked in Redis keyed by a hash of the (case-sensitive) username and **reset on a successful credential verification**; the first few failures (a configurable threshold) are free, after which each further failure delays the next permitted attempt by a doubling window up to a **low cap**. It is deliberately a bounded **backoff, never a hard lockout** — a legitimate user is delayed by at most the cap — so an attacker who knows a username can only briefly slow the owner, not lock them out. A backed-off attempt is rejected **before** any database hit or PBKDF2 work (so spamming during an active window neither extends it nor costs server CPU) and surfaces as a 429 with a `Retry-After` header and a message distinct from invalid-credentials. The keying is per-account, not per-(account × IP): per-(account × IP) would let each distributed bot keep its own counter and add little over the IP limiter. Like the IP limiter the parameters are config-bound (`LoginBackoff` section) with safe defaults and validated as sane on start. The backoff arithmetic is a pure, unit-tested policy; the system clock (an injected `TimeProvider`) and the Redis update sit in a thin guard around it. Because the threat model is a **concurrent** distributed guess, that update must not lose increments — the guard records a failure through an **optimistic compare-and-set retry** (`ICacheService.CompareAndSet`, the conditional-write counterpart to `CompareAndDelete`): it re-reads and recomputes whenever a concurrent failure wins the swap, so the streak climbs at the true attempt rate rather than letting a plain read-modify-write clobber a racing increment.
