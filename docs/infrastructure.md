# Build & Test Infrastructure

This document covers the build/test/CI tooling that supports the backend — TypeScript client codegen, the integration-test container strategy, and the Dockerized API stack used for end-to-end runs. These are deliberately kept out of [backend.md](./backend.md) so that file stays focused on the application architecture itself.

## Standalone TypeScript codegen

The reflection-based code generator (`Game.Api/CodeGen/ApiCodeGenerator`) that emits the frontend's TypeScript API client only needs the API **assembly** — it inspects controller/socket-command type metadata and writes `.ts` files, with no database, cache, or DI graph involved. Historically it was reachable only as a side effect of running the API in `Development`, so regenerating types meant standing up the full web host (and its Postgres/Redis dependencies), which is awkward in CI and impossible in restricted environments.

It is now also runnable as a standalone command that exits before the web host is built:

```
dotnet run --project Game.Api -- codegen [outputDirectory]
```

- **`CodeGenCommand`** (intercepted at the top of `Startup.Main`, before `WebApplication.CreateBuilder`) builds a throwaway console `ILogger`, resolves the target directory, runs `ApiCodeGenerator.GenerateCode`, and returns. Because it short-circuits before host construction, it is independent of `ASPNETCORE_ENVIRONMENT` and never touches an out-of-process dependency. An explicit output directory may be passed; otherwise it defaults to the committed frontend location.
- **`CodeGenPaths`** centralizes path resolution so the dev-startup hook and the standalone command compute the same target directory. The default directory is found by walking up from the running assembly to the repository root (the directory containing `Game.sln`) and appending the canonical `UI/src/lib/api/types` segments. Resolving the root from the assembly rather than the current working directory keeps the command location-independent (the dev hook continues to derive the root from `ContentRootPath`).

The output is byte-identical to the dev-time generation, so CI can run the command and assert a clean `git diff` to catch api model/client drift.

### Generated domain constants (not just DTOs)

The codegen also emits a small amount of **static domain data** that the frontend would otherwise hand-mirror — closing a latent parity gap where a backend-only change could silently disagree with the frontend (#282). The first such case is the engine's static attribute-modifier table:

- **`StaticModifierWriter`** emits `STATIC_ATTRIBUTE_MODIFIERS` (`UI/src/lib/api/types/attribute-modifiers.ts`) from the domain's single ordered source of truth, `Game.Core.Attributes.Modifiers.StaticAttributeModifiers.All` (the same list `AttributeCollection` seeds itself from). A `Derived` modifier renders its `derivedSource`; the others omit it, so the table matches the frontend's discriminated `AttributeModifier` union (where a base modifier carries no `derivedSource`). The frontend file `$lib/battle/attribute-modifier.ts` keeps only that union — a value-free, frontend-specific type — and re-exports the generated table typed as `readonly AttributeModifier[]`, which makes the assignment a compile-time check that the generated values still conform to the union.
- **Domain enums opt into mirroring with `[ClientMirrored]`.** Some enums must reach `enums.ts` even though no API/socket DTO references them (the reflection walk only discovers types on the wire contract). Rather than a hardcoded list in the codegen, the enum declaration in `Game.Core` carries the `Game.Core.ClientMirroredAttribute` marker, and `ApiCodeGenerator.GetClientMirroredEnumDescriptors` scans the `Game.Core` assembly for every enum bearing it — so mirroring a new enum is a one-line attribute at its declaration (where the "this is client-visible" decision belongs), not an edit to the generator. The `EModifierType`/`EAttributeModifierSource` enums the static-modifier table references use this, as does `EEquipmentSlot` (formerly hand-declared in `UI/src/lib/engine/player/inventory-manager.ts`, now generated and re-exported from there). Marking an enum the walk already reaches is harmless — the interface writer de-duplicates by type name.
- **Gameplay constants are mirrored the same way, via `[ClientMirrored]` on a constants class.** `Game.Core.GameConstants` is the single backend declaration of the constants that must stay identical on both sides of the boundary — the battle tick size (`MS_PER_TICK`) and default max duration (`DEFAULT_MAX_BATTLE_MS`), the loadout cap (`MAX_SELECTED_SKILLS`), and the exp/stat-point curve (`EXP_PER_LEVEL`, `STAT_POINTS_PER_LEVEL`) — all of which were previously hand-copied as magic numbers on the frontend with no parity guard (#306). The same `[ClientMirrored]` attribute (its usage widened to classes as well as enums) marks the class; `ApiCodeGenerator.GetClientMirroredConstantFields` reflects each public `const` and `StaticModifierWriter`'s sibling `ConstantsWriter` emits them to `UI/src/lib/api/types/game-constants.ts` as `export const SCREAMING_SNAKE = value` (PascalCase → SCREAMING_SNAKE_CASE). The frontend consumes them directly from `$lib/api/types/game-constants` (the deep-path pattern the modifier table uses), deleting the duplicated literals — `logical-engine.ts`'s `tickSize`, `battler.ts`'s loadout cap, `player-manager.ts`'s exp curve, and the parity tests' `msPerTick`. Because the cap is one global constant, `MaxSelectedSkills` was also **dropped from the `PlayerData` wire payload** (redundant once generated) and the skills-selection UI now reads the generated `MAX_SELECTED_SKILLS` rather than a per-player wire field.
- **`ERole` is the deliberate exception and stays hand-maintained.** The generated TS enums are numeric (matching the backend), but the frontend `ERole` (`UI/src/lib/api/roles.ts`) is **string-keyed** (`Admin = 'Admin'`) because the JWT `role` claim carries the role *name*, which is what `hasRole` compares against. A generated numeric `ERole` would not serve that name-based check, so `ERole` is intentionally not `[ClientMirrored]` and the string-keyed frontend copy is kept and documented as the exception — the lower-friction choice over teaching codegen to emit a name-valued variant for a single enum.

Because all of this lands under `UI/src/lib/api/types`, the existing CI codegen-drift check already guards it: a retune of a coefficient (or an enum value) that isn't regenerated fails the `git diff`, so the cross-implementation parity no longer rests on a human keeping the two files in step.

## Integration-Test Containers in Constrained Environments

Integration tests get a real PostgreSQL + Redis via **Testcontainers** (`PostgresContainerFixture`, `RedisContainerFixture`, composed by `IntegrationTestContainers`). Testcontainers depends on Docker **bridge networking**, which is unavailable in constrained sandboxes such as the Claude Code web environment (old kernel, no iptables, no usable bridge). There, Testcontainers cannot create networks or map ports and every integration test fails before it runs. See [anthropics/claude-code#29515](https://github.com/anthropics/claude-code/issues/29515).

The workaround keeps the normal path untouched and adds an opt-in reuse path:

- **`.claude/hooks/session-start.sh`** runs at Claude Code session start. When Docker is fully functional — detected by an **`overlay2`** storage driver plus bridge networking — it is a **no-op** and removes any stale marker, so local dev and CI keep using Testcontainers. (The guard checks `overlay2` specifically: a sandbox-compatible `dockerd` the hook itself started uses `vfs`, which must not be mistaken for a real Docker install.) Otherwise it starts a sandbox-compatible `dockerd` (`--storage-driver=vfs --iptables=false --ip6tables=false`) and launches PostgreSQL and Redis with **host networking** on fixed ports (5499 / 6399, using the same images as the fixtures). If the images can't be pulled or started (e.g. Docker Hub rate limits with no cached image), it falls back to **natively-installed** PostgreSQL and Redis, creating the test user/database as needed. Either way it writes a `.container-info.json` marker at the repo root. It is best-effort and never fails the session.
- **`PreexistingContainerInfo.TryLoad()`** walks up from the test assembly's base directory to find that marker. When present, `PostgresContainerFixture`/`RedisContainerFixture` reuse those connection strings and skip Testcontainers entirely (and don't dispose containers they don't own). The Postgres fixture still applies EF migrations — idempotent against a reused database.

**Why a marker file rather than environment variables:** the hook runs in a separate process at session start, so its environment doesn't propagate to later `dotnet test` invocations; a file at a well-known path is the portable handoff. This mirrors the `.container-info.json` pattern from the upstream issue.

**Isolation:** the reuse path shares one PostgreSQL database and one Redis instance across all integration tests. _Within_ an assembly this is safe because every integration test runs in the single `Integration` xUnit collection (so they are serialized) and `ApplicationIntegrationTestBase.CleanupAsync` truncates player/game data and flushes Redis before each test. _Across_ assemblies it is not: `dotnet test <solution>` runs `Game.Api.Tests` and `Game.Application.Tests` as parallel processes, so their per-test truncate/flush would intermittently corrupt each other against the shared services. To prevent this, in reuse mode `IntegrationTestContainers` acquires a **cross-process file lock** (`CrossProcessLock` — a `FileShare.None` OS lock on a fixed file under the temp directory) for the lifetime of the collection fixture, serializing the integration suites across assemblies. Because the lock is an OS file handle, a crashed run releases it automatically (no stale lock). Normal (Testcontainers) runs skip the lock entirely — each assembly gets its own isolated containers and parallelizes freely.

## Dockerized API stack for end-to-end (Playwright) runs

`docker-compose.yml` (repo root) stands up the **Game.Api** service plus its PostgreSQL and Redis dependencies so the frontend's Playwright suite can run a real backend in CI and in the constrained Claude sandbox. It is the e2e counterpart to the integration-test fixtures above (which only provide the database/cache, not a running API).

- **Scope is API + Postgres + Redis only.** The SvelteKit frontend is deliberately excluded: Playwright already starts the dev server itself (`playwright.config.ts` `webServer`), and Vite proxies `/api` + `/socket` to the API on `localhost:5008`. The compose stack just has to make the API reachable there. The CI `test-e2e` job is a **matrix across `chromium`/`firefox`/`webkit`**: each browser runs as its own parallel job that brings the stack up, waits for the API to answer, and runs `playwright test --project=<browser>`. Running each project as its own invocation is deliberate — a single invocation covering all three projects intermittently leaves a browser worker hung at end-of-run teardown, which Playwright only force-kills after a ~5-minute grace; one project per invocation tears down cleanly. The matrix gives that clean teardown _and_ cuts wall-clock to roughly a single browser's runtime (an earlier version ran the three invocations serially in a shell loop). Locally, `npm run test:e2e` still runs every project in one invocation — use `npx playwright test --project=chromium` (etc.) to avoid the teardown delay.
- **Each matrix job builds the API image with `docker compose up -d --build`.** Because the matrix jobs run in parallel (and CI minutes are free on this public repo), the redundant per-job build stays off the serial critical path. `Game.Api/Dockerfile` restores from the `.csproj` manifests before `COPY . .` — the standard .NET layering that lets Docker's local cache reuse the NuGet restore layer on incremental rebuilds. _An earlier revision of this job built the image via buildx with a GitHub Actions layer cache (`type=gha`) and had compose reuse the loaded image; measured runs showed the cache-blob transfer plus image-load overhead was slower than a plain build on the hosted runners, so it was dropped in favour of simplicity._
- **Host networking, not bridge.** Every service uses `network_mode: host`. The remote/restricted sandbox runs a `dockerd` with `--iptables=false` and no usable bridge (see the constrained-environment section above and `.claude/hooks/session-start.sh`), so bridge networks and published ports don't work there. Host networking keeps everything on `localhost`, which works in that sandbox and on GitHub Actions Linux runners alike. The images (`postgres:18-alpine`, `redis:7-alpine`) match the test fixtures so the sandbox's pre-pulled images are reused. Ports default to dedicated values (5544 / 6544) to avoid colliding with the integration-test services (5499 / 6399) or a native install. The trade-off: host networking requires a Linux Docker host — use `docker-compose.local.yml` (see below) on Docker Desktop (Windows/Mac).
- **Migrations are applied on startup via a config flag, decoupled from the dev codegen.** The API previously only migrated (and ran the TypeScript client codegen, which writes into the frontend source tree) when `IsDevelopment()`. `DataAccessOptions.MigrateOnStartup` now drives migration independently: when `true`, `Startup` applies EF migrations on boot in any environment (the migrations carry only the **intrinsic** enum-backed seed data — see [Reference Data](backend.md#reference-data)), while the codegen stays Development-only. The compose API service runs as `Production` with `DataAccessOptions__MigrateOnStartup=true`, so it converges an empty database to the current schema without the dev-only tooling. Migration runs before Kestrel begins listening, so any HTTP response from the API doubles as a readiness signal (the CI job polls `/api/Login/Status` for any status code, including the expected `401`).
- **`docker-compose.local.yml` — bridge-networked variant for Docker Desktop (Windows/Mac).** `docker-compose.yml` cannot run on Docker Desktop because host networking is Linux-only. `docker-compose.local.yml` (repo root) is a drop-in alternative that uses standard Docker bridge networking: each service gets a network-scoped hostname matching its service name (`postgres`, `redis`), ports are published to localhost, and the API's connection strings reference those hostnames instead of `localhost`. Default published ports are 5432 (PostgreSQL), 6379 (Redis), and 5008 (API — the Vite proxy target). Run it with `docker compose -f docker-compose.local.yml up -d --build`; the seeding step is the same as above (replace `docker compose` with `docker compose -f docker-compose.local.yml`).
- **Static reference data is seeded out-of-band by `e2e-seed.sql`.** Migrations seed only intrinsic reference data; the **static** content (Skills, Enemies, Zones, …) is normally authored through the admin tools and is absent from a freshly-migrated database. That matters because `LoginController.CreateAccount` gives every new player the starter skills `0/1/2`, so without those `Skill` rows the very first signup fails on `FK_PlayerSkills_Skills_SkillId` (HTTP 500) and the whole Playwright suite — which logs in for every test — can't get past the login page. The `e2e-seed.sql` script (repo root) seeds the minimal, coherent slice the suite needs (the three starter skills, plus one zone populated with enemies so the Enemies admin catalogue has rows and the fight screen is playable, and a dedicated zone boss — deliberately weak so a fresh player wins quickly — so the "Challenge Boss" flow can be exercised end-to-end). Rows are ordered by dependency (Enemies before the Zones that reference them via `BossEnemyId`). The CI job applies it with `docker compose exec -T postgres psql … -f - < e2e-seed.sql` **after** the readiness check (so migrations have run), then **restarts the API** so it reloads its reference-data caches from the now-seeded database before Playwright starts. The restart is needed because the API eagerly loads those caches at startup (#357): the seed lands after that first boot, so the cached reads (e.g. `Skills.GetSkill` during login) would otherwise never see it. In production this static content already exists before the app starts, or is authored through the admin tools, which invalidate the caches — the e2e stack is the only place it is injected directly post-boot, hence the restart. Every statement is idempotent (`ON CONFLICT DO NOTHING`), and because the script is deliberately decoupled from the app, local and production databases are never touched by it. Keeping the seed in SQL (rather than EF `HasData`) honors the intrinsic-vs-static split: the app and its migrations stay free of authored game content.

## Trusted reverse proxies (`X-Forwarded-For`)

The login-tracking IP (an audit surface) is read from `HttpContext.Connection.RemoteIpAddress`, never by hand-parsing `X-Forwarded-For`. To stop a direct client forging that IP, the forwarded-headers middleware is enabled **only when a trusted proxy is configured**, and rewrites the remote IP solely for requests whose socket peer is on the allowlist.

- Configure proxies under the **`ForwardedHeaders`** section: `KnownProxies` (a list of individual IPs, e.g. `"10.0.0.1"`, `"::1"`) and `KnownNetworks` (a list of CIDR ranges, e.g. `"10.0.0.0/8"`). Both default to **empty**.
- **The default is "trust nothing":** with neither list set, the middleware is not added at all and a spoofed `X-Forwarded-For` is ignored — the recorded IP is the socket peer. (An empty allowlist would otherwise make ASP.NET Core's middleware *skip* its known-proxy check and trust the header unconditionally, so it is deliberately left off rather than configured-but-empty.)
- A real deployment behind a load balancer / ingress must set the relevant proxy IPs or CIDRs so the player's true client IP is recorded.

## Battle-simulation performance tests

The backend re-simulates **every** battle after the client reports it (the anti-cheat replay — see [game-design.md](./game-design.md#battle-mechanics)), so the battle hot path must stay cheap as new mechanics are added. `Game.Core.Tests/Battle/Performance` guards that with tests that measure **relatively** rather than against absolute wall-clock times, because raw machine speed varies wildly between a dev box and a CI runner (#283).

- **Scaling gates are the primary, machine-independent guard.** Each gate compares a simple baseline battle against a more complex one that differs in exactly one dimension by a known factor — skill count (`SkillCountScaling_StaysWithinLinearBounds`) and damage-multiplier count (`DamageMultiplierScaling_StaysWithinLinearBounds`). Both battles run on the same machine in the same process, so the **ratio cancels out machine speed**; what it actually asserts is that added work scales roughly *linearly*, not super-linearly. Costs are normalised to time-per-tick (the battles are built to run to the tick cap, so the tick count is fixed and known), and the budget is a generous multiple of the work factor (`work × 2.0`): an accidental O(n²) in a per-skill or per-multiplier loop lands far above it while ordinary CI noise stays far below. This is the answer to "systems vary in performance" — the gates never reference an absolute time.
- **One coarse absolute backstop catches uniform regressions** (`WorstCaseBattle_SimulatesWellWithinRealtimeBudget`). A ratio cannot see a regression that inflates *every* battle equally (e.g. a new per-tick allocation), so a single worst-case battle (8 skills/side, 8 multipliers each, full default tick cap) is checked against a deliberately huge absolute ceiling. It is the one machine-dependent gate; its margin is an order of magnitude (a slow runner simulates the battle in tens of ms against a 500 ms ceiling), so a ~10× uniform regression trips it but contention cannot. Subtler uniform regressions are meant to be spotted by watching the figures every test logs, not caught by this gate.
- **Flake-resistance is built into the harness** (`PerformanceMeasurement`): a warm-up phase keeps JIT/cache effects out of the measured region; combatant construction (battlers are stateful and must be rebuilt per run) happens *untimed*; each sample times a batch and the **minimum** across samples is the reported figure, since timing noise can only ever add time. Every measurement also logs its raw cost/ratio so thresholds can be tightened from data over time.
- **They run on the PR gate but carry an escape hatch.** The tests are tagged `[Trait("Category", "Performance")]`; if runner contention ever makes even the coarse backstop flake, excluding them is one flag — `dotnet test … -- --filter-not-trait "Category=Performance"` (the flag is passed to the xUnit v3 native runner the suite uses, after the `--` separator) — with no code change.
- **Intentionally backend-only.** Unlike the battle-simulation *parity* tests, these assert nothing about correctness or determinism (only cost), so — although the simulation itself is mirrored on the client — there is deliberately **no frontend mirror** to keep in sync. JS-vs-C# performance is not comparable anyway, and the client is not on the anti-cheat critical path.

## Code coverage

Both tiers gate test coverage with a **per-area ratchet**: each area carries a floor seeded at its current level that only ever rises, so a regression fails the build while no global percentage has to be hand-maintained. The design deliberately leans on the existing architecture rather than trying to detect "is this real logic?" line by line — strict floors where logic concentrates, *measure-only* elsewhere, and explicit, reviewable exclusions for no-logic code (generated clients, DTOs/entities, EF migrations, styles, barrel files). An exclusion is a visible line in a diff, never a silent gap.

### Frontend (Vitest)

Per-folder floors live in `coverageThresholds` (`UI/coverage.config.ts`, shared with `UI/vite.config.ts`) as glob keys — high on the logic layers (`src/lib/battle`, `src/stores`, `src/lib/{common,api,card-game,engine}`) and at-current on the display-heavy `src/routes`/`src/components`, so those can't backslide without forcing tests for pure-display markup. Vitest enforces the floors itself: `npm run test:unit:coverage` locally, and `test:unit:ci` (run by the `test-frontend` CI job) carries `--coverage`, so a breach fails the job with no extra workflow wiring. No-logic code is excluded outright (generated `lib/api/types`, barrel `index.ts`, `*.scss`, test scaffolding); `coverageConfigDefaults.exclude` is spread because Vitest *replaces* rather than extends the default exclude list. A unit test (`src/tests/coverage-config.test.ts`) asserts every source file is gated by exactly one area or an explicit exclusion, so a new top-level folder can't slip in ungated. Re-seed floors after coverage rises with `node UI/scripts/coverage-floors.mjs`.

### Backend (dotnet-coverage + ReportGenerator)

`build/coverage.ps1` is the single command (local and CI; runs on Windows PowerShell 5.1 and `pwsh` 7): it collects **one merged Cobertura across the whole solution's test run** via `dotnet-coverage`, builds the ReportGenerator report (text + JSON summary + HTML), then runs `build/coverage-gate.ps1`. The `test-backend` CI job invokes it as `./build/coverage.ps1 -NoBuild` (reusing the build step's binaries); a test failure *or* a gate breach fails the step.

- **Why dotnet-coverage, not the in-process extension.** `dotnet-coverage` profiles the test host out-of-band, so it is immune to the version coupling between `Microsoft.Testing.Extensions.CodeCoverage` and the `xunit.v3` / Microsoft.Testing.Platform pair — the in-process extension throws `TypeLoadException` against the MTP version `xunit.v3` pins. The tool (and ReportGenerator) are pinned in `.config/dotnet-tools.json` (`dotnet tool restore`).
- **Why the whole-solution merge matters.** `Game.Core` reads ~85% from its own unit tests but ~97% once the integration suites' coverage is folded in, so the gate is only honest against the merged run.
- **Gated vs measure-only** (`build/coverage-floors.json`): gated — `Game.Core` (strict), `Game.Application` (ratchet), and `Game.DataAccess` (ratchet); measure-only — `Game.Api`, `Game.Infrastructure`, `Game.Abstractions`. `Game.DataAccess` is gated because it holds the highest-risk-for-data-loss logic (write-behind sync, retry policy, the change-set/reconciler stack) and the integration suites already cover it densely, so its floor is seeded at-current to block regressions even though it has no dedicated unit-test project. The measure-only set is the thin HTTP adapters, the out-of-process-coupled infra tier, and the interface-and-contract assembly — code the [testing guidelines](backend.md#testing-guidelines) cover through integration tests rather than unit-worthy logic, so it is reported but not gated. Exclusions are centralized in the ReportGenerator filters (test/support assemblies, compiler-generated regex, EF migrations, `obj/`). Re-seed floors from `coverage/backend-report/Summary.json` after coverage rises.
