# Build & Test Infrastructure

This document covers the build/test/CI tooling that supports the backend â€” TypeScript client codegen, the integration-test container strategy, and the Dockerized API stack used for end-to-end runs. These are deliberately kept out of [backend.md](./backend.md) so that file stays focused on the application architecture itself.

## Standalone TypeScript codegen

The reflection-based code generator (`Game.Api/CodeGen/ApiCodeGenerator`) that emits the frontend's TypeScript API client only needs the API **assembly** â€” it inspects controller/socket-command type metadata and writes `.ts` files, with no database, cache, or DI graph involved. Historically it was reachable only as a side effect of running the API in `Development`, so regenerating types meant standing up the full web host (and its Postgres/Redis dependencies), which is awkward in CI and impossible in restricted environments.

It is now also runnable as a standalone command that exits before the web host is built:

```
dotnet run --project Game.Api -- codegen [outputDirectory]
```

- **`CodeGenCommand`** (intercepted at the top of `Startup.Main`, before `WebApplication.CreateBuilder`) builds a throwaway console `ILogger`, resolves the target directory, runs `ApiCodeGenerator.GenerateCode`, and returns. Because it short-circuits before host construction, it is independent of `ASPNETCORE_ENVIRONMENT` and never touches an out-of-process dependency. An explicit output directory may be passed; otherwise it defaults to the committed frontend location.
- **`CodeGenPaths`** centralizes path resolution so the dev-startup hook and the standalone command compute the same target directory. The default directory is found by walking up from the running assembly to the repository root (the directory containing `Game.sln`) and appending the canonical `UI/src/lib/api/types` segments. Resolving the root from the assembly rather than the current working directory keeps the command location-independent (the dev hook continues to derive the root from `ContentRootPath`).

The output is byte-identical to the dev-time generation, so CI can run the command and assert a clean `git diff` to catch api model/client drift.

## Integration-Test Containers in Constrained Environments

Integration tests get a real PostgreSQL + Redis via **Testcontainers** (`PostgresContainerFixture`, `RedisContainerFixture`, composed by `IntegrationTestContainers`). Testcontainers depends on Docker **bridge networking**, which is unavailable in constrained sandboxes such as the Claude Code web environment (old kernel, no iptables, no usable bridge). There, Testcontainers cannot create networks or map ports and every integration test fails before it runs. See [anthropics/claude-code#29515](https://github.com/anthropics/claude-code/issues/29515).

The workaround keeps the normal path untouched and adds an opt-in reuse path:

- **`.claude/hooks/session-start.sh`** runs at Claude Code session start. When Docker is fully functional â€” detected by an **`overlay2`** storage driver plus bridge networking â€” it is a **no-op** and removes any stale marker, so local dev and CI keep using Testcontainers. (The guard checks `overlay2` specifically: a sandbox-compatible `dockerd` the hook itself started uses `vfs`, which must not be mistaken for a real Docker install.) Otherwise it starts a sandbox-compatible `dockerd` (`--storage-driver=vfs --iptables=false --ip6tables=false`) and launches PostgreSQL and Redis with **host networking** on fixed ports (5499 / 6399, using the same images as the fixtures). If the images can't be pulled or started (e.g. Docker Hub rate limits with no cached image), it falls back to **natively-installed** PostgreSQL and Redis, creating the test user/database as needed. Either way it writes a `.container-info.json` marker at the repo root. It is best-effort and never fails the session.
- **`PreexistingContainerInfo.TryLoad()`** walks up from the test assembly's base directory to find that marker. When present, `PostgresContainerFixture`/`RedisContainerFixture` reuse those connection strings and skip Testcontainers entirely (and don't dispose containers they don't own). The Postgres fixture still applies EF migrations â€” idempotent against a reused database.

**Why a marker file rather than environment variables:** the hook runs in a separate process at session start, so its environment doesn't propagate to later `dotnet test` invocations; a file at a well-known path is the portable handoff. This mirrors the `.container-info.json` pattern from the upstream issue.

**Isolation:** the reuse path shares one PostgreSQL database and one Redis instance across all integration tests. _Within_ an assembly this is safe because every integration test runs in the single `Integration` xUnit collection (so they are serialized) and `ApplicationIntegrationTestBase.CleanupAsync` truncates player/game data and flushes Redis before each test. _Across_ assemblies it is not: `dotnet test <solution>` runs `Game.Api.Tests` and `Game.Application.Tests` as parallel processes, so their per-test truncate/flush would intermittently corrupt each other against the shared services. To prevent this, in reuse mode `IntegrationTestContainers` acquires a **cross-process file lock** (`CrossProcessLock` â€” a `FileShare.None` OS lock on a fixed file under the temp directory) for the lifetime of the collection fixture, serializing the integration suites across assemblies. Because the lock is an OS file handle, a crashed run releases it automatically (no stale lock). Normal (Testcontainers) runs skip the lock entirely â€” each assembly gets its own isolated containers and parallelizes freely.

## Dockerized API stack for end-to-end (Playwright) runs

`docker-compose.yml` (repo root) stands up the **Game.Api** service plus its PostgreSQL and Redis dependencies so the frontend's Playwright suite can run a real backend in CI and in the constrained Claude sandbox. It is the e2e counterpart to the integration-test fixtures above (which only provide the database/cache, not a running API).

- **Scope is API + Postgres + Redis only.** The SvelteKit frontend is deliberately excluded: Playwright already starts the dev server itself (`playwright.config.ts` `webServer`), and Vite proxies `/api` + `/socket` to the API on `localhost:5008`. The compose stack just has to make the API reachable there. The CI `test-e2e` job is a **matrix across `chromium`/`firefox`/`webkit`**: each browser runs as its own parallel job that brings the stack up, waits for the API to answer, and runs `playwright test --project=<browser>`. Running each project as its own invocation is deliberate â€” a single invocation covering all three projects intermittently leaves a browser worker hung at end-of-run teardown, which Playwright only force-kills after a ~5-minute grace; one project per invocation tears down cleanly. The matrix gives that clean teardown _and_ cuts wall-clock to roughly a single browser's runtime (an earlier version ran the three invocations serially in a shell loop). Locally, `npm run test:e2e` still runs every project in one invocation â€” use `npx playwright test --project=chromium` (etc.) to avoid the teardown delay.
- **Each matrix job builds the API image with `docker compose up -d --build`.** Because the matrix jobs run in parallel (and CI minutes are free on this public repo), the redundant per-job build stays off the serial critical path. `Game.Api/Dockerfile` restores from the `.csproj` manifests before `COPY . .` â€” the standard .NET layering that lets Docker's local cache reuse the NuGet restore layer on incremental rebuilds. _An earlier revision of this job built the image via buildx with a GitHub Actions layer cache (`type=gha`) and had compose reuse the loaded image; measured runs showed the cache-blob transfer plus image-load overhead was slower than a plain build on the hosted runners, so it was dropped in favour of simplicity._
- **Host networking, not bridge.** Every service uses `network_mode: host`. The remote/restricted sandbox runs a `dockerd` with `--iptables=false` and no usable bridge (see the constrained-environment section above and `.claude/hooks/session-start.sh`), so bridge networks and published ports don't work there. Host networking keeps everything on `localhost`, which works in that sandbox and on GitHub Actions Linux runners alike. The images (`postgres:18-alpine`, `redis:7-alpine`) match the test fixtures so the sandbox's pre-pulled images are reused. Ports default to dedicated values (5544 / 6544) to avoid colliding with the integration-test services (5499 / 6399) or a native install. The trade-off: host networking requires a Linux Docker host â€” use `docker-compose.local.yml` (see below) on Docker Desktop (Windows/Mac).
- **Migrations are applied on startup via a config flag, decoupled from the dev codegen.** The API previously only migrated (and ran the TypeScript client codegen, which writes into the frontend source tree) when `IsDevelopment()`. `DataAccessOptions.MigrateOnStartup` now drives migration independently: when `true`, `Startup` applies EF migrations on boot in any environment (the migrations carry only the **intrinsic** enum-backed seed data â€” see [Reference Data](#reference-data)), while the codegen stays Development-only. The compose API service runs as `Production` with `DataAccessOptions__MigrateOnStartup=true`, so it converges an empty database to the current schema without the dev-only tooling. Migration runs before Kestrel begins listening, so any HTTP response from the API doubles as a readiness signal (the CI job polls `/api/Login/Status` for any status code, including the expected `401`).
- **`docker-compose.local.yml` â€” bridge-networked variant for Docker Desktop (Windows/Mac).** `docker-compose.yml` cannot run on Docker Desktop because host networking is Linux-only. `docker-compose.local.yml` (repo root) is a drop-in alternative that uses standard Docker bridge networking: each service gets a network-scoped hostname matching its service name (`postgres`, `redis`), ports are published to localhost, and the API's connection strings reference those hostnames instead of `localhost`. Default published ports are 5432 (PostgreSQL), 6379 (Redis), and 5008 (API â€” the Vite proxy target). Run it with `docker compose -f docker-compose.local.yml up -d --build`; the seeding step is the same as above (replace `docker compose` with `docker compose -f docker-compose.local.yml`).
- **Static reference data is seeded out-of-band by `e2e-seed.sql`.** Migrations seed only intrinsic reference data; the **static** content (Skills, Enemies, Zones, â€¦) is normally authored through the admin tools and is absent from a freshly-migrated database. That matters because `LoginController.CreateAccount` gives every new player the starter skills `0/1/2`, so without those `Skill` rows the very first signup fails on `FK_PlayerSkills_Skills_SkillId` (HTTP 500) and the whole Playwright suite â€” which logs in for every test â€” can't get past the login page. The `e2e-seed.sql` script (repo root) seeds the minimal, coherent slice the suite needs (the three starter skills, plus one zone populated with enemies so the Enemies admin catalogue has rows and the fight screen is playable, and a dedicated zone boss â€” deliberately weak so a fresh player wins quickly â€” so the "Challenge Boss" flow can be exercised end-to-end). Rows are ordered by dependency (Enemies before the Zones that reference them via `BossEnemyId`). The CI job applies it with `docker compose exec -T postgres psql â€¦ -f - < e2e-seed.sql` **after** the readiness check (so migrations have run) and before Playwright starts. Seeding once the API is already listening is fine because the in-memory reference caches are populated lazily on first request, not at startup: the readiness probe is an unauthenticated `/api/Login/Status` that `401`s before touching them, so the first real population happens during the test run (after the seed), and the admin tools fetch with `refreshCache=true` regardless. Every statement is idempotent (`ON CONFLICT DO NOTHING`), and because the script is deliberately decoupled from the app, local and production databases are never touched by it. Keeping the seed in SQL (rather than EF `HasData`) honors the intrinsic-vs-static split: the app and its migrations stay free of authored game content.

## Battle-simulation performance tests

The backend re-simulates **every** battle after the client reports it (the anti-cheat replay â€” see [game-design.md](./game-design.md#battle-mechanics)), so the battle hot path must stay cheap as new mechanics are added. `Game.Core.Tests/Battle/Performance` guards that with tests that measure **relatively** rather than against absolute wall-clock times, because raw machine speed varies wildly between a dev box and a CI runner (#283).

- **Scaling gates are the primary, machine-independent guard.** Each gate compares a simple baseline battle against a more complex one that differs in exactly one dimension by a known factor â€” skill count (`SkillCountScaling_StaysWithinLinearBounds`) and damage-multiplier count (`DamageMultiplierScaling_StaysWithinLinearBounds`). Both battles run on the same machine in the same process, so the **ratio cancels out machine speed**; what it actually asserts is that added work scales roughly *linearly*, not super-linearly. Costs are normalised to time-per-tick (the battles are built to run to the tick cap, so the tick count is fixed and known), and the budget is a generous multiple of the work factor (`work Ă— 2.0`): an accidental O(nÂ˛) in a per-skill or per-multiplier loop lands far above it while ordinary CI noise stays far below. This is the answer to "systems vary in performance" â€” the gates never reference an absolute time.
- **One coarse absolute backstop catches uniform regressions** (`WorstCaseBattle_SimulatesWellWithinRealtimeBudget`). A ratio cannot see a regression that inflates *every* battle equally (e.g. a new per-tick allocation), so a single worst-case battle (8 skills/side, 8 multipliers each, full default tick cap) is checked against a deliberately huge absolute ceiling. It is the one machine-dependent gate; its margin is an order of magnitude (a slow runner simulates the battle in tens of ms against a 500 ms ceiling), so a ~10Ă— uniform regression trips it but contention cannot. Subtler uniform regressions are meant to be spotted by watching the figures every test logs, not caught by this gate.
- **Flake-resistance is built into the harness** (`PerformanceMeasurement`): a warm-up phase keeps JIT/cache effects out of the measured region; combatant construction (battlers are stateful and must be rebuilt per run) happens *untimed*; each sample times a batch and the **minimum** across samples is the reported figure, since timing noise can only ever add time. Every measurement also logs its raw cost/ratio so thresholds can be tightened from data over time.
- **They run on the PR gate but carry an escape hatch.** The tests are tagged `[Trait("Category", "Performance")]`; if runner contention ever makes even the coarse backstop flake, excluding them is one flag â€” `dotnet test â€¦ -- -trait- "Category=Performance"` (the xUnit v3 native runner the suite uses) â€” with no code change.
- **Intentionally backend-only.** Unlike the battle-simulation *parity* tests, these assert nothing about correctness or determinism (only cost), so â€” although the simulation itself is mirrored on the client â€” there is deliberately **no frontend mirror** to keep in sync. JS-vs-C# performance is not comparable anyway, and the client is not on the anti-cheat critical path.
