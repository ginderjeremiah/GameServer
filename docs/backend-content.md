# Content Pipeline (export → seed → lint → calibration)

The static content graph is authored at runtime in the Workbench and lives only in Postgres; this pipeline mirrors it to source control, reseeds fresh databases from that mirror, lint-checks the whole graph in CI, and backs the tuning reports. Split out of [backend.md](./backend.md#reference-data) — read that first for the reference-data architecture (intrinsic vs static sets, id-as-index caches, retirement) these mechanisms build on.

## Content export (source-controlled snapshot)

A freshly-migrated DB has no authored content. Authoring model **(A) DB-primary + one-way export** (spike #1390) mirrors the authored graph to source-controlled JSON so it can be diffed, reviewed, reproduced, and rolled back. `IContentExporter` (`Game.Application/Content`) serializes each authored reference set — `skills`, `tags`, `items`, `item-mods`, `enemies`, `challenges`, `zones`, `classes`, `paths`, `proficiencies`, `skill-recipes`, `lessons` (Path and Proficiency are distinct id-spaces, so each gets its own file) — to one canonical JSON file under `content/`, reading the same nested **read contracts** the client and Workbench receive (not the entities). `tags` is the one set not held in an in-memory cache (read straight from the DB, hence the export is async) and the one with a **non-zero-based** identity resolved by lookup rather than index; its `TagCategory` parent is intrinsic (enum-seeded via `HasData`), so it stays out of the export like the other intrinsic sets. The format is tuned for review-friendly, byte-stable diffs: arrays id-ordered, child collections in a declared stable order, enums as names, `RetiredAt` as a UTC ISO-8601 instant, fixed 2-space/LF formatting, retired rows included. The JSON is **generated, never hand-edited**.

A **CI drift guard** (`ContentExportDriftGuardTests`) seeds a freshly-migrated DB from the committed `content/*.json` through the seeder below, re-derives the export, and asserts byte-equality — a round-trip, so a hand-edit, a stale snapshot, or a seeder/exporter asymmetry fails the build. To land a content change, run the exporter and commit the diff — set `CONTENT_EXPORT_REGEN=1` when running that test to rewrite the files.

## Seeding a fresh DB from the export

The reverse direction (spike #1390, decision 3): `IContentSeeder` (`Game.DataAccess/Content`) reconstructs the whole static graph on a freshly-migrated database from the export, so dev / CI / recovery get a real content baseline instead of an empty catalogue. It is a **dedicated bulk seeder, not a replay through the admin repos** — it writes rows directly with their **explicit ids** (a metadata-driven `EntityRowInserter` generates the INSERTs so it can preserve id `0`, which EF's normal insert path treats as store-generated for a zero-based identity column), preserves `RetiredAt`, inserts in FK/dependency order, and advances each identity sequence past its seeded max. It is **idempotent** — it no-ops when the database already holds content (Skills present), so it never overwrites an authored dev DB. `IContentImportReader` (`Game.Application/Content`) deserializes the export through the exporter's canonical JSON options; the files are copied next to the app so the seeder resolves them from `AppContext.BaseDirectory/content`. Startup runs it after migrating and before the eager cache load (so the caches build from the seed), gated on `DataAccessOptions.SeedContentOnStartup` — always on in Development, opt-in elsewhere (the Docker/e2e stack enables it), off in production unless explicitly enabled for a recovery. This **subsumes the old `e2e-seed.sql`**: the e2e stack now seeds itself at boot, and only the e2e-only admin-role fixture (`e2e-admin-provision.sql`) is applied separately by CI.

## Progression-graph lint (whole-graph reachability)

Per-entity admin saves validate one side of a reference at save time; nothing re-checks the **whole graph**, where the dangling end lives on the *other* record and a reference valid at authoring can be silently broken later by **retiring its target**. `IProgressionGraphChecker` (`Game.Application/Content`) walks a source-agnostic `ContentGraph` (the export contract sets, tags included) and returns `ContentGraphFinding`s split by severity: **Error** = a genuine break the runtime doesn't paper over (a dangling reference; live content wedged permanently unusable — a zone gated by a retired challenge, an item gated by a frozen proficiency, weapon stranding, >1 Home zone), **Warning** = unreachable/dead content the runtime tolerates (an orphan skill no source grants, a live combat zone with no enemies, a zone with no unlock path from the start, an enemy attribute distribution nothing in its kit consumes). Only **live** records are checked as sources — a retired record is out of circulation. The checker is pure (no DB); a CI test (`ContentGraphLintTests`) loads the committed `content/*.json` and fails the build on any Error **and on any Warning not on the test's `AcceptedWarnings` allowlist** — the committed seed slice is held lint-clean, and any tolerated work-in-progress gap must be an explicit, documented allowlist entry rather than a silent accumulation (#1435). The same checker also backs a read-only admin **Content Health** view: `IContentHealthService` (`Game.Application/Content`) builds a `ContentGraph` from the **live reference caches** (rather than the committed export), runs the checker, and projects the findings onto the `ContentHealthReport` admin contract, served over `GET /api/AdminTools/GetContentHealth` (Admin-role, read-only, no cache reload).

## Combat-rating calibration report (tuning tool, not a runtime system)

`CombatRatingCalibrator` (`Game.Application/Content/Calibration`) is a re-runnable tuning report, not a lint or a gate: given already-resolved enemies/zones/reference builds, it compares the **legacy** `DefeatRewards.SumCoreAttributes` power measure (retained only for this comparison — the live reward path prices via `CombatRating` since #1532) against the `CombatRating` (#1531, spike #1526) per authored enemy, places each zone's spawn table on the `r = enemy/player` axis under both measures, and recommends the `k`/proficiency-pie constants for re-calibrating the reward curve — anchored on whichever sample the current game already treats as matched, and using real `BattleSimulator` runs (not a formula approximation) for the XP/hour figures. `CombatRatingCalibrationIntegrationTests` wires it against the real seeded content (`ISkills`/`IEnemies`/`IZones`/`IClasses`) and logs the report; nothing here writes to `ServerGameConstants` or authored content — a human reads the output and applies the constants by hand.
