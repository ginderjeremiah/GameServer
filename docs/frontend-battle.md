# Frontend Battle Simulation — Parity Mechanics

Deep-dive companion to [frontend.md → Battle simulation & parity](./frontend.md#battle-simulation--parity) and the backend's [backend-battle.md](./backend-battle.md). The frontend simulates battles in real time as anti-cheat: the backend replays the battle the client reports, so the two simulators must agree tick-for-tick. Everything below is a parity surface — change it only in lockstep with the backend and the shared parity matrix.

## The single stepper and its three consumers

The per-tick arithmetic has a **single source** — the pure `battleStep` (`$lib/battle/battle-step.ts`): the player's ready skills fire, then (only if the enemy survives) the enemy fires back, operating on real `Battler`/`Skill` objects. Three consumers drive that one stepper: `BattleEngine.logicalUpdate` (the live loop, driven by the worker tick source — see [frontend.md → Engine tick sources](./frontend.md#engine-tick-sources-worker-driven-logical-loop) — layering combat-log/stage UI on top), `BattleSimulator` (a headless runner, the analogue of the backend's `BattleSimulator`), and `BattleEngine.replayToOffset` (#1597) — when the server hands back a battle already in progress (a stale reconnect, #1595, or an away-window boundary that fell inside an uncredited battle, #1596), the offset arrives as `elapsedOffsetMs` on the enemy instance and the engine fast-forwards headlessly to it (no log lines/floats/mechanic triggers for the skipped ticks) before continuing live, mirroring the backend `BattleSimulator.Simulate`'s own tick loop and exit value so the two replay identically.

The parity suite drives the production `BattleSimulator` (not a hand-rolled copy), so a divergence from the backend fails parity rather than silently weakening the replay; keep its scenario matrix row-for-row aligned with the backend's.

## The seeded battle RNG

**The seeded battle RNG is a parity surface too.** One `Mulberry32` — seeded from `enemyInstance.seed` in `BattleEngine.reset`, and from the headless `BattleSimulator`'s seed — is threaded into `battleStep`, where the player-only crit/parry/dodge rolls draw from it in a fixed, outcome-independent order (1 crit draw per player fire, then 3 draws per enemy fire — parry, dodge, and the parry counter's crit — all taken unconditionally). Both sides must seed identically and draw in the same order or every fight diverges; see [backend-battle.md](./backend-battle.md#battle-runtime--seeded-crit--dodge--parry-player-only) for the rules.

## Shared scalar formulas and composition inputs

**The scalar formulas are shared with the display surfaces.** A skill's raw damage, the Toughness mitigation curve, the typed amplification/resistance steps, and the cooldown multiplier live as pure functions in `$lib/battle/battle-formulas.ts`; the battle classes delegate to them, and any UI showing battle numbers (skills page, skill tooltips) must consume the same functions rather than keeping a display-only copy — so a formula change propagates everywhere and stays under the parity suite's guard. The damage-type taxonomy itself (the `DAMAGE_TYPE_*` tables in `$lib/api/types/damage-types.ts`, mirrored by `$lib/battle/damage-types.ts`) and the attribute coefficients are **codegen'd from the C# domain** with a CI drift check, so the amp/resist map can't silently desync from the backend's `DamageTypes.Applies`. Sharing formulas isn't enough on its own — the *inputs* to those formulas must match too: the player's live `BattleAttributes` (allocation + equipped gear, class locked base, proficiency bonuses, and the class signature passive composed last) is assembled by the single `composePlayerBattleAttributes`/`playerBattleModifiers` pair in `$lib/battle/player-battle-composition.ts`, consumed by the battle engine, the skills screen, and the attribute-breakdown screen alike, rather than each hand-rolling its own composition.

## Mid-battle attributes: a modifier graph, not a snapshot

**Mid-battle attributes are a modifier graph, not a frozen snapshot.** `BattleAttributes` retains its modifiers (bucketed per attribute) instead of flattening to numbers at battle start. Totals are memoised and, on add/remove, recomputed **incrementally** — only the changed attribute and the attributes that transitively derive from it — via the same lazy cascade the backend `AttributeCollectionNode` runs, rather than rebuilding every attribute from the full modifier list. Each single-attribute value folds through the shared `foldAttributeValue` kernel, the value-only counterpart of the `computeAttributes` reduction the breakdown screen uses, so both stay bit-identical (the FE/BE replay parity rests on it) while a heavy DoT/buff stack pays per change only for the attributes that actually moved.

## The per-spawn player reset

**The per-spawn player reset skips redundant re-derivation.** An idle farm re-spawns enemies continuously, but the player's battler only needs its full attribute graph (and skill set) rebuilt when something actually changed. `BattleEngine` re-derives the player only when its inputs differ since the last derive — the memoised `InventoryManager.equipmentStats`, `PlayerManager.selectedSkills`, the class locked-base and proficiency battle modifiers, the attribute distribution, and the level — and otherwise cheaply re-arms the existing battler (clears effects, resets health + skill charges). Those getters are memoised on the managers and invalidated by the managers' own mutation methods (equip/unequip/mod, and `PlayerManager.setSelectedSkills` — the single loadout mutation path). The change-detection compares the inputs **by reference**, so every producer must **reassign** its cached array on change, never mutate it in place.
