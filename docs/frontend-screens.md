# Frontend Screen Design Notes

This document collects the per-screen design decisions for the in-game screens and the boot flow. It is split from [frontend.md](./frontend.md) (which covers the cross-cutting architecture — auth, reference data, the Workbench, theming, toasts) so the main doc stays focused. The decisions below are feature-specific; the cross-cutting patterns they build on live in `frontend.md`.

A few patterns recur across these screens, so they're stated once here rather than per screen:

- **Reactive view-model split.** Each screen pairs a pure, heavily-unit-tested logic module with a `statify`-reactive `*View` class that only wires state to `$derived`; the `+page.svelte` constructs the view and renders presentational subcomponents. This keeps the logic testable without rendering — the same plain-class-plus-`statify` split the battle engine uses.
- **Diff-only save.** Editing screens keep a `draft` working copy against a persisted baseline, derive the dirty set by comparison, and persist only the changes — re-seeding the baseline on success.
- **Empty vs. error are distinct.** A screen with no data yet shows a friendly empty state; a *failed* load is never collapsed into that (which would tell a returning player to go play when their data merely failed to load) — it surfaces a `toastError` and a separate error state, per the toast guidance that load failures are reported, not swallowed.

## Inventory screen (gear & mods)

The Inventory screen (`src/routes/game/screens/inventory`) is the gear/mod loadout: an equipped rail, a filterable/sortable item grid, an equipped-totals panel, and a detail drawer for inspecting/modding an item.

- **The `InventoryManager` is the single source of truth and the single mutation path.** The view holds only UI state (sort/filter/selection/drag/page) and reads authoritative data *through* the manager, delegating every action (equip, unequip, apply/remove mod, favorite) to it — it keeps no shallow copies and re-implements no mutation logic. (An earlier copy-and-mirror approach was two parallel sources kept in hand-sync, which is exactly the divergence it caused.)
- **Optimistic apply with rollback.** Each mutation applies to the authoritative state optimistically (instant UI), then persists; a failed persist rolls back to the captured prior state, so local state can't diverge from what was saved. (Favorite is the exception — a low-stakes flag kept optimistically and re-synced on the next toggle/reload.)
- **Mutations are serialized, not just rolled back.** The view fires mutations fire-and-forget, so two rapid actions could overlap. The snapshot/apply/persist windows are single-flight-serialized so each mutation captures its baseline only after the previous one fully settles — the same "collapse concurrency" spirit as the auth refresh (see [frontend.md](./frontend.md) → _Authentication_). Otherwise an in-flight op's stale rollback baseline could restore over a later change. Queueing the second action beats dropping it.
- **The `items` array is a republished view of the `unlockedItems` map.** The manager's source of truth is the `unlockedItems` `Map`; the view's `$derived` getters iterate the `items` array, which the manager **rebuilds on every mutation** so a changed item *set* (unlock/equip) surfaces reactively. The items are made field-reactive by `statify` on the way in — safe because the display-only `BattleAttributes` they carry keeps its modifier graph in `#private` fields that `statify` can't see, so item reactivity never reaches the battle hot path (see [frontend.md](./frontend.md) → _Battle simulation & parity_).
- **One shared item tooltip, driven by context.** `Inventory.svelte` owns a single `ItemTooltip` and its controller (`createItemTooltip`), published via context so the equipped rail and the item grid drive the one panel instead of each mounting their own — the same context-controller pattern as `AttributeTooltip` (see [frontend.md](./frontend.md)). The grid keeps the suppression rule (no tooltip while an item is selected or dragged) in its own hover handler, since the rail has no such constraint.

## Challenge screen (the Type Rail)

The Challenges screen (`src/routes/game/screens/challenges`) is a **type-driven master/detail** view: a left rail lists each `ChallengeType` (ring meter + done/total) plus an "Overview"; selecting a type swaps the right pane to that type's challenge cards, while "Overview" shows overall progress and a per-type grid.

- **Rewards are sealed until unlocked.** Every item/mod is unlocked by completing a challenge (random drops were scrapped — see the game-design doc), so a locked reward is a *sealed* teaser: rarity tier + category hinted, name `???`. On completion it *reveals* in rarity colour, and hovering opens the same tooltip you'd see inspecting it in your inventory.
- **One tooltip, driven by context.** Rather than a bespoke anchored tooltip, the screen reuses the **global tooltip system**: a single `RewardTooltip` dispatcher renders the right variant — the real `ItemTooltip` for revealed items (single source of truth), a `ModTooltip` for revealed mods, and masked sealed variants (name/values redacted, but the *count* of stat rows and mod slots stays truthful). The hover controller is provided through Svelte context so nested reward affordances don't thread handlers down the tree.
- **Completion arrives live, not just on reload.** When a challenge completes the backend pushes a `ChallengeCompleted` socket command (see `backend.md` → _Challenge-completion notifications_); the engine marks it complete and unlocks each reward into the live managers, so a newly-unlocked item/mod is equippable and a skill selectable immediately, and completion-gated UI (zone navigation) updates without a refresh. The push also raises a **success toast** naming the challenge and its reward (with a *View* action onto this screen) — a cross-screen toast rather than the arena-anchored Zone-Cleared overlay, since challenges complete during the continuously-running battle regardless of the active screen.

## Options screen (the settings shell)

The Options screen (`src/routes/game/screens/options`) reimagines the old logging pop-up as an **extensible two-pane settings shell**: a category rail + the active category's content + a Save/Discard footer.

- **Config-driven and extensible.** The category list, log groups, and per-type metadata are plain arrays; adding a settings category or log type is a one-line edit, not new UI. Only **Logging** is built; Display/Audio/Gameplay/Account render as disabled `soon` items to demonstrate the shell extends cleanly.
- **Optimistic local apply.** The combat-log writer filters by the player manager's log preferences, so saving pushes the draft onto the manager immediately — the change takes effect for the session regardless of the network result; a persist failure still applies locally and surfaces a toast warning.
- **Live preview reuses the real log row.** The preview renders a sample event stream through the actual `LogRow`, filtered by the *draft* preferences, so it looks exactly like the real combat log and reflects each toggle before you save.
- **All colour flows from the root `--log-*` vars** (the theming rule in frontend.md): one `var(--log-*)` reference map is shared by the real log, the preview, and the per-type rows, with every neutral/alpha shade a `color-mix` over a base token.

## Loading screen (the boot manifest)

The loading screen (`src/routes/loading`) runs the boot sequence that fetches the static reference data the game needs and surfaces it as a sequential "manifest" the player watches tick through.

- **The reference-data sets are a single table.** Each set is one row (key + label + its `Get*` socket command + a `loaded()` check + load/hydrate/current accessors), so adding/removing a set is a one-line edit. The table and its version/cache helpers live in `$lib/engine/reference-data.ts` so the resume path can share them; concurrent fetches of the same key are de-duplicated through a process-wide in-flight cache so a re-mount mid-load can't double-fetch.
- **Sets load over the socket, version-checked against a persistent cache.** `start()` fetches the per-set content hashes, hydrates from `localStorage` every set whose cached hash matches, and only then fetches the genuinely stale/missing sets — caching each freshly-loaded set under its new version. If the version check fails the cache is bypassed and everything is fetched fresh, so a transient error never serves stale data (the cache itself is best-effort). Because the socket has no built-in per-command timeout, each call is bounded so an unreachable backend surfaces the error/retry UI instead of hanging.
- **The manifest window keeps its own DOM state** — the wheel handler, the rAF easing toward the active row, and the per-row opacity falloff live inside the window component, driven only by props, keeping that view concern out of the load model.

## Session resume (the boot gate)

A page refresh used to always route through the loading screen, even though both the credentials (the JWT pair in `localStorage`) and the reference data (the persistent cache) survive a refresh. The **boot gate** in the root layout now resumes a restorable session straight into `/game`, falling back to the loading screen only when something genuinely needs downloading. The decision logic is the pure `resumeSession()`.

- **One boot decision, in the root layout.** On first load the layout runs `resumeSession()` once (in `onMount`, so it never fires on in-app navigation) and routes accordingly: session restored + whole cache current → game; session restored but a set must download → loading; no tokens / can't restore → login.
- **A `game` resume keeps the player on the route they refreshed on.** A fully-restored session is already where it should be, so the destination→navigation mapping (the pure `bootRedirect`) only hands off to `/game` from the transient boot-only routes (`/`, `/loading`) and otherwise stays put — so refreshing `/admin` lands back on `/admin`. The socket auto-connects on the first reference-data command, so staying on `/admin` doesn't strand its data.
- **Skipping the loading screen is all-or-nothing.** The cache resumes into the game only when *every* set is present and current (one version check); otherwise it leaves the store untouched and the loading screen does its normal version-check-and-download. A failed version check also declines, matching the "never serve stale data" rule.
- **A lightweight splash, kept SSR-safe.** The brief restore step shows a minimal pulsing `BootSplash` rather than flashing the loading manifest or login form. The splash flag starts `false` to match the server render and flips on only on the client (the login form stays server-rendered for the e2e hydration strategy), and gates the route content while booting so no protected route renders prematurely.
- **Scope: refresh / resume only.** A fresh username/password login still plays the loading manifest (the intended first-time transition); the login submit path is unchanged.

## Welcome-back gate (offline rewards)

The welcome-back gate sits inside the `/game` route, between reaching the route (from the loading screen **or** the silent resume path) and the game shell. It runs the offline-rewards check (`GetOfflineProgress`, see [backend](./backend.md) / spike `docs/spikes/879-offline-rewards.md`) **before** the idle loop starts, then either passes straight through or shows an away-summary the player dismisses to enter. Logic lives in the reactive `WelcomeBackView`; the engine/socket touch points are injected so the flow is unit-testable without the live engine.

- **The gate owns when the idle loop starts.** `startGame()` is no longer called at game-route init; the gate defers it until the summary is dismissed (or runs it immediately when there's nothing to show), so simulated and live battles never overlap. To make that deferral possible, the engines' unmount cleanup moved out of `startGame` into a single `onDestroy(stopEngines)` on the game page (idempotent, so safe even if the loops never ran) — `startGame` no longer registers lifecycle hooks and can be called after an `await`.
- **No progress ⇒ no gate.** An empty result (away under the backend's 5-minute floor, a fresh character, or a failed check) passes straight through to the game with no screen — the common case stays a clean boot.
- **The summary renders from the command's DTO; state re-syncs before entering.** The figures (away duration, idle/boss mode, battles won/lost/drawn, exp/levels/stat-points, completed challenges + what they unlocked) come straight off `IOfflineProgressModel`, resolving reward names through the shared `resolveUnlockReward`. When there's progress the gate re-pulls the authoritative player aggregate (`refreshPlayer` → `Login/Status`) so the engine builds the live battler from the reward-updated state; statistics/challenges/inventory are then loaded fresh by `startGame` on entry.
- **It reconciles the persisted loop mode.** The live `autoFight` always starts false, so the gate re-arms it from the response's `autoChallengeBoss` (via `enemyManager.reconcilePersistedMode`) and seeds the mode-sync dedup baseline — restoring a returning boss-farmer's toggle. This runs on **every** return (even sub-threshold), and is the reconciliation the persisted-mode sync (#1040) deliberately left to this gate. It restores the toggle (pre-armed intent) but does not auto-engage the boss; auto-resuming the boss loop on return is a deliberate non-goal (a balance/UX call for later).

## Attributes screen (the build page)

The Attributes screen (`src/routes/game/screens/attributes`) is a **point-buy** on the six core attributes with a live preview of the derived stats they feed. It has two in-page **density modes** sharing one allocation model: **Guided** (default) makes the hexagon radar the hero with steppers and plain-language "feeds" tags and no derived-stats panel; **Theorycraft** densifies into an allocation table with per-point marginal yields, a compact radar, and a grouped derived-stats panel.

- **Game logic is the single source of truth for derived stats.** The screen redefines no formula — derived values compute through the real `BattleAttributes`, the same model the battle simulation uses. Only the derived stats the game actually produces today are surfaced; deprecated/unimplemented ones are omitted and will appear automatically once they gain real formulas. Feeds tags, marginal yields, and "fed by" lines are all auto-derived by bumping an attribute and diffing, so they never go stale.
- **Points move freely — no respec cost.** The backend accepts negative deltas (requiring only that no attribute drops below 0 and the pool isn't overspent), so any attribute can be decremented back to 0 to refund. The mock's "Respec · 250 g" affordance was dropped (there is no gold/respec system).
- **Diff-only save over the socket.** Persists only the changed per-attribute deltas via the `UpdatePlayerStats` **socket command** — on the socket (not HTTP) so it serializes with the idle battle loop and can't lose a read-modify-write race against a background battle save (see [backend.md → HTTP vs WebSocket](./backend.md#http-vs-websocket-communication)). On success it writes the returned allocations back onto the player manager so other screens see the change immediately; the chosen mode is persisted per player.
- **Themeable accents stay frontend; everything else flows from the backend.** The six attribute hues are `--attr-*` CSS variables read only through `attributeColor`, following the rarity/category pattern. The attribute's name, short code, and `isHarmful` flag are display metadata **promoted onto the backend `Attribute` model** and read off `staticData.attributes` — the colour is the only attribute facet the frontend still owns.
- **The radar is draggable, not just clickable.** A vertex is dragged radially to set its attribute directly (outward allocates up to the remaining budget, inward refunds to 0), making the hexagon the primary allocation control. The pointer-to-value mapping is pure and unit-tested, reconciled through the same bounded inc/dec the steppers use so the radar can never overspend. The gesture uses pointer events (mouse/pen/touch). **The radar scale is pinned for the duration of a drag** — the scale grows with the build, so an unpinned scale would rescale mid-gesture and inflate the value under the pointer (a positive-feedback loop that made over-allocating near the boundary easy), so it freezes at gesture start and recomputes on release.
- **Allocation-related stats only.** The derived-stat preview reflects the player's *attribute allocations only*, not gear (the marginal yields are identical either way since the formulas are linear in the core attributes); full combat totals including gear live on the Attribute Breakdown screen.

## Attribute Breakdown screen (the inspector)

The Attribute Breakdown screen (`src/routes/game/screens/attribute-breakdown`) is a **read-only inspector** — a separate nav entry beside the point-buy Attributes screen. It decomposes every surfaced attribute into *where its value comes from* (engine base, allocated points, gear, applied mods, derived formulas): a left rail of stacked source bars grouped by display taxonomy, and a right panel that blows the selected attribute up into a by-source breakdown plus a literal **apply-order trace**.

- **The frontend attribute pipeline mirrors the backend.** Two `$lib/battle` modules faithfully port `Game.Core.Attributes` — the modifier enums + static modifiers, and `computeAttributes`/`groupBySource` (additive modifiers before multiplicative; a `Derived` modifier scaled by the *final* value of its source attribute). These enums are domain concepts, not API DTOs, so they're hand-maintained. The screen assembles the player's real modifiers exactly as `Player.GetAllModifiers` does. Current real data is additive-only (no multiplicative item yet), so the multiplicative apply-order machinery is present but dormant.
- **One frontend attribute calculator.** `BattleAttributes` is a thin wrapper over the same `computeAttributes` pipeline, so the base/derived formula constants have a single home rather than being duplicated across two calculators. A regression test asserts the two produce identical totals for every implemented attribute, so the breakdown can never silently disagree with the simulation.
- **The breakdown is self-selecting, off backend metadata.** Membership is not a hardcoded allow-list: an attribute is surfaced when it has at least one *non-combat contributor*, grouped by the reference-data display taxonomy and formatted with its reference `decimals`. So obsolete/unimplemented attributes drop out and new ones appear automatically once they gain real contributors.
- **Themeable source accents** — the five source hues are `--source-*` CSS variables read only through the `source-display` helper, following the rarity/attribute pattern.

## Skills screen (the loadout inspector)

The Skills screen (`src/routes/game/screens/skills`) is the player-facing **loadout editor**: a left rail of every unlocked (and optionally aspirational locked) skill beside a large inspector, with an editable equipped band below. The player chooses which skills — and in what order — are equipped, capped at the generated `MAX_SELECTED_SKILLS` constant.

- **Display damage *is* the battle formula.** The page computes its numbers with the shared pure battle formulas, against the same `BattleAttributes` the simulation uses (allocations + equipment stats). One definition, no display-only copy: a change to the battle arithmetic propagates here and stays guarded by the parity suite (see [frontend.md](./frontend.md) → _Battle simulation & parity_).
- **One atomic set-loadout write.** Each committed edit (equip/unequip/swap/reorder) replaces the **whole** ordered loadout through the single `SetSelectedSkills` socket command, not discrete calls. It applies optimistically (mirrored onto the player manager so other screens see it), then reverts and toasts on error. There is no "Save" button — composing the loadout *is* saving.
- **Locked skills are aspirational, not owned.** The catalogue is every non-retired skill plus any retired skill the player still owns. Locked (un-owned) skills are hidden by default and revealed by the filter toggle; their "unlock by" line is derived on the client from the challenge catalogue, so no extra backend read is needed.
- **Compare-vs is a static-defense slider with current-zone enemy presets.** Effective DPS/burst and per-skill values subtract a single Compare-vs flat **Defense**, driving defense-aware sorting. Enemy quick-pick pills are sourced from the player's current zone (every non-retired spawn plus the dedicated boss); the pill defense is **derived client-side from cached reference data** (building a `BattleAttributes` from the enemy's attribute distribution at a level, exactly mirroring the backend enemy build), so no round-trip is needed. Because defense scales linearly with level, each spawn pill is evaluated at the zone's range midpoint (the expected encounter) and the boss pill at its fixed level. With no zone data loaded it falls back to slider-only — no placeholder enemies are fabricated.
- **Themeable attribute accents** — skill glows, scaling bars, and equipped-card chips are keyed to the skill's scaling attributes via the `--attr-*` helpers. The sort/filter modal is a page-local overlay (a live, non-blocking filter), not the global promise-based `modal` store (reserved for blocking dialogs).

## Statistics screen (tracked metrics)

The Statistics screen (`src/routes/game/screens/stats`) presents the player's tracked statistics grouped **by statistic**: the types split into category tabs with per-type cards (grand total + per-entity breakdown). Clicking an entity row **deep-links into that entity's Codex dossier**, where the per-entity record lives now (the old in-place "by entity" dossier was retired — see the Codex section below).

- **The statistic→entity-kind mapping comes from the backend, not the frontend.** Each statistic's entity breakdown is read from the statistic-type reference data (`entityType`, in `staticData.statisticTypes` — see [backend.md](backend.md#reference-data)), merged with a frontend presentation table that carries only what the backend doesn't model (unit, aggregation, comparison direction, category, display order). So the frontend no longer hard-codes which kind a statistic breaks down by — a backend change flows through automatically. Statistics whose entity type is `None` render as a headline with no breakdown.
- **Real data, sourced two ways by lifetime.** Statistic *values* are per-player and change, so they're fetched fresh on mount; the type *metadata* and the entity lists are static reference data loaded once. Entity ids resolve against `staticData` by index, the zero-based convention the rest of the app uses.
- (Empty-vs-error handling follows the shared rule above.)

## Codex screen (the reference glossary)

The Codex screen (`src/routes/game/screens/codex`) is a **read-only reference glossary** of the game's content behind a top-level tab bar (Enemies / Zones / Skills). Each tab is a master/detail: a list/rail/table beside a detail dossier. **All three tabs are built.** It's a port of a Claude Design handoff onto live reference/runtime data — none of the prototype's mock catalogues survive.

- **Reference facts come from the stores; nothing is fabricated.** The dossiers read `staticData.enemies/zones/skills/challenges`, the per-enemy player record via the Statistics screen's `StatisticsData.statsForEntity`, per-zone clears via `statistics.isZoneCleared`, and challenge progress/completion from `playerChallenges`. Level bands, spawn shares and the dossier projections live in pure modules (`enemy-level`, `codex-display`) unit-tested without rendering.
- **Enemy stat scaling reuses the real battle build.** The Attributes sub-tab scales an enemy to a chosen level (a slider for normal enemies; a fixed level for bosses); each primary attribute is `base + perLevel × level` straight from its `attributeDistribution`, and the derived **Max Health / Defense** are computed through the same `BattleAttributes` composition the battle uses (`enemy-stats`, generalising the Skills screen's `enemyDefense`) — so the numbers match combat by construction. The per-level slope of a derived stat is its linear two-point delta. "Show scaling" reveals the base/per-level breakdown.
- **The Zones tab is a progression rail + zone dossier.** The left rail lists zones in authored `order` with a status node — **cleared** (a clear recorded), **locked** (a `Zone.unlockChallengeId` the player hasn't completed), else **unlocked** — resolved by the pure `resolveZoneStatus`. The dossier shows the level band, spawn-pool size, the zone boss card (`bossEnemyId`/`bossLevel`), the spawn table (enemies whose `spawns` include the zone, shares via the shared `zoneTotalWeight`/`spawnShare` helpers), and the unlock condition (the gating challenge's name, sealed when locked). The zone section accent is `--accent`.
- **The Skills tab is a skill catalogue + skill dossier.** A skill table (name / base damage / cooldown / used-by count) beside a dossier showing base damage, cooldown, the attributes the skill's damage scales with (`damageMultipliers`, tinted via `attributeColor`), its authored effects (worded via the shared `$lib/common/skill-effect-display` helper, so the buff/debuff direction matches every other effect surface) and the enemies that use it (`skillPool`), whose pills cross-link into the enemy dossier. It's a **reference catalogue, deliberately distinct from the Skills _screen_** — no equip/loadout and none of that screen's player-build DPS math. The skill section accent is `--attr-intellect`.
- **Per-entity statistics live in the Codex, not the Statistics screen.** Every dossier (enemy, zone and skill) surfaces the player's record against that entity via the shared `StatisticsPanel` — the enemy dossier as its Statistics sub-tab, the zone and skill dossiers as a "Your record" section. The Statistics screen **deep-links an entity into the matching dossier** (via the navigation store — see [frontend.md](./frontend.md) → _Cross-screen navigation_; the enemy payload also targets the Statistics sub-tab) instead of rendering its own in-place dossier, which has been retired entirely.
- **The enemy table is searchable + sortable.** The search box matches an enemy's name, kind and spawn-zone names; the sort control orders by level or name. Both are pure helpers in `codex-display` (`matchesEnemySearch`/`sortEnemyRows`) the view-model wires reactive state to.
- **Cross-links are within-screen tab switches.** The Zones dossier's boss card and spawn rows, and the Skills dossier's used-by pills, open the enemy dossier (`CodexView.openEnemy` sets the tab + selection — no navigation store needed inside one screen). The reverse direction (the enemy dossier's Spawns/Skills rows linking into the Zones/Skills tabs) is still non-interactive, tracked as a follow-up.

## Card Game screen (the Initiative Loom)

The Card Game screen (`src/routes/game/screens/card-game`) is **"The Initiative Loom"** — a real-time card-boss duel imported from a Claude Design handoff, a deliberate *active* break from the idle grind: the present (a glowing **NOW** line) is a moving deadline, cards are multi-tick **spans** on a scrolling timeline, and each resolves the instant its right edge crosses NOW.

This is the **demo phase** of a staged plan (integrate as a demo → build the backend → admin-configure → wire to real data). It runs entirely client-side: HP, deck, and enemy cadences are hardcoded sample values, and a labelled **Sandbox** strip (AGI/DEX/LUCK sliders) stands in for the player's real attributes.

- **Pure mechanics, separated from rendering.** The simulation lives in `$lib/card-game`, not the screen folder: a deterministic, framework-free core (attribute formulas, half-open span coverage, no-overlap queue placement, strike damage) and a plain game class that owns state and steps it via `advance(dt)`. Randomness is injected so the engine is fully unit-testable without a browser; the screen makes it reactive via `statify` and drives `advance` from a single rAF loop — the same split the battle engine uses.
- **The loom runs on tempo attributes, so a STR-or-INT build stays playable.** Optimised builds end up high in Strength *or* Intellect but not both, so the loom's skill lives in the tempo attributes everyone has — Agility drives draw speed, Dexterity the hand cap, Luck the crit cadence — while offense is intended to scale off `max(STR, INT)` so pure-STR and pure-INT hit equally hard (the demo uses fixed card damage; the `max(STR,INT)` scaling is the documented intent for the data-wiring phase).
- **Unified right-edge resolution and half-open coverage.** Every entity resolves when its right edge crosses NOW, so the moment reads identically on both sides; coverage spans are half-open `[start, end)` so an impact lands centered in a cell and adjacent queued spans tile seamlessly. These load-bearing rules are unit-tested directly.
- **Theming reuses existing tokens; one new semantic var.** The port maps onto the existing palette, adding only a semantic `--gold` for crit ticks and the draw-timer fill — it shares a hue with `--rarity-legendary` but is its own concept (timing reward, not item rarity), per the "variables by semantic intent" rule.
- **Cross-device input is `pointer*`, not `mouse*`.** The cast/aim drag and press-and-hold Reflex run on pointer events with `touch-action: none`, so the loom is playable by touch/pen (`pointercancel` aborts an in-flight drag). The hand card carries no `role`/`tabindex` (dragging to aim has no keyboard analogue, and casting is reachable via the `1–7` hotkeys), so it doesn't falsely announce a button AT can't focus.

## Fight screen (Challenge Boss)

The Fight screen (`src/routes/game/screens/fight`) layers a deliberate **boss encounter** onto the existing Mirror combatant layout. Each zone has one dedicated boss (authored via the admin Workbench); an always-available affordance slot holds either the **Challenge trigger** (a banner naming the boss + an Auto-fight toggle) or, while engaged, the **boss bar**. Engaging swaps the enemy card for a heightened gold boss card, turns on an atmosphere vignette, and on victory plays a zone-cleared overlay. The boss design is the Svelte 5 port of the `fight-boss.jsx` handoff (its Banner trigger is the finalized treatment).

- **Two mutually-exclusive loops, never running at once.** "Available" *is* the normal idle farm loop with the trigger shown; "engaged" runs the dedicated-boss fight with the idle loop paused, gated by a `mode: 'idle' | 'boss'`. Challenge sends the `ChallengeBoss` socket command (the backend deterministically builds the boss at its fixed level with full loadout and implicitly abandons any idle fight); Retreat drops back to idle; a boss victory with auto-fight on re-challenges immediately (no cooldown), off hands back to idle; a loss records it and returns to idle. A `transitioning` guard plus a pause keep the outgoing battle's resolving stage change from being mistaken for an outcome of the incoming one.
- **Controls are last-input-wins: a press that differs from the in-flight transition supersedes it.** While a challenge/retreat is mid-flight (including parked in its enemy-fetch retry backoff), pressing the *opposite* control takes over — the new press wins and the superseded transition abandons its result rather than the player having to wait it out then press again. A press matching the in-flight transition's destination is a no-op (it lets that transition continue) rather than re-sending the command, since each `ChallengeBoss`/`NewEnemy` makes the backend abandon-and-respawn the current fight (a phantom abandoned-battle record). Supersession is gated by a transition generation (mirroring the enemy-fetch generation) so it preserves the single-in-flight-fetch / no-double-spawn anti-cheat guarantees and the boss-victory re-entrancy checks.
- **Victory and loss resolve through the same backend paths as idle** — a boss victory sends `DefeatEnemy` (shared victory helper), a loss sends `BattleLost` (which returns the post-loss cooldown). The `IsBossBattle` marker the backend persists at challenge time is what makes a victory count as a dedicated-boss clear, so the frontend never has to tell the backend "this was a boss".
- **The boss is deterministic, so FE/BE battle parity is straightforward** — fixed level + full authored loadout (no rolls) means the shared boss-loadout scenario is mirrored identically in both parity suites.
- **A shared, reactive statistics store backs the "Cleared" seal.** Per-zone clears are promoted to a small `statistics` store loaded once at boot, so the fight screen shows the seal for already-cleared zones and the Statistics screen reuses the same source (one fetch). On a boss victory the engine optimistically marks the zone cleared so the seal appears immediately, reconciled on the next load.
- **A dedicated `--boss-accent` token, not a reuse of `--gold`** — the heightened gold treatment is its own semantic token, distinct from the card-game timing `--gold` it currently shares a hue with, so a theme can restyle boss fights independently. Retreat reuses the destructive `--enemy-accent`.
- **The Zone-Cleared overlay is kept truthful** — with zone-navigation locking in place (see the zone-progression section in [game-design.md](./game-design.md)), it shows the "Next zone unlocked" line only when the clear actually flipped the next zone's gate open. The affordance simply doesn't render for a zone without a boss.

### Combat feedback (XP bar · floating numbers · battle timer · enemy cooldown)

Three real-time feedback treatments layer onto the Mirror layout (ported from the `Fight Screen Final` handoff — see [designs](./designs/designs.md)):

- **Player XP bar in the card header.** The player card's header carries the account `LV` plus a slim gold `XpBar` filling toward the next level; the enemy card keeps the plain `LV · n`. `XpBar` is a shared, specialised multi-layer bar (like `HpBar`, not the generic `Bar` — its shimmer is clipped to the fill) and is intentionally reusable so the Attributes screen can mirror it (right-anchored) later. It derives its fill from `playerManager.nextLevelThreshold` (now public) so the display never re-derives `level * EXP_PER_LEVEL`, and flashes gold on a level-up.
- **Floating combat numbers** ride over each card via `CombatFloaters`, driven by a new engine pub/sub hook (`onCombatFloat`, alongside `onBattleStageChanged`). `BattleEngine.logicalUpdate` emits one `CombatFloatEvent` per skill activation — the same flags as the combat-log outcome — so the number pops over whichever side was struck/defended, colour- and label-coded by outcome (crit/dodge/block/hit). The layer is `aria-hidden`; the combat log remains the accessible record. The float motion is the shared `dmg-rise` keyframe.
- **Battle timer above the VS badge.** `BattleTimer` is a presentational readout of `battleEngine.timeElapsed` against `DEFAULT_MAX_BATTLE_MS` (2:00), with the track flipping to `--warning` in the final 20s and a `TIMEOUT · draw` state at the cap. It sits in a center column over the VS diamond, clear of the zone-nav/boss band. The live engine enforces the cap as a real outcome: reaching it with both battlers alive ends the fight as a draw (the `BattleStage.Drawn` stage, recorded as a `BattlesAbandoned` "neither died" outcome — see [game-design.md](./game-design.md#battle-outcomes-battleswon--battleslost--battlesabandoned)), not just a pinned readout.
- **Enemy-cooldown countdown, swapped into the same slot.** Between enemies the engine sits in the `Loading` stage counting `battleEngine.loadingTime` down from `loadingTotal` (the server-returned cooldown the next foe spawns after); the center column swaps the battle clock for `EnemyCooldown`, a mono `next enemy` seconds countdown over a track that fills toward 100% as the foe approaches, themed `--enemy-accent` to read as "incoming". The two readouts share the column and never show at once — the clock during `Active`, the cooldown during `Loading`.

### Skill-effect indicators

The player-facing surface of skill effects is three display-only views, all driven by one shared derivation helper (`$lib/common/skill-effect-display`) so wording and buff/debuff signalling stay identical everywhere an effect is shown: a **badge** on a skill carrying effects (neutral accent — it signals "has effects", not direction), **tooltip "On hit" lines** (magnitude tinted by direction), and **active-effect chips** on each battler card (tinted buff/debuff, with a render-interpolated countdown that depletes smoothly between logical ticks and refills on re-application).

- **Buff vs. debuff is a single heuristic, not per-effect authoring** — an effect is a buff when it raises a beneficial attribute or lowers a harmful one (only `DamageTakenPerSecond` is harmful-when-raised today). The buff/debuff accents are themeable tokens read only through the helper.
- **The chip data is a reactive projection that never holds a modifier reference** — the modifier each effect added is kept off the reactive view in a private map keyed by effect id, because a reactive proxy would break the reference identity the attribute collection matches on to remove a modifier (the same constraint `BattleAttributes` documents). The render countdown mirrors the cooldown bar (a per-frame interpolation), leaving the parity-relevant logical timing to the per-tick advance. Effect logs ship under a dedicated `SkillEffect` log type (see [game-design.md](./game-design.md#logs)).

## Combat log panel (resizable)

The combat log (`src/components/log-panel`) is a persistent fixture pinned below the active screen, not a screen of its own. It is **user-resizable**: a thin grab strip on its top edge lets the player drag it taller to read more history, shrinking the scrollable screen area above it.

- **A view-model owns the height, the component owns the DOM** — the reactive height/dragging state, the gesture lifecycle, and the pure clamp + persistence live in the view-model (unit-testable without a DOM); the component only feeds DOM-dependent inputs (pointer position, container height).
- **Flex layout, not a fixed body height** — the height is an inline style and the body flexes to fill it, so one number resizes the whole panel; the drag is clamped so neither the log nor the screen above it can collapse.
- **Pointer events + window tracking** — one `pointerdown` path covers mouse/pen/touch; move/up listeners live on `window` so the gesture keeps tracking off the thin handle, and a body class locks the resize cursor and suppresses selection for the duration.
- **Height persists in `localStorage`** — written on resize-end and restored after mount (not in the constructor) so SSR and first client render agree, then re-clamped against the live container so a height saved on a larger viewport can't overflow a smaller one.
