# Frontend Screen Design Notes

This document collects the per-screen design decisions for the in-game screens and the boot flow. It is split out from [frontend.md](./frontend.md) (which covers the cross-cutting frontend architecture — auth, reference data, the Workbench, theming, toasts) so that the main doc stays focused. The decisions below are feature-specific; the cross-cutting patterns they build on (the reactive view-model pattern, themeable accent variables, the empty-vs-error/toast rule) are documented in `frontend.md`.

## Inventory screen (gear & mods)

The in-game Inventory screen (`src/routes/game/screens/inventory`) is the gear/mod loadout: an equipped rail, a filterable/sortable item grid, an equipped-totals panel, and a detail drawer for inspecting/modding an item.

Key decisions and rationale:

- **The `InventoryManager` is the single source of truth and the single mutation path.** `InventoryView` (`inventory-view.svelte.ts`) holds only UI state (sort/filter/selection/drag/page) and reads the authoritative item data **through** the manager's reactive `items` / `equippedSlots` / `equipmentStats`, delegating every action (equip, unequip, apply/remove mod, favorite) to it. It does **not** keep its own shallow copies or re-implement the manager's mutation logic — the earlier copy-and-mirror approach (#352) was two parallel sources kept in hand-sync, which is exactly the divergence #342 hit. The manager owns the item objects; the view is a pure projection.
- **Optimistic apply with rollback.** Each manager mutation applies to the authoritative item/slot state synchronously (so the UI updates instantly), then persists; a failed persist rolls the change back to the captured prior state so the local state can never diverge from what was saved. (Favorite is the one exception — a low-stakes flag kept optimistically and re-synced on the next toggle/reload rather than rolled back.)
- **Reactivity without making the items reactive.** `Item`s stay plain objects (they carry a display-only `BattleAttributes`, which must not become a reactive proxy — see [frontend.md](./frontend.md) → _Battle simulation & parity_). Instead the `statify`-reactive manager **republishes** a fresh `items` array (and re-references `equippedSlots`) on every mutation; that reassignment is the reactivity signal the view's `$derived` getters depend on, so in-place item-field edits surface without deep item reactivity and without touching the battle hot path (which still reads `equipmentStats` only).

## Challenge screen (the Type Rail)

The in-game Challenges screen (`src/routes/game/screens/challenges`) is a **type-driven master/detail** view: a left rail lists each `ChallengeType` (ring meter + done/total) plus an "Overview" entry; selecting a type swaps the right pane to a hero banner + that type's challenge cards with an in-page Progress/Rarity/Name sort, while "Overview" shows overall progress, a pulsing "next up" reward, and a per-type grid. It reuses the existing screen chrome (dark frame, diamond mark, Geist/Geist Mono) and is decomposed into many small components per the component-size guideline.

Key decisions and rationale:

- **Rewards are sealed until unlocked.** Every item/mod is unlocked by completing a challenge (random drops were scrapped — see the game-design doc), so a locked reward is shown as a _sealed_ teaser: its rarity tier + category are hinted but the name is `???`. On completion it _reveals_ — the name appears in rarity colour and hovering it opens the same tooltip you'd see inspecting it in your inventory. Reward resolution, reveal-gating and progress maths live in the pure helpers in `challenges-view.svelte.ts` (heavily unit-tested); the reactive `ChallengesView` class only wires state to `$derived`, mirroring `InventoryView`.
- **One tooltip, driven by context.** Rather than re-implementing the design mock's bespoke anchored tooltip, the screen reuses the **global tooltip system** (`registerTooltipComponent`). A single `RewardTooltip` dispatcher renders the right variant — the real `ItemTooltip.svelte` for revealed items (single source of truth), a new `ModTooltip` for revealed mods, and masked `SealedItem`/`SealedModTooltip` for sealed rewards (name/values redacted, but the _count_ of stat rows and mod slots stays truthful). The hover controller is provided through Svelte **context** (`reward-tooltip-context.ts`) so deeply-nested reward affordances don't have to thread handlers down the tree.
- **A failed progress load is kept distinct from "no progress yet."** The challenge catalogue comes from `staticData`, so a player with no recorded progress is a normal render (every challenge at zero) — but a failed `Challenges/Player` fetch would otherwise collapse into that same all-zeros view. `ChallengesView` carries an `error` flag: on a fetch rejection the screen surfaces a `toastError` and a dedicated error state instead, mirroring the Statistics screen's empty-vs-error rule (per the toast-system guidance that load failures should be reported rather than silently swallowed).
- **Completion arrives live, not just on reload.** When a challenge completes the backend pushes a `ChallengeCompleted` socket command (see `backend.md` → _Challenge-completion notifications_); the engine's `handleChallengeCompleted` (`engine.ts`) marks it complete in the `playerChallenges` store and unlocks each reward into the live managers (`inventoryManager.addUnlockedItem`/`addUnlockedMod`, `playerManager.addUnlockedSkill`). So a newly-unlocked item/mod is equippable and a skill selectable immediately, and completion-gated UI (zone navigation) updates without a refresh — the fix for an unlocked reward only appearing after a page reload (#408).

## Options screen (the settings shell)

The in-game Options screen (`src/routes/game/screens/options`) reimagines the old logging pop-up as an **extensible two-pane settings shell** living inside the existing game frame: a category rail (left) + the active category's content + a Save/Discard footer. It reuses the established chrome (dark frame, diamond mark, Geist/Geist Mono) and is decomposed into small components per the component-size guideline.

Key decisions and rationale:

- **Config-driven and extensible.** The category list (`SETTINGS_CATS`), log groups (`LOG_GROUPS`) and per-type metadata (`LOG_TYPES`, which mirrors `ELogType`) are plain arrays in `options-view.svelte.ts`. Adding a future settings category or a new log type is a one-line edit, not new UI. Only **Logging** is built; Display/Audio/Gameplay/Account render as disabled `soon` rail items to demonstrate the shell extends cleanly.
- **Diff-only save, mirroring the old `LogManager`.** `OptionsView` keeps a `draft` working copy against a persisted `baseline` and derives the dirty set by comparison (the same model as the inventory/challenge view-models). Save sends only the changed `ILogPreference`s to `Player/SaveLogPreferences`. The diff and the changed-set are computed from plain `$state` (not the `$derived` getters) so they are safe to read from non-reactive call sites like `save()`.
- **Optimistic local apply.** The combat-log writer (`log.ts`) filters by `playerManager.logPreferences`, so saving pushes the draft onto the player manager immediately — the change takes effect for the session regardless of the network result. On a persist failure the change still applies locally and a toast warning is surfaced.
- **Live preview reuses the real log row.** The right-hand preview renders a self-contained sample event stream through the actual `LogRow` component, filtered by the _draft_ preferences — so it looks exactly like the real combat log and visibly reflects each toggle before you save.
- **All colour flows from the root vars.** The semantic combat-log palette lives in the root `--log-*` CSS variables (`+layout.svelte`); `logColors` in `log-kind.ts` is just the `var(--log-*)` reference map shared by the real log, the preview and the per-type rows. Every neutral/alpha shade in the screen is `color-mix(in srgb, var(--…) N%, transparent)` over a base token (off-white `--text-primary`, `--white`, `--black`, the `--eyebrow` accent) or an existing semantic var (`--text-secondary/-tertiary/-muted`, `--border-subtle/-light/-medium`) — no hard-coded `rgba(…)`, so a theme can restyle the whole screen by overriding the variables alone.

## Loading screen (the boot manifest)

The loading screen (`src/routes/loading`) runs the boot sequence that fetches the static reference data the game needs and surfaces it as a sequential "manifest" the player can watch tick through. Like the Options/Challenge screens, it is decomposed into small components per the component-size guideline rather than living as one large `+page.svelte`.

Key decisions and rationale:

- **Load orchestration is a reactive view-model.** `LoadingView` (`loading-view.svelte.ts`) owns the manifest state machine (`phase`, `items`, `activeIndex`, the `completed`/`progressPct`/`title` deriveds) and the `start`/`retryFailed`/`enterGame` actions — mirroring `OptionsView`/`InventoryView`. `+page.svelte` just constructs it and renders the presentational subcomponents (`DiamondMark`, `LoadingHeader`, `ProgressBar`, `ManifestWindow` + `ManifestRow`, `ErrorBanner`, `EnterButton`). This makes the load logic unit-testable without rendering.
- **The reference-data sets are a single table.** Each set is one row in the `REFERENCE_DATA` array (built by the typed `refSource` helper: `key` + `label` + its `Get*` socket `command` + a `loaded()` in-memory check + `load`/`hydrate`/`current` accessors), so adding/removing a set is a one-line edit rather than a copy-pasted block. The table and its version/cache helpers (`fetchReferenceVersions`, `hydrateFromCacheIfCurrent`, `cacheSet`, `dedupedFetch`, `hydrateAllFromCache`) live in `$lib/engine/reference-data.ts` so the resume path can share them; `LoadingView` is now just the manifest state machine over that table. Concurrent fetches of the same key are de-duplicated through a process-wide in-flight cache so a re-mount mid-load can't double-fetch.
- **Sets load over the socket, version-checked against a persistent cache.** During the "checking" beat `start()` calls `GetReferenceDataVersions`, hydrates from `localStorage` (`reference-cache.ts`) every set whose cached content hash matches the server's, and only then fetches the genuinely stale/missing sets over their `Get*` socket commands — caching each freshly-loaded set under its new version. If the version check fails the cache is bypassed and every set is fetched fresh, so a transient error never serves stale data; the cache itself is best-effort (a corrupt entry or full quota degrades to a normal fetch). Because the socket has no built-in per-command timeout, each socket call is bounded by `withTimeout` so an unreachable backend surfaces the error/retry UI instead of hanging.
- **The manifest window keeps its own DOM state.** The centred sliding window — the wheel handler, the `requestAnimationFrame` easing toward the active row, and the per-row opacity falloff — lives inside `ManifestWindow`, driven only by the `items`/`activeIndex`/`phase` props, keeping that view concern out of the load model.

## Session resume (the boot gate)

A page refresh used to always route the player through the loading screen, even though both the credentials (the JWT pair in `localStorage`) and the reference data (the persistent cache above) already survive a refresh. The **boot gate** in the root layout (`routes/+layout.svelte`) now resumes a restorable session straight into `/game`, falling back to the loading screen only when something genuinely needs downloading. The decision logic is the pure `resumeSession()` in `$lib/engine/session.ts`.

Key decisions and rationale:

- **One boot decision, in the root layout.** On first load the layout runs `resumeSession()` once (in `onMount`, so it never fires on in-app client navigation) and routes accordingly: `game` (session restored + the **whole** reference cache is current) → the game; `loading` (session restored but a set must be (re)downloaded) → the loading screen; `login` (no tokens, or the session can't be restored) → the login form. This replaces the old per-page resume that lived in the login screen's `onMount` and always went to `/loading`. The layout's `$effect` redirect is now only a post-boot safety net (it stays inert until the boot decision resolves).
- **A `game` resume keeps the player on the route they refreshed on (#322).** The destination → navigation mapping is the pure `bootRedirect(destination, currentPath)` (`$lib/engine/boot-redirect.ts`, kept separate from the resume machinery so it stays dependency-free and testable). A fully-restored session is _already where it should be_, so `bootRedirect` only hands off to `/game` from the transient **boot-only routes** (`/` login, `/loading`) and otherwise returns `null` to **stay put** — so refreshing `/admin` lands back on `/admin` instead of bouncing to `/game`. Staying on `/admin` doesn't strand its data: the socket auto-connects on the first reference-data command, so `startGame()` (only run by the game page) isn't required. A `login` resume still sends a protected route back to the form; a `loading` resume always routes to the loading screen.
- **Skipping the loading screen is all-or-nothing.** `hydrateAllFromCache()` resumes into the game only when every reference-data set is present and current in the cache (verified via a single `GetReferenceDataVersions` check); it commits the cached sets to memory only in that case, and otherwise leaves the store untouched and returns `false` so the loading screen does its normal version-check-and-download. A failed version check also returns `false`, so a transient error never resumes against stale data — matching the loading screen's own "never serve stale data" rule.
- **A lightweight splash, kept SSR-safe.** The brief restore step (a `Login/Status` + one version check, sub-second when warm) shows a minimal pulsing `BootSplash` rather than flashing the loading manifest or the login form. The splash flag starts `false` so it matches the server render and is flipped on only on the client — the login form stays server-rendered for the e2e hydration strategy (`data-hydrated`), and a session-less load shows the form with no splash. The splash gates the route content while booting, so no protected route renders prematurely.
- **Scope: refresh / resume only.** The skip applies to restoring an existing session on page load. A fresh username/password login still plays the loading manifest (the intended first-time / new-character transition), so the login submit path is unchanged.

## Attributes screen (the build page)

The in-game Attributes screen (`src/routes/game/screens/attributes`) is a **point-buy** on the six core attributes (STR/END/INT/AGI/DEX/LUK) with a live preview of the derived stats they feed. It has two in-page **density modes** that share one allocation model, so the toggle reads as a single page densifying rather than two screens: **Guided** (the default) makes the hexagon radar the hero with attribute steppers and plain-language "feeds" tags and _no_ derived-stats panel, so it can't get cluttered as stats are added; **Theorycraft** densifies into an allocation table with per-point marginal yields, a compact radar, and a scrollable grouped derived-stats panel. It reuses the established chrome (Geist/Geist Mono, mono eyebrow + title, the `--accent` system) and is decomposed into small components per the component-size guideline. The pure logic + reactive `AttributesView` live in `attributes-view.svelte.ts` (heavily unit-tested), mirroring `OptionsView`/`InventoryView`.

Key decisions and rationale:

- **Game logic is the single source of truth for derived stats.** The screen does **not** redefine any formula: `deriveStats()` computes derived values through the real `BattleAttributes` (`$lib/battle`), the same model the battle simulation uses, so the page always matches what the game actually computes. Only the derived stats the game produces today are surfaced (`MaxHealth`, `Defense`, `CooldownRecovery`); the deprecated `DropBonus` and the not-yet-implemented crit/dodge/block attributes are intentionally omitted and will appear automatically once they gain real formulas. The mock's illustrative formulas/derived stats were discarded in favour of this. **Feeds tags, per-point marginal yields, and the "fed by" lines are all auto-derived** by bumping an attribute and diffing, so they never go stale when a formula changes.
- **Points move freely — no respec cost.** The backend (`PlayerStatPoints.TryUpdateAttributes`) accepts negative deltas and only requires that no attribute drops below 0 and the pool isn't overspent, so any attribute can be decremented back to 0 to refund its points. The design mock's "Respec · 250 g" affordance and locked baseline were dropped accordingly (there is no gold/respec system); the budget meter shows the live remaining pool and the bar simply caps at full when refunds push it above the baseline.
- **Diff-only save, mirroring the existing endpoint.** `AttributesView` keeps a working `draft` against the persisted `committed` baseline and persists only the changed per-attribute deltas via `Player/UpdatePlayerStats`. On success it writes the returned allocations and the new `statPointsUsed` back onto `playerManager` (so battles and other screens see the change immediately) and re-seeds the baseline; on failure it surfaces a toast and keeps the pending changes. The chosen mode is persisted to `localStorage` per player.
- **Themeable attribute accents.** The six core-attribute hues are declared as `--attr-*` CSS variables in `+layout.svelte` and read only through the helpers in `$lib/common/attribute-display` (`attributeColor`/`attributeTint`/`attributeCode`), following the rarity/challenge-type/category pattern, so a theme can restyle every attribute surface by overriding the variables alone. The radar morph respects `prefers-reduced-motion` (it snaps instead of animating).
- **The radar is draggable, not just clickable.** A vertex is dragged radially to set its attribute directly — outward allocates points (up to the remaining budget), inward refunds them down to 0 — which makes the hexagon the primary allocation control rather than a one-point-per-click affordance. The pointer-to-value mapping is the pure, unit-tested `radarValueAtPointer` (projects the pointer onto the axis so perpendicular drift is ignored), and the result is reconciled through `AttributesView.setValue`, which clamps it into the same bounded `inc`/`dec` the steppers use — so the radar can never overspend or refund below the saved baseline's floor. A motion-free press is still the quick **+1** click (the trailing synthetic click after a real drag is suppressed); the gesture uses pointer events with `touch-action: none` so it works for mouse, pen, and touch.
- **Allocation related stats only** the derived-stat preview reflects the player's **attribute allocations only**, not equipment or other modifiers — the marginal yields and deltas are identical either way (the formulas are linear in the core attributes), but the absolute totals exclude gear. The full combat totals (allocations + `InventoryManager.equipmentStats`) are purposefully omitted (these can be found in the attribute breakdown screen)

## Attribute Breakdown screen (the inspector)

The in-game Attribute Breakdown screen (`src/routes/game/screens/attribute-breakdown`) is a **read-only inspector** — distinct from, and a separate nav entry beside, the point-buy Attributes screen above. It decomposes every implemented attribute into **where its value comes from** (engine base, allocated stat points, equipped gear, applied item mods, derived formulas): a left rail of all attributes (each a full-width stacked source bar + total, grouped Core/Derived) and a right panel that blows the selected attribute up into a by-source breakdown plus a literal **apply-order trace** (the order the engine resolves it). It reuses the established chrome and is decomposed into small components.

Key decisions and rationale:

- **The frontend attribute pipeline mirrors the backend.** Rather than inventing a breakdown, two new `$lib/battle` modules faithfully port `Game.Core.Attributes`: `attribute-modifier.ts` (the `EModifierType` / `EAttributeModifierSource` enums + `STATIC_ATTRIBUTE_MODIFIERS`, mirroring `StaticAttributeModifiers`) and `attribute-collection.ts` (`computeAttributes` / `groupBySource`, mirroring `AttributeCollection` — additive modifiers applied before multiplicative, a `Derived` modifier scaled by the **final** value of its source attribute). These enums are domain concepts, not API DTOs, so they are hand-maintained on the frontend (not codegen'd). The screen assembles the player's real modifiers exactly as `Player.GetAllModifiers` does (stat points → `PlayerStatPoints`, equipped items → `Item`, applied mods → `ItemMod`) from `playerManager.attributes` + `inventoryManager.equippedSlots`. Current real data is **additive-only** (no item is multiplicative yet), so the multiplicative apply-order machinery is present but dormant.
- **One frontend attribute calculator.** `BattleAttributes` (the battle calculator) is now a thin wrapper over the same `computeAttributes` pipeline (#117), so the base/derived formula constants have a single home (`STATIC_ATTRIBUTE_MODIFIERS`) rather than being duplicated across two calculators. A regression test (`attribute-collection.test.ts` → _parity with BattleAttributes_) still asserts the two produce identical totals for every implemented attribute, guarding the wiring so the breakdown can never silently disagree with the simulation.
- **Only implemented attributes are surfaced** (the 6 core + `MaxHealth`/`Defense`/`CooldownRecovery`), matching the point-buy screen's philosophy. The obsolete `DropBonus` and the unimplemented crit/dodge/block attributes are omitted (they have no real contributors) and would appear automatically once they gain modifiers — so the mock's "show inactive/obsolete" toggle was dropped.
- **Themeable source accents.** The five source hues are declared as `--source-*` CSS variables in `+layout.svelte` and read only through the `source-display` helper (`sourceColor`/`sourceTint`/`sourceLabel`), following the rarity/attribute pattern.

## Skills screen (the loadout inspector)

The in-game Skills screen (`src/routes/game/screens/skills`) is the player-facing **loadout editor**: a left **rail** of every unlocked (and, optionally, aspirational locked) skill beside a large **inspector**, with an editable **equipped band** below. The player chooses which skills — and in what order — are equipped for battle, capped at the generated `MAX_SELECTED_SKILLS` constant. It ports the `Skills Inspector — Game Theme` mock (`docs/designs/game/skills/`) and reuses the established chrome (mono eyebrow + title, the `--accent` system, the themeable `--attr-*` accents). The pure logic + reactive `SkillsView` live in `skills-view.svelte.ts` (heavily unit-tested), mirroring `AttributesView`/`InventoryView`; the screen is decomposed into a rail, inspector, equipped band, sort/filter modal, and a shared icon tile per the component-size guideline.

Key decisions and rationale:

- **Display damage _is_ the battle formula.** The page computes its numbers with the shared pure battle formulas (`$lib/battle/battle-formulas`, #347): `calculateSkillDamage` (the function `Skill.calculateDamage` delegates to), `applyDefense` (the `Battler.takeDamage` defense clamp) and `cooldownMultiplier` (the `Battler.cdMultiplier` read) — computed against the same `BattleAttributes` the simulation uses, the player's allocation **plus** `inventoryManager.equipmentStats`. One definition, no display-only copy: a change to the battle arithmetic propagates here automatically and stays guarded by the battle parity suite (see [frontend.md](./frontend.md) → _Battle simulation & parity_).
- **One atomic set-loadout write.** Each committed edit (equip / unequip / swap / reorder) replaces the **whole** ordered loadout through the single `SetSelectedSkills` socket command (the write path from #263), not discrete select/deselect/reorder calls. `SkillsView.commit` applies the new loadout optimistically (and mirrors it onto `playerManager.unlockedSkills` so battles and other screens see it without a reload), then reverts and toasts if the command errors. There is no separate "Save" button — composing the loadout _is_ saving, matching the mock's live feel and the optimistic `SetItemFavorite` precedent.
- **Locked skills are aspirational, not owned.** The catalogue is every non-retired skill plus any retired skill the player still owns (retirement keeps owned records resolvable). Locked (un-owned) skills are hidden by default and revealed by the sort/filter modal's toggle; their "unlock by" line is derived on the client from the challenge catalogue (`rewardSkillId`), so no extra backend read is needed.
- **Compare-vs is a static-defense slider with current-zone enemy presets.** The header's **EFF. DPS/burst** and every per-skill effective value subtract a single Compare-vs flat **Defense** (0 → the biggest hit's raw damage), driving defense-aware sorting — the manual fallback. Above it, the `CompareBar` shows **enemy quick-pick pills** sourced from the player's current zone (`playerManager.currentZone`): every non-retired enemy that spawns there plus the zone's dedicated boss (the boss reads in `--boss-accent`). Clicking a pill snaps the slider to that enemy's flat **Defense** and marks it selected; dragging the slider deselects the pill (#340). The pill defense is **derived client-side from cached reference data** (`enemyDefense` builds a `BattleAttributes` from the enemy's `attributeDistribution` resolved at a level — `baseAmount + amountPerLevel·level` — and reads the derived `Defense`, exactly mirroring the backend enemy build), so no server round-trip is needed. **Level-evaluation rule:** spawns roll uniformly across the zone's `[levelMin, levelMax]` and defense scales linearly with level, so each spawn pill is evaluated at the **range midpoint** (`zoneSpawnLevel` — the expected, hence average, encounter); the boss pill uses the zone's fixed `bossLevel`. When the zone's reference data isn't loaded (or it has no spawns/boss), the bar falls back to the slider-only note — no placeholder enemies are fabricated, honouring the "no mock data in production" stance.
- **Themeable attribute accents.** Skill icon glows, scaling bars, and the equipped-card attribute chips are keyed to the skill's scaling attributes via the `--attr-*` helpers (`attributeColor`/`attributeCode`) in `$lib/common`, so a theme restyles them by overriding the variables alone. The sort/filter modal is a page-local overlay (a live, non-blocking filter panel) rather than the global promise-based `modal` store, which is reserved for blocking confirm/acknowledge dialogs.

## Statistics screen (tracked metrics)

The in-game Statistics screen (`src/routes/game/screens/stats`) fills the former `Stats` placeholder. It presents the player's tracked statistics (`EStatisticType`) two cross-linking ways behind a top-level **By statistic / By entity** toggle: _By statistic_ groups the 14 types into Combat/Survival/Exploration/Time category tabs with per-type cards (each showing its grand total and a per-entity breakdown); _By entity_ picks an enemy/zone/skill and shows every statistic referencing it with that entity's rank. Clicking an entity in a card pivots to its dossier; clicking a stat in a dossier jumps back to its category. Both sub-navs share one `UnderlineTabs` so the screen reads as one system. The pure query logic (`StatisticsData`) + reactive `StatisticsView` live in `statistics-view.svelte.ts`.

Key decisions and rationale:

- **The statistic→entity-kind mapping comes from the backend, not the frontend.** Each statistic's entity breakdown is read from the statistic-type reference data (`IStatisticType.entityType`, loaded into `staticData.statisticTypes` — see [backend.md](backend.md#reference-data)); `buildStatTypes()` merges that with a frontend `STAT_PRESENTATION` table that carries only the concerns the backend doesn't model (unit, aggregation, comparison direction, category, display order). So the frontend no longer hard-codes which kind a statistic breaks down by — a backend change to `StatisticType.GetEntityType` flows through automatically. Statistics whose entity type is `None` (e.g. `PlayerDeaths`, `DamageTaken`, `TotalBattleTime`) render as a headline with no per-entity breakdown.
- **Real data, sourced two ways by lifetime.** Statistic _values_ are per-player and change, so they are fetched fresh from `GET /api/Statistics` when the screen mounts; the statistic-type _metadata_ and the enemy/zone/skill _entities_ are static reference data loaded once on the loading screen (`staticData`). The pure `StatisticsData` takes the merged catalogue + values + entity lists as plain injected lists (so it stays unit-testable), and the reactive `StatisticsView` assembles them from the store. Entity ids resolve against `staticData.enemies/zones/skills` by index, the same zero-based convention the rest of the app uses.
- **Empty vs. error are kept distinct.** A player who has fought no battles has no recorded statistics; the screen shows a friendly empty state (`StatsEmpty`) once the fetch resolves rather than a wall of zeros, and the query helpers return zero headlines / empty breakdowns without erroring. A _failed_ fetch is deliberately **not** collapsed into that empty state (which would tell a returning player to go fight enemies when their data merely failed to load) — it surfaces a `toastError` and a separate error state, per the toast-system guidance that load failures should be reported rather than silently swallowed.
- **`FastestVictory` reconciled on the backend.** It is _recorded_ per enemy (and as a global min), so `StatisticType.GetEntityType` now declares it `Enemy` (previously `None`); the screen simply follows the server metadata. This is safe for challenge evaluation because challenges look a statistic up by its id + the challenge's own target entity, never by the declared entity type.

## Card Game screen (the Initiative Loom)

The in-game Card Game screen (`src/routes/game/screens/card-game`) fills the former `Card Game` placeholder with **"The Initiative Loom"** — a real-time card-boss duel imported from a Claude Design handoff. It is a deliberate _active_ break from the idle grind: the present (a glowing **NOW** line) is a moving deadline, cards are multi-tick **spans** laid onto a scrolling timeline, and each span resolves the instant its right edge crosses NOW. Two overlay lanes (enemy strikes with your blocks on top; your strikes with the boss's guards + crit ticks on the bottom), a draw-over-time deck economy, a held **Reflex** slow-time, an enemy **CHANNEL**, and `1–7`/drag/`space` hotkeys. It reuses the established chrome (dark frame, diamond mark, Geist/Geist Mono) and is decomposed into small components per the component-size guideline.

This is the **demo phase** of a staged plan (integrate as a demo → build the backend → admin-configure → wire to real data). It runs entirely client-side: HP, the deck, and enemy cadences are hardcoded sample values, and a labelled **Sandbox** strip (AGI/DEX/LUCK sliders) stands in for the player's real attributes so the scaling is feelable. The legacy canvas card game (`UI/CardGame`, removed with the rest of the legacy UI) is superseded.

Key decisions and rationale:

- **Pure mechanics, separated from rendering, mirroring the `lib/engine` pattern.** The simulation lives in `$lib/card-game`, not the screen folder: `loom-core.ts` holds the deterministic, framework-free rules (the attribute formulas, half-open span coverage, the no-overlap queue placement, strike-damage), and `loom-game.ts` is a plain `LoomGame` class that owns all state and steps it via `advance(dt)`. Randomness is injected (`rng`) so the engine is fully unit-testable without a browser, and the screen makes it reactive by wrapping it in `statify` (the documented battle-sim pattern) and driving `advance` from a single `requestAnimationFrame` loop — the same plain-class-plus-`statify` split the battle engine uses. The screen's `CardGameView` view-model only adds the UI concerns (board geometry, the in-flight drag, the placement preview).
- **The loom runs on tempo attributes, so a STR-or-INT build stays playable.** A known constraint is that optimised builds end up high in **Strength _or_ Intellect but not both**. The loom's skill therefore lives entirely in the tempo attributes everyone has — **Agility** drives draw speed (`max(1.0, 2.0 − 0.005·AGI)`), **Dexterity** the hand cap (`4 + ⌊log₁₀(DEX)⌋`), **Luck** the crit cadence (`max(6, 26 − ⌊LUCK⌋)`) — while offense is intended to scale off `max(STR, INT)` so a pure-STR and a pure-INT build hit equally hard (the demo uses fixed card damage; the `max(STR,INT)` scaling is the documented intent for the data-wiring phase).
- **Unified right-edge resolution and half-open coverage.** Every entity — enemy strikes/channels and your strikes alike — resolves when its right edge crosses NOW (`resolve === end`), so the moment reads identically on both sides. Coverage spans are half-open `[start, end)`, which makes an impact land _centered_ in a cell instead of straddling an edge and lets adjacent queued spans tile into one seamless wall. The two player queues (block-chain, strike-chain) place tail-to-tail and never overlap (`freeStart`); the background grid scrolls phased at half-ticks so attack edges fall mid-cell while blocks and crit columns snap to grid lines. These rules are the load-bearing bits and are unit-tested directly.
- **Theming reuses existing tokens; one new semantic var.** The port maps 1:1 onto the existing palette — `--accent` (NOW line), `--enemy-accent`/`--log-enemy` (the boss), `--health-remaining-color` (your green), `--rarity-epic` (skill/Channel cards), and all six `--attr-*` hues — with Geist/Geist Mono via `--sans`/`--mono`. The only addition is a semantic `--gold` (in `_colors.scss` + `+layout.svelte`) for the crit ticks and the draw-timer fill: it shares a hue with `--rarity-legendary` but is its own concept (timing reward, not item rarity), per the "use variables by semantic intent" rule. Every neutral/alpha shade is `color-mix(...)` over a base token — no hard-coded `rgba`.
- **Testing matches the prototype's verified scenarios.** `loom-core`/`loom-game` tests assert the same behaviours the design prototype verified (queue chaining with no overlap, off-kilter half-open coverage, crit doubling, a guarded strike reduced to 40%, the enemy resolving at `end`, DEX-scaled hand cap, draw-over-time refill, win/lose). The screen itself is auth-gated, so it carries a mount **smoke-test** rather than a full e2e for now; battle-sim parity with a backend is a follow-up once the duel is backed by real data.

## Fight screen (Challenge Boss)

The Fight screen (`src/routes/game/screens/fight`) layers a deliberate **boss encounter** onto the existing Mirror combatant layout (the boss feature is built on the Mirror layout only). Each zone has one dedicated boss (`Zone.bossEnemyId`/`bossLevel`, authored via the admin Workbench); an always-available affordance slot sits between the zone nav and the arena and holds either the **Challenge trigger** (a boxed, centered, single-row banner naming the boss, with an Auto-fight toggle) or, while engaged, the **boss bar** (with Auto-fight + Retreat). Engaging swaps the enemy `BattlerCard` for a heightened gold `BossBattlerCard` (Zone-Boss ribbon, phase-pip HP bar, gold skill glow), turns on a `BossAtmosphere` vignette, and on victory plays a `ZoneClearedOverlay`. The components live under `fight/boss/`; `BossView` (`boss-view.svelte.ts`) is the thin presentation view-model and the engine work lives in `EnemyManager`. The boss design is the Svelte 5 port of the `fight-boss.jsx` Claude Design handoff (its Banner trigger is the finalized treatment; the handoff's Inline/Plinth trigger explorations were dropped).

Key decisions and rationale:

- **Two mutually-exclusive loops in `EnemyManager`, never running at once.** The "available" state _is_ the normal idle farm loop running with the trigger shown; "engaged" runs the dedicated-boss fight with the idle loop paused. `mode: 'idle' | 'boss'` gates `watchBattleStage`. **Challenge** (`challengeBoss`) sends the `ChallengeBoss` socket command (the backend deterministically builds the boss at `bossLevel` with its full skill loadout and implicitly abandons any idle fight) and engages; **Retreat** drops back to the idle loop; a boss **victory** with auto-fight on re-challenges immediately (no cooldown gate), and with it off hands back to idle; a boss **loss** records it, turns auto-fight off, and returns to the available/idle state. A `transitioning` guard plus a `battleEngine.pause()` keep the outgoing battle's resolving stage change from being mistaken for an outcome of the incoming one.
- **Victory and loss resolve through the same backend paths as idle.** A boss victory sends `DefeatEnemy` (shared `claimVictory` helper grants exp), and a boss **loss** sends the dedicated `BattleLost` command — immediate, and it returns the post-loss cooldown the screen honors. The `IsBossBattle` marker the backend persists at `ChallengeBoss` time is what makes a victory count as a _dedicated-boss clear_ (not an inference from `enemy.IsBoss`), so the FE never has to tell the BE "this was a boss".
- **The boss is deterministic, so FE/BE battle parity is straightforward.** Fixed level + full authored loadout (no rolls) means the shared `bossFullLoadout` scenario in the battle-simulation parity suite is mirrored identically in the frontend (`battle-simulation-parity.test.ts`) and backend (`BattleSimulatorParityTests`) — same inputs, same `totalMs`/outcome — per the project's battle-parity rule.
- **A shared, reactive statistics store backs the "Cleared" seal.** Per-zone `ZonesCleared` previously lived only in the screen-scoped `StatisticsView`; it is promoted to a small `statistics` store (`$stores`) loaded once at game boot, so the fight screen can show the seal for already-cleared zones and the Statistics screen reuses the same source (one fetch). On a boss victory the engine optimistically `markZoneCleared`s the zone so the seal appears immediately, reconciled on the next load.
- **A dedicated `--boss-accent` token, not a reuse of `--gold`.** The heightened gold treatment (`#e8c878`) is declared as its own semantic token in `_colors.scss`/`+layout.svelte`, distinct from the card-game timing `--gold` it currently shares a hue with, so a theme can restyle boss fights independently — the same "variables by semantic intent" rule the editor change-state and card-game tokens follow. The Retreat action reuses the destructive `--enemy-accent`. The shared `Skills` component gained an optional `accent` override (defaulting to the existing behaviour) so the boss card's skills glow gold.
- **The Zone-Cleared overlay is kept truthful.** With zone-navigation locking now in place (#190 — see the zone-progression section in [game-design.md](./game-design.md)), the overlay celebrates the clear (`{boss} defeated`, the Cleared seal, the zone name) and shows the handoff mockup's "Next zone unlocked" line only when the clear actually flipped the next zone's gate open. Reference data (including a zone's boss) is authored through the admin tools — the e2e stack seeds one deliberately weak boss via `e2e-seed.sql` so the flow is exercised end-to-end — and the affordance simply does not render for a zone without a boss.

### Skill-effect indicators (#335)

The player-facing surface of skill effects (the runtime is areas C/D of the #180 spike) is three display-only views, all driven by one shared derivation helper (`$lib/common/skill-effect-display`) so the wording and buff/debuff signalling stay identical everywhere an effect is shown:

- **Skill-effect badge** — a small marker (`components/SkillEffectBadge`) on a battle skill button (`Skills.svelte`) whose authored `effects` list is non-empty. A neutral `--effect-accent` (it signals "carries effects", not direction). The loadout/selection surface badge is deferred with that page (#264).
- **Tooltip "On hit" lines** — the battle `SkillTooltip` lists each authored effect (`EffectLines`), e.g. `+15 Strength · self · 5s`, with the magnitude tinted by direction. Names come from reference data; the line text is pure (`describeEffect`).
- **Active-effect chips** — a row on each battler card (`ActiveEffectChips`, on `BattlerCard` and `BossBattlerCard`) showing one chip per timed effect currently on that battler, tinted buff/debuff, with a **render-interpolated countdown** that depletes smoothly between the 40ms logical ticks and visibly refills on re-application (refresh stacking).

Two decisions hold this together:

- **Buff vs. debuff is a single heuristic, not per-effect authoring.** `effectDirection` classifies an effect as a buff when it raises a beneficial attribute or lowers a harmful one; only `DamageTakenPerSecond` is "harmful when raised" today (`HealthRegenPerSecond` and the rest are beneficial). The `--effect-buff`/`--effect-debuff` accents are themeable tokens read only through the helper.
- **The chip data is a reactive projection that never holds a modifier reference.** `Battler.activeEffects` is a public (`statify`-reactive) `ActiveEffectView[]` the chips render; the modifier each effect added is kept off it in a private `#effectModifiers` map keyed by the effect's id, because a reactive proxy would break the reference identity `BattleAttributes.removeModifier` matches on (the same constraint `BattleAttributes` documents). The render countdown mirrors the skill cooldown bar: a per-frame `updateRenderEffects` interpolates `renderRemainingMs` toward the next tick (driven by `BattleEngine.renderUpdate`), leaving the parity-relevant logical `remainingMs` to the per-tick `advanceEffects`. **Effect logs** (application/expiry messages, and the DoT-tick aggregation gated on #334) are deferred — they need a new persisted `ELogType` and are tracked separately.

## Combat log panel (resizable)

The combat log (`src/components/log-panel`) is a persistent fixture pinned below the active screen inside `.main-content` (`routes/game/+page.svelte`), not a screen of its own. It is **user-resizable**: a thin grab strip straddling its top edge (the `ns-resize`/↕ affordance) lets the player drag the panel taller to read more history at once, shrinking the screen area above it (which is `flex: 1; overflow: auto`, so it just scrolls when squeezed).

Key decisions and rationale:

- **A view-model owns the height, the component owns the DOM.** `LogPanelView` (`log-panel-view.svelte.ts`) holds the reactive `height`/`dragging` state and the gesture lifecycle (`beginResize`/`moveResize`/`endResize`); the pure `clampLogPanelHeight` and the persistence live there so they're unit-testable without a DOM. The component only feeds in DOM-dependent inputs — the pointer `clientY` and the container height (`panelEl.parentElement.clientHeight`) — mirroring the card-game board's drag split.
- **Flex layout, not a fixed body height.** The panel's height is an inline style driven by the view-model and `.log-body` flexes to fill it (replacing the old fixed `132px`), so one number resizes the whole panel. The drag is clamped to `[MIN_LOG_PANEL_HEIGHT, available − MIN_SCREEN_HEIGHT]` so neither the log nor the screen above it can collapse.
- **Pointer events + window tracking.** A single `pointerdown` path covers mouse/pen/touch (`touch-action: none` on the handle); move/up listeners live on `window` so the gesture keeps tracking off the thin handle, and a `body.log-resizing` class locks the `ns-resize` cursor and suppresses text selection for the duration.
- **Height persists in local storage.** The chosen size is written on resize-end and restored after mount (not in the constructor) so the SSR markup and first client render agree — the same hydration-mismatch avoidance the boot gate uses for roles. The restored value is clamped against the live container (and re-clamped on window resize), so a height saved on a larger viewport can't overflow a smaller one.
