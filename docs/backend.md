# File Structure and Projects

The backend follows an onion-style architecture across several projects (each in a folder of the same name):

- **`Game.Core`** — the domain layer: domain objects/services, game mechanics, and the battle simulation. The heart of the game logic.
- **`Game.Application`** — orchestration only: coordinates repositories and domain services to fulfil use cases. No domain logic.
- **`Game.Api`** — ASP.NET Core controllers, WebSocket command handlers, and API models. Also hosts a reflection-based codegen that generates the frontend's API client. Handles HTTP/WebSocket requests and input validation only.
- **`Game.Abstractions`** — shared interfaces (repositories, infrastructure services) and the read/write contracts (`Game.Abstractions.Contracts`). EF entities do **not** live here.
- **`Game.DataAccess`** — repository implementations, the entity↔domain/contract mappers (`Mapping/`), and the in-memory reference-data caches.
- **`Game.Infrastructure`** — the Redis cache/pub-sub implementations, the EF `GameContext`, the **EF entity models** (`Game.Infrastructure.Entities`), and migrations.

Four `*.Tests` projects cover the layers (`Api`/`Application`/`Core`/`Infrastructure`); data-tier tests live under `Game.Application.Tests/DataAccess`, and `Game.Abstractions` (interfaces only) has none. Shared test plumbing lives in two non-shipping projects: `Game.TestInfrastructure` (integration-test fixtures, container/marker plumbing) and `Game.Core.TestInfrastructure` (domain-object builders). `Game.Infrastructure.Tests` is deliberately lightweight — only in-process logic with no out-of-process dependency; anything coupled to Redis or the database is covered by integration tests instead.

# General Backend Guidelines

- Repositories create and persist domain objects and dispatch domain events before persisting. They contain no domain logic and never return entity models or DB DTOs — mapping between domain objects, entities, and DTOs lives in `Game.DataAccess/Mapping`.
- Domain logic lives in `Game.Core`. The application layer only orchestrates; the API layer only handles requests, validates input, and returns responses.

## Testing Guidelines

Unit tests follow the classical (Detroit) school — the project has no unmanaged dependencies, so avoid test doubles. A class that depends on an out-of-process dependency (database, cache) should hold no logic worth unit-testing; if it does, move that logic into a domain class and cover the dependency interaction with integration tests instead.

Judge branch coverage per-branch rather than chasing the number: defensive guards (e.g. config that is always supplied) and branches reachable only by disrupting process-wide shared infrastructure are accepted as measure-only, with the real logic pinned by integration tests against the DI-resolved interface.

### Running tests (Microsoft.Testing.Platform)

The test projects run on **Microsoft.Testing.Platform** (MTP / xunit.v3), not the legacy VSTest runner. The gotcha: VSTest's `--filter` syntax is **silently ignored** (it sits before `--`, so the wrapper swallows it and the *whole* suite runs). Pass MTP filters after `--` instead:

```
dotnet test Game.Api.Tests/Game.Api.Tests.csproj -- --filter-class "*SessionServiceTests"
```

(`--filter-class` / `-method` / `-namespace` / `-trait`, each with a `--filter-not-*` negation; run the binary with `--help` to list them.) A filtered run is a dev convenience — CI runs the whole solution under coverage, so run the full suite before pushing.

# Important Architectural Design Decisions

## Entity model location and isolation

EF entity models are a persistence detail of the data tier and are **invisible to `Game.Api` and `Game.Application`**. They live in `Game.Infrastructure.Entities`, next to `GameContext`. `Game.DataAccess` references `Game.Infrastructure` with `PrivateAssets="Compile"`, so the entity types don't flow transitively up to `Game.Api`, and `Game.Application` never references `Game.Infrastructure` at all. Within the data tier itself the entity seam stays `internal` (the low-level insert/update/delete tracker and the entity-returning reference lookups); the public `Game.Abstractions` repository interfaces speak only domain models and contracts. An architecture test asserts that neither `Game.Api` nor `Game.Application` references `Game.Infrastructure`, failing fast if the wiring ever regresses.

## Reference Data

Reference data is either **intrinsic** or **static**:

- **Intrinsic** reference data is encoded in the application as an enum or similar (e.g. `EAttribute`, `EItemCategory`). It is persisted only for data integrity and is essentially never queried — it is served straight from the enum. Intrinsic sets may carry derived display metadata (classification, formatting) on the domain/contract directly, sourced from the enum with no database columns; adding such a field changes the set's version hash, so clients re-download it once.
- **Static** reference data (items, item mods, skills, enemies, zones, challenges, classes, paths/proficiencies, skill recipes, lessons) is authored content stored in the database and **cached in-memory as immutable lists, indexed by id**. The Id column is seeded from **0** so an entity's id doubles as its array index on both tiers — an O(1) lookup, but one that requires the ids stay contiguous.

The caches are immutable snapshots **loaded eagerly at startup** (a database problem surfaces as a boot failure, not a first-request failure) and refreshed by build-then-swap (see [Reference-data cache reload](#reference-data-cache-reload-build-then-swap)). There is no lazy-fill path.

> **Contiguity is guaranteed by retirement, not convention.** A hard delete of a non-terminal record would open a permanent gap that silently mis-resolves every lookup above it, so top-level records are **retired, not deleted** (below). As a backstop against a seed/migration mistake, each snapshot **asserts `Id == index` as it is built** and throws otherwise, so a corrupt set fails the build-then-swap (keeping the prior good snapshot, or a boot failure on the startup load) rather than serving mis-resolved data.

**Player-load missing-reference policy: loud fail-fast, never silent drop.** Retirement + contiguity keep every owned reference (item, mod, skill) resolvable forever, so an unresolvable one when the player aggregate loads can only mean a content-data mistake (a seed/migration that removed a referenced id). The load then **fails loudly** with a diagnosable exception (player id, catalog, missing id) rather than silently dropping the player-owned row — player data is never discarded, and the message makes the mistake obvious from logs instead of an opaque player-lockout.

### Retiring reference data (content lifecycle)

A top-level reference record carries a nullable `RetiredAt` timestamp — the same flag-plus-audit shape as the User soft-delete. Retiring takes a record out of circulation without erasing it; reinstating clears the timestamp.

- **The slot is permanent** — a retired record is never removed or renumbered, so the cache stays gap-free and every index lookup keeps resolving correctly.
- **It stays resolvable by id**, so existing references (an equipped item, a battle snapshot, an authored reward) keep working. Retirement is a catalogue/authoring concern only — it is not carried on the lean battle models, so a retired item a player owns fights identically.
- **It is out of circulation** — retirement removes a record from every forward-looking path (spawn tables, challenge completion, zone navigation, a path/tier's proficiency XP accrual, and an unviable zone's relocation) while never revoking what a player already earned: a completed challenge keeps its reward, an authored reference still resolves by id, and a frozen track's already-accrued levels and granted skills are untouched (retirement only stops *further* accrual). The per-entity mechanics (spawn-table exclusion, challenge/zone gating, the path-freeze reverse index, and zone-viability resolution) live in [backend-admin.md → Content-lifecycle retirement guards](./backend-admin.md#content-lifecycle-retirement-guards).
- **Authoring guards (backend-enforced, reported via `AdminSaveResult.Failure`).** Some retirements would author live content into a permanently broken state the runtime deliberately does not paper over — a live zone left with no spawnable enemy, or a live proficiency gateway whose prerequisite path is frozen — so they are **rejected at save time** as anti-tamper (a tampered admin client can't bypass a frontend check). The specific guards live in [backend-admin.md → Content-lifecycle retirement guards](./backend-admin.md#content-lifecycle-retirement-guards).
- **No hard delete** of a reference record: the admin "delete" affordance is Retire/Reinstate, and the reference admin repos reject a top-level delete outright.

**The Home sanctuary is the one deliberately-empty zone.** A zone flagged `IZones.IsHomeZone` (entity + contract only — the battle never reads it, resolved like `IsZoneRetired`) is a no-combat place the player idles in without fighting. It is exempt from the empty-zone relocation above: it never spawns enemies (`AdminZones`/`AdminEnemies` reject a boss or spawn-table membership on a Home zone) and the `StartBattle` zone-change anti-cheat refuses a move into it, so a player's persisted `CurrentZoneId` is never Home — and the offline-rewards path keeps replaying their last real combat zone unchanged. The sanctuary is **singular**: `AdminZones.SaveZones` rejects any save that would leave more than one non-retired Home zone (a second Home add or a combat zone flipped to Home), checking the whole post-save catalogue rather than the changed record. See [game-design.md](./game-design.md#home-zone--a-no-combat-sanctuary).

**Tags are the deliberate exception.** They carry their own (non-zero-based) identity and are resolved by lookup, not by index, so they never suffer the mis-resolution bug retirement exists to prevent. Their admin "delete" stays a real hard delete (graceful for an in-use tag — the join rows cascade and the caches reload so the removed tag id disappears on the next read).

### Reference-data read contracts

The read path never exposes EF entities:

- **Reference reads return read contracts** (`Game.Abstractions.Contracts`) — one published language shared by the game client's loading screen and the admin Workbench. The internal cache stays as entities (the domain mappers need their fidelity) and is projected to contracts, preserving the id-as-index ordering.
- **Gameplay reads return lean `Game.Core` domain models**, not entities, and hand back **shared, pre-materialized instances** by id rather than re-mapping a fresh graph per call (these reads are on the per-battle hot path). This is safe because the reference-data domain models are structurally immutable value objects — `init`-only properties and `IReadOnlyList<>` collections, enforced by the compiler and guarded by a reflection test — so a consumer cannot grow or reassign the shared graph and corrupt the cache for every player. The level-parameterized enemy read is the same idea adapted: a level-independent template is shared and cloned at the requested level.
- **Admin writes keep entities inside the data tier** — the entity-returning lookups live on `internal` data-tier interfaces consumed only by the Content Authoring admin repositories (see [backend-admin.md](./backend-admin.md#admin-tools-api-surface-content-authoring-context)).

### Reference-data cache reload (build-then-swap)

Cache busting is an **eager build-then-swap** (stale-while-revalidate), not a null-and-lazily-refill (see the [cache-reload spike](./spikes/356-reference-data-cache-reload.md)). Readers never observe an empty or torn snapshot, and never pay a refill query inline.

- A **singleton holder** per set owns the current immutable snapshot and exposes a lock-free read plus `ReloadAsync()`. A reload builds the whole new snapshot off to the side (on its own DbContext) and publishes it with a single atomic reference swap, so a **failed reload leaves the old snapshot in place**. Derived structures (e.g. the per-zone spawn tables built from the enemy list) are bundled into the snapshot so they swap atomically and a reader can never see a new list against stale derived data. A per-holder semaphore serializes reloads to preserve read-your-writes.
- **An admin write triggers an awaited reload** after the write commits (so the Workbench reads its own writes with zero gap) and **broadcasts a cross-instance invalidation over the Redis backplane**. Other instances debounce the notification and run one background reload sweep; the publishing instance skips its own message. A failed background sweep is retried with backoff and never disturbs readers. The local reload is bounded by a timeout so a wedged database connection can't hold the admin request open indefinitely. The broadcast publish itself is **awaited, not fire-and-forget** — unlike the write-behind wake signal, it has no durable write backing it, so a genuine send failure must surface (logged at the admin-write site) rather than vanish silently. Delivery to subscribers is still Redis pub/sub's ordinary at-most-once guarantee, so a subscriber missing the message mid-reconnect only self-heals on the next notification or restart — a known residual gap (#1888).

> **Snapshot read-once idiom (repo reads).** Because the swap replaces the snapshot reference atomically, a reference repo must read `holder.Current` **once per logical operation** and read every field/derived structure off that captured snapshot — never re-read `holder.Current` mid-operation (e.g. via a property getter), or one call could mix an old and a new snapshot. The thin repo readers expose a single `Snapshot => holder.Current` accessor and bind it once; methods that need more than one read capture it into a local first (`var snapshot = holder.Current;`).

### Content pipeline (export → seed → lint → calibration)

The authored content graph is mirrored one-way to source-controlled JSON (`content/*.json`, generated never hand-edited, with a CI byte-equality drift guard), reseeded onto any freshly-migrated database at startup (`IContentSeeder`, idempotent), whole-graph lint-checked in CI and in the admin Content Health view (`IProgressionGraphChecker` — dangling references and wedged content are Errors, tolerated dead content needs an explicit allowlist entry), and tuned via the re-runnable combat-rating calibration report. The mechanics live in [backend-content.md](./backend-content.md).

### Per-field authoring metadata & battle-input fields

Beyond the architecture above, the authored sets carry a catalogue of per-field design decisions — the recurring authoring-metadata pattern (entity + contract only unless the battle reads it; version-hash bump; backfilled default) and the individual fields built on it (designer notes, skill acquisition flags, words of power, skill recipes, skill rarity, skill damage portions, item-granted skills, weapon type + the no-stranding invariant, the weapon-match gate, and the proficiency gear gate). These are only needed when working on a specific field, so they live in [backend-reference-fields.md](./backend-reference-fields.md).

## Player-facing read projections

Player-facing reads also avoid exposing EF entities, but — unlike reference data — they reuse the existing **gameplay domain models** rather than dedicated contracts (those models are already lean enough), with the API DTOs projecting from them.

- **Statistics, challenge progress, and proficiency progress** (level + XP per proficiency) go through the single `IPlayerProgressRepository`, which also owns the progress write path, giving one source of truth for player-progress data. The write-behind/dirty-set plumbing, the row-presence-vs-zero-value semantics, and the challenge-evaluation reverse index live in [backend-persistence.md → Write-behind player cache](./backend-persistence.md#write-behind-player-cache).
- **The skill loadout is sent as the full unlocked set** (each skill with its `Selected`/`Order`), mirroring how inventory is sent as the full unlocked item set; the client derives the ordered equipped loadout from it. The loadout cap is a generated client constant, not a per-player wire field.
- **The loadout is edited through one atomic socket command** that replaces the whole equipped set in order (handling select, deselect, and reorder in one path). Like the other player mutations it validates as **anti-cheat** — no duplicate ids, count within the cap, every id already unlocked — and rejects with no state mutation on any violation.

## Tutorial lessons (reference set + player-owned read state)

Tutorial content (spike #1392) is an authored **`Lesson` reference set** like any other static catalogue — cache holder, `GetLessons` socket command, export/seed/lint coverage. The per-player `locked → unread → read` lifecycle is persisted on the **Player aggregate's write-behind** (`LessonUnlockedEvent`/`LessonReadEvent`), deliberately **not** the `PlayerProgress` aggregate — lesson state is player-owned UI state, not battle-derived progress. There is also **no server push**: the client evaluates the triggers (see [frontend.md → Tutorials](./frontend.md#tutorials)), so unlock and read arrive as client-initiated socket commands (`UnlockLesson`/`MarkLessonRead`).

## HTTP vs WebSocket Communication

**The goal: once the session is live, every game-client read and player-state mutation goes over the socket — only the auth flow stays on HTTP.** This is what lets the player's state be held in-memory on the backend and updated in real time without re-synchronizing against Redis or the database on each request. Users connect to one instance at a time, and a player's socket commands are handled sequentially. The goal has been reached; the HTTP that remains is deliberately scoped to:

- the **auth and character-selection flow** (login, account creation, refresh, logout, status, active-session, device-info, plus the pre-game player surface: `Players`, `CreatePlayer`, `CharacterCreationData`, `SelectPlayer`, and the in-game `SwitchPlayer`) — it runs before a socket exists (or, for `SwitchPlayer`, deliberately after the client tears its socket down), and the active-session check must deliberately *not* open one;
- admin **persistence** endpoints; and
- the admin-only reads that have no socket equivalent yet (tags, user/role lookups, and the Ops/diagnostics reads — dead-letter queues, Content Health).

**Player-state mutations belong on the socket because the socket serializes them with the battle loop.** Player writes are read-modify-write against the in-memory aggregate; the socket read loop processes one command per player at a time, so a write is naturally serialized with the idle battle-completion commands that mutate the same cached player. The same write over HTTP would run concurrently on a separate thread and could silently clobber a background battle save. Every player-mutating action is therefore a socket command and validates as anti-cheat (rejecting with no mutation) — e.g. a stat allocation only accepts the core attributes that actually have an allocation row, which blocks allocating into derived attributes.

**The per-player sequential guarantee is enforced by a per-socket command lock**, because server-initiated pushes arrive over the Redis backplane and run *off* the read loop — two paths that could otherwise mutate the same cached player concurrently. Each command runs under a per-command timeout whose cancellation token is plumbed down through the application services and the data tier, and a separate per-socket send lock guards WebSocket output. The lock/timeout mechanics live in [backend-sockets.md](./backend-sockets.md#per-socket-command-serialization--timeouts).

## Socket connection lifecycle

A player may only be connected to one instance at a time (a TTL'd Redis presence key + a takeover signal to the replaced socket, with the `Login/ActiveSession` HTTP endpoint letting the client warn *before* triggering a takeover), and a stopping instance drains its live sockets cleanly so rolling deploys just reconnect clients to a healthy instance. Both `SocketContext.Close` and `SendData` self-bound and fall back to `WebSocket.Abort()` so a wedged peer can never pin the send lock, the handler, or the in-memory `Player`. The presence/takeover, shutdown-drain, and wedged-peer mechanics live in [backend-sockets.md](./backend-sockets.md).

## Reference-data caching and versioning

The frontend caches each reference-data set in the browser and re-downloads only the sets that changed (see `frontend.md`). The `GetReferenceDataVersions` socket command supports this by returning a **content hash per set** in one round-trip. The hash is computed by hashing the same serialized models the client actually receives (SHA-256 over the camelCase JSON), so the version changes if and only if the client-visible data changes — there is no separate version counter to maintain or invalidate. The versioned set is exactly the set of reference-data socket commands (discovered by assembly scan), so a newly-added one is versioned automatically.

`GetReferenceDataVersions` is the **first** command the loading screen issues on every connect, so the hash is **memoized per set, keyed on the immutable cache snapshot it was computed from** — the serialize-and-hash runs once per cache swap rather than once per connect. Keying on the snapshot instance makes invalidation automatic: a build-then-swap publishes a new snapshot object (a memo miss), so an admin write can never serve a stale hash. Intrinsic (enum-derived) sets have no holder to swap, so they key on a process-stable sentinel and hash once for the process lifetime.

## Battle (setup, runtime & skill effects)

Setting up and running a battle is a domain decision, not an orchestration one: the application layer only snapshots the player and the chosen enemy loadout, sets the active battle, and delegates the game-logic choices to `Game.Core`. **Parity invariant:** battle logic runs on both the frontend and backend and the results must be identical — the backend replays the client's reported battle as anti-cheat — so these mechanics are mirrored on both sides and pinned by a parity matrix; they must never diverge. The tick-by-tick detail — encounter setup, snapshot reconstruction, zone-unlock anti-cheat, the RNG seed, and the skill-effect/DoT-HoT runtime — lives in [backend-battle.md](./backend-battle.md).

Two adjacent server-side subsystems hang off battle completion: **offline rewards** replay the away window through a server-side batch simulation of the same domain logic (mechanics in [backend-battle.md → Offline rewards simulation](./backend-battle.md#offline-rewards-simulation), player-facing rules in [game-design.md](./game-design.md#offline-rewards)), and **combat rating** condenses a battler into the scalar `√(Offense × Survivability)` measure that prices XP bounties and proficiency accrual (model owned by [game-design.md → Experience Rewards](./game-design.md#experience-rewards); calibration tooling in [backend-content.md](./backend-content.md)).

Two distinct constant classes back this: **`GameConstants`** holds values that must stay identical on both sides (tick size, loadout cap, …) and is `[ClientMirrored]` so codegen emits them to the TypeScript client, while **`ServerGameConstants`** holds server-only anti-cheat/reward clamps the client never evaluates (e.g. the exp-reward caps) and is deliberately not mirrored. Put a constant in the right one — a client-mirrored value referenced as a server-only one (or vice versa) is a recurring footgun.

### Damage types (taxonomy foundation)

Damage carries a discriminator (spike #1320). A skill/DoT instance has exactly one leaf `EDamageType` (physical, the four elements, the bleed/poison/burn DoT forms, and the **weapon leaves** below); per-type and per-category **amplification** (attacker) and **resistance** (defender) ride the normal attribute-modifier pipeline as `EAttribute` percentage values (decimal-percentage convention, base `0`), so any modifier source — proficiency, item, mod, skill-effect, class passive — can grant a typed buff with no new plumbing. The single piece of machinery is the lookup indirection in `Game.Core.Attributes.DamageTypes`: a static `Applies(type) → keys` map resolves a leaf type to the amp/resist **keys** (the leaf type itself plus cross-cutting `Elemental`/`DoT` categories) whose attributes apply to a hit, in a **fixed iteration order parity depends on**. That map and the key→attribute pairing **codegen to the client** (`$lib/api/types/damage-types.ts`, consumed through the hand-written `$lib/battle/damage-types.ts` wrapper) like `StaticAttributeModifiers`, so the two simulators cannot drift. The pipeline is **live**: every hit folds the applicable amplification attributes into outgoing damage and the defender's resistance attributes into mitigation (`Battler.AmplifyDamage` and friends, mirrored in the client simulator), and combat rating prices typed resistances into its survivability measure.

**Weapon-type leaves (spike #1340).** Martial masteries (Swordsmanship, Axe mastery, …) are ordinary type-routed leaves — `Sword`/`Axe`/`Bow`/`Club`/`Dagger`/`Unarmed` — sitting under the `Physical` **category key** (`Applies()` = `[<weapon>, Physical]`), parallel to how the elements sit under `Elemental`. They are **statically per-skill typed** (a martial skill is authored with a fixed weapon type, not inherited from the equipped weapon) and purely **additive** on the shipped #1320 foundation (no migration beyond seeding the new amplification attributes). The one asymmetry with the elemental keys: a weapon key is **amplification-only** — there is no per-weapon resistance, so `DamageTypes.KeyInfo.Resistance` is nullable and a weapon hit resolves to `PhysicalResistance` via the shared `Physical` key (physical resistance stays the only physical mitigation lever; per-weapon resistance is a deferred opt-in). The enums grow **append-only**, so a weapon leaf no longer shares a numeric value with its key (the `EDamageType.Fire === EDamageTypeKey.Fire` coincidence holds only for the original #1320 leaves); a leaf type's own key is recovered as `Applies(type)[0]` (always first), which both simulators rely on. `Unarmed` is the no-weapon default. Offense proficiency routing reaches the weapon paths through the same `Applies → ActivityKeys.ForDamageKey` fold (a sword hit trains the weapon path **and** the `Physical`/Martial path).

## Caching, Pub/Sub & write-behind persistence

Most reference data is cached in-memory (see [Reference Data](#reference-data)), but **player data uses a write-behind cache where Redis is the source of truth**: a live player's aggregate is loaded once onto the connection-scoped session, mutated in-memory per socket command, and persisted to Postgres asynchronously through a Redis queue drained by a background service. Player progress (statistics + challenges) is a second such write-behind aggregate under its own key, and Redis pub/sub doubles as the WebSocket backplane and the carrier for cross-instance reference-data invalidations. The deep mechanics — the sliding TTLs, the domain-event → batched-publish path, dead-lettering and retry, drain serialization, and worker ownership — live in [backend-persistence.md](./backend-persistence.md).

On the `ICacheService` cache contract every value-accepting setter carries an expiry, and they fall into two camps. `Set` and `SetAndForget` take a **nullable** value where a **null value deletes the key** — the de-facto null-means-delete behaviour their generic `<T>` overloads rely on. `GetSet` (the atomic set-with-expiry) is **non-null**: writing a TTL implies writing a value, so a null would error its server-side Lua `SET` — it has no delete path to expose.

## Server-push socket commands

Some game state changes as a side effect of battle-completion handling, decoupled from the socket command that triggered the battle — so the command's response can't carry it, and it would otherwise surface only on the next page load (the client doesn't evaluate challenges or proficiency accrual itself; the server is authoritative). Instead the aggregate raises a domain event and an API-layer handler pushes a server-initiated socket command to the player over the Redis backplane, so it reaches them on whichever instance holds their connection and covers **every** completion path (idle victory, won-abandon, even a loss that completes a counting challenge). The gameplay pushes today are `ChallengeCompleted` and `ProficiencyXpGained` (plus the `ServerCommandFailed`/`SocketReplaced` lifecycle notices). Being server-initiated, a push carries a typed response but no request parameters, so codegen emits no client request type.

A server push that **faults** is escalated rather than silently dropped: the poisoned payload is dead-lettered (preserved on a holding queue for inspection, mirroring the player write-behind dead-letter path) and the affected socket is sent a one-way `ServerCommandFailed` notice carrying the failed command's name, so the client re-syncs the authoritative state the push would have updated (a failed `ChallengeCompleted` force-reloads challenge progress, which gates zone navigation) instead of silently diverging. Faults are **not retried**: server pushes have no transient failure mode (a poisoned payload can't be fixed by a retry) and re-executing across a released-lock window isn't generally safe. Like the player write-behind dead-letter queue, the socket command dead-letter queue has an admin inspect/replay Ops surface (#1542); its depth is also logged on every escalation so an accumulating backlog is visible without opening the Workbench. The fault-classification taxonomy (timeout vs. teardown cancellation vs. malformed parameters vs. a merely-failed delivery) and the replay mechanics live in [backend-persistence.md → Queue processing resilience](./backend-persistence.md#queue-processing-resilience).

## Proficiency XP accrual (battle-completion hook)

Proficiency XP is computed **server-side at battle completion, off the tick loop** — like statistics and challenge rewards, never inside the deterministic 40 ms simulation. This is the deliberate boundary that keeps it off the frontend/backend battle-parity surface: the accrual reads the simulation's typed damage books and combat ratings after the fact, and only the eventual proficiency *attribute bonus* is parity-sensitive (both sides compose it from player state at snapshot time — see [Attribute composition](#attribute-composition-battle-assembly-parity)). The math is a pure domain function (`Game.Core.Proficiencies.ProficiencyAccrual`) that **both** the live path (`ProficiencyRewardService`, per victory) and the offline simulator (in-loop per simulated victory) run, so the two paths accrue identically; openness is **derived, never stored**, and the only persisted milestone effect is the idempotent skill grant. The player-facing mechanic is owned by [game-design.md](./game-design.md#proficiency-xp--effect-based-power-normalized-accrual); the execution paths, milestone/skill-grant flow, and derived-openness rules live in [backend-proficiency.md](./backend-proficiency.md).

## Attribute composition (battle-assembly parity)

The proficiency *attribute bonus* is one of three frontend↔backend **attribute-composition parity surfaces** assembled into the battler from player state at snapshot time. Each is composed into the modifier pipeline at a **fixed position** (float addition isn't associative, so order is part of the contract) and pinned **bit-exactly** by a mirrored parity test, since the backend replays the client's battle as anti-cheat:

- **Proficiency attribute bonus** — authored per-level/milestone payouts (`EAttributeModifierSource.Proficiency`), composed with the base set (allocations + gear) *before* the statics; **frozen into `BattleSnapshot`** at battle start.
- **Class locked base** (spike #1126 area D) — the level-scaled, non-reallocatable half of the hybrid attribute model (`BaseAmount + AmountPerLevel × level`, `EAttributeModifierSource.AttributeDistribution`), composed after gear and before the proficiency bonuses; the free pool is the reduced per-level stat-point grant.
- **Class signature passive** (spike #1126 area E) — the flat or attribute-scaled combat-identity bonus (`EAttributeModifierSource.Class`), composed **last**, after the statics, so a scaled passive reads fully-resolved attribute values.

The locked base and signature passive also feed the `DefeatRewards` combat-rating measure. The composition-order contract, the `double`-arithmetic rule that keeps the decimal-authored values bit-identical to the frontend's IEEE-754 recompute, the live-frontend wiring, and the named parity tests live in [backend-attributes.md](./backend-attributes.md).

## Authentication

JWT bearer auth, not server sessions. Access tokens are signed (HMAC-SHA256), short-lived (15 min), and carry the user id + role claims; refresh tokens are opaque, long-lived (48h), single-use, and rotate on every refresh. Refresh tokens live server-side in Redis, keyed by a SHA-256 hash of the token — chosen because auth already depends on Redis, so it adds no new durability assumption.

**The selected player is a token claim.** An account can own multiple characters, so which one a request acts as is carried as a `player_id` claim on the access token (and on the refresh-token record, so it survives a refresh) — set once a character is chosen, absent on a pre-selection token. Both the socket handshake and the session-rehydration path read the bound player from this claim rather than defaulting to the account's first player, so an evicted/uncached session re-binds to the *selected* character. Login no longer binds a player; see [Account/login orchestration](./backend-auth.md#accountlogin-orchestration).

The **token is the sole authorization source of truth** (roles are read off the validated principal, never the session cache; likewise the selected player), so a role change only takes effect on next login. A global fallback policy requires every endpoint to be authenticated unless explicitly marked anonymous. WebSockets authenticate via an `access_token` query-string parameter, since browsers can't set headers on the WebSocket handshake. The frontend owns token persistence, pre-emptive refresh, and the single-flight refresh path (see `frontend.md`).

**The validated token is likewise the sole authority for whether a caller is *authenticated* — never the gameplay session cache.** That cache (`Session_{userId}` → the in-flight `PlayerState`) is a volatile presentation convenience holding the in-flight battle state and the user→player binding; it can be evicted (Redis flush, sliding-TTL lapse) or simply never established on a given instance. A valid token with no cached session is therefore *authenticated-but-uncached*, not anonymous: the session is rehydrated in memory by binding the player named in the token's selected-player claim instead of reporting "not logged in" (a pre-selection token with no such claim is authenticated but has no player, surfaced as a graceful error). The claim is authoritative — a cached session bound to a different character than the claim names is re-bound to the claim's player. `SessionService.Authenticated` is gated only on the token-derived `UserId`; a cache hit/miss governs `HasPlayerSession`, never authentication. This rehydration is **in-memory only** — it re-binds the request's `PlayerState` but does **not** write the session cache, since that write would land on the concurrent HTTP thread and player-state writes belong on the socket (above). The cache is (re)populated only by login (which establishes the binding before any socket exists) and by the socket's write-behind path.

`SessionLoaderMiddleware` records only the token-derived identity (the `UserId` and the selected-player claim) on every authenticated request; the `GetSession` read (and the rehydration on a miss) is deferred to the consumers that actually need player state — the socket handshake, the `Status`/`ActiveSession` auth endpoints, and `SwitchPlayer`'s departed-character credit — via `SessionInitializer`. So the rest of the authenticated HTTP surface (admin tooling, refresh, device-info) pays no per-request session-cache read.

**Passwords** use **PBKDF2-HMAC-SHA256** behind an `IPasswordHasher` abstraction (tunable work factor, per-hash salt, app-wide pepper), with constant-time verify and equivalent dummy work on an unknown username so timing never reveals whether an account exists. **Login is split around player selection** — `Login` verifies credentials and returns a pre-selection token plus the account's character summaries, then an authenticated `SelectPlayer` binds the chosen character and rotates the token to carry the `player_id` claim; an in-game `SwitchPlayer` changes the live character losslessly, and characters are created post-signup under a per-account cap. The anonymous auth endpoints are protected by a **per-IP rate limiter** and, layered on top, a **per-account exponential backoff** that is deliberately a bounded delay, **never a hard lockout**. The orchestration, hashing, and abuse-mitigation mechanics live in [backend-auth.md](./backend-auth.md).

## CORS allowed origins (deployment config)

The browser CORS policy's allowed origins are **configuration-bound, not hardcoded** — they are deployment-specific. The list supports multiple origins and is validated at startup (must be non-empty), so a misconfigured environment fails fast rather than silently rejecting every browser request. The local dev origin ships in the Development config.

## Unhandled exception handling

A centralized exception-handling middleware is the **single place** unhandled exceptions are logged and turned into a response. It converts any exception escaping the downstream pipeline into the project's consistent `ApiResponse` error envelope with a 500 status, so clients get the same `{ errorMessage }` shape they get for business failures rather than the host's default response. Internal details (the exception message) are surfaced **only in Development**; other environments get a generic message so nothing leaks. A client-disconnect cancellation is not treated as a server error, and an already-started response (e.g. an upgraded WebSocket) is left to unwind rather than corrupted. It runs **inside `RequestLoggingMiddleware`** so the 500 is set before the "Request Ended" event logs the status; `RequestLoggingMiddleware` therefore stays focused on structured start/end events and never logs the exception itself.

## Access Roles and Admin Authorization

Authenticated users are gated out of admin tooling unless they hold the `Admin` role. **Roles live in the signed auth token, not the session store** — the token is the source of truth for what a request is authorized to do, which keeps the check self-contained and avoids a second lookup; the trade-off is that a role change only takes effect on the user's next login. The admin authorization filter reads the role straight off the cryptographically validated principal, **never** off the gameplay session cache: that cache is a presentation concern decoupled from the token's validity, so keying off it would wrongly reject a valid admin whose session key happens to be absent (evicted, never established, or aged out).

## Admin tooling & user administration

Reference-data administration is modelled as a distinct **Content Authoring** bounded context (thin admin controllers → admin repositories that own persistence; the EF entity never surfaces in `Game.Api`), and user/role management is a separate **Identity / User Admin** context that shares the `User` aggregate with the auth flow. Both speak `Game.Abstractions.Contracts` types rather than entities, and every admin write reloads the in-memory reference caches. Users are archived (a soft delete that frees the username) or banned (blocked but name reserved). The API surface, the change-set/reconciler persistence patterns and their zero-based-id gotchas, the archive-vs-ban model, and the connection-tracking tables live in [backend-admin.md](./backend-admin.md).
