# File Structure and Projects

The backend follows an onion-style architecture across several projects (each in a folder of the same name):

- **`Game.Core`** â€” the domain layer: domain objects/services, game mechanics, and the battle simulation. The heart of the game logic.
- **`Game.Application`** â€” orchestration only: coordinates repositories and domain services to fulfil use cases. No domain logic.
- **`Game.Api`** â€” ASP.NET Core controllers, WebSocket command handlers, and API models. Also hosts a reflection-based codegen that generates the frontend's API client. Handles HTTP/WebSocket requests and input validation only.
- **`Game.Abstractions`** â€” shared interfaces (repositories, infrastructure services) and the read/write contracts (`Game.Abstractions.Contracts`). EF entities do **not** live here.
- **`Game.DataAccess`** â€” repository implementations, the entityâ†”domain/contract mappers (`Mapping/`), and the in-memory reference-data caches.
- **`Game.Infrastructure`** â€” the Redis cache/pub-sub implementations, the EF `GameContext`, the **EF entity models** (`Game.Infrastructure.Entities`), and migrations.

Each layer has a matching `*.Tests` project. `Game.Infrastructure.Tests` is deliberately lightweight â€” only in-process logic with no out-of-process dependency; anything coupled to Redis or the database is covered by integration tests instead.

# General Backend Guidelines

- Repositories create and persist domain objects and dispatch domain events before persisting. They contain no domain logic and never return entity models or DB DTOs â€” mapping between domain objects, entities, and DTOs lives in `Game.DataAccess/Mapping`.
- Domain logic lives in `Game.Core`. The application layer only orchestrates; the API layer only handles requests, validates input, and returns responses.

## Testing Guidelines

Unit tests follow the classical (Detroit) school â€” the project has no unmanaged dependencies, so avoid test doubles. A class that depends on an out-of-process dependency (database, cache) should hold no logic worth unit-testing; if it does, move that logic into a domain class and cover the dependency interaction with integration tests instead.

Judge branch coverage per-branch rather than chasing the number: defensive guards (e.g. config that is always supplied) and branches reachable only by disrupting process-wide shared infrastructure are accepted as measure-only, with the real logic pinned by integration tests against the DI-resolved interface.

### Running tests (Microsoft.Testing.Platform)

The test projects run on **Microsoft.Testing.Platform** (MTP / xunit.v3), not the legacy VSTest runner. The gotcha: VSTest's `--filter` syntax is **silently ignored** (it sits before `--`, so the wrapper swallows it and the *whole* suite runs). Pass MTP filters after `--` instead:

```
dotnet test Game.Api.Tests/Game.Api.Tests.csproj -- --filter-class "*SessionServiceTests"
```

(`--filter-class` / `-method` / `-namespace` / `-trait`, each with a `--filter-not-*` negation; run the binary with `--help` to list them.) A filtered run is a dev convenience â€” CI runs the whole solution under coverage, so run the full suite before pushing.

# Important Architectural Design Decisions

## Entity model location and isolation

EF entity models are a persistence detail of the data tier and are **invisible to `Game.Api` and `Game.Application`**. They live in `Game.Infrastructure.Entities`, next to `GameContext`. `Game.DataAccess` references `Game.Infrastructure` with `PrivateAssets="Compile"`, so the entity types don't flow transitively up to `Game.Api`, and `Game.Application` never references `Game.Infrastructure` at all. Within the data tier itself the entity seam stays `internal` (the low-level insert/update/delete tracker and the entity-returning reference lookups); the public `Game.Abstractions` repository interfaces speak only domain models and contracts. An architecture test asserts that neither `Game.Api` nor `Game.Application` references `Game.Infrastructure`, failing fast if the wiring ever regresses.

## Reference Data

Reference data is either **intrinsic** or **static**:

- **Intrinsic** reference data is encoded in the application as an enum or similar (e.g. `EAttribute`, `EItemCategory`). It is persisted only for data integrity and is essentially never queried â€” it is served straight from the enum. Intrinsic sets may carry derived display metadata (classification, formatting) on the domain/contract directly, sourced from the enum with no database columns; adding such a field changes the set's version hash, so clients re-download it once.
- **Static** reference data (items, item mods, skills, enemies, zones, challenges) is authored content stored in the database and **cached in-memory as immutable lists, indexed by id**. The Id column is seeded from **0** so an entity's id doubles as its array index on both tiers â€” an O(1) lookup, but one that requires the ids stay contiguous.

The caches are immutable snapshots **loaded eagerly at startup** (a database problem surfaces as a boot failure, not a first-request failure) and refreshed by build-then-swap (see [Reference-data cache reload](#reference-data-cache-reload-build-then-swap)). There is no lazy-fill path.

> **Contiguity is guaranteed by retirement, not convention.** Because lookups index by id, a hard delete of a non-terminal record would open a permanent gap that silently mis-resolves every lookup above it. Top-level reference records are therefore **retired, not deleted** (see below). As a backstop against the one remaining way a gap could appear (a seed/migration mistake), each snapshot **asserts `Id == index` as it is built** and throws otherwise, so a corrupt set fails the build-then-swap (keeping the prior good snapshot, or surfacing as a boot failure on the startup load) rather than serving silently mis-resolved data.

**Player-load missing-reference policy: loud fail-fast, never silent drop.** When the player aggregate is loaded, its owned references (items, item mods, skills) are resolved against these catalogs. Retirement + contiguity keep every owned slot resolvable forever, so an unresolvable owned reference can only mean a content-data mistake (a seed/migration that actually removed a referenced id). The chosen policy **guarantees referential integrity**: the load fails loudly with a diagnosable exception naming the player id, catalog, and missing id â€” it does **not** silently skip/drop the player-owned row (player data is never discarded). In practice it should never fire; when it does, the message makes the content-data mistake obvious from logs instead of surfacing as an opaque player-lockout.

### Retiring reference data (content lifecycle)

A top-level reference record carries a nullable `RetiredAt` timestamp â€” the same flag-plus-audit shape as the User soft-delete. Retiring takes a record out of circulation without erasing it; reinstating clears the timestamp.

- **The slot is permanent** â€” a retired record is never removed or renumbered, so the cache stays gap-free and every index lookup keeps resolving correctly.
- **It stays resolvable by id**, so existing references (an equipped item, a battle snapshot, an authored reward) keep working. Retirement is a catalogue/authoring concern only â€” it is not carried on the lean battle models, so a retired item a player owns fights identically.
- **It is out of circulation** â€” a retired enemy is dropped from the random per-zone spawn tables, a retired challenge can no longer be completed by a player who has not already completed it (challenge evaluation skips it), a retired zone is non-navigable (the `StartBattle` zone-change anti-cheat refuses a move into one, and its boss cannot be challenged), and a retired **path** is frozen for XP accrual â€” it is excluded from the skillâ†’contribution reverse index the proficiency cache builds, so no skill routes XP into it and the whole track accrues nothing, levels nothing, and grants no further skills (the path is the freeze unit, since a path's tiers progress as a contiguous frontier walk). A player whose current zone is no longer **viable** â€” retired, or left with no spawnable enemies â€” is lazily relocated to the nearest viable, unlocked zone on their next encounter, so the idle loop never stalls or throws on an empty spawn table. Viability combines the zone-retirement flag (`IZones.IsZoneRetired`) and spawn-table presence (`IEnemies.HasSpawnableEnemies`). Already-earned references are never revoked: a completed challenge keeps its reward, an authored reference still resolves by id, and a frozen track's already-accrued levels and granted skills are untouched (retirement only stops *further* accrual).
- **Authoring guard â€” no spawnable enemies in a live zone.** The admin enemy save path (`AdminEnemies.SaveEnemies`) rejects retiring an enemy that is the **last active spawn of a live (non-retired) zone**, so a live zone is never authored into the no-spawnable-enemies state to begin with â€” the workflow is to retire the *zone* first, then its enemies. The guard is backend-enforced (a tampered admin client can't bypass a frontend check) and reports through the standard `AdminSaveResult.Failure`. It deliberately scopes to *live* zones; the runtime relocation above is the defense-in-depth backstop for the cases it doesn't cover (a player standing in, reconnecting into, or persisted against an already-retired zone).
- **No hard delete** of a reference record: the admin "delete" affordance is Retire/Reinstate, and the reference admin repos reject a top-level delete outright.

**Tags are the deliberate exception.** They carry their own (non-zero-based) identity and are resolved by lookup, not by index, so they never suffer the mis-resolution bug retirement exists to prevent. Their admin "delete" stays a real hard delete (graceful for an in-use tag â€” the join rows cascade and the caches reload so the removed tag id disappears on the next read).

### Reference-data read contracts

The read path never exposes EF entities:

- **Reference reads return read contracts** (`Game.Abstractions.Contracts`) â€” one published language shared by the game client's loading screen and the admin Workbench. The internal cache stays as entities (the domain mappers need their fidelity) and is projected to contracts, preserving the id-as-index ordering.
- **Gameplay reads return lean `Game.Core` domain models**, not entities, and hand back **shared, pre-materialized instances** by id rather than re-mapping a fresh graph per call (these reads are on the per-battle hot path). This is safe because the reference-data domain models are structurally immutable value objects â€” `init`-only properties and `IReadOnlyList<>` collections, enforced by the compiler and guarded by a reflection test â€” so a consumer cannot grow or reassign the shared graph and corrupt the cache for every player. The level-parameterized enemy read is the same idea adapted: a level-independent template is shared and cloned at the requested level.
- **Admin writes keep entities inside the data tier** â€” the entity-returning lookups live on `internal` data-tier interfaces consumed only by the Content Authoring admin repositories (see [backend-admin.md](./backend-admin.md#admin-tools-api-surface-content-authoring-context)).

### Reference-data cache reload (build-then-swap)

Cache busting is an **eager build-then-swap** (stale-while-revalidate), not a null-and-lazily-refill (see the [cache-reload spike](./spikes/356-reference-data-cache-reload.md)). Readers never observe an empty or torn snapshot, and never pay a refill query inline.

- A **singleton holder** per set owns the current immutable snapshot and exposes a lock-free read plus `ReloadAsync()`. A reload builds the whole new snapshot off to the side (on its own DbContext) and publishes it with a single atomic reference swap, so a **failed reload leaves the old snapshot in place**. Derived structures (e.g. the per-zone spawn tables built from the enemy list) are bundled into the snapshot so they swap atomically and a reader can never see a new list against stale derived data. A per-holder semaphore serializes reloads to preserve read-your-writes.
- **An admin write triggers an awaited reload** after the write commits (so the Workbench reads its own writes with zero gap) and **broadcasts a cross-instance invalidation over the Redis backplane**. Other instances debounce the notification and run one background reload sweep; the publishing instance skips its own message. A failed background sweep is retried with backoff and never disturbs readers.

### Skill acquisition flags (authoring intent vs. reference reality)

A skill carries an `ESkillAcquisition` `[Flags]` bitmask (`Player`/`Item`/`Enemy`) declaring which channels are *allowed* to grant it. The flag is **intent**; the references that actually grant a skill (an item's `GrantedSkillId`, a proficiency milestone/seed skill, an enemy's skill pool) are **reality**; backend authoring validation bridges the two as **anti-tamper** where a reference is authored (a frontend picker filter alone is insufficient): the enemy skill-set save rejects any skill that isn't `Enemy`-flagged, and the item save rejects a `GrantedSkillId` whose target isn't `Item`-flagged. `Player` denotes a milestone-grantable skill (**challenges no longer grant skills** â€” spike #982). The flag lives on the **entity + contract** (so admin tooling and the codex can read it) but **not** on the lean `Game.Core` battle model â€” the battle never reads provenance. Adding it changes the skills set's version hash, so clients re-download skills once. Existing rows backfill to `Player`.

### Skill rarity (entity + contract + lean core model)

A skill carries an `ERarity` tier (the `Rarity` FK, inferred by convention like `Item.Rarity`). Unlike the acquisition flag, rarity lives on the **entity + contract *and* the lean `Game.Core.Skills.Skill` model** â€” the deliberate divergence is forward-looking: the battle still never reads rarity (no battle-parity surface, no frontend battle-model change), but the proficiency-XP accrual reads it server-side at battle completion via `ISkills.GetSkill(id).Rarity` (the gameplay read returns the core model), so the field must sit there. It is authoring + display only today. Adding it to the contract changes the skills set's version hash, so clients re-download skills once; existing rows backfill to `Common` (the migration default; the model carries no default so EF always writes the authored value on insert).

### Item-granted skills (battle assembly)

An item carries an optional `GrantedSkillId` â€” a nullable FK to a skill it grants while equipped. It lives on the **entity + contract** and on the lean `Game.Core.Items.Item` (as an **id only**, keeping the item cache independent of the skill cache); adding it to the contract changes the items set's version hash, so clients re-download items once. Granted skills are **innate** â€” additive to and exempt from the selected-loadout cap. The battler's skill set is assembled in `BattleSnapshot.ToBattler` from the **already-captured `EquippedItems`** (no snapshot field added â€” granted skills are *derived*, like item attributes): the rule is **gather the granted ids from all active grant sources â†’ de-duplicate by id (first wins) â†’ order**, with player-selected skills first (their order) then the grants in `EEquipmentSlot` order (Helmâ†’Accessory). The order/dedupe is the shared `BattleLoadout.OrderSkillIds` rule, mirrored tick-for-tick on the frontend for battle parity. Writing the gather as "all active sources" (not "iterate equipped items") makes a future set-bonus grant an additive source, not a rewrite. This is a deliberate player-favouring break in the player/enemy skill-count symmetry â€” enemies can't equip items (see [game-design.md](./game-design.md#skills)).

## Player-facing read projections

Player-facing reads also avoid exposing EF entities, but â€” unlike reference data â€” they reuse the existing **gameplay domain models** rather than dedicated contracts (those models are already lean enough), with the API DTOs projecting from them.

- **Statistics, challenge progress, and proficiency progress** (level + XP per proficiency) go through the single `IPlayerProgressRepository`, which also owns the progress write path, giving one source of truth for player-progress data. Each is a `(PlayerId, â€¦)`-keyed row folded into the `PlayerProgress` aggregate with its own dirty set, persisted as one batched, absolute-value `ProgressUpdated` write-behind event whose handler upserts each kind idempotently (a re-applied event converges). A statistic's "no data yet" state is the **absence of its row**, never a magic `0` value â€” a stored value (0 included) is always a genuine recording. Min/max aggregation and the "at most" challenge path therefore key off row presence (`TryGetStatisticValue`), so a legitimate `0` minimum (e.g. an instant `FastestVictory`) records and counts correctly instead of being treated as empty. Challenge evaluation on battle completion is scoped to the challenges whose tracked statistic actually changed that battle â€” via a reverse index (statistic â†’ challenges) bundled with the cached challenge snapshot â€” rather than re-scanning the whole authored catalog on every battle (a per-player hot path).
- **The skill loadout is sent as the full unlocked set** (each skill with its `Selected`/`Order`), mirroring how inventory is sent as the full unlocked item set; the client derives the ordered equipped loadout from it. The loadout cap is a generated client constant, not a per-player wire field.
- **The loadout is edited through one atomic socket command** that replaces the whole equipped set in order (handling select, deselect, and reorder in one path). Like the other player mutations it validates as **anti-cheat** â€” no duplicate ids, count within the cap, every id already unlocked â€” and rejects with no state mutation on any violation.

## HTTP vs WebSocket Communication

**The goal: once the session is live, every game-client read and player-state mutation goes over the socket â€” only the auth flow stays on HTTP.** This is what lets the player's state be held in-memory on the backend and updated in real time without re-synchronizing against Redis or the database on each request. Users connect to one instance at a time, and a player's socket commands are handled sequentially. The goal has been reached; the HTTP that remains is deliberately scoped to:

- the **auth flow** (login, account creation, refresh, logout, status, active-session, device-info) â€” it runs before a socket exists, and the active-session check must deliberately *not* open one;
- admin **persistence** endpoints; and
- the admin-only reads that have no socket equivalent yet (tags, user/role lookups).

**Player-state mutations belong on the socket because the socket serializes them with the battle loop.** Player writes are read-modify-write against the in-memory aggregate; the socket read loop processes one command per player at a time, so a write is naturally serialized with the idle battle-completion commands that mutate the same cached player. The same write over HTTP would run concurrently on a separate thread and could silently clobber a background battle save. Every player-mutating action is therefore a socket command and validates as anti-cheat (rejecting with no mutation) â€” e.g. a stat allocation only accepts the core attributes that actually have an allocation row, which blocks allocating into derived attributes.

**The per-player sequential guarantee is enforced by a per-socket command lock**, because server-initiated pushes (challenge-completion, socket-replaced) arrive over the Redis backplane and run *off* the read loop â€” so two paths could otherwise mutate the same cached player concurrently, the exact lost-update class the serialization exists to prevent. Each command also runs under a per-command timeout, and that cancellation token is plumbed down through the application services and the data tier (EF/Npgsql honour it natively; the Redis client is wrapped to honour it cooperatively) so a wedged command unwinds promptly and releases the lock rather than stalling every later command. A separate per-socket send lock guards WebSocket output, since overlapping sends on one socket are illegal.

## Single active connection (session takeover warning)

A player may only be connected to one instance at a time: the socket manager records the player's current socket id in Redis and, when a new socket replaces an existing one, signals the old socket to close. To let the client warn the user **before** that takeover happens, the authenticated `Login/ActiveSession` endpoint reports whether the player already has a live socket â€” deliberately over **HTTP, not a socket command**, because opening a socket to ask the question would itself trigger the replacement it is meant to warn about. Because the check is presence-based it covers "another tab" and "another device" uniformly.

The presence key carries a short TTL refreshed on every inbound socket message (the client heartbeats every 10s), so a live connection stays fresh while a connection that vanished without a clean close expires within the TTL. The key is claimed atomically (set-with-TTL in one operation) and rolled back if the post-claim setup fails, so a partial registration can't leave a ghost session that blocks the player.

## Graceful socket drain on shutdown

A stopping instance drains its live sockets cleanly rather than leaving them to be force-killed â€” the missing lifecycle piece for an app designed to run as multiple scalable instances (rolling deploys, scale-down). A singleton registry tracks every live socket and uses its shutdown hook to drain them in two phases: first stop the per-socket inactivity watchdogs (teardown now owns the close), then send each socket a normal-closure close frame and await the loops within a bounded timeout, aborting any still-blocked receive past the deadline so the loop unwinds rather than waiting out the host timeout. The client treats the normal closure as a cue to reconnect â€” which a rolling deploy routes to a healthy instance â€” so no client change was needed. The "draining has started" gate and the snapshot are flipped together under one lock, so a handshake that completes after the drain begins is closed on the spot with the same reason instead of slipping past the snapshot and lingering until the process is killed.

## Reference-data caching and versioning

The frontend caches each reference-data set in the browser and re-downloads only the sets that changed (see `frontend.md`). The `GetReferenceDataVersions` socket command supports this by returning a **content hash per set** in one round-trip. The hash is computed by hashing the same serialized models the client actually receives (SHA-256 over the camelCase JSON), so the version changes if and only if the client-visible data changes â€” there is no separate version counter to maintain or invalidate. The versioned set is exactly the set of reference-data socket commands (discovered by assembly scan), so a newly-added one is versioned automatically.

`GetReferenceDataVersions` is the **first** command the loading screen issues on every connect, so the hash is **memoized per set, keyed on the immutable cache snapshot it was computed from** â€” the serialize-and-hash runs once per cache swap rather than once per connect. Keying on the snapshot instance makes invalidation automatic: a build-then-swap publishes a new snapshot object (a memo miss), so an admin write can never serve a stale hash. Intrinsic (enum-derived) sets have no holder to swap, so they key on a process-stable sentinel and hash once for the process lifetime.

## Battle (setup, runtime & skill effects)

Setting up and running a battle is a domain decision, not an orchestration one: the application layer only snapshots the player and the chosen enemy loadout, sets the active battle, and delegates the game-logic choices to `Game.Core`. **Parity invariant:** battle logic runs on both the frontend and backend and the results must be identical â€” the backend replays the client's reported battle as anti-cheat â€” so these mechanics are mirrored on both sides and pinned by a parity matrix; they must never diverge. The tick-by-tick detail â€” encounter setup, snapshot reconstruction, zone-unlock anti-cheat, the RNG seed, and the skill-effect/DoT-HoT runtime â€” lives in [backend-battle.md](./backend-battle.md).

Two distinct constant classes back this: **`GameConstants`** holds values that must stay identical on both sides (tick size, loadout cap, â€¦) and is `[ClientMirrored]` so codegen emits them to the TypeScript client, while **`ServerGameConstants`** holds server-only anti-cheat/reward clamps the client never evaluates (e.g. the exp-reward caps) and is deliberately not mirrored. Put a constant in the right one â€” a client-mirrored value referenced as a server-only one (or vice versa) is a recurring footgun.

## Caching, Pub/Sub & write-behind persistence

Most reference data is cached in-memory (see [Reference Data](#reference-data)), but **player data uses a write-behind cache where Redis is the source of truth**: a live player's aggregate is loaded once onto the connection-scoped session, mutated in-memory per socket command, and persisted to Postgres asynchronously through a Redis queue drained by a background service. Player progress (statistics + challenges) is a second such write-behind aggregate under its own key, and Redis pub/sub doubles as the WebSocket backplane and the carrier for cross-instance reference-data invalidations. The deep mechanics â€” the sliding TTLs, the domain-event â†’ batched-publish path, dead-lettering and retry, drain serialization, and worker ownership â€” live in [backend-persistence.md](./backend-persistence.md).

On the `ICacheService` cache contract every value-accepting setter carries an expiry, and they fall into two camps. `Set` and `SetAndForget` take a **nullable** value where a **null value deletes the key** â€” the de-facto null-means-delete behaviour their generic `<T>` overloads rely on. `GetSet` (the atomic set-with-expiry) is **non-null**: writing a TTL implies writing a value, so a null would error its server-side Lua `SET` â€” it has no delete path to expose.

## Challenge-completion notifications (server push)

Completing a challenge unlocks its rewards as a side effect of battle-completion handling, decoupled from the socket command that triggered the battle â€” so the command's response can't carry the unlock, and it would otherwise surface only on the next page load (the client doesn't evaluate challenges itself; the server is authoritative). Instead the player aggregate raises a challenge-completed event and an API-layer handler pushes a server-initiated socket command to the player over the Redis backplane, so it reaches them on whichever instance holds their connection and covers **every** completion path (idle victory, won-abandon, even a loss that completes a counting challenge). Being server-initiated, the push carries a typed response but no request parameters, so codegen emits no client request type.

A server push that **faults** is escalated rather than silently dropped: the poisoned payload is dead-lettered (preserved on a holding queue for inspection, mirroring the player write-behind dead-letter path) and the affected socket is sent a one-way `ServerCommandFailed` notice carrying the failed command's name, so the client re-syncs the authoritative state the push would have updated (a failed `ChallengeCompleted` force-reloads challenge progress, which gates zone navigation) instead of silently diverging. Faults are **not retried**: the only server pushes have no transient failure mode (a poisoned payload can't be fixed by a retry) and re-executing across a released-lock window isn't generally safe. Faults are classified inside the command executor, which keeps a per-command **timeout** (`"Command timed out."`) distinct from a **lifetime/teardown cancellation** (a normal unwind, never an `"Internal Server Error"`); a server push reports its outcome so the pub/sub processor owns the dead-letter-plus-notify escalation.

## Proficiency XP accrual (battle-completion hook)

Proficiency XP is computed **server-side at battle completion, off the tick loop** â€” like statistics and challenge rewards, never inside the deterministic 40 ms simulation. This is the deliberate boundary that keeps it off the frontend/backend battle-parity surface: the simulation produces the `BattleStats` (which skills fired) and the difficulty multiplier is the same `DefeatRewards` curve factor already computed server-side; the accrual reads those after the fact. Only the eventual proficiency *attribute bonus* is parity-sensitive (both sides compute it from player state at snapshot time), not the XP. The mechanic itself â€” difficulty-scaled pie, routed to each path's frontier tier, split by attention Ă— on-tier efficiency (absolute falloff) â€” is in [game-design.md](./game-design.md#proficiency-xp--victory-based-difficulty-scaled-absolute-falloff).

The accrual lives in one place â€” `ProficiencyRewardService` (mirroring `ChallengeRewardService`) â€” and **both** completion paths call it, so live and offline accrue identically: the live path runs it in `BattleStatisticsEventHandler` (reacting to `BattleCompletedEvent`, whose `DifficultyMultiplier` the victory path threads on), and the offline consolidation runs it per won battle in `BattleService`. It resolves each fired skill's path contribution (`IProficiencies.ContributionsForSkill` â†’ `(PathId, HomeTier, Weight)`) to the path's **frontier tier** (`IProficiencies.GetPath(id).Frontier`, read off the player's current proficiency levels) and the home-tier **falloff**, then delegates the pure attention/efficiency split to `ProficiencyXpCalculator`. Because the frontier is re-resolved from progress each battle, the offline batch â€” replaying the same loadout over the away window â€” evolves identically to live. It writes through the same `PlayerProgress` aggregate / write-behind path as statistics and challenges (absolute values, so a re-applied event converges). The live path raises a `ProficiencyXpGainedEvent` that an API-layer notifier pushes to the client exactly like the challenge-completion push; the offline path suppresses the push and instead **folds each won battle's accrual across the away window** (`ProficiencyGainAccumulator` â€” XP summed per proficiency, level/residual XP final, milestones/skills/opened nodes unioned) onto the **welcome-back summary**, which carries the gains and opened nodes (reusing the live push's models so the gate renders offline and live the same way) and counts them in `HasProgress`. The pie magnitude is a server-only `ServerGameConstants` value; the per-tier XP curve, cap, and the path's falloff base are authored content.

### Milestone & open effects (derived openness, persisted only as skill grants)

A proficiency's **openness is derived, never stored**: a node is open iff it has accrued XP, is a `StartsUnlocked` root, has its previous in-path tier maxed, or has all its cross-path prerequisites maxed. There is no per-player "opened" flag and no migration â€” the same level + structure derivation the client uses (spike #982 decision 14), so XP routing needs no open-gate (firing a contributing skill *is* what opens a path). The **only** persisted effects of the milestone/open layer are therefore idempotent **skill grants** via `Player.UnlockSkill`, applied inside `ProficiencyRewardService` right where a tier levels: a crossed milestone grants its `RewardSkillId`; maxing a tier opens its in-path successor (`Path.NextTier`) and any cross-path gateway whose prerequisites are now all maxed (`IProficiencies.DependentsOf`), each granting the opened tier's `SeedSkillId` so a fresh node has a full-pace training vehicle (decision 8). Because the grants are idempotent and fire only on the level/max transition, the offline batch re-running the accrual per battle never double-grants. Roots are seeded at character creation by granting their seed skills in `NewPlayerFactory` (no per-player rows). The prerequisite graph is validated **acyclic** both at admin-authoring time (`AdminProficiencies.SetPrerequisites`) and as a build-time cache invariant (`ProficiencyPrerequisiteGraph`), since a cycle would soft-lock every node on it. The live push (`ProficiencyXpGainedEvent`) carries the granted skills and opened nodes alongside the XP; the offline path suppresses it.

### Proficiency attribute bonuses (frozen in the battle snapshot)

A proficiency's per-level/milestone payouts pay out as **attribute bonuses** through the same modifier pipeline as stat allocations and gear: each authored `ProficiencyModifier` becomes an `AttributeModifier` with the new `EAttributeModifierSource.Proficiency` source, and a player at level *L* contributes the cumulative bonus of every authored payout level at or below *L* (`Proficiency.ModifiersForLevel`). This is the **one frontendâ†”backend parity surface** the proficiency system adds â€” both sides compose the bonus from player state at snapshot time â€” so the rule is mirrored in `UI/src/lib/battle/proficiency-modifiers.ts` and pinned by `ProficiencyBonusParityTests` / `proficiency-bonus-parity.test.ts`. **The composition order is part of the contract:** the proficiency modifiers must sit with the base set (stat allocations + gear) *before* the static engine modifiers (`StaticAttributeModifiers`), so the additive accumulation order â€” and therefore the result down to the last bit â€” is identical on both sides. The backend gets this for free (`BattleSnapshot.GetModifiers` concatenates the proficiency modifiers onto the base set, and `AttributeCollection` appends the statics last); the frontend `Battler` feeds them through `BattleAttributes.setData` for the same reason rather than appending them after the statics. The parity scenarios assert **bit-exactly** (a tolerance would hide a last-bit ordering divergence) and include a derived attribute that carries a static additive base (MaxHealth) to pin that order.

The bonuses are **frozen into `BattleSnapshot` at battle start** (`ProficiencyLevels`), exactly like the captured stat allocations and gear â€” a level gained while idling takes effect on the next battle, and an offline window fights at a stationary power even as its victories accrue XP. Because proficiency progress lives on the separate `PlayerProgress` aggregate (not the battle aggregate), `BattleService` reads the levels through the lean `IPlayerProgressRepository.GetProficiencies` accessor when it builds the snapshot and threads a proficiency resolver into `BattleSnapshot.ToBattler`/`GetModifiers` (so the same bonus also flows into the `DefeatRewards` power measurement â€” a proficiency-boosted player is measured at its true power). Until proficiencies are authored and opened, every player's captured set is empty, so the bonus is a no-op and the anti-cheat replay is unchanged. **The live frontend wiring** (reading the player's proficiency levels + the reference per-level modifiers into the live player battler at assembly) ships with the client proficiency data delivery (#1119) â€” the `playerProficiencies` store composes the bonuses via the shared `proficiency-modifiers.ts` (#1117) and the battle engine feeds them through the same per-spawn change-detection as equipment/attributes; #1117 ships the shared computation and the parity vectors.

## Authentication

JWT bearer auth, not server sessions. Access tokens are signed (HMAC-SHA256), short-lived (15 min), and carry the user id + role claims; refresh tokens are opaque, long-lived (48h), single-use, and rotate on every refresh. Refresh tokens live server-side in Redis as a hash â€” chosen because auth already depends on Redis, so it adds no new durability assumption.

**The selected player is a token claim.** An account can own multiple characters, so which one a request acts as is carried as a `player_id` claim on the access token (and on the refresh-token record, so it survives a refresh) â€” set once a character is chosen, absent on a pre-selection token. Both the socket handshake and the session-rehydration path read the bound player from this claim rather than defaulting to the account's first player, so an evicted/uncached session re-binds to the *selected* character. Login no longer binds a player; see [Account/login orchestration](#accountlogin-orchestration).

The **token is the sole authorization source of truth** (roles are read off the validated principal, never the session cache; likewise the selected player), so a role change only takes effect on next login. A global fallback policy requires every endpoint to be authenticated unless explicitly marked anonymous. WebSockets authenticate via an `access_token` query-string parameter, since browsers can't set headers on the WebSocket handshake. The frontend owns token persistence, pre-emptive refresh, and the single-flight refresh path (see `frontend.md`).

**The validated token is likewise the sole authority for whether a caller is *authenticated* â€” never the gameplay session cache.** That cache (`Session_{userId}` â†’ the in-flight `PlayerState`) is a volatile presentation convenience holding the in-flight battle state and the userâ†’player binding; it can be evicted (Redis flush, sliding-TTL lapse) or simply never established on a given instance. A valid token with no cached session is therefore *authenticated-but-uncached*, not anonymous: the session is rehydrated in memory by binding the player named in the token's selected-player claim instead of reporting "not logged in" (a pre-selection token with no such claim is authenticated but has no player, surfaced as a graceful error). The claim is authoritative â€” a cached session bound to a different character than the claim names is re-bound to the claim's player. `SessionService.Authenticated` is gated only on the token-derived `UserId`; a cache hit/miss governs `HasPlayerSession`, never authentication. This rehydration is **in-memory only** â€” it re-binds the request's `PlayerState` but does **not** write the session cache, since that write would land on the concurrent HTTP thread and player-state writes belong on the socket (above). The cache is (re)populated only by login (which establishes the binding before any socket exists) and by the socket's write-behind path.

`SessionLoaderMiddleware` records only the token-derived `UserId` on every authenticated request; the `GetSession` read (and the rehydration on a miss) is deferred to the consumers that actually need player state â€” the socket handshake and the `Status`/`ActiveSession` auth endpoints â€” via `SessionInitializer`. So the rest of the authenticated HTTP surface (admin tooling, refresh, device-info) pays no per-request session-cache read.

### Password hashing

Passwords use **PBKDF2-HMAC-SHA256** behind an `IPasswordHasher` abstraction. A hash is stored in a self-contained, tunable format carrying the iteration count and a per-hash salt (default 600k iterations, the OWASP floor); an application-wide pepper is folded in via HMAC and validated at startup so a missing pepper fails fast. Verification is constant-time, and a valid credential stored with an outdated work factor is transparently re-hashed on next login. An **unknown username** still runs equivalent derivation work via `IPasswordHasher.VerifyDummy` (against a fixed dummy hash), so the login response time does not reveal whether an account exists â€” a constant-time hash compare alone cannot cover the present-vs-absent-account branch.

### Account/login orchestration

Account creation and the login/select/refresh/logout flows are orchestrated by `AccountService` in the application layer; the controller is a thin HTTP adapter that maps the result and wires the request-scoped session (a presentation concern). The new-player defaults (starter skills, base attributes, default preferences) are a domain concern built by a `NewPlayerFactory` in `Game.Core`, so the application layer neither encodes the defaults nor constructs entity graphs. Access-token issuance sits behind an `IAccessTokenService` abstraction so the application layer stays free of the JWT libraries, while the concrete implementation lives at the presentation edge alongside the bearer-validation pipeline.

**Login is split around player selection.** `Login` verifies credentials and returns the issued (pre-selection) token pair plus a summary of each character on the account â€” it binds no player. A second authenticated `SelectPlayer(playerId, refreshToken)` call then validates that the player belongs to the account (anti-cheat), loads it, binds the session, and **rotates the token pair** to carry the chosen player as a `player_id` claim (consuming the caller's refresh token only after the validations pass, so a rejected selection leaves it intact). The single-active-session takeover check (`ActiveSession`) is a per-player presence check, so it runs *after* selection. This is the seam for multiple characters per account (the in-game character switcher and lossless switching land on top of it).

**Switching characters in-game is lossless.** A `SwitchPlayer(playerId, refreshToken)` flow lets a logged-in account change its live character without re-logging in. Before binding the new character (via the same `SelectPlayer` path), it **credits the departed character** â€” the one the caller's token currently binds â€” for its elapsed idle time through the offline-rewards simulator, resolving its in-flight battle and **dropping the â‰Ą5-minute floor** (that floor is a login-time concern; a deliberate switch credits any elapsed time). The credit runs over HTTP off the departed character's battle loop, so the client must tear its game socket down first; the target's ownership is validated up front so a rejected switch never mutates the departed character. That socket-teardown expectation is also **enforced server-side**: the credit is skipped (and logged) when the departed character still has a live socket, since its battle loop is already serializing its own saves under the per-socket command lock â€” running the off-lock HTTP credit then would reintroduce the lost-update race the lock exists to prevent. The switch itself still proceeds. The entered character's own welcome-back catch-up then runs through the normal offline-rewards path on reconnect. This makes one-live-at-a-time outcome-equivalent to parallel progression â€” the parked character loses nothing while another is played. The switcher lists the account's characters to pick from via an authenticated `Players` endpoint (the login-time summaries are returned only by `Login`, and that handoff is one-shot, so an in-session switch re-fetches the current list).

**Characters are created post-signup with their own names, bounded by a cap.** Beyond the one character made at signup, an authenticated account adds characters via `CreatePlayer(name)` (HTTP, part of the pre-game select flow â€” it binds no session). Each character carries its own user-supplied name (validated by the `PlayerName` domain rule: trimmed, non-blank, â‰¤ the 20-char `Player.Name` limit, no control characters; names need not be globally unique). A **config-bound per-account cap** (`PlayerCreation:MaxPlayersPerAccount`, default 6) is enforced server-side as anti-cheat â€” the count check and insert run under a lock on the owning user row and commit in-tier, so concurrent creations can't slip past the cap.

At most one active account may hold a given username, enforced by a **partial unique index** (`Username WHERE ArchivedAt IS NULL`) so two concurrent creations can't both slip past the up-front availability check and insert duplicate active rows. The filter excludes archived users, preserving username reuse after archival. Account creation therefore commits its own insert in the data tier (rather than deferring to the per-request unit of work) so the index's unique-violation surfaces as a clean "username taken" result instead of a 500 raised after the action returns.

### Auth endpoint rate limiting

The anonymous auth endpoints (login, refresh, account creation) are throttled by a **per-client-IP fixed-window** rate limiter (ASP.NET Core's built-in `AddRateLimiter`), blunting credential stuffing, refresh-token brute force, and the PBKDF2 resource-exhaustion vector. The three endpoints share one `"auth"` policy, so the limit is a **combined per-IP budget** an attacker can't multiply across endpoints. The partition key is the client IP resolved the same trusted-proxy way as login tracking (`ClientIp.Resolve`), so a spoofed `X-Forwarded-For` can't shard a caller across partitions. A throttled request returns the standard `{ errorMessage }` envelope (429) with a `Retry-After` header. Unlike the CORS origins, the limits carry **safe positive defaults** so an unconfigured deployment is still protected â€” config only tunes them â€” and are validated as positive on start so an accidental zero fails fast rather than silently disabling the guard.

### Per-account login backoff

Layered on top of the per-IP limiter as defence-in-depth against a **slow, distributed** credential guess on one account that stays under any single IP's rate limit, `Login` applies a **per-account exponential backoff**. Consecutive failures are tracked in Redis keyed by a hash of the (case-sensitive) username and **reset on a successful credential verification**; the first few failures (a configurable threshold) are free, after which each further failure delays the next permitted attempt by a doubling window up to a **low cap**. It is deliberately a bounded **backoff, never a hard lockout** â€” a legitimate user is delayed by at most the cap â€” so an attacker who knows a username can only briefly slow the owner, not lock them out. A backed-off attempt is rejected **before** any database hit or PBKDF2 work (so spamming during an active window neither extends it nor costs server CPU) and surfaces as a 429 with a `Retry-After` header and a message distinct from invalid-credentials. The keying is per-account, not per-(account Ă— IP): per-(account Ă— IP) would let each distributed bot keep its own counter and add little over the IP limiter. Like the IP limiter the parameters are config-bound (`LoginBackoff` section) with safe defaults and validated as sane on start. The backoff arithmetic is a pure, unit-tested policy; the system clock (an injected `TimeProvider`) and the Redis update sit in a thin guard around it. Because the threat model is a **concurrent** distributed guess, that update must not lose increments â€” the guard records a failure through an **optimistic compare-and-set retry** (`ICacheService.CompareAndSet`, the conditional-write counterpart to `CompareAndDelete`): it re-reads and recomputes whenever a concurrent failure wins the swap, so the streak climbs at the true attempt rate rather than letting a plain read-modify-write clobber a racing increment.

## CORS allowed origins (deployment config)

The browser CORS policy's allowed origins are **configuration-bound, not hardcoded** â€” they are deployment-specific. The list supports multiple origins and is validated at startup (must be non-empty), so a misconfigured environment fails fast rather than silently rejecting every browser request. The local dev origin ships in the Development config.

## Unhandled exception handling

A centralized exception-handling middleware is the **single place** unhandled exceptions are logged and turned into a response. It converts any exception escaping the downstream pipeline into the project's consistent `ApiResponse` error envelope with a 500 status, so clients get the same `{ errorMessage }` shape they get for business failures rather than the host's default response. Internal details (the exception message) are surfaced **only in Development**; other environments get a generic message so nothing leaks. A client-disconnect cancellation is not treated as a server error, and an already-started response (e.g. an upgraded WebSocket) is left to unwind rather than corrupted. It runs **inside `RequestLoggingMiddleware`** so the 500 is set before the "Request Ended" event logs the status; `RequestLoggingMiddleware` therefore stays focused on structured start/end events and never logs the exception itself.

## Access Roles and Admin Authorization

Authenticated users are gated out of admin tooling unless they hold the `Admin` role. **Roles live in the signed auth token, not the session store** â€” the token is the source of truth for what a request is authorized to do, which keeps the check self-contained and avoids a second lookup; the trade-off is that a role change only takes effect on the user's next login. The admin authorization filter reads the role straight off the cryptographically validated principal, **never** off the gameplay session cache: that cache is a presentation concern decoupled from the token's validity, so keying off it would wrongly reject a valid admin whose session key happens to be absent (evicted, never established, or aged out).

## Admin tooling & user administration

Reference-data administration is modelled as a distinct **Content Authoring** bounded context (thin admin controllers â†’ admin repositories that own persistence; the EF entity never surfaces in `Game.Api`), and user/role management is a separate **Identity / User Admin** context that shares the `User` aggregate with the auth flow. Both speak `Game.Abstractions.Contracts` types rather than entities, and every admin write reloads the in-memory reference caches. Users are archived (a soft delete that frees the username) or banned (blocked but name reserved). The API surface, the change-set/reconciler persistence patterns and their zero-based-id gotchas, the archive-vs-ban model, and the connection-tracking tables live in [backend-admin.md](./backend-admin.md).
