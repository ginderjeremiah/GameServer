# File Structure and Projects

The game is structured is similar to an onion architecture with multiple projects. The backend projects (all in folders of the same name) are:

- `Game.Abstractions`: This project contains shared abstractions and interfaces used across the backend, such as repository interfaces, infrastructure-service interfaces, and the read/write contracts (`Game.Abstractions.Contracts`). The EF entity models are **not** here — they live in `Game.Infrastructure` (see _Entity model location_ below).
- `Game.Api`: This project contains the ASP.NET Core Web API controllers, models, and related code for handling HTTP requests from the frontend, as well as WebSocket command handlers for real-time communication. It also includes a reflection-based code generator for automatically generating API client interfaces for the frontend based on the API controllers and WebSocket handlers.
- `Game.Api.Tests`: This project contains unit and integration tests for the `Game.Api` project, ensuring that all API endpoints and WebSocket handlers function correctly and that the battle logic is consistent with the frontend implementation.
- `Game.Application`: This project contains the application layer of the backend, including services for orchestrating game logic.
- `Game.Application.Tests`: This project contains unit and integration tests for the application layer, ensuring that all orchestration logic is correctly implemented and consistent with the frontend.
- `Game.Core`: This project contains the core domain logic of the game, including domain objects/services and the battle simulation logic. This is where the main game mechanics are implemented.
- `Game.Core.Tests`: This project contains unit tests for the core domain logic, ensuring that all game mechanics, including battle simulation, character progression, and item logic, are correctly implemented and consistent with the frontend.
- `Game.DataAccess`: This project contains the data access layer, including implementations of repositories from `Game.Abstractions` that interact with the database using Entity Framework and the cache and pub/sub implementations provided by `Game.Infrastructure`.
- `Game.Infrastructure`: This project contains the implementation of the cache and pub/sub interfaces defined in `Game.Abstractions` (currently using Redis), as well as the database context, the **EF entity models** (`Game.Infrastructure.Entities`), and migrations for EF Core.
- `Game.Infrastructure.Tests`: This project contains classical unit tests for the in-process logic in `Game.Infrastructure` that has no out-of-process dependency (currently the `BackgroundWorker` lifecycle plus the cache/pub-sub factory and subscribe-rollback failure paths). It is deliberately lightweight — it references only `Game.Infrastructure`, not `Game.TestInfrastructure` — because, per the testing guidelines below, anything coupled to Redis or the database is covered by integration tests instead of here.

# General Backend Guidelines

- Repositories should create and persist domain objects and are responsible for dispatching domain events before persisting changes. They should not contain any domain logic nor should they directly return entity models or database DTOs. Mappers should be used to convert between domain objects, entity models, and database DTOs and placed in the Mapping folder in the root of the `Game.DataAccess` project.
- All domain logic should be implemented in the `Game.Core` project, and the application layer should only be responsible for orchestrating calls to the repositories and domain objects/services to fulfill the use case. The API layer should only be responsible for handling HTTP/websocket requests, validating input, and returning responses, and should not contain any domain or orchestration logic.

## Testing Guidelines

Unit tests should be written according to the classical (Detroit) school of testing as much as possible. This project does not have any unmanaged dependencies, so you should be avoiding test doubles as much as possible in both unit and integration tests. If a service under test involves an out-of-process dependency such as the database or cache, it should only need to be tested through integration tests. Any classes that depend on out-of-process dependencies should NOT contain any logic worth unit testing. If you find that a service contains domain logic and depends on an out-of-process dependency, you should refactor the code to move said logic into an appropriate domain class that can be unit tested without the dependency, and then write integration tests only to validate the interaction with the out-of-process dependency.

When deciding whether to cover a branch, evaluate per-branch whether a worthwhile test exists rather than chasing the coverage number. The Redis primitives in `Game.Infrastructure` (`RedisMultiplexerFactory`, `RedisPubSubService`, `RedisQueue`) are the canonical example (#387, #399): their genuine logic is pinned by integration tests against the DI-resolved interface (`RedisQueueIntegrationTests`, `RedisPubSubServiceIntegrationTests` — typed queue round-trips, FIFO order, duplicate-handle-id rejection, and the `UnSubscribe` remove/unknown branches), while the residual uncovered branches are accepted as **measure-only**: the `RedisMultiplexerFactory` null-connection-string guards (defensive — config always supplies one) and the cache↔pub/sub multiplexer reuse-vs-connect branches (incidentally exercised at startup; provoking them deterministically would require disposing the process-wide shared multiplexer mid-suite, which is riskier than the branch is worth).

### Running tests (Microsoft.Testing.Platform)

The test projects run on **Microsoft.Testing.Platform** (MTP — xunit.v3 with `TestingPlatformDotnetTestSupport`), not the legacy VSTest runner. The practical gotcha: **the VSTest `dotnet test --filter "…"` syntax is silently ignored** — MTP rejects it as `Unknown option '--filter'`, and because the flag sits *before* `--` it is swallowed by the `dotnet test` wrapper, so the **entire suite runs** instead of erroring. Filter like this instead:

- **Via `dotnet test`** — pass MTP filter options *after* `--`:
  - `dotnet test Game.Api.Tests/Game.Api.Tests.csproj -- --filter-class "*SessionServiceTests"`
  - simple filters: `--filter-class`, `--filter-method`, `--filter-namespace`, `--filter-trait "name=value"` (each has a `--filter-not-*` negation). Repeated filters of the same type OR together; different types AND together.
  - query filter: `--filter-query "/assembly/namespace/class/method[trait=value]"` (`*` wildcards; [xUnit query filter language](https://xunit.net/docs/query-filter-language)). Simple and query filters cannot be mixed.
- **Running the built test binary directly** (faster — skips the `dotnet test` wrapper) uses xunit.v3's *native single-dash* CLI:
  - `dotnet Game.Api.Tests/bin/Debug/net10.0/Game.Api.Tests.dll -class "Game.Api.Tests.Unit.SessionServiceTests"`
  - native flags: `-class` / `-method` / `-namespace` / `-trait` (append `-` to exclude, e.g. `-class-`) or `-filter "/asm/ns/class/method"`; run the binary with `--help` to list them.

A filtered run is a dev convenience; CI runs the whole solution under coverage via `build/coverage.ps1`, so run the full suite before pushing.

# Important Architectural Design Decisions

## Entity model location and isolation

The EF entity models are a persistence implementation detail of the data tier, **invisible to `Game.Api` and `Game.Application`**. This was completed in the entity-relocation capstone (#138, the last of the spike-#130 series; see `docs/spikes/130-hide-entity-models.md`).

- **Entities live in `Game.Infrastructure.Entities`,** next to `GameContext` — not in `Game.Abstractions`. The supporting `IZeroBasedIdentityEntity` marker and `NotLoadedException` moved with them (into `Game.Infrastructure`); the unused `IAuditableEntity` marker and the vestigial `PlayerState` entity (#142) were deleted as dead code.
- **Hiding mechanism — no `internal` needed.** `Game.DataAccess` references `Game.Infrastructure` with `PrivateAssets="Compile"`, so Infrastructure's compile-time types do **not** flow transitively to `Game.Api` (which references `Game.DataAccess`), and `Game.Application` never references `Game.Infrastructure` at all. The entities themselves stay `public` (EF and the test infrastructure need that) but are simply unreachable from the upper layers.
- **The data tier's own entity seam is internal.** `IEntityStore` (the low-level insert/update/delete tracker) is `internal` to `Game.DataAccess`, and the entity-returning reference lookups are exposed only on internal data-tier interfaces (`IItemEntityCache` etc. — see [Reference-data read contracts](#reference-data-read-contracts)). The public `Game.Abstractions` repo interfaces speak only domain models and contracts.
- **Architecture test (regression guard).** `Game.Api.Tests/Architecture/EntityIsolationTests` asserts that neither the `Game.Api` nor the `Game.Application` assembly references `Game.Infrastructure`. Because every entity type lives in `Game.Infrastructure`, "no reference to that assembly" is equivalent to "no dependency on the entity model", and it fails fast if the `PrivateAssets`/reference wiring ever regresses.

## Reference Data

Reference data in this app can be split into 2 categories: intrinsic and static. Intrinsic reference data is fundamentally encoded into the application and often has an associated enum or similar construct in the code (e.g. `EAttribute` or `EItemCategory`). These are persisted into the database for data integrity, but they do not need to be (and very rarely are) queried from the database. Static reference data, on the other hand, is not fundamentally encoded into the application but is typically cached in-memory as a `List` such that the Id of the entity is its index. IMPORTANT: The Id column for these tables is seeded to start at 0 to facilitate this indexing. This includes things like items, item modifications, skills, enemies, zones, and challenges. This data is stored in the database and queried on application startup to be cached in-memory for fast access. Since this data is static and does not change often, it is not a problem to cache it in-memory, and it allows for much faster access than querying the database every time it is needed.

**Intrinsic reference data carries derived display metadata on the domain/contract, not in the database (#529).** The `Attribute` reference set surfaces classification/display fields (`AttributeType` — the `EAttributeType` Primary/Secondary/Status taxonomy — plus `IsPercentage`, `IsHarmful`, `Code`, `DisplayOrder`, `Decimals`) mapped per-attribute from the `EAttribute` enum on the `Game.Core` domain `Attribute` and projected onto the `Game.Api` model so they codegen onto `IAttribute`. This promotes metadata that previously lived as hardcoded frontend constants into one backend source of truth. Because the `Attributes` table is integrity-only (served from the enum, never queried), these derived, never-queried fields live on the domain/contract only with **no migration** — no columns are added. Adding them changes the attribute reference-data version hash, so clients re-download the set once, which is expected. The display taxonomy stays deliberately distinct from the `Attribute.IsCore`/`CoreAttributes` power-calc invariant (`Primary` is expected to equal `CoreAttributes` but the two are not collapsed).

**Caches are immutable snapshots loaded eagerly and replaced by build-then-swap reload (#356 → #357, #358).** A **singleton snapshot holder** per set (`ReferenceCacheHolder<TSnapshot>` and its six subclasses in `Game.DataAccess/Repositories/Caching`) owns the immutable cached snapshot; the scoped reference repos are thin readers over `holder.Current`. `Startup` awaits `IServiceProvider.InitializeReferenceCachesAsync()` (after migrations, before the host serves traffic) to load every holder up front, so a database problem at boot surfaces as a startup failure rather than on the first player request, and there is no lazy-fill path — reads of an unloaded holder throw rather than refilling inline. See [Reference-data cache reload (build-then-swap)](#reference-data-cache-reload-build-then-swap) for the reload mechanics.

> **The Id-as-index optimization requires contiguous ids — guaranteed by retirement, not convention.** Both tiers look these sets up *by array index* (`Items.GetItem`/`LookupItem` index the `OrderBy(Id)` cache directly; the frontend does `staticData.items[id]`). That is only correct while the data stays append-only, since the index equals the Id only when there are no gaps. A **hard delete** of a non-terminal record would open a permanent Id gap that silently mis-resolves every lookup above it (and orphan persisted player-state ids and battle snapshots that reference it). To make this structurally impossible, top-level reference records are **retired, not deleted** (#289 — see [Retiring reference data](#retiring-reference-data-content-lifecycle)).

### Retiring reference data (content lifecycle)

A top-level reference record (item, item mod, skill, enemy, zone, challenge) carries a nullable `RetiredAt` timestamp — the same flag-plus-audit shape as the User `ArchivedAt`/`BannedAt` soft-delete. Retiring takes a record *out of circulation* without erasing it; reinstating clears the timestamp. The guarantees:

- **The slot is permanent.** A retired record is never removed or renumbered, so the cache stays gap-free and every index lookup keeps resolving the right record — the contiguity invariant is enforced structurally rather than relying on "only delete the highest Id".
- **It stays resolvable by id.** `GetItem`/`LookupItem`/`GetEnemy`/… still return a retired record, so existing references — a player's equipped item or applied mods, a battle snapshot, an authored boss/reward/unlock-gate FK — keep working. Retirement is a catalogue/authoring concern only; it is **not** carried on the lean `Game.Core` battle models, so a retired item a player owns fights identically.
- **It is out of circulation.** The current concrete rule: a retired enemy is excluded from the random per-zone spawn tables (`EnemiesCacheHolder.BuildZoneEnemyTables`) so it no longer rolls as an idle encounter. (There is no gameplay path that picks randomly from *all* reference data — items/mods/skills are authored challenge rewards, enemies/bosses are authored per-zone — so the circulation surface is small.)
- **No hard delete of a reference record.** The admin "delete" affordance is replaced by Retire/Reinstate (an ordinary `RetiredAt` edit through `AddEdit*`). The six reference admin repos pass no `delete` handler to `ChangeSetProcessor`, so a stray top-level `Delete` change throws instead of corrupting data. Child collections (attributes, mod slots, spawn rows, …) are still added/edited/deleted normally.

`RetiredAt` is on the read contracts so the client and Workbench can show the state; it changes the reference-data version hash when toggled (an expected re-download). The Workbench's authoring pickers exclude retired records from *new* authoring while keeping an already-referenced retired value visible (#295 — see `frontend.md` → the Workbench section). Still deferred (#296): per-entity edge cases (idling in a retired zone, an in-progress challenge whose target/reward is retired, retiring a zone/challenge used as an unlock gate) and referenced-by warnings on retire.

**Tags deliberately keep the hard-delete lifecycle (#297).** Tags are *not* one of the six zero-based-id reference sets — they carry their own (non-zero-based) identity and are resolved by `.find`/Map on both tiers, so they never suffer the index mis-resolution bug retirement exists to prevent. Adding the `RetiredAt` lifecycle to them would be cost without that benefit (a label is not referenced by persisted player state or battle snapshots the way an item/skill is), so the admin "delete" stays a real hard delete. It is graceful for an **in-use** tag: the `ItemTags`/`ItemModTags` join FKs are `ON DELETE CASCADE`, so deleting a tag drops its join rows rather than throwing, and `AdminCacheReloadFilter` reloads the item/mod caches so the next read no longer carries the removed tag id (the change also flows to clients through the items/mods version hash). Those two joins are modeled as **explicit entities** (`ItemTag`/`ItemModTag`) rather than EF implicit many-to-many tables (#503): the `Item.Tags`/`ItemMod.Tags` skip navigations still back the reference-data reads, but the admin `SetTags` reconciles a single record's assignments by inserting/deleting one navigation-free join row per change (diffed from `ITagAssignmentQueries`' id-only reads) instead of eager-loading a tag's full membership just to drop one row. The Workbench's staged-delete UX surfaces the tag's item/mod usage counts (the Usage tab + list meta) before the admin commits the removal.

### Reference-data read contracts

The reference-data **read** path does not expose EF entities. The repository read methods return dedicated **read contracts** in `Game.Abstractions.Contracts` — a single published language shared by the game client's loading screen and the admin Workbench (which share the `Get*` socket commands). The entity→contract mapping lives in `Game.DataAccess` (`Mapping/*` for the cached in-memory lists; EF-translatable inline projections for the small `IAsyncEnumerable` reads), consistent with the data tier owning DTO mapping.

- **Reads return contracts:** the cached `All()` family (`IItems`/`ISkills`/`IEnemies`/`IZones`/`IItemMods`) and the small reference sets (`ITags`/`ITagCategories`) return `Game.Abstractions.Contracts` types. The internal cache stays as entities (the domain mappers need their full fidelity); `All()` projects that cache to contracts, preserving the zero-based Id-as-index ordering.
- **Gameplay reads return lean domain models, not entities:** `GetItem`/`GetSkill`/`GetDomainEnemy`/`GetItemMod` return `Game.Core` domain models; the EF entity stays an implementation detail of this layer, reachable only through the internal entity-cache interfaces (consumed by the Content Authoring admin repos), never the public read contracts.
- **Gameplay reads return shared, pre-materialized instances (#449).** `GetItem`/`GetItemMod`/`GetSkill` are on the per-battle hot path — invoked once per equipped item / applied mod / equipped skill during `PlayerMapper.ToCore` (cache miss) and `BattleSnapshot.ToBattler` (every battle validation). Rather than re-running the mapper to build a fresh domain graph on every call, the `Items`/`ItemMods`/`Skills` snapshots bundle the entity list **with** its pre-mapped `Game.Core` model list (the `ItemSnapshot`/`ItemModSnapshot`/`SkillSnapshot` records — the same build-and-publish-together shape as the `EnemySnapshot` derived structures), and the read returns the shared instance by id. This is safe because the reference-data domain models (`Item`/`ItemMod`/`ItemModSlot`/`Skill`/`SkillEffect`) are **structurally immutable value objects** — every property is `init`-only and every collection is an `IReadOnlyList<>` (#547), so the read-only invariant is enforced by the compiler rather than only by review. A consumer therefore cannot reassign or grow the shared graph (e.g. set `ItemModSlot.ItemMod`, or `Add` to `Item.ModSlots`) and silently corrupt the cache for every player. Consumers already treated them read-only — the battle path composes modifiers into a separate `AttributeCollection`, applied mods live on the player's `UnlockedItemSlot` (never on the shared `Item.ModSlots`), and `BattleSkill` only reads its `Skill` — and the type now makes that the only thing the compiler permits. The `AttributeModifier` elements held inside those collections are likewise structurally immutable — `init`-only on a reference type, the same pattern (#603) — closing the one-level-deeper variant of the same cross-player corruption: a consumer holding a shared modifier can no longer mutate its `Amount`/`Attribute`/`Type`. It stays a reference type (not a value-equality `record`/struct) deliberately, because `AttributeCollection.RemoveModifier` and the battle-effect bookkeeping remove a *specific instance* by reference identity, which value equality would change. `Game.Core.Tests/Items/ReferenceDataImmutabilityTests` reflection-guards the invariant (for these models and `AttributeModifier`) so a later edit can't quietly flip an `init` back to `set`. The **enemy read is the same idea adapted to a level-parameterized model (#584):** an `Enemy` can't be one shared instance because it carries the encounter's level *and* mutable battle-skill selection, so `EnemySnapshot` bundles a level-independent `EnemyTemplate` per enemy (its pre-mapped attribute distributions and available-skill graph, aligned by id with the entity list) and `GetDomainEnemy`/`GetRandomDomainEnemy` clone it at the requested level — reusing those shared immutable blocks instead of re-mapping the skill graph on every battle setup. Because the template is built inside the enemy snapshot, `Enemies` no longer reaches into the skill cache per call; its snapshot loads the skills it maps directly (the holders reload independently, so reading another set's cache mid-sweep could observe a stale snapshot). `IChallenges.All()` is the same idea for the whole list — it returns the immutable snapshot directly instead of copying it on every battle's challenge evaluation.
- **Admin writes keep entities inside the data tier:** the entity-returning lookups (`LookupItem`/`LookupItemMod`/`LookupSkill`/`GetEnemy`/`GetRandomEnemy`/`LookupZone`) plus the id-returning tag-assignment queries (`GetTagIdsForItem`/`GetTagIdsForItemMod`/`GetExistingTagIds`) live on **internal data-tier interfaces** in `Game.DataAccess` (`IItemEntityCache`/`IItemModEntityCache`/`ISkillEntityCache`/`IEnemyEntityCache`/`IZoneEntityCache`/`ITagAssignmentQueries`), not on the public `Game.Abstractions` repo interfaces. Each reference repo implements both its public read interface and the internal entity-cache interface, and one shared scoped instance backs both. Their only consumers are the **Content Authoring admin repositories** in `Game.DataAccess` (see [Admin Tools API surface](./backend-admin.md#admin-tools-api-surface-content-authoring-context)). Neither `Game.Api` nor `Game.Application` reaches an entity through them: `BattleService` reads the `Zone` contract via `IZones.GetZone`, and `PlayerService` checks mod existence via `IItemMods.ValidateItemModId` (#137).
- **`IModel`** (the cross-boundary model marker) lives in `Game.Abstractions` so the contracts there satisfy the `IModel`-constrained `ApiResponse`/`Change` generics. The contracts keep the same property names/shape as the previous API DTOs, so the serialized wire shape — and therefore the reference-data version hash — is unchanged.

### Reference-data cache reload (build-then-swap)

Cache busting is an **eager build-then-swap** (stale-while-revalidate), not a null-and-lazily-refill — the direction settled in spike [#356](./spikes/356-reference-data-cache-reload.md) and implemented across #357 (eager startup load) and #358 (this design). Readers never observe an empty cache, never pay a refill query inline, and never see a torn snapshot.

- **A singleton holder owns each set's immutable snapshot.** `ReferenceCacheHolder<TSnapshot>` (`Game.DataAccess/Repositories/Caching`) holds the current snapshot and exposes `Current` (a lock-free read) and `ReloadAsync()`. The six subclasses each build their snapshot from a fresh `GameContext` resolved through `IServiceScopeFactory`, so a reload's DbContext lifetime is its own rather than borrowing whichever request scope is in flight (the awkward static-state-on-a-scoped-service seam the old design had is gone). The scoped reference repos are thin readers that project `holder.Current`.
- **The snapshot bundles derived structures so they swap atomically.** `EnemySnapshot` carries the enemy list **and** the per-zone spawn tables derived from it; building and publishing them together means a reader can never see a new enemy list against stale spawn tables. `ReloadAsync` builds the whole snapshot off to the side and publishes it with a single atomic reference assignment (`Volatile.Read`/`Write` — a generic field can't use the `volatile` keyword), so readers keep serving the previous snapshot until the swap and a **failed reload leaves the old snapshot in place**. A per-holder `SemaphoreSlim` serializes concurrent reloads; serializing (rather than coalescing onto an in-flight load) is what preserves read-your-writes — an awaited reload always starts a fresh query after the write it must reflect.
- **The admin filter awaits the reload, and must run after the commit.** `AdminCacheReloadFilter` resolves `IEnumerable<IReloadableReferenceCache>` (the holders, registered as that interface) and awaits `ReloadAsync()` on each after a successful admin write, so the Workbench's read-your-writes holds while players read the old snapshot with zero gap. Because the reload queries the database on a *fresh* context, it must run **after** the global `CommitFilter` has persisted the write; it therefore declares `Order = int.MinValue` so it is the outermost action filter and its post-action reload executes after the commit. Controllers apply it through the `[ReloadReferenceCaches]` attribute (a `ServiceFilterAttribute` subclass that bakes in that order), so a new admin controller can't silently default to order 0 and reload before the commit. A reload failure after a committed write propagates (the response is an error; the write persisted, so the admin can retry). The filter deliberately reloads **all** sets on every admin write (it does not know which entity changed); the sets are small and admin writes rare, so a Workbench save reloading each set several times is accepted (coalescing the *local* awaited reloads per request is a future optimization; the cross-instance broadcasts below already coalesce).
- **Cross-instance invalidation rides the Redis backplane (#359).** After a successful admin write (the write is committed once the filter's post-action code runs), the filter also broadcasts a reference-data-changed notification through the existing `IPubSubService` via `IReferenceDataChangeNotifier`, implemented by the `ReferenceCacheSynchronizer` hosted service (`Game.DataAccess`). Every instance subscribes at startup; a **foreign** notification signals a `CoalescingReferenceCacheReloader`, which debounces (default 500ms) and then runs one background reload sweep over all holders — so a Workbench save's burst of admin writes collapses into one (or few) sweeps per instance, on top of the per-holder reload gate. The payload is the publisher's process id, which the publishing instance uses to **skip its own message** (its caches were already reloaded synchronously in the filter). A failed background sweep is retried with exponential backoff (`ReferenceCacheReloadPolicy`) and never disturbs readers — the old snapshot keeps serving; once retries are exhausted the instance stays stale until the next notification, an accepted cost since pub/sub delivery is itself fire-and-forget best-effort (and a fresh boot always loads fresh data).

## Player-facing read projections

Player-facing reads do not expose EF entities to `Game.Api` either, but — unlike reference data — they reuse the existing **gameplay domain models** rather than dedicated read contracts (those domain models are already lean and lossless enough for these reads).

- **Statistics and challenge progress go through the single `IPlayerProgressRepository`,** which already owns the progress write path. Its `GetStatistics`/`GetChallenges` read methods return the `Game.Core.Progress` domain models (`PlayerStatistic`/`PlayerChallenge`), and the entity→domain mapping lives in `Game.DataAccess` (`Mapping/PlayerProgressMapper`, shared with `Load`). The API DTOs (`Game.Api/Models/Progress/*`) project from those domain models, so the read path references no entities. Consolidating onto `IPlayerProgressRepository` retired the duplicate entity-returning read repos (`IPlayerStatistics`/`IPlayerChallenges`), giving a single source of truth for player-progress data.
- **`Save` is self-sufficient — no implicit "load first" ordering.** Both `Load` and `Save` fetch the player's existing rows through a shared private helper that lazily loads and caches the tracked-entity snapshot for the lifetime of the scoped repository. The common battle-completion flow (load → mutate → save) reuses the cached snapshot, so no redundant query is issued; but a standalone `Save` simply fetches the snapshot on demand and upserts correctly rather than dereferencing an un-loaded field. This removed the prior temporal coupling between `Save` and a preceding `Load` (and its null-forgiving `!` usages) — #164.
- **Inventory and log-preference reads were already domain-sourced:** `PlayerData.FromPlayer` projects them from the `Player` aggregate. Their leftover entity-coupled mapping (`InventoryItem`'s entity constructor and `LogPreference.FromSource`) was dead and has been removed.
- **The skill loadout is sent as the full unlocked set, not just the equipped ids.** `PlayerData.UnlockedSkills` projects every `player.Skills` entry as an `UnlockedSkill { SkillId, Selected, Order }` — the direct parallel of `InventoryItem`'s `Equipped`/`EquipmentSlotId` (a skill's `Order` is null when unequipped, like an unequipped item's slot). Because `player.SelectedSkills` is already the deterministic `(Order, SkillId)` ordering (`PlayerMapper.ToCore`), an equipped skill's `Order` is just its index in that list. The client derives the ordered equipped loadout from this richer set, so there is no redundant equipped-id field. The loadout cap is **not** on the wire — it is a generated client constant (`MAX_SELECTED_SKILLS`, emitted from `Game.Core.GameConstants`; see [infrastructure.md](./infrastructure.md)), so the client renders "X / N" without hardcoding it and without a redundant per-player payload field (#306).
- **The loadout is edited through one atomic write command.** `SetSelectedSkills(orderedSkillIds)` — a socket command mirroring `SaveLogPreferences`, aligned with the WebSocket-everything direction — replaces the whole equipped set in order, handling select, deselect, and reorder through a single path. It delegates to `Player.TrySetSelectedSkills`, which enforces the loadout rules as **anti-cheat**: no duplicate ids, count ≤ `GameConstants.MaxSelectedSkills`, and every id already in the player's unlocked set; any violation rejects the change with no state mutation. On success it raises a single `SelectedSkillsChangedEvent` whose write-behind handler (`DataProviderSynchronizer.HandleSelectedSkillsChanged`) clears the player's `Selected`/`Order` flags and rebuilds them from the ordered list (delete-then-rebuild, idempotent on retry — the same shape as the mod-applied handler). This single-writer guarantee is why `PlayerSkill` can carry both `Selected` and `Order` without the two drifting.

## HTTP vs WebSocket Communication

The backend uses a mix of HTTP and WebSockets for communication with the frontend, with WebSockets carrying the real-time communication such as battle updates. The long-standing goal — move all **game-client** communication onto the socket except the auth flow — has now been reached (#473 and its sub-issues): once the session is live every game-client read and player-state mutation goes over the socket, which is what lets the Player state be kept in-memory on the backend and updated in real-time without having to synchronize against the Redis cache or DB on each request. Users may only connect to one instance at a time for simplicity, and any websocket commands are handled sequentially for each player. The HTTP that remains is deliberately scoped to:

- the **auth flow** — `Login`, `Login/CreateAccount`, `Login/Refresh`, `Login/Logout`, `Login/ActiveSession`, `Login/Status`, `Login/DeviceInfo` — which authenticates the caller (or operates on the refresh token) before a socket exists, and in the `Login/ActiveSession` case must deliberately *not* open one (see [Single active connection](#single-active-connection-session-takeover-warning));
- admin **persistence** endpoints (the `/api/AdminTools/*` writes); and
- the admin-only reference/user reads that have no socket equivalent yet — `Tags`, `Tags/TagCategories`, `AdminTools/GetUsers`, `AdminTools/GetRoles`.

**Player-state mutations belong on the socket — serializing them with the battle loop avoids lost updates.** Player writes are read-modify-write against the write-behind cache (load the cached `Player`, mutate, save it back); the socket read loop processes one command per player at a time (`SocketHandler.ReadLoop`), so a write on the socket is naturally serialized with the idle battle-completion commands (`DefeatEnemy`/`BattleLost`) that mutate the same cached player. An equivalent write over **HTTP** runs on a separate request thread *concurrently* with that loop, so a background battle save that loaded the player before the HTTP write but persisted after it silently clobbers the change — the client got a success response, yet a refresh shows the old value. This is why every player-mutating gameplay action is a socket command: `SetSelectedSkills`, `SaveLogPreferences`, `SetItemFavorite`, `UpdatePlayerStats` (the attribute allocation that #432 fixed), and the inventory writes `EquipItem`/`UnequipItem`/`ApplyMod`/`RemoveMod` (#463). With those moved, `PlayerController` had no actions left and was removed (#566) — player data reaches the client through the auth responses (`Login` / `Login/Status`), not a standalone `GET /api/Player`. The two remaining game-client reads — the player's tracked statistics and per-challenge completion — likewise moved off HTTP onto the player-scoped `GetPlayerStatistics` / `GetPlayerChallenges` socket commands (#563, #564), and their dead `GET /api/Statistics` and `GET /api/Challenges/Player` endpoints were removed (#567), so no game-client read remains on HTTP.

Like `TrySetSelectedSkills`, the stat-allocation validation in `PlayerStatPoints.TryUpdateAttributes` is **anti-cheat that rejects with no state mutation**: an update targeting an attribute the player has no allocation row for is rejected (`false`) rather than silently succeeding as a no-op or upserting a new row (#488). Only the six core attributes are seeded as allocation rows (`NewPlayerFactory.AttributeCount`), so this makes "you can only allocate into a core attribute that exists" an enforced contract instead of an invisible invariant, and it blocks allocating into the derived attributes.

**The per-player sequential guarantee is enforced by a per-socket command lock, because server-initiated commands run off the read loop (#478).** The read loop is not the only path into a player's commands: a server push (`ChallengeCompleted`, `SocketReplaced`) arrives over the Redis backplane and the pub/sub processor (`SocketManagerService.RegisterSocketCommandListener`) runs it on a thread-pool thread, concurrently with the read loop — both call `SocketHandler.ExecuteCommand` over the same cached `Player`. A `ChallengeCompleted` push mutating the player (`Player.CompleteChallenge`) could therefore interleave a read-modify-write with a `DefeatEnemy` in the read loop — the exact lost-update class the loop's serialization was meant to prevent. So `ExecuteCommand` takes a **per-socket `SemaphoreSlim(1,1)` command lock**, making the documented "handled sequentially for each player" guarantee real for *both* paths. Separately, a **per-socket send lock** (`SocketContext`) guards every `WebSocket` output (`SendData` and the close frame), since `WebSocket.SendAsync` forbids overlapping sends and the two paths (plus the ping/close paths) could otherwise interleave a multi-chunk frame or throw. The two locks are distinct because `ExecuteCommand` holds the command lock while calling `SendData`. (Both `SemaphoreSlim`s are intentionally left undisposed — like `ReferenceCacheHolder`'s gate — since they never materialize a wait handle and disposing risks an `ObjectDisposedException` against an in-flight pub/sub command during socket teardown.)

**A wedged command can't stall the queue forever — each runs under a per-command timeout (#483).** `ExecuteCommand` runs the command under a `CancellationTokenSource` budget (`DefaultCommandTimeout` = 25s — below the client's 30s per-request backstop in `api-socket.ts` and above StackExchange.Redis's 5s command timeout. Npgsql defaults to a 30s command timeout, so a genuinely slow DB call can be cut off a few seconds early; that's an accepted trade-off, since a >25s player command is already pathological and the client abandons it at 30s regardless). The token is threaded through `AbstractSocketCommand.ExecuteAsync`, so a cancellation-aware command unwinds itself, while one that ignores the token is bounded by a `Task.WaitAsync` on the same budget. On expiry the client immediately gets a `"Command timed out."` response so the read/pub-sub loop keeps draining — but the command lock is **not** released to the next command. It is handed to a continuation that releases it only once the abandoned task finally settles, so a still-in-flight read-modify-write of the shared cached `Player` can't race the next command (the lost-update class the lock exists to prevent). The abandoned command still commits its write (so it isn't lost), but `RunCommand` never sends — `ExecuteCommand` owns the single send — so the client never receives a second, late response for an id it already saw time out. The remaining exposure — a stuck command that genuinely *cannot* observe the token still queues later commands behind the lock until it clears (or the client reconnects, which takes over and closes the socket) — is bounded but, by design, not eliminated: releasing the lock would restore the very race the per-socket lock was added for. The `Task.WaitAsync` backstop is what makes "cannot observe the token" the worst case rather than the common one.

**The budget reaches the actual I/O — cooperative cancellation is plumbed through the data tier (#558).** The token is threaded from `AbstractSocketCommand.ExecuteAsync` down through the application services (`PlayerService`/`BattleService`), the player and progress repositories, `IUnitOfWork.CommitAsync`, and the `ICacheService`/`IPubSubService` async members. Each layer takes an optional `CancellationToken` defaulting to `CancellationToken.None`, so the non-command callers (startup, the HTTP `CommitFilter` — which forwards `HttpContext.RequestAborted` — and the write-behind consumer) are unaffected. The two dependencies honour it differently: **EF Core / Npgsql honour the token natively** on `SaveChangesAsync(ct)` and the async queries, so a wedged DB call unwinds promptly; **StackExchange.Redis exposes no token** on its database operations, so `RedisService`/`RedisPubSubService` honour it *partially* by wrapping each call in `Task.WaitAsync(ct)` — the await unwinds promptly on a cancelled budget while the underlying Redis command settles in the background (`WaitAsync(CancellationToken.None)` is a zero-overhead no-op, so the default-token callers pay nothing). The payoff: when a command wedges on one of these now-cancellation-aware paths, `RunCommand` itself unwinds on the budget, so the lock-release continuation runs *promptly* instead of waiting out the dependency's own timeout — the "later commands queue behind a stuck dependency" trade-off above shrinks to only the paths that still can't observe the token (the `Task.WaitAsync` backstop still covers those). The fire-and-forget cache writes (`SetAndForget`/`ExpireAndForget`) and the `Subscribe`/`UnSubscribe` lifecycle calls deliberately take no token: the former return before the command settles, and the latter are app-lifetime registrations, not per-command I/O.

As a step in that direction, the read-only reference data the loading screen needs (zones, enemies, items, skills, item mods, attributes, challenges, challenge types, statistic types) is exposed over the socket via `Get*` commands (`Game.Api/Sockets/Commands`) that mirror the corresponding `GET` endpoints. These share the `AbstractReferenceDataCommand<TModel>` base, which wraps each repository/intrinsic collection in an `ApiSocketResponse<IEnumerable<TModel>>`. Both the loading screen and the admin Workbench now load these sets over the socket (see [Reference-data caching and versioning](#reference-data-caching-and-versioning)), and the duplicate HTTP `GET` endpoints have been removed (#64). The admin reads can share the same cached socket commands because every admin write reloads these in-memory caches server-side (`AdminCacheReloadFilter`), so the next read serves the freshly written data — the mechanism the admin `Challenges` reads already relied on, now applied to every reference set (see [Reference-data cache reload](#reference-data-cache-reload-build-then-swap)). The only reference reads still served *exclusively* over HTTP are `Tags` and `Tags/TagCategories` (#318 removed the dead `ZoneEnemies`, `SlotsForItem`, `TagsForItem(Mod)`, `ItemCategories`, and `ItemModTypes` endpoints whose data had moved onto the socket contracts or codegen'd enums). (Statistic-type metadata is socket-only via `GetStatisticTypes`; its HTTP twin `GET /api/Statistics/StatisticTypes` was removed once the client migrated to the socket command, #565.)

Statistic-type metadata (`StatisticType { id, entityType, bossOnly, name }`) is intrinsic reference data computed from `EStatisticType` in `Core.Progress.StatisticType.GetAll()` (the DB seed exists only for integrity — there is no repository read). It is exposed over the socket via the `GetStatisticTypes` command so the Statistics screen can resolve which entity kind each statistic breaks down by instead of hard-coding it. `StatisticType.GetEntityType` is the single source of truth for that mapping and must agree with the per-entity rows `PlayerProgress.RecordBattleCompleted` actually writes; `EntityType` is descriptive only, so changing it does not affect challenge evaluation (challenges resolve a statistic by its id + the challenge's target entity id, never by the declared `EntityType`). `BossOnly` is likewise a descriptive domain fact derived alongside `EntityType` (`GetBossOnly` — only `BossesDefeated`, which `RecordBattleCompleted` increments solely on dedicated-boss victories): it is the single source of truth for the "this statistic only counts bosses" rule, consumed by both the Statistics screen's display hint and the admin challenge editor, which restricts an enemy-scoped challenge's target picker to bosses when the tracked statistic is boss-only rather than special-casing `BossesDefeated`.

## Single active connection (session takeover warning)

A player may only be connected to one instance at a time: `SocketManagerService.RegisterSocket` records the player's current socket id in Redis (`PlayerSocket_{playerId}`) and, when a new socket replaces an existing one, emits `SocketReplaced` to the old socket to close it. To let the client warn the user **before** that takeover happens, `Login/ActiveSession` (an authenticated `GET`) reports whether the player already has a live socket via `SocketManagerService.HasActiveSocket` — a read of the same presence key, so the knowledge of the key's format stays in one place. The check is deliberately **HTTP, not a socket command**: opening a socket to ask the question would itself trigger the replacement it is meant to warn about. Because it is presence-based (rather than device- or tab-scoped), it covers both "another tab" and "another device" uniformly — any other live connection for the player.

- **The presence key is liveness-accurate via a heartbeat-refreshed TTL.** The key is written with a 30s TTL (`SocketPresenceTtl`) at register and refreshed on **every inbound socket message** — the client heartbeats every 10s (`api-socket.ts`), the same `_lastResponse` signal the 60s inactivity check already uses — so a live connection keeps its key fresh while a connection that vanished without a clean close (a tab/device closed without a WebSocket close frame, e.g. WebKit) lets the key expire within the TTL instead of lingering. The refresh is an `Expire` (extend-TTL) rather than a re-set, so it only ever prolongs the currently-registered key and never resurrects one a newer connection has taken over. Net effect: the warning fires immediately for a genuinely-open second tab/device, and the residual false-positive window — re-logging-in within ~30s of a non-graceful close and being warned about your own just-closed session — is bounded by the TTL (tunable; tolerates two missed heartbeats) and documented as acceptable for a best-effort, dismissible warning.
- **The claim is written atomically and rolled back on a partial registration (#691).** The key is set with its TTL in one operation (`ICacheService.GetSet(key, value, expiry)` — a Lua read-old-then-write, mirroring `CompareAndDelete`), never a `GetSet`-then-`Expire` pair that a fault between the two calls could leave TTL-less (a permanent ghost session). And because the post-claim steps (notifying the replaced socket, subscribing, starting the drain loops) can still throw, `RegisterSocket` wraps them so a failure `CompareAndDelete`s the key (only while it is still ours, so a concurrent newer owner's claim survives) and tears down any subscription before propagating — leaving no "registered but dead" presence that would block the player without ever draining.
- **Possible future tightening:** a `BroadcastChannel`-based same-browser probe could give instant, zero-window detection of other tabs in the *same* browser (a crashed tab simply can't answer), layered on top of this server-side presence as polish — see the follow-up spike (#221).

## Graceful socket drain on shutdown

A stopping instance drains its live player sockets cleanly rather than leaving them to be force-killed when the process dies (#526) — the missing lifecycle piece for an app explicitly designed to run as multiple scalable instances. Without it, each per-socket read loop blocks on `ReceiveAsync` indefinitely, so on a rolling deploy or scale-down the host cannot signal those loops to stop or tell connected clients to reconnect elsewhere; the connections just hang until the OS/orchestrator kills the process, exactly the session churn the single-active-connection model is otherwise careful about.

- **`SocketConnectionRegistry` is the single long-lived owner of the connection set.** Because `SocketManagerService` is transient (one per handshake), the registry — a singleton registered as an `IHostedService` — tracks every live `SocketHandler` (register on `RegisterSocket`, unregister on teardown) and uses its `StopAsync` as the host's shutdown hook. This replaces the handlers' former untracked, fire-and-forget `Task.Run` loops: `Register` starts the loops and the handler exposes a `Completion` task so the drain can **await** them.
- **The drain is two-phase, carried by two cancellation tokens threaded into each handler's loops.** Outside of shutdown neither token is cancelled, so behaviour is unchanged. (1) A *stopping* token is cancelled first, stopping the per-socket inactivity watchdogs — their job is moot once the drain owns teardown, and leaving them running would just race the shutdown close. (2) The registry then sends every open socket a clean close frame (the dedicated `ESocketCloseReason.ServerShuttingDown`) and awaits the loops within a bounded `_drainTimeout` (5s, well under the host's default 30s shutdown timeout). If a client never completes the closing handshake within that window, a *drain-deadline* token is cancelled, aborting the still-blocked `ReceiveAsync` so the loop unwinds rather than waiting out the host timeout. The per-command timeout token (#483) is orthogonal — it bounds a single command; these bound the socket's whole lifetime.
- **The close frame is a normal closure, so the client reconnects rather than hangs.** `SocketContext.Close` already sends `WebSocketCloseStatus.NormalClosure`; the client's `handleClose` (`api-socket.ts`) settles in-flight requests with an error (callers retry) and re-opens the socket on its next 10s heartbeat — which a rolling deploy routes to a healthy instance. No client change was needed; `ServerShuttingDown` only enriches the close-frame *description*. The socket I/O methods (`SendData`/`ReadMessage`/`Close`) now take a `CancellationToken` (defaulting to none) so the loops can thread the drain token through to the underlying WebSocket calls, which previously passed `CancellationToken.None`.
- **`Close` settles `WaitSocketClosed` only after the close frame has been sent (#605).** The result is set in `Close`'s `finally`, *after* `CloseAsync`, not before it — otherwise the middleware resuming on `WaitSocketClosed` could fall out of its `using var webSocket` scope and dispose the socket while the frame was still flushing. The settle stays best-effort: the `finally` releases waiters even if the send faults or the drain token cancels it.

## Reference-data caching and versioning

The frontend caches each reference-data set in the browser and re-downloads only the sets that changed (see `frontend.md`). The `GetReferenceDataVersions` socket command supports this by returning a content version per set in one round-trip.

- **The version is a content hash of the API models, computed on demand.** Each `AbstractReferenceDataCommand` exposes `ComputeVersion()` (via `IReferenceDataCommand`), which hashes the same serialized model collection the command returns (`ReferenceDataVersioning` — SHA-256 over the camelCase JSON, through the concrete model type so every property is included). Hashing what the client actually receives means the version changes if and only if the client-visible data changes, so no separate version counter has to be maintained or invalidated. It is recomputed per call — the command is hit roughly once per client boot — so there is nothing to invalidate when an admin edits reference data; memoizing it keyed off cache refresh is a possible future optimization if it ever shows up in profiling.
- **The versioned set is exactly the set of reference-data commands.** `GetReferenceDataVersions` is injected with `IEnumerable<IReferenceDataCommand>` (registered by `AddReferenceDataCommands`, which scans the assembly), so a newly-added `Get*` reference-data command is versioned automatically — there is no second list of datasets to keep in sync.

## Battle (setup, runtime & skill effects)

Setting up and running a battle is a domain decision, not an orchestration one: `BattleService` only snapshots the player and the chosen enemy loadout, sets the active battle, and delegates the game-logic choices to `Game.Core` (`BattleFactory`, the `Zone` aggregate, `BattleSnapshot`). The loadout and a non-time-derived RNG `seed` are snapshotted at battle start and reconstructed at validation time, and skills can carry authored `SkillEffect`s that apply as timed attribute modifiers on the live attribute graph. **Parity invariant:** battle logic runs on both the frontend and backend and the results must be identical — the backend replays the client's reported battle as anti-cheat — so these mechanics are mirrored on both sides and pinned by a parity matrix; they must never diverge. The tick-by-tick detail — enemy/boss encounter setup, snapshot reconstruction, zone-unlock anti-cheat, the seed, and the skill-effect/DoT-HoT runtime — lives in [backend-battle.md](./backend-battle.md).

## Caching, Pub/Sub & write-behind persistence

Most reference data is cached in-memory on the backend (see [Reference Data](#reference-data)), but **player data uses a write-behind cache where Redis is the source of truth**: a live player's aggregate is loaded once onto the connection-scoped session, mutated in-memory per socket command, and persisted to Postgres asynchronously through a Redis queue (`PlayerUpdateQueue`) drained by the `DataProviderSynchronizer` hosted service. Player progress (statistics + challenges) is a second such write-behind aggregate under its own key, and Redis pub/sub doubles as the WebSocket backplane and the carrier for cross-instance reference-data cache invalidations. The deep mechanics — the cached-aggregate sliding TTLs, the domain-event -> envelope -> batched-`LPUSH` publish path, dead-lettering and retry, drain serialization/coalescing, the startup drain, dispatcher draining and handler isolation, the `RedisMultiplexerFactory` thread-pool floor, and `BackgroundWorker`/pub-sub-worker ownership and disposal — live in [backend-persistence.md](./backend-persistence.md).

## Challenge-completion notifications (server push)

Completing a challenge unlocks its rewards as a side effect of `BattleCompletedEvent` handling, decoupled from the socket command that triggered the battle — so the command's response cannot carry the unlock, and it would otherwise surface only on the next page load (the client does not evaluate challenges itself; the server is authoritative). Instead, `Player.CompleteChallenge` raises a `ChallengeCompletedEvent` (challenge id + any reward ids), and `ChallengeCompletedNotifier` (an API-layer `IDomainEventHandler`, registered in `Startup` because it depends on the socket infrastructure) pushes a `ChallengeCompleted` socket command to the player via `SocketManagerService.EmitSocketCommand`. Riding the Redis backplane like `SocketReplaced`, it reaches the player on whichever instance holds their connection and covers **every** completion path (idle victory, won-abandon, even a loss that completes a counting challenge), not just the explicit-victory response. The client applies it by unlocking the item/mod/skill locally so the reward is immediately usable. The push command is server-initiated only: it carries a typed response but no `Parameters` property, so codegen emits no client request type.

**Server-initiated command failures are logged and the queue keeps draining — there is no client surfacing or dead-lettering today (#656).** A server push (`ChallengeCompleted`, `SocketReplaced`) is drained from the per-socket pub/sub queue by `SocketManagerService.GetSocketCommandProcessor` and run through the same `SocketHandler.ExecuteCommand` (and thus the same per-socket command lock + per-command timeout) as a client command. `ExecuteCommand` is itself the primary failure handler: its own `try/catch` absorbs the per-command timeout *and* turns any genuine command fault (including an unknown/poisoned command name failing to resolve) into an `"Internal Server Error"` response, so for a server push that error response goes nowhere (no client awaits it) and the fault is logged at error inside `ExecuteCommand` rather than re-surfaced. The processor's own `catch` is therefore a **backstop** for the narrow residual that escapes `ExecuteCommand` — and since no lifetime cancellation token is plumbed into command execution today, in practice that backstop is unhit. It still distinguishes the two cases for when a token *is* threaded through later: a lifetime `OperationCanceledException` is logged at debug as a normal teardown, while any other fault is logged at error; either way it keeps draining the rest of the queue (mirroring the write-behind consumer's "process the rest, never silently drop" philosophy). The upshot of the contract: a server push that faults gets **no client ack/error** and there is **no dead-lettering or backoff**, so a persistently-failing push (a poisoned payload that always throws) re-logs on every redelivery with no escalation. Two deferred gaps are tracked as follow-up: (1) escalating that persistent-failure case (surface-to-client / bounded retry / dead-letter), and (2) that `ExecuteCommand` currently mislabels a lifetime `OperationCanceledException` as an `"Internal Server Error"` instead of treating it as a teardown — the cancellation distinction belongs there too, not only in the processor backstop.

**The dequeue itself is inside the loop's fault isolation, not just command execution (#691).** The processor wraps the `queue.GetNextAsync()` call as well as `ExecuteCommand`, so a fault *reading the next message* (a malformed payload, a Redis blip) is logged and skipped rather than escaping the processor delegate and terminating the drain loop — which would silently drop every subsequent server push for that socket. A malformed payload is already popped by `GetNextAsync` before its deserialization throws, so skipping advances the queue; a transient blip is retried on the next pass.

## Authentication (JWT access + refresh tokens)

Authentication uses standard ASP.NET Core JWT bearer auth rather than the legacy cookie-backed pseudo-session token.

- **Short-lived signed access tokens + long-lived rotating refresh tokens.** `AccountService.Login` (called by `LoginController.Login`) verifies credentials and returns the issued tokens plus the player aggregate; the controller maps that to a `LoginResult` carrying an `AuthTokens` pair plus the player data. The **access token** is a JWT signed (HMAC-SHA256) behind the `IAccessTokenService` abstraction (`JwtTokenService`) using the configured `Jwt:SigningKey`, with a 15-minute lifetime (`AuthConstants.AccessTokenLifetime`); it carries the user id (`sub`) and `role` claims. The **refresh token** is an opaque, high-entropy random value with a 48-hour lifetime (`AuthConstants.RefreshTokenLifetime`). Clients send the access token as an `Authorization: Bearer` header; when it expires they call `Login/Refresh` to mint a new pair.
- **Standard claims pipeline.** `AddJwtBearer` validates the token (issuer/audience/signature/lifetime) and projects it onto `HttpContext.User` (with `RoleClaimType = "role"` so `User.IsInRole` works). A global `FallbackPolicy` requires every endpoint to be authenticated unless marked `[AllowAnonymous]` (login, account creation, refresh, logout). `SessionLoaderMiddleware` bridges the validated principal's user id to the request-scoped `SessionService`, loading the cached player state; inbound claim mapping is disabled so claim types stay verbatim (`sub`/`role`). Authorization reads roles straight off the validated principal, not the session (see [Access Roles and Admin Authorization](#access-roles-and-admin-authorization)).
- **Refresh tokens are stored server-side in Redis, keyed by a SHA-256 hash of the value (`IRefreshTokenStore`/`RefreshTokenStore`).** Only the hash is persisted, the TTL enforces the lifetime, and consumption is an atomic get-and-delete so each refresh token is **single-use**: `Login/Refresh` rotates it (issues a brand-new pair and invalidates the old refresh token), and `Login/Logout` revokes it. Redis is the natural home because auth already depends on the Redis player session (a flushed cache logs users out regardless), so this adds no new durability assumption and avoids a DB migration.
- **WebSockets authenticate via an `access_token` query-string parameter.** Browsers cannot set an `Authorization` header on the WebSocket handshake, so the `/socket` endpoint reads the access token from the query string (wired through `JwtBearerEvents.OnMessageReceived`) — the canonical ASP.NET Core pattern for bearer auth over WebSockets.
- **The frontend consumes this scheme through its API client layer** — token persistence, pre-emptive refresh, the single-flight refresh path, and the socket query param all live in `UI/src/lib/api` (see `frontend.md` → Authentication).

### Password hashing (PBKDF2)

Passwords are hashed with **PBKDF2-HMAC-SHA256** behind the `IPasswordHasher` abstraction (`Game.Abstractions/Auth`), implemented by `Pbkdf2PasswordHasher` (`Game.Application/Auth`). This replaced a bespoke iterated-SHA-512 construction (a fast, GPU-friendly hash assembled by hand) with a purpose-built, deliberately-slow KDF.

- **Self-contained, tunable format.** A hash is stored as `$pbkdf2-sha256$<iterations>$<base64-salt>$<base64-key>`, so the work factor and a per-hash salt live alongside the key — the stored hash is fully self-contained. The iteration count is configured via `PasswordHashingOptions.Iterations` (default 600k, the OWASP floor for PBKDF2-HMAC-SHA256); tests lower it via `PasswordHashing:Iterations`.
- **Salt and pepper.** A fresh 128-bit random salt is generated for every hash and embedded in the output (so `Hash`/`Verify` take no salt parameter and there is no separate salt column). An application-wide pepper (`HashPepper` secret) is folded in via a standard HMAC pre-hash for defence-in-depth; it is injected through `PasswordHashingOptions` rather than process-global mutable static state, and validated on startup so a missing pepper still fails fast.
- **Transparent work-factor upgrade on login.** `Verify` returns `SuccessRehashNeeded` for a valid credential stored with an outdated work factor. `AccountService.Login` then re-hashes the verified plaintext (with a fresh salt) and persists it via `IUsers.UpdatePasswordHash` (a targeted `ExecuteUpdate`), so existing accounts upgrade on next login without a forced reset. Verification comparisons are constant-time (`CryptographicOperations.FixedTimeEquals`).

### Account/login orchestration lives in the application layer

Account creation and the login/refresh/logout flows are orchestrated by `AccountService` (`Game.Application/Services`), not the controller. `LoginController` is a thin HTTP adapter: it calls the service, maps the result to an `ApiResponse`, and wires the **request-scoped** `SessionService` (creating the session on login, clearing it on logout) — session identity is a presentation concern, so it stays in the API layer. `AccountService` owns the data orchestration: username availability, password hashing, delegating the new account's user-and-initial-player insert to the Identity context, credential verification, loading the player aggregate, and refresh-token rotation/revocation via `IRefreshTokenStore`. `Game.Application` references no EF entities (#137); it speaks only Identity contracts (`NewAccount`/`AccountCredentials`) and domain types.

- **The new-player defaults are a domain concern, built by `NewPlayerFactory` (`Game.Core/Players`).** "What does a new player look like?" — the starter skill set, the base attribute spread, and the default log preferences (plus the `StarterSkillCount`/`AttributeCount`/`StartingAttributeAmount`/`StartingZoneId` constants) — is a game rule, so it lives in `Game.Core` rather than the orchestration layer. `NewPlayerFactory.Create(name)` returns a `NewPlayer` blueprint (raw defaults, no identity or resolved reference-data objects). `AccountService` hashes the password into a `NewAccount` credential blueprint and hands both blueprints to `IUsers.CreateAccount` (the Identity context owns the `User` aggregate), which maps them to the linked user+player entity graph (`PlayerMapper.ToEntity`) and queues it in the unit of work. `AccountService` therefore neither encodes the defaults nor constructs entity graphs — it only orchestrates. Unlike `SavePlayer`'s write-behind path, account creation persists straight through the unit of work because a freshly created player has no cached state or domain events yet.
- **Access-token issuance is abstracted behind `IAccessTokenService` (`Game.Abstractions/Auth`).** The application layer must issue the access token as part of the login/refresh use case, but the concrete JWT implementation (`JwtTokenService`) sits at the presentation edge alongside the bearer-validation pipeline it mirrors. Depending on the abstraction keeps `Game.Application` free of the JWT libraries; the API composition root registers `JwtTokenService` as the implementation. Token-lifetime policy is shared via `AuthConstants` (`Game.Abstractions/Auth`) so the access-token signer and the refresh-token issuer reference one source.

## CORS allowed origins (deployment config)

The browser CORS policy's allowed origins are **configuration-bound, not hardcoded** (#585). The allowed frontend origins are deployment-specific, so `Startup` binds `CorsOptions` (an `AllowedOrigins` list) from the `Cors` configuration section and applies them via `UseCors(...).WithOrigins(...)`.

- **Multiple origins are supported** — `AllowedOrigins` is a list, so an environment can permit several frontend hosts.
- **Fail fast, not silently broken.** The options are registered with `Validate(... Count > 0).ValidateOnStart()`, so an environment with no configured origin fails at startup rather than silently rejecting every browser request. The local dev origin (`http://localhost:5174`) ships in `appsettings.Development.json` so local behaviour is unchanged, and the integration-test host supplies it in `GameServerFactory`.

## Access Roles and Admin Authorization

Authenticated users are gated out of the admin tooling unless they hold the `Admin` role.

- **Roles live in the auth token, not the session store.** `LoginController.Login` loads the user's roles and bakes their names into the access token's `role` claims (and into the refresh token's stored session data, so a rotated token keeps them), so the token is the source of truth for what a request is authorized to do. This keeps the role check self-contained in the signed token and avoids a second lookup; the trade-off is that a role change only takes effect on the user's next login (acceptable for an admin grant, and revisitable if finer-grained revocation is ever needed).
- **Authorization derives from the validated principal, never the gameplay session cache (#649).** `AdminRoleAuthorizationFilter` reads `HttpContext.User.IsInRole(nameof(ERole.Admin))` directly from the cryptographically validated `ClaimsPrincipal` (the bearer handler projects the token's `role` claims onto it, with `RoleClaimType = "role"`). It must **not** key off `SessionService`: that service is populated only on a `Session_{userId}` cache hit, which is a presentation/gameplay concern (the cached `PlayerState`) decoupled from the token's validity — a valid admin whose session key is absent (evicted under memory pressure, never established, or aged out once #537 adds a sliding TTL to that key) would otherwise be wrongly 403'd. Because the principal is the only role source, `SessionService` carries no roles at all; conflating the two is the bug class this filter exists to avoid.

## Admin tooling & user administration

Reference-data administration is modelled as a distinct **Content Authoring** bounded context (thin admin controllers -> admin repositories that own persistence; the EF entity never surfaces in `Game.Api`), and user/role management is the separate **Identity / User Admin** context that shares the `User` aggregate with the auth flow. Both speak `Game.Abstractions.Contracts` types rather than entities, and every admin write reloads the in-memory reference caches via `AdminCacheReloadFilter` (see [Reference-data cache reload](#reference-data-cache-reload-build-then-swap)). Users are archived (a soft delete that frees the username) or banned (blocked but name reserved) as nullable timestamps on `User`, and connections are tracked across three tables each keyed by the signal that distinguishes them. The admin API surface, the change-set/reconciler persistence patterns and their two zero-based-id gotchas, the archive-vs-ban model, and the connection-tracking tables live in [backend-admin.md](./backend-admin.md).
