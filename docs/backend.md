# File Structure and Projects

The game is structured is similar to an onion architecture with multiple projects. The backend projects (all in folders of the same name) are:

- `Game.Abstractions`: This project contains shared abstractions and interfaces used across the backend, such as repository interfaces, infrastructure-service interfaces, and the read/write contracts (`Game.Abstractions.Contracts`). The EF entity models are **not** here â€” they live in `Game.Infrastructure` (see _Entity model location_ below).
- `Game.Api`: This project contains the ASP.NET Core Web API controllers, models, and related code for handling HTTP requests from the frontend, as well as WebSocket command handlers for real-time communication. It also includes a reflection-based code generator for automatically generating API client interfaces for the frontend based on the API controllers and WebSocket handlers.
- `Game.Api.Tests`: This project contains unit and integration tests for the `Game.Api` project, ensuring that all API endpoints and WebSocket handlers function correctly and that the battle logic is consistent with the frontend implementation.
- `Game.Application`: This project contains the application layer of the backend, including services for orchestrating game logic.
- `Game.Application.Tests`: This project contains unit and integration tests for the application layer, ensuring that all orchestration logic is correctly implemented and consistent with the frontend.
- `Game.Core`: This project contains the core domain logic of the game, including domain objects/services and the battle simulation logic. This is where the main game mechanics are implemented.
- `Game.Core.Tests`: This project contains unit tests for the core domain logic, ensuring that all game mechanics, including battle simulation, character progression, and item logic, are correctly implemented and consistent with the frontend.
- `Game.DataAccess`: This project contains the data access layer, including implementations of repositories from `Game.Abstractions` that interact with the database using Entity Framework and the cache and pub/sub implementations provided by `Game.Infrastructure`.
- `Game.Infrastructure`: This project contains the implementation of the cache and pub/sub interfaces defined in `Game.Abstractions` (currently using Redis), as well as the database context, the **EF entity models** (`Game.Infrastructure.Entities`), and migrations for EF Core.
- `Game.Infrastructure.Tests`: This project contains classical unit tests for the in-process logic in `Game.Infrastructure` that has no out-of-process dependency (currently the `BackgroundWorker` Start/Kill lifecycle). It is deliberately lightweight â€” it references only `Game.Infrastructure`, not `Game.TestInfrastructure` â€” because, per the testing guidelines below, anything coupled to Redis or the database is covered by integration tests instead of here.

# General Backend Guidelines

- Repositories should create and persist domain objects and are responsible for dispatching domain events before persisting changes. They should not contain any domain logic nor should they directly return entity models or database DTOs. Mappers should be used to convert between domain objects, entity models, and database DTOs and placed in the Mapping folder in the root of the `Game.DataAccess` project.
- All domain logic should be implemented in the `Game.Core` project, and the application layer should only be responsible for orchestrating calls to the repositories and domain objects/services to fulfill the use case. The API layer should only be responsible for handling HTTP/websocket requests, validating input, and returning responses, and should not contain any domain or orchestration logic.

## Testing Guidelines

Unit tests should be written according to the classical (Detroit) school of testing as much as possible. This project does not have any unmanaged dependencies, so you should be avoiding test doubles as much as possible in both unit and integration tests. If a service under test involves an out-of-process dependency such as the database or cache, it should only need to be tested through integration tests. Any classes that depend on out-of-process dependencies should NOT contain any logic worth unit testing. If you find that a service contains domain logic and depends on an out-of-process dependency, you should refactor the code to move said logic into an appropriate domain class that can be unit tested without the dependency, and then write integration tests only to validate the interaction with the out-of-process dependency.

# Important Architectural Design Decisions

## Entity model location and isolation

The EF entity models are a persistence implementation detail of the data tier, **invisible to `Game.Api` and `Game.Application`**. This was completed in the entity-relocation capstone (#138, the last of the spike-#130 series; see `docs/spikes/130-hide-entity-models.md`).

- **Entities live in `Game.Infrastructure.Entities`,** next to `GameContext` â€” not in `Game.Abstractions`. The supporting `IZeroBasedIdentityEntity` marker and `NotLoadedException` moved with them (into `Game.Infrastructure`); the unused `IAuditableEntity` marker and the vestigial `PlayerState` entity (#142) were deleted as dead code.
- **Hiding mechanism â€” no `internal` needed.** `Game.DataAccess` references `Game.Infrastructure` with `PrivateAssets="Compile"`, so Infrastructure's compile-time types do **not** flow transitively to `Game.Api` (which references `Game.DataAccess`), and `Game.Application` never references `Game.Infrastructure` at all. The entities themselves stay `public` (EF and the test infrastructure need that) but are simply unreachable from the upper layers.
- **The data tier's own entity seam is internal.** `IEntityStore` (the low-level insert/update/delete tracker) is `internal` to `Game.DataAccess`, and the entity-returning reference lookups are exposed only on internal data-tier interfaces (`IItemEntityCache` etc. â€” see [Reference-data read contracts](#reference-data-read-contracts)). The public `Game.Abstractions` repo interfaces speak only domain models and contracts.
- **Architecture test (regression guard).** `Game.Api.Tests/Architecture/EntityIsolationTests` asserts that neither the `Game.Api` nor the `Game.Application` assembly references `Game.Infrastructure`. Because every entity type lives in `Game.Infrastructure`, "no reference to that assembly" is equivalent to "no dependency on the entity model", and it fails fast if the `PrivateAssets`/reference wiring ever regresses.

## Reference Data

Reference data in this app can be split into 2 categories: intrinsic and static. Intrinsic reference data is fundamentally encoded into the application and often has an associated enum or similar construct in the code (e.g. `EAttribute` or `EItemCategory`). These are persisted into the database for data integrity, but they do not need to be (and very rarely are) queried from the database. Static reference data, on the other hand, is not fundamentally encoded into the application but is typically cached in-memory as a `List` such that the Id of the entity is its index. IMPORTANT: The Id column for these tables is seeded to start at 0 to facilitate this indexing. This includes things like items, item modifications, skills, enemies, zones, and challenges. This data is stored in the database and queried on application startup to be cached in-memory for fast access. Since this data is static and does not change often, it is not a problem to cache it in-memory, and it allows for much faster access than querying the database every time it is needed.

> **The Id-as-index optimization requires contiguous ids â€” guaranteed by retirement, not convention.** Both tiers look these sets up *by array index* (`Items.GetItem`/`LookupItem` index the `OrderBy(Id)` cache directly; the frontend does `staticData.items[id]`). That is only correct while the data stays append-only, since the index equals the Id only when there are no gaps. A **hard delete** of a non-terminal record would open a permanent Id gap that silently mis-resolves every lookup above it (and orphan persisted player-state ids and battle snapshots that reference it). To make this structurally impossible, top-level reference records are **retired, not deleted** (#289 â€” see [Retiring reference data](#retiring-reference-data-content-lifecycle)).

### Retiring reference data (content lifecycle)

A top-level reference record (item, item mod, skill, enemy, zone, challenge) carries a nullable `RetiredAt` timestamp â€” the same flag-plus-audit shape as the User `ArchivedAt`/`BannedAt` soft-delete. Retiring takes a record *out of circulation* without erasing it; reinstating clears the timestamp. The guarantees:

- **The slot is permanent.** A retired record is never removed or renumbered, so the cache stays gap-free and every index lookup keeps resolving the right record â€” the contiguity invariant is enforced structurally rather than relying on "only delete the highest Id".
- **It stays resolvable by id.** `GetItem`/`LookupItem`/`GetEnemy`/â€¦ still return a retired record, so existing references â€” a player's equipped item or applied mods, a battle snapshot, an authored boss/reward/unlock-gate FK â€” keep working. Retirement is a catalogue/authoring concern only; it is **not** carried on the lean `Game.Core` battle models, so a retired item a player owns fights identically.
- **It is out of circulation.** The current concrete rule: a retired enemy is excluded from the random per-zone spawn tables (`Enemies.BuildZoneEnemyTables`) so it no longer rolls as an idle encounter. (There is no gameplay path that picks randomly from *all* reference data â€” items/mods/skills are authored challenge rewards, enemies/bosses are authored per-zone â€” so the circulation surface is small.)
- **No hard delete of a reference record.** The admin "delete" affordance is replaced by Retire/Reinstate (an ordinary `RetiredAt` edit through `AddEdit*`). The six reference admin repos pass no `delete` handler to `ChangeSetProcessor`, so a stray top-level `Delete` change throws instead of corrupting data. Child collections (attributes, mod slots, spawn rows, â€¦) are still added/edited/deleted normally.

`RetiredAt` is on the read contracts so the client and Workbench can show the state; it changes the reference-data version hash when toggled (an expected re-download). Deferred follow-ups: excluding retired records from the admin authoring pickers, per-entity edge cases (idling in a retired zone, an in-progress challenge whose target/reward is retired, retiring a zone/challenge used as an unlock gate), and referenced-by warnings on retire.

**Tags deliberately keep the hard-delete lifecycle (#297).** Tags are *not* one of the six zero-based-id reference sets â€” they carry their own (non-zero-based) identity and are resolved by `.find`/Map on both tiers, so they never suffer the index mis-resolution bug retirement exists to prevent. Adding the `RetiredAt` lifecycle to them would be cost without that benefit (a label is not referenced by persisted player state or battle snapshots the way an item/skill is), so the admin "delete" stays a real hard delete. It is graceful for an **in-use** tag: the `ItemTags`/`ItemModTags` join FKs are `ON DELETE CASCADE`, so deleting a tag drops its join rows rather than throwing, and `AdminCacheInvalidationFilter` invalidates the item/mod caches so the next read no longer carries the removed tag id (the change also flows to clients through the items/mods version hash). The Workbench's staged-delete UX surfaces the tag's item/mod usage counts (the Usage tab + list meta) before the admin commits the removal.

### Reference-data read contracts

The reference-data **read** path does not expose EF entities. The repository read methods return dedicated **read contracts** in `Game.Abstractions.Contracts` â€” a single published language shared by the game client's loading screen and the admin Workbench (which share the `Get*` socket commands). The entityâ†’contract mapping lives in `Game.DataAccess` (`Mapping/*` for the cached in-memory lists; EF-translatable inline projections for the small `IAsyncEnumerable` reads), consistent with the data tier owning DTO mapping.

- **Reads return contracts:** the cached `All()` family (`IItems`/`ISkills`/`IEnemies`/`IZones`/`IItemMods`) and the small reference sets (`ITags`/`ITagCategories`) return `Game.Abstractions.Contracts` types. The internal cache stays as entities (the domain mappers need their full fidelity); `All()` projects that cache to contracts, preserving the zero-based Id-as-index ordering.
- **Gameplay reads are unchanged:** `GetItem`/`GetSkill`/`GetDomainEnemy`/`GetItemMod` still return lean `Game.Core` domain models. Cross-repo entity access within the data tier (e.g. `Enemies` building domain enemies from skill entities) goes through the internal `ISkillEntityCache`, never the public `ISkills` read contract.
- **Admin writes keep entities inside the data tier:** the entity-returning lookups (`LookupItem`/`LookupItemMod`/`LookupSkill`/`GetEnemy`/`GetRandomEnemy`/`LookupZone`, and the `GetTagEntitiesFor*`/`GetTags` tag-graph methods) live on **internal data-tier interfaces** in `Game.DataAccess` (`IItemEntityCache`/`IItemModEntityCache`/`ISkillEntityCache`/`IEnemyEntityCache`/`IZoneEntityCache`/`ITagEntityQueries`), not on the public `Game.Abstractions` repo interfaces. Each reference repo implements both its public read interface and the internal entity-cache interface, and one shared scoped instance backs both. Their only consumers are the **Content Authoring admin repositories** in `Game.DataAccess` (see [Admin Tools API surface](#admin-tools-api-surface-content-authoring-context)). Neither `Game.Api` nor `Game.Application` reaches an entity through them: `BattleService` reads the `Zone` contract via `IZones.GetZone`, and `PlayerService` checks mod existence via `IItemMods.ValidateItemModId` (#137).
- **`IModel`** (the cross-boundary model marker) lives in `Game.Abstractions` so the contracts there satisfy the `IModel`-constrained `ApiResponse`/`Change` generics. The contracts keep the same property names/shape as the previous API DTOs, so the serialized wire shape â€” and therefore the reference-data version hash â€” is unchanged.

## Player-facing read projections

Player-facing reads do not expose EF entities to `Game.Api` either, but â€” unlike reference data â€” they reuse the existing **gameplay domain models** rather than dedicated read contracts (those domain models are already lean and lossless enough for these reads).

- **Statistics and challenge progress go through the single `IPlayerProgressRepository`,** which already owns the progress write path. Its `GetStatistics`/`GetChallenges` read methods return the `Game.Core.Progress` domain models (`PlayerStatistic`/`PlayerChallenge`), and the entityâ†’domain mapping lives in `Game.DataAccess` (`Mapping/PlayerProgressMapper`, shared with `Load`). The API DTOs (`Game.Api/Models/Progress/*`) project from those domain models, so the read path references no entities. Consolidating onto `IPlayerProgressRepository` retired the duplicate entity-returning read repos (`IPlayerStatistics`/`IPlayerChallenges`), giving a single source of truth for player-progress data.
- **`Save` is self-sufficient â€” no implicit "load first" ordering.** Both `Load` and `Save` fetch the player's existing rows through a shared private helper that lazily loads and caches the tracked-entity snapshot for the lifetime of the scoped repository. The common battle-completion flow (load â†’ mutate â†’ save) reuses the cached snapshot, so no redundant query is issued; but a standalone `Save` simply fetches the snapshot on demand and upserts correctly rather than dereferencing an un-loaded field. This removed the prior temporal coupling between `Save` and a preceding `Load` (and its null-forgiving `!` usages) â€” #164.
- **Inventory and log-preference reads were already domain-sourced:** `PlayerData.FromPlayer` projects them from the `Player` aggregate. Their leftover entity-coupled mapping (`InventoryItem`'s entity constructor and `LogPreference.FromSource`) was dead and has been removed.
- **The skill loadout is sent as the full unlocked set, not just the equipped ids.** `PlayerData.UnlockedSkills` projects every `player.Skills` entry as an `UnlockedSkill { SkillId, Selected, Order }` â€” the direct parallel of `InventoryItem`'s `Equipped`/`EquipmentSlotId` (a skill's `Order` is null when unequipped, like an unequipped item's slot). Because `player.SelectedSkills` is already the deterministic `(Order, SkillId)` ordering (`PlayerMapper.ToCore`), an equipped skill's `Order` is just its index in that list. The client derives the ordered equipped loadout from this richer set, so there is no redundant equipped-id field. `PlayerData.MaxSelectedSkills` carries the loadout cap (`Player.MaxSelectedSkills`, the single source of truth) so the client renders "X / N" without hardcoding it.
- **The loadout is edited through one atomic write command.** `SetSelectedSkills(orderedSkillIds)` â€” a socket command mirroring `SaveLogPreferences`, aligned with the WebSocket-everything direction â€” replaces the whole equipped set in order, handling select, deselect, and reorder through a single path. It delegates to `Player.TrySetSelectedSkills`, which enforces the loadout rules as **anti-cheat**: no duplicate ids, count â‰¤ `Player.MaxSelectedSkills`, and every id already in the player's unlocked set; any violation rejects the change with no state mutation. On success it raises a single `SelectedSkillsChangedEvent` whose write-behind handler (`DataProviderSynchronizer.HandleSelectedSkillsChanged`) clears the player's `Selected`/`Order` flags and rebuilds them from the ordered list (delete-then-rebuild, idempotent on retry â€” the same shape as the mod-applied handler). This single-writer guarantee is why `PlayerSkill` can carry both `Selected` and `Order` without the two drifting.

## HTTP vs WebSocket Communication

The backend uses a mix of HTTP and WebSockets for communication with the frontend. Right now, HTTP is used for some operations and data fetching, while WebSockets are used for real-time communication such as battle updates. In the future, the goal is to move towards using WebSockets for all communication (other than login/account creation) between the frontend and backend, as this will also enable the Player state to be kept in-memory on the backend and updated in real-time without having to worry about synchronizing with the redis cache or db. Users will only be able to connect to one instance at a time for simplicity and any websocket commands will be handled sequentially for each player. However, HTTP will still be used for admin **persistence** endpoints (the `/api/AdminTools/*` writes), the few reference lookups that have no socket equivalent, and other operations not directly related to the core game.

As a step in that direction, the read-only reference data the loading screen needs (zones, enemies, items, skills, item mods, attributes, challenges, challenge types, statistic types) is exposed over the socket via `Get*` commands (`Game.Api/Sockets/Commands`) that mirror the corresponding `GET` endpoints. These share the `AbstractReferenceDataCommand<TModel>` base, which wraps each repository/intrinsic collection in an `ApiSocketResponse<IEnumerable<TModel>>`. Both the loading screen and the admin Workbench now load these sets over the socket (see [Reference-data caching and versioning](#reference-data-caching-and-versioning)), and the duplicate HTTP `GET` endpoints have been removed (#64). The admin reads can share the same cached socket commands because every admin write invalidates these in-memory caches server-side (`AdminCacheInvalidationFilter`), so the next read re-reads from the database â€” the mechanism the admin `Challenges` reads already relied on, now applied to every reference set. The only HTTP reference reads still active are `Tags` and `Tags/TagCategories` (#318 removed the dead `ZoneEnemies`, `SlotsForItem`, `TagsForItem(Mod)`, `ItemCategories`, and `ItemModTypes` endpoints whose data had moved onto the socket contracts or codegen'd enums).

Statistic-type metadata (`StatisticType { id, entityType, bossOnly, name }`) is intrinsic reference data computed from `EStatisticType` in `Core.Progress.StatisticType.GetAll()` (the DB seed exists only for integrity â€” there is no repository read). It is exposed both as `GET /api/Statistics/StatisticTypes` and the `GetStatisticTypes` socket command so the Statistics screen can resolve which entity kind each statistic breaks down by instead of hard-coding it. `StatisticType.GetEntityType` is the single source of truth for that mapping and must agree with the per-entity rows `PlayerProgress.RecordBattleCompleted` actually writes; `EntityType` is descriptive only, so changing it does not affect challenge evaluation (challenges resolve a statistic by its id + the challenge's target entity id, never by the declared `EntityType`). `BossOnly` is likewise a descriptive domain fact derived alongside `EntityType` (`GetBossOnly` â€” only `BossesDefeated`, which `RecordBattleCompleted` increments solely on dedicated-boss victories): it is the single source of truth for the "this statistic only counts bosses" rule, consumed by both the Statistics screen's display hint and the admin challenge editor, which restricts an enemy-scoped challenge's target picker to bosses when the tracked statistic is boss-only rather than special-casing `BossesDefeated`.

## Single active connection (session takeover warning)

A player may only be connected to one instance at a time: `SocketManagerService.RegisterSocket` records the player's current socket id in Redis (`PlayerSocket_{playerId}`) and, when a new socket replaces an existing one, emits `SocketReplaced` to the old socket to close it. To let the client warn the user **before** that takeover happens, `Login/ActiveSession` (an authenticated `GET`) reports whether the player already has a live socket via `SocketManagerService.HasActiveSocket` â€” a read of the same presence key, so the knowledge of the key's format stays in one place. The check is deliberately **HTTP, not a socket command**: opening a socket to ask the question would itself trigger the replacement it is meant to warn about. Because it is presence-based (rather than device- or tab-scoped), it covers both "another tab" and "another device" uniformly â€” any other live connection for the player.

- **The presence key is liveness-accurate via a heartbeat-refreshed TTL.** The key is written with a 30s TTL (`SocketPresenceTtl`) at register and refreshed on **every inbound socket message** â€” the client heartbeats every 10s (`api-socket.ts`), the same `_lastResponse` signal the 60s inactivity check already uses â€” so a live connection keeps its key fresh while a connection that vanished without a clean close (a tab/device closed without a WebSocket close frame, e.g. WebKit) lets the key expire within the TTL instead of lingering. The refresh is an `Expire` (extend-TTL) rather than a re-set, so it only ever prolongs the currently-registered key and never resurrects one a newer connection has taken over. Net effect: the warning fires immediately for a genuinely-open second tab/device, and the residual false-positive window â€” re-logging-in within ~30s of a non-graceful close and being warned about your own just-closed session â€” is bounded by the TTL (tunable; tolerates two missed heartbeats) and documented as acceptable for a best-effort, dismissible warning.
- **Possible future tightening:** a `BroadcastChannel`-based same-browser probe could give instant, zero-window detection of other tabs in the *same* browser (a crashed tab simply can't answer), layered on top of this server-side presence as polish â€” see the follow-up spike (#221).

## Reference-data caching and versioning

The frontend caches each reference-data set in the browser and re-downloads only the sets that changed (see `frontend.md`). The `GetReferenceDataVersions` socket command supports this by returning a content version per set in one round-trip.

- **The version is a content hash of the API models, computed on demand.** Each `AbstractReferenceDataCommand` exposes `ComputeVersion()` (via `IReferenceDataCommand`), which hashes the same serialized model collection the command returns (`ReferenceDataVersioning` â€” SHA-256 over the camelCase JSON, through the concrete model type so every property is included). Hashing what the client actually receives means the version changes if and only if the client-visible data changes, so no separate version counter has to be maintained or invalidated. It is recomputed per call â€” the command is hit roughly once per client boot â€” so there is nothing to invalidate when an admin edits reference data; memoizing it keyed off cache refresh is a possible future optimization if it ever shows up in profiling.
- **The versioned set is exactly the set of reference-data commands.** `GetReferenceDataVersions` is injected with `IEnumerable<IReferenceDataCommand>` (registered by `AddReferenceDataCommands`, which scans the assembly), so a newly-added `Get*` reference-data command is versioned automatically â€” there is no second list of datasets to keep in sync.

## Battle setup (enemy encounter)

Setting up a battle is a domain decision, not an orchestration one. `BattleService.StartBattle`/`StartBossBattle` (Application) only orchestrate â€” obtain the zone, snapshot the player and the chosen enemy loadout, set the active battle â€” and delegate the game-logic choices to `Game.Core`:

- **The `Game.Core.Zones.Zone` aggregate owns the per-zone encounter-level decisions.** It exposes the `[LevelMin, LevelMax]` idle range (and `RollEncounterLevel()`), and the dedicated boss's identity + fixed level (`BossEnemyId`/`BossLevel`/`HasBoss`). The data tier serves it via `IZones.GetDomainZone` (the gameplay/domain read, mapped by `ZoneMapper.ToCore`), alongside the existing `IZones.GetZone` contract read used by the reference-data catalogue. This replaced an unreferenced, mis-shaped `Zone` remnant (#149): the responsibility for "what level do you fight here, and who is the boss" now lives on one domain type instead of being threaded as loose ints.
- **`BattleFactory.CreateBattleEnemy(zone, resolver)` rolls a random idle encounter** within the zone's range and returns a battle-ready enemy with a capped, randomly-selected loadout (`Enemy.SelectBattleSkills()`, â‰¤4 skills). Enemy _selection_ (which enemy spawns) still lives in the data-access layer (the cached per-zone spawn table behind `IEnemies.GetRandomDomainEnemy`), so the factory takes a `Func<int, Enemy>` resolver rather than depending on a repository â€” keeping `Game.Core` free of data-access references.
- **`BattleFactory.CreateBossEnemy(zone, resolver)` builds the dedicated boss deterministically** for the "Challenge Boss" action: resolved at the zone's fixed `BossLevel` (no random roll) with its **full authored loadout** (`Enemy.SelectAllBattleSkills()`, no cap, no shuffle). The greater determinism is exactly what frontend/backend battle parity wants. The boss challenge is always available (no cooldown gate) and challenging does not change the player's current zone; `StartBossBattle` returns `null` when the zone has no boss authored, which the `ChallengeBoss` socket command surfaces as an error.
- **Zone-unlock is enforced at battle start (anti-cheat).** A zone carries an optional `Zone.UnlockChallengeId` (#190 â€” see the [game-design.md](./game-design.md#zone-progression--locking-zones-behind-challenges) zone-progression section). `BattleService` refuses to enter a locked zone: a `StartBattle` *transition* into one is ignored (the player keeps fighting their current zone) and a `StartBossBattle` against one returns `null`. The unlock rule lives on the domain `Zone.IsUnlocked(completedChallengeIds)`; the orchestration reads completion via `IPlayerProgressRepository.GetCompletedChallengeIds`. An ungated zone (the common case) pays no extra read, and a gated zone costs one indexed completion lookup incurred only on a real zone transition or a boss challenge â€” never per idle tick.
- **The chosen loadout is snapshotted rather than re-rolled.** The selected skill ids are stored on `PlayerState.ActiveEnemySkillIds` (symmetric with the player's `BattleSnapshot`) and sent to the client; validation (`EndBattleVictory`/`Loss`/`AbandonBattle` â†’ `SimulateBattle`) reconstructs the enemy from that snapshot via `Enemy.SetBattleSkills(ids)`. Both sides therefore use the exact same loadout â€” required because battle logic runs on both the frontend and the backend.
- **The player side of the battle is captured and reconstructed in the domain.** `BattleSnapshot.FromPlayer` projects the player's battle-relevant state (level, stat allocations, equipped item + applied-mod ids, skill ids) into a minimal id-based snapshot at battle start, and `BattleSnapshot.ToBattler(resolveItem, resolveMod, resolveSkill)` rebuilds the player `Battler` from it at validation time â€” resolving the ids against the catalogs through caller-supplied resolvers (the same `BattleFactory` resolver pattern that keeps `Game.Core` free of data-access). Critically, reconstruction **reuses the same modifier-composition rules as the live `Player` aggregate** â€” `StatAllocation.ToModifier` and `Item.GetAttributeModifiers`, the building blocks of `Player.GetAllModifiers` â€” so a battle validated from a snapshot composes the identical attributes the player has live; this is the player half of the frontend/backend battle-parity guarantee, so the rules cannot be allowed to diverge. `Game.Application` (`BattleService`) only fetches the reference data and hands over the resolvers (#208).
- **A dedicated-boss battle is marked, not inferred.** `PlayerState.IsBossBattle` (+ `BattleZoneId`, the challenged zone) is captured at battle start and threaded through `BattleCompletedEvent`, so the resolve path knows a victory was a dedicated-boss clear rather than inferring it from `enemy.IsBoss` + `CurrentZoneId`. `PlayerProgress` keys the zone-clear off that marker, so only the challenge path clears a zone. The clear _semantics_ (#188) treat `ZonesCleared` as a count of distinct zones ever cleared â€” the per-zone entry is a binary flag and the global counter bumps only on a zone's first clear â€” while the per-boss/global `BossesDefeated` farm counter increments on every dedicated-boss victory (see the Zone Clears section in [game-design.md](./game-design.md)).
- **An abandon that resolved as a win is a full victory.** `AbandonBattle` (reached only when the player starts a new battle while one is active) re-simulates the in-progress fight capped at the elapsed wall-clock time. If the enemy died within that window the result is a genuine win, so it pays out through the same `RecordVictory` helper as `EndBattleVictory` â€” exp **and** the win/zone-clear/challenge credit â€” instead of booking the win while withholding the exp (#206). Both win paths share that one helper so they cannot drift. An abandon that did **not** resolve as a win is split by outcome: if the player died within the elapsed window it is a loss, otherwise (neither combatant died) it is an _incomplete abandon_, tracked as the distinct `BattlesAbandoned` statistic rather than a loss (#202 â€” see the [game-design.md](./game-design.md#battle-outcomes-battleswon--battleslost--battlesabandoned) battle-outcomes split).
- **The battle `seed` is reserved for the simulation's RNG, not for setup.** It is generated at battle start, persisted on `PlayerState.BattleSeed`, and sent to the client as the shared starting point for any randomness _inside_ the simulation (crit/dodge/etc., not yet implemented). Setup (level roll and skill selection) deliberately uses ambient randomness instead, so it never advances the seed's stream â€” otherwise the server (consuming draws during setup) and the client (which receives the loadout ready-made) would begin the simulation from divergent RNG states. (The boss path is already deterministic, so this only matters for the random idle encounter.) The RNG primitive that will consume the seed is already ported on both sides (`Game.Core/Battle/Mulberry32.cs` and `UI/src/lib/engine/mulberry32.ts`) â€” intentional forward-scaffolding for the seeded crit/dodge work (#178), not dead code. It has no live consumer yet, so a cross-implementation parity vector (`Mulberry32ParityTests` â†” `mulberry32-parity.test.ts`) is the only thing guarding the two ports against silent drift; that vector must stay mirrored, exactly like the battle-simulation parity matrix.

## Caching and Pub/Sub

As mentioned above, most of the reference data is cached in-memory on the backend for fast access. However, player data is cached in Redis and uses a write-behind caching strategy where the cache is the source of truth for player data, and changes are persisted to the database asynchronously. The application uses Redis pub/sub to trigger persistence of player data to the database whenever it changes and as a backplane for WebSocket communication.

The write-behind path is wired through the domain-event pipeline. When the `Player` aggregate mutates it raises domain events; on save, `PlayerRepository` dispatches them in-process and `PlayerPersistencePublisher` (an `IDomainEventHandler` registered per event type in `AddDataAccess`) serializes each one into a `DomainEventEnvelope` and pushes it onto the `PlayerUpdateQueue` Redis list. The `DataProviderSynchronizer` hosted service consumes that queue and applies each change to the database. The publisher is registered only for the player events that represent persistent state changes; events that are purely in-process reactions (e.g. `PlayerLeveledUpEvent`, whose persistent effect is already carried by `PlayerCoreUpdatedEvent`) are intentionally not registered, so they are never enqueued. This keeps the "which events get persisted" decision in the data-access layer alongside the `DataProviderSynchronizer` switch that consumes them, rather than encoding a persistence concern onto the domain events themselves.

Queue processing in `DataProviderSynchronizer` is resilient and never silently drops an event:

- **A malformed payload is dead-lettered, not retried.** A payload that fails to deserialize (or whose inner event payload is malformed) can never succeed, so it is logged as a warning and moved to the `PlayerUpdateDeadLetterQueue` Redis list for inspection instead of being discarded.
- **A valid event that fails on an unexpected error is retried with exponential backoff, then dead-lettered.** Transient failures (e.g. a database connection blip, deadlock, or timeout) often succeed on a second attempt, so each event is retried per `PlayerUpdateRetryPolicy` (default: 3 attempts, 200ms base backoff). The non-final attempts log a warning; once the retries are exhausted the event is logged as an error and pushed onto the dead-letter queue rather than dropped, so the change is preserved for later inspection/replay. Retrying is safe because the handlers are idempotent (existence checks, absolute `ExecuteUpdate`s, and delete-then-insert), so re-applying a partially-applied event converges to the same state.
- In every case the rest of the queued events are still processed.

The dead-letter queue is intentionally a passive holding list â€” there is no automatic redelivery consumer yet, and the queue read is still a destructive pop, so an event in flight when the process crashes is lost (a crash-safe reliable-queue / peek-then-acknowledge read remains future work). The retry-then-dead-letter path covers the common transient-error case and keeps a durable record of anything that exhausts its retries.

## Redis connection thread-pool floor

`RedisMultiplexerFactory` raises the process thread-pool minimum (to at least `MinThreadPoolThreads`, never lowering a host that already runs higher) before connecting any multiplexer. StackExchange.Redis completes async commands on the .NET thread pool, whose default minimum equals the CPU count; once busy threads exceed that minimum the CLR adds threads only ~1 per 500ms. On small, busy hosts (a CI runner hosting the API + Postgres + Redis, or multiple integration-test assemblies at once) that ramp-up stall could leave a reply unread in the socket past the 5s command timeout, surfacing as a spurious `RedisTimeoutException` (`WORKER: (Busy=N, Min=processorCount)`). Centralizing the bump in the single connection chokepoint covers production and every test process that touches Redis without scattering startup tweaks across entry points.

## BackgroundWorker disposal and pub/sub worker ownership

`BackgroundWorker` (`Game.Infrastructure`) owns an `AutoResetEvent`, so it is `IDisposable`. `Kill()` and `Dispose()` are deliberately distinct: `Kill()` stops the worker from being scheduled again while keeping the instance queryable, whereas `Dispose()` also releases the OS wait handle (and implies `Kill()`). `Start()` publishes `IsRunning = true` **before** signaling the worker loop so the loop's trailing `IsRunning = false` always happens-after it â€” otherwise a fast (already-signaled) run could complete and reset the flag before the late `true` landed, wedging `IsRunning` stuck `true` until the next run. The flag is `volatile` so the transition is published across the threads that read it.

`RedisPubSubService` is **not** `IDisposable`. Its worker registry is `static` (process-wide) but the service is registered **transient**, so disposing one instance would tear down workers owned by every other instance. Each worker is therefore disposed at its own teardown point â€” when its subscription is removed (`UnSubscribe`) or when it is discarded because its id was already taken â€” which is the correct ownership boundary. (The single app-lifetime queue subscription created without an id, e.g. `DataProviderSynchronizer`, is intentionally not tracked or disposed; it lives for the process.)

## Admin Tools API surface (Content Authoring context)

The frontend admin tools were consolidated into a single entity-driven Workbench (see `docs/frontend.md`). That UI saves a whole record (identity + related collections) at once, but the backend keeps persisting each relationship through its own admin endpoint â€” the Workbench just orchestrates them.

Reference-data administration is modelled as a distinct **Content Authoring** bounded context (its own model over the shared database), implemented as namespaces within the existing projects rather than new projects. The EF entity is this context's persistence shape and is an implementation detail of `Game.DataAccess` â€” it never surfaces in `Game.Api`. (User administration is the separate Identity / User Admin context â€” see [User Administration](#user-administration-archive-vs-ban).)

- **Admin controllers are thin HTTP adapters.** `Game.Api/Controllers/Admin/*` accept an admin write contract, call an admin repository, and map the result to an `ApiResponse`. They reference no `Game.Abstractions.Entities` and no `IEntityStore`.
- **Admin write contracts live in `Game.Abstractions.Contracts.Admin`** (`Change<T>`, `EChangeType`, `AddEditAttributesData`, `SetTagsData`, and the `Set*Data` relationship DTOs). The whole-record `Change<T>` payloads reuse the reference-data read contracts in `Game.Abstractions.Contracts` (`Enemy`/`Item`/`Skill`/`Zone`/`Tag`/`ItemModSlot`/`Challenge`), so reading and authoring share one serialized shape. `Challenge` is the single read+write contract: the `GetChallenges` socket maps the gameplay domain `Challenge` (which `IChallenges` still serves for challenge evaluation) onto it, and the admin write path persists it.
- **Admin repositories own the persistence.** The `IAdmin*` interfaces (`Game.Abstractions/DataAccess/Admin`) are implemented in `Game.DataAccess/Repositories/Admin`. They encapsulate the change-set ordering (`ChangeSetProcessor`, also relocated here), the fresh-entity construction, and the `IEntityStore` calls; they reuse the cached reference repos for existence/diff lookups. A relationship setter returns `false` when its target record is missing, and the controller maps that to the user-facing "not found" message.
- **Cache invalidation stays a cross-cutting filter, discovered through `ICacheInvalidatable`.** `AdminCacheInvalidationFilter` invalidates the in-memory reference caches after every admin write so the next read re-reads from the database. Each cached reference repository implements `ICacheInvalidatable` (`Game.Abstractions.DataAccess`) and is registered under that interface as well as its own, so the filter (and the test bases, via `Game.TestInfrastructure`'s `ReferenceCacheCleaner.InvalidateAll`) resolves `IEnumerable<ICacheInvalidatable>` from DI instead of hand-maintaining a list of the caches â€” a newly-added cache is covered automatically (#312). The filter never touches entities, so it stayed at the presentation edge rather than moving into the repositories.

Two persistence gotchas the admin repositories respect (and any new admin write must too):

- **Updates must not be inferred from the key value.** `IEntityStore.Update` tracks the passed entity as `Modified` directly (`_context.Entry(entity).State = EntityState.Modified`) instead of going through `DbContext.Update`. `DbContext.Update` infers Add-vs-Edit from whether the key is "set", which misfires for our zero-based-identity tables (see [Reference Data](#reference-data)): the first row's `Id == 0` is the CLR/identity-column default, so EF would treat an edit of record 0 as a brand-new insert and silently duplicate it.
- **Don't update/delete cached entities directly.** Reference data is read with `AsNoTracking().Include(...)`, so a fetched child (e.g. a `SkillDamageMultiplier`) carries a loaded back-reference to its parent and siblings. Passing that instance to `Update`/`Delete` would drag the whole graph into the change tracker and re-insert it. Relationship endpoints therefore build fresh, navigation-free entities keyed only by their own ids (as `SetEnemySkills`/`SetZoneEnemies`/`SetSkillMultipliers` do) rather than mutating the cached objects.

## Authentication (JWT access + refresh tokens)

Authentication uses standard ASP.NET Core JWT bearer auth rather than the legacy cookie-backed pseudo-session token.

- **Short-lived signed access tokens + long-lived rotating refresh tokens.** `AccountService.Login` (called by `LoginController.Login`) verifies credentials and returns the issued tokens plus the player aggregate; the controller maps that to a `LoginResult` carrying an `AuthTokens` pair plus the player data. The **access token** is a JWT signed (HMAC-SHA256) behind the `IAccessTokenService` abstraction (`JwtTokenService`) using the configured `Jwt:SigningKey`, with a 15-minute lifetime (`AuthConstants.AccessTokenLifetime`); it carries the user id (`sub`) and `role` claims. The **refresh token** is an opaque, high-entropy random value with a 48-hour lifetime (`AuthConstants.RefreshTokenLifetime`). Clients send the access token as an `Authorization: Bearer` header; when it expires they call `Login/Refresh` to mint a new pair.
- **Standard claims pipeline.** `AddJwtBearer` validates the token (issuer/audience/signature/lifetime) and projects it onto `HttpContext.User`. A global `FallbackPolicy` requires every endpoint to be authenticated unless marked `[AllowAnonymous]` (login, account creation, refresh, logout). `SessionLoaderMiddleware` bridges the validated principal to the request-scoped `SessionService` (user id + roles + cached player state); inbound claim mapping is disabled so claim types stay verbatim (`sub`/`role`).
- **Refresh tokens are stored server-side in Redis, keyed by a SHA-256 hash of the value (`IRefreshTokenStore`/`RefreshTokenStore`).** Only the hash is persisted, the TTL enforces the lifetime, and consumption is an atomic get-and-delete so each refresh token is **single-use**: `Login/Refresh` rotates it (issues a brand-new pair and invalidates the old refresh token), and `Login/Logout` revokes it. Redis is the natural home because auth already depends on the Redis player session (a flushed cache logs users out regardless), so this adds no new durability assumption and avoids a DB migration.
- **WebSockets authenticate via an `access_token` query-string parameter.** Browsers cannot set an `Authorization` header on the WebSocket handshake, so the `/socket` endpoint reads the access token from the query string (wired through `JwtBearerEvents.OnMessageReceived`) â€” the canonical ASP.NET Core pattern for bearer auth over WebSockets.
- **Frontend consumption is a follow-up.** This change is backend-only; persisting the tokens in the browser and sending them as a bearer header / socket query param is tracked separately (see issue history).

### Password hashing (PBKDF2)

Passwords are hashed with **PBKDF2-HMAC-SHA256** behind the `IPasswordHasher` abstraction (`Game.Abstractions/Auth`), implemented by `Pbkdf2PasswordHasher` (`Game.Application/Auth`). This replaced a bespoke iterated-SHA-512 construction (a fast, GPU-friendly hash assembled by hand) with a purpose-built, deliberately-slow KDF.

- **Self-contained, tunable format.** A hash is stored as `$pbkdf2-sha256$<iterations>$<base64-salt>$<base64-key>`, so the work factor and a per-hash salt live alongside the key â€” the stored hash is fully self-contained. The iteration count is configured via `PasswordHashingOptions.Iterations` (default 600k, the OWASP floor for PBKDF2-HMAC-SHA256); tests lower it via `PasswordHashing:Iterations`.
- **Salt and pepper.** A fresh 128-bit random salt is generated for every hash and embedded in the output (so `Hash`/`Verify` take no salt parameter and there is no separate salt column). An application-wide pepper (`HashPepper` secret) is folded in via a standard HMAC pre-hash for defence-in-depth; it is injected through `PasswordHashingOptions` rather than process-global mutable static state, and validated on startup so a missing pepper still fails fast.
- **Transparent work-factor upgrade on login.** `Verify` returns `SuccessRehashNeeded` for a valid credential stored with an outdated work factor. `AccountService.Login` then re-hashes the verified plaintext (with a fresh salt) and persists it via `IUsers.UpdatePasswordHash` (a targeted `ExecuteUpdate`), so existing accounts upgrade on next login without a forced reset. Verification comparisons are constant-time (`CryptographicOperations.FixedTimeEquals`).

### Account/login orchestration lives in the application layer

Account creation and the login/refresh/logout flows are orchestrated by `AccountService` (`Game.Application/Services`), not the controller. `LoginController` is a thin HTTP adapter: it calls the service, maps the result to an `ApiResponse`, and wires the **request-scoped** `SessionService` (creating the session on login, clearing it on logout) â€” session identity is a presentation concern, so it stays in the API layer. `AccountService` owns the data orchestration: username availability, password hashing, delegating the new account's user-and-initial-player insert to the Identity context, credential verification, loading the player aggregate, and refresh-token rotation/revocation via `IRefreshTokenStore`. `Game.Application` references no EF entities (#137); it speaks only Identity contracts (`NewAccount`/`AccountCredentials`) and domain types.

- **The new-player defaults are a domain concern, built by `NewPlayerFactory` (`Game.Core/Players`).** "What does a new player look like?" â€” the starter skill set, the base attribute spread, and the default log preferences (plus the `StarterSkillCount`/`AttributeCount`/`StartingAttributeAmount`/`StartingZoneId` constants) â€” is a game rule, so it lives in `Game.Core` rather than the orchestration layer. `NewPlayerFactory.Create(name)` returns a `NewPlayer` blueprint (raw defaults, no identity or resolved reference-data objects). `AccountService` hashes the password into a `NewAccount` credential blueprint and hands both blueprints to `IUsers.CreateAccount` (the Identity context owns the `User` aggregate), which maps them to the linked user+player entity graph (`PlayerMapper.ToEntity`) and queues it in the unit of work. `AccountService` therefore neither encodes the defaults nor constructs entity graphs â€” it only orchestrates. Unlike `SavePlayer`'s write-behind path, account creation persists straight through the unit of work because a freshly created player has no cached state or domain events yet.
- **Access-token issuance is abstracted behind `IAccessTokenService` (`Game.Abstractions/Auth`).** The application layer must issue the access token as part of the login/refresh use case, but the concrete JWT implementation (`JwtTokenService`) sits at the presentation edge alongside the bearer-validation pipeline it mirrors. Depending on the abstraction keeps `Game.Application` free of the JWT libraries; the API composition root registers `JwtTokenService` as the implementation. Token-lifetime policy is shared via `AuthConstants` (`Game.Abstractions/Auth`) so the access-token signer and the refresh-token issuer reference one source.

## Access Roles and Admin Authorization

Authenticated users are gated out of the admin tooling unless they hold the `Admin` role.

- **Roles live in the auth token, not the session store.** `LoginController.Login` loads the user's roles and bakes their names into the access token's `role` claims (and into the refresh token's stored session data, so a rotated token keeps them), so the token is the source of truth for what a request is authorized to do. `SessionLoaderMiddleware` copies the validated principal's roles onto the request-scoped `SessionService` (which exposes `IsInRole`, used by `AdminRoleAuthorizationFilter`). This keeps the role check self-contained in the signed token and avoids a second lookup; the trade-off is that a role change only takes effect on the user's next login (acceptable for an admin grant, and revisitable if finer-grained revocation is ever needed).

## User Administration (Archive vs Ban)

`AdminUsersController` (under `Game.Api/Controllers/Admin`) exposes the admin user-management surface: search the roster (`GetUsers`), list the grantable roles (`GetRoles`), replace a user's roles (`SetUserRoles`), and the two lifecycle actions below. It shares the `/api/AdminTools/[action]` route prefix like the other admin controllers, but a few of its conventions differ on purpose:

- **Identity / User Admin is its own bounded context, and its repository reads speak contracts, not EF entities.** This context shares the `User` aggregate with the auth/login flow, so it sits alongside it rather than under Content Authoring. `IUsers`/`IRoles` return `Game.Abstractions.Contracts.Identity` types: the admin roster (`SearchUsers` â†’ `AdminUser`, with nested `PlayerSummary`/`Role`; `GetRoles` â†’ `Role`) is the published read language the Workbench consumes, and `GetUser` returns an `AccountCredentials` lookup for the login use case. The entity â†’ contract projection lives in `Game.DataAccess` (`Users`/`Roles`), so the `User`/`Role` EF entities never surface in `Game.Api` or in the `AccountService` login path. `AccountCredentials` is deliberately **not** an `IModel`: it carries the password hash and is internal to the data tier + application layer, never serialized across the API boundary. Account creation is the matching **write** path: `IUsers.CreateAccount(NewAccount, NewPlayer)` takes a `NewAccount` credential blueprint (also a non-`IModel` Identity contract) plus the new-player blueprint and builds the linked user+player entity graph in the data tier, so `AccountService` constructs no entities (#137).
- **Archiving and banning are two distinct, non-destructive states, modeled as nullable timestamps on `User` (`ArchivedAt`, `BannedAt`).** Timestamps (rather than booleans) double as the flag _and_ an audit of when the action happened, and follow the idiomatic soft-delete shape.
  - **Archive = soft delete that frees the username.** An archived user is excluded from login (`IUsers.GetUser`) and from username-availability checks (`IUsers.CheckIfUsernameExists`, which gates account creation). Because the username check ignores archived rows, the username becomes available for a brand-new account â€” there is intentionally **no DB unique index** on `Username`, so an archived row and a new active row can share a username; the active one is the single non-archived match the queries resolve to. The admin roster, by contrast, does **not** hide archived users by default: `SearchUsers`/`CountUsers` take a tri-state `archived` filter (`null` = all users, `false` = active only, `true` = archived only), so an admin can still find and manage archived accounts.
  - **Ban = block while keeping the username reserved.** A banned user still counts toward username availability (so the name cannot be reused) and remains visible in the admin roster for later management. Surfacing the ban to the user at login time is intentionally out of scope for now.
- **Role changes take effect on next login.** As noted under [Access Roles and Admin Authorization](#access-roles-and-admin-authorization), roles live in the signed auth token, so a `SetUserRoles` change is reflected only after the affected user logs in again.

## User Connection Tracking

Connections are tracked with three tables rather than a full ledger, each keyed by the signal that actually distinguishes it:

- **`BrowserInfo` â€” the user-agent profile, deduplicated by `UserAgent`** (unique index). Holds only the server-observable signals carried on every request: the user-agent and the low-entropy client-hint headers (`Sec-CH-UA`/`-Mobile`/`-Platform`). It is genuinely shared across the many devices that report the same user-agent. Client hints are filled in only when missing, so a later request with fewer hints never clears earlier values.
- **`Device` â€” the device, deduplicated by `DeviceFingerprintHash`** (unique index). This is the most device-unique signal, so device-specific data lives here: the fingerprint, the capabilities (`deviceMemory`/`hardwareConcurrency`), and a FK to its `BrowserInfo` (the fingerprint incorporates the user-agent, so that mapping is stable). The fingerprint is computed client-side; the capabilities are reported by the frontend after login (they aren't on a regular request) via the authenticated `Login/DeviceInfo` endpoint.
- **`UserLogin` â€” a connection origin.** A surrogate identity key plus a unique index on `(UserId, IpAddress, DeviceId)`; `LastConnection` is updated in place for that combination rather than appended. (A unique index, not a composite PK, keeps the store-generated `Device` key out of the dependent's primary key â€” which EF cannot cleanly propagate when inserting both rows together.)

Keying `UserLogin` on the device means the per-request tracker needs the fingerprint on **every** request, but browsers don't send one on their own. So the frontend computes a deterministic fingerprint once (a SHA-256 of stable client signals, cached in `localStorage`) and the request layer attaches it as an `X-Device-Fingerprint` header on every authenticated request. This is the deliberate trade-off chosen over storing device data on a user-agent-keyed row (which would let two devices sharing a user-agent overwrite each other).

- **Recording happens in `LoginTrackingMiddleware`, after `SessionLoaderMiddleware`,** so it fires on authenticated **HTTP** requests (not per WebSocket command â€” those never re-enter the request pipeline, and a WebSocket handshake can't set the custom header anyway, so it's skipped). A request without the fingerprint header is skipped (no device to key on). Recording is **best-effort**: failures are logged and swallowed so tracking never affects the response. Because it runs outside the per-action `CommitFilter`, the tracking service commits its own unit of work. The trade-off is a small write per authenticated HTTP request; acceptable while most game traffic is on the socket, and throttleable later if HTTP volume grows.
