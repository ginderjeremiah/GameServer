# File Structure and Projects

The backend follows an onion-style architecture across several projects (each in a folder of the same name):

- **`Game.Core`** — the domain layer: domain objects/services, game mechanics, and the battle simulation. The heart of the game logic.
- **`Game.Application`** — orchestration only: coordinates repositories and domain services to fulfil use cases. No domain logic.
- **`Game.Api`** — ASP.NET Core controllers, WebSocket command handlers, and API models. Also hosts a reflection-based codegen that generates the frontend's API client. Handles HTTP/WebSocket requests and input validation only.
- **`Game.Abstractions`** — shared interfaces (repositories, infrastructure services) and the read/write contracts (`Game.Abstractions.Contracts`). EF entities do **not** live here.
- **`Game.DataAccess`** — repository implementations, the entity↔domain/contract mappers (`Mapping/`), and the in-memory reference-data caches.
- **`Game.Infrastructure`** — the Redis cache/pub-sub implementations, the EF `GameContext`, the **EF entity models** (`Game.Infrastructure.Entities`), and migrations.

Each layer has a matching `*.Tests` project. `Game.Infrastructure.Tests` is deliberately lightweight — only in-process logic with no out-of-process dependency; anything coupled to Redis or the database is covered by integration tests instead.

# General Backend Guidelines

- Repositories create and persist domain objects and dispatch domain events before persisting. They contain no domain logic and never return entity models or DB DTOs — mapping between domain objects, entities, and DTOs lives in `Game.DataAccess/Mapping`.
- Domain logic lives in `Game.Core`. The application layer only orchestrates; the API layer only handles requests, validates input, and returns responses.

## Testing Guidelines

Unit tests follow the classical (Detroit) school — the project has no unmanaged dependencies, so avoid test doubles. A class that depends on an out-of-process dependency (database, cache) should hold no logic worth unit-testing; if it does, move that logic into a domain class and cover the dependency interaction with integration tests instead.

Judge branch coverage per-branch rather than chasing the number: defensive guards (e.g. config that is always supplied) and branches reachable only by disrupting process-wide shared infrastructure are accepted as measure-only, with the real logic pinned by integration tests against the DI-resolved interface.

### Running tests (Microsoft.Testing.Platform)

The test projects run on **Microsoft.Testing.Platform** (MTP / xunit.v3), not the legacy VSTest runner. The gotcha: VSTest's `--filter` syntax is **silently ignored** (it sits before `--`, so the wrapper swallows it and the *whole* suite runs). Pass MTP filters after `--` instead:

```
dotnet test Game.Api.Tests/Game.Api.Tests.csproj -- --filter-class "*SessionServiceTests"
```

(`--filter-class` / `-method` / `-namespace` / `-trait`, each with a `--filter-not-*` negation; run the binary with `--help` to list them.) A filtered run is a dev convenience — CI runs the whole solution under coverage, so run the full suite before pushing.

# Important Architectural Design Decisions

## Entity model location and isolation

EF entity models are a persistence detail of the data tier and are **invisible to `Game.Api` and `Game.Application`**. They live in `Game.Infrastructure.Entities`, next to `GameContext`. `Game.DataAccess` references `Game.Infrastructure` with `PrivateAssets="Compile"`, so the entity types don't flow transitively up to `Game.Api`, and `Game.Application` never references `Game.Infrastructure` at all. Within the data tier itself the entity seam stays `internal` (the low-level insert/update/delete tracker and the entity-returning reference lookups); the public `Game.Abstractions` repository interfaces speak only domain models and contracts. An architecture test asserts that neither `Game.Api` nor `Game.Application` references `Game.Infrastructure`, failing fast if the wiring ever regresses.

## Reference Data

Reference data is either **intrinsic** or **static**:

- **Intrinsic** reference data is encoded in the application as an enum or similar (e.g. `EAttribute`, `EItemCategory`). It is persisted only for data integrity and is essentially never queried — it is served straight from the enum. Intrinsic sets may carry derived display metadata (classification, formatting) on the domain/contract directly, sourced from the enum with no database columns; adding such a field changes the set's version hash, so clients re-download it once.
- **Static** reference data (items, item mods, skills, enemies, zones, challenges) is authored content stored in the database and **cached in-memory as immutable lists, indexed by id**. The Id column is seeded from **0** so an entity's id doubles as its array index on both tiers — an O(1) lookup, but one that requires the ids stay contiguous.

The caches are immutable snapshots **loaded eagerly at startup** (a database problem surfaces as a boot failure, not a first-request failure) and refreshed by build-then-swap (see [Reference-data cache reload](#reference-data-cache-reload-build-then-swap)). There is no lazy-fill path.

> **Contiguity is guaranteed by retirement, not convention.** A hard delete of a non-terminal record would open a permanent gap that silently mis-resolves every lookup above it, so top-level records are **retired, not deleted** (below). As a backstop against a seed/migration mistake, each snapshot **asserts `Id == index` as it is built** and throws otherwise, so a corrupt set fails the build-then-swap (keeping the prior good snapshot, or a boot failure on the startup load) rather than serving mis-resolved data.

**Player-load missing-reference policy: loud fail-fast, never silent drop.** Retirement + contiguity keep every owned reference (item, mod, skill) resolvable forever, so an unresolvable one when the player aggregate loads can only mean a content-data mistake (a seed/migration that removed a referenced id). The load then **fails loudly** with a diagnosable exception (player id, catalog, missing id) rather than silently dropping the player-owned row — player data is never discarded, and the message makes the mistake obvious from logs instead of an opaque player-lockout.

### Retiring reference data (content lifecycle)

A top-level reference record carries a nullable `RetiredAt` timestamp — the same flag-plus-audit shape as the User soft-delete. Retiring takes a record out of circulation without erasing it; reinstating clears the timestamp.

- **The slot is permanent** — a retired record is never removed or renumbered, so the cache stays gap-free and every index lookup keeps resolving correctly.
- **It stays resolvable by id**, so existing references (an equipped item, a battle snapshot, an authored reward) keep working. Retirement is a catalogue/authoring concern only — it is not carried on the lean battle models, so a retired item a player owns fights identically.
- **It is out of circulation** — a retired enemy is dropped from the random per-zone spawn tables, a retired challenge can no longer be completed by a player who has not already completed it (challenge evaluation skips it), a retired zone is non-navigable (the `StartBattle` zone-change anti-cheat refuses a move into one, and its boss cannot be challenged), and a retired **path** is frozen for XP accrual — it is excluded from the skill→contribution reverse index the proficiency cache builds, so no skill routes XP into it and the whole track accrues nothing, levels nothing, and grants no further skills (the path is the freeze unit, since a path's tiers progress as a contiguous frontier walk). The same cache also drops a retired path's proficiencies from the gated side of the prerequisite (gateway-opening) index, so a retired tier authored as a cross-path gateway is never opened when its live prerequisites max. A player whose current zone is no longer **viable** — retired, or left with no spawnable enemies — is lazily relocated to the nearest viable, unlocked zone on their next encounter, so the idle loop never stalls or throws on an empty spawn table. Viability combines the zone-retirement flag (`IZones.IsZoneRetired`) and spawn-table presence (`IEnemies.HasSpawnableEnemies`). Already-earned references are never revoked: a completed challenge keeps its reward, an authored reference still resolves by id, and a frozen track's already-accrued levels and granted skills are untouched (retirement only stops *further* accrual).
- **Authoring guards (backend-enforced, reported via `AdminSaveResult.Failure`).** Some retirements would author live content into a permanently broken state the runtime deliberately does not paper over — a live zone left with no spawnable enemy, or a live proficiency gateway whose prerequisite path is frozen — so they are **rejected at save time** as anti-tamper (a tampered admin client can't bypass a frontend check). The specific guards live in [backend-admin.md → Content-lifecycle retirement guards](./backend-admin.md#content-lifecycle-retirement-guards).
- **No hard delete** of a reference record: the admin "delete" affordance is Retire/Reinstate, and the reference admin repos reject a top-level delete outright.

**The Home sanctuary is the one deliberately-empty zone.** A zone flagged `IZones.IsHomeZone` (entity + contract only — the battle never reads it, resolved like `IsZoneRetired`) is a no-combat place the player idles in without fighting. It is exempt from the empty-zone relocation above: it never spawns enemies (`AdminZones`/`AdminEnemies` reject a boss or spawn-table membership on a Home zone) and the `StartBattle` zone-change anti-cheat refuses a move into it, so a player's persisted `CurrentZoneId` is never Home — and the offline-rewards path keeps replaying their last real combat zone unchanged. The sanctuary is **singular**: `AdminZones.SaveZones` rejects any save that would leave more than one non-retired Home zone (a second Home add or a combat zone flipped to Home), checking the whole post-save catalogue rather than the changed record. See [game-design.md](./game-design.md#home-zone--a-no-combat-sanctuary).

**Tags are the deliberate exception.** They carry their own (non-zero-based) identity and are resolved by lookup, not by index, so they never suffer the mis-resolution bug retirement exists to prevent. Their admin "delete" stays a real hard delete (graceful for an in-use tag — the join rows cascade and the caches reload so the removed tag id disappears on the next read).

### Reference-data read contracts

The read path never exposes EF entities:

- **Reference reads return read contracts** (`Game.Abstractions.Contracts`) — one published language shared by the game client's loading screen and the admin Workbench. The internal cache stays as entities (the domain mappers need their fidelity) and is projected to contracts, preserving the id-as-index ordering.
- **Gameplay reads return lean `Game.Core` domain models**, not entities, and hand back **shared, pre-materialized instances** by id rather than re-mapping a fresh graph per call (these reads are on the per-battle hot path). This is safe because the reference-data domain models are structurally immutable value objects — `init`-only properties and `IReadOnlyList<>` collections, enforced by the compiler and guarded by a reflection test — so a consumer cannot grow or reassign the shared graph and corrupt the cache for every player. The level-parameterized enemy read is the same idea adapted: a level-independent template is shared and cloned at the requested level.
- **Admin writes keep entities inside the data tier** — the entity-returning lookups live on `internal` data-tier interfaces consumed only by the Content Authoring admin repositories (see [backend-admin.md](./backend-admin.md#admin-tools-api-surface-content-authoring-context)).

### Reference-data cache reload (build-then-swap)

Cache busting is an **eager build-then-swap** (stale-while-revalidate), not a null-and-lazily-refill (see the [cache-reload spike](./spikes/356-reference-data-cache-reload.md)). Readers never observe an empty or torn snapshot, and never pay a refill query inline.

- A **singleton holder** per set owns the current immutable snapshot and exposes a lock-free read plus `ReloadAsync()`. A reload builds the whole new snapshot off to the side (on its own DbContext) and publishes it with a single atomic reference swap, so a **failed reload leaves the old snapshot in place**. Derived structures (e.g. the per-zone spawn tables built from the enemy list) are bundled into the snapshot so they swap atomically and a reader can never see a new list against stale derived data. A per-holder semaphore serializes reloads to preserve read-your-writes.
- **An admin write triggers an awaited reload** after the write commits (so the Workbench reads its own writes with zero gap) and **broadcasts a cross-instance invalidation over the Redis backplane**. Other instances debounce the notification and run one background reload sweep; the publishing instance skips its own message. A failed background sweep is retried with backoff and never disturbs readers. The local reload is bounded by a timeout so a wedged database connection can't hold the admin request open indefinitely.

> **Snapshot read-once idiom (repo reads).** Because the swap replaces the snapshot reference atomically, a reference repo must read `holder.Current` **once per logical operation** and read every field/derived structure off that captured snapshot — never re-read `holder.Current` mid-operation (e.g. via a property getter), or one call could mix an old and a new snapshot. The thin repo readers expose a single `Snapshot => holder.Current` accessor and bind it once; methods that need more than one read capture it into a local first (`var snapshot = holder.Current;`).

> **A recurring authoring-metadata pattern** runs through the reference fields below. A field the **battle never reads** lives on the **entity + contract only**, *not* the lean `Game.Core` battle model (the entity→core mapper ignores it; the entity→contract mapper and the admin editor carry it); a field the **battle does read** also lives on the lean Core model. Either way, adding a contract field changes that set's **version hash**, so clients re-download the set once, and existing rows **backfill** to a stated default. Each subsection below calls out only its own departures from this pattern (its default, where it lives, and whether the battle reads it).

### Skill acquisition flags (authoring intent vs. reference reality)

A skill carries an `ESkillAcquisition` `[Flags]` bitmask (`Player`/`Item`/`Enemy`/`Synthesis`) declaring which channels are *allowed* to grant it. The flag is **intent**; the references that actually grant a skill (an item's `GrantedSkillId`, a proficiency milestone reward, an enemy's skill pool, a synthesis recipe's result) are **reality**; backend authoring validation bridges the two as **anti-tamper** where a reference is authored (a frontend picker filter alone is insufficient): the enemy skill-set save rejects any skill that isn't `Enemy`-flagged, the item save rejects a `GrantedSkillId` whose target isn't `Item`-flagged, and the recipe save rejects a result skill that isn't `Synthesis`-flagged. `Player` denotes a milestone-grantable skill (**challenges no longer grant skills** — spike #982). The flag is entity + contract only (the battle never reads provenance); existing rows backfill to `Player`.

### Skill words of power (entity + contract)

A skill carries an Aetheric conlang **word of power** — `Word` (the romanization rendered as glyphs), `Pronunciation`, and `Translation` — the same decorative-flavour trio proficiencies carry, surfaced most prominently on the **Synthesis screen** as a synthesized skill's discovered identity. Like `IconPath`, it is display/authoring metadata the battle never reads, so it stays off the parity surface; existing rows backfill to `""` (the migration default).

### Skill recipes (synthesis reference data)

A `SkillRecipe` is authored, zero-based reference data (spike #1125) combining a set of owned input skills (under optional proficiency-level conditions) into one `Synthesis`-flagged result skill — synthesis is **non-consumptive and deterministic**, so a recipe carries no cost beyond possession + conditions. It is cached like the other reference sets (`SkillRecipesCacheHolder` → snapshot → version key), with inputs/conditions as child collections and a cache-built **reverse index** (input skill → the recipes it feeds) that drives the client's hinted reveal; retired recipes drop out of the index but stay resolvable by id. The recipe graph (result skill → its input skills) is validated **acyclic** both at admin-authoring time (`AdminSkillRecipes`) and as a build-time cache invariant (`SkillRecipeGraph`), since a cycle would make a skill its own ancestor (unreachable) — the same shape as the proficiency prerequisite check, sharing `DirectedGraphCycleDetector`. The same save also enforces **one skill = one producing (non-retired) recipe** as an authoring invariant: the synthesis graph keys revealed results by skill id, so two live recipes producing the same skill would collapse to one node, and the gameplay reveal/dossier model assumes a single producer.

The player forges a recipe through the **`SynthesizeSkill` socket command** (a player mutation, so it lives on the socket like the other player writes). Validation is **authoritative anti-cheat** — a tampered client controls only the recipe id, so `Player.TrySynthesizeSkill` (orchestrated by `SynthesisService`) re-checks every fact against the live recipe and player state: the recipe must be live (not retired), the player must own **every input as an *unlocked* skill** (innate item-granted skills live only in battle assembly, never in `Player.Skills`, so they're excluded — no equip-to-synthesize-then-unequip), and every proficiency-level condition must be met (a missing proficiency counts as level 0, read from the `PlayerProgress` aggregate like the gear gate). On success it `UnlockSkill`s the result (**idempotent** — re-synthesis is a no-op) and returns the result skill id in the command response; being player-initiated it needs no separate server push. The result is **not** re-validated as `Synthesis`-flagged at command time — that flag is authoring intent, guaranteed by the `AdminSkillRecipes` save, and the lean battle `Skill` deliberately carries no acquisition flag. Synthesis is non-consumptive, so inputs are never removed. The client delivery/UI are separate sub-issues.

### Skill rarity (entity + contract; display-only)

A skill carries an `ERarity` tier (the `Rarity` FK, inferred by convention like `Item.Rarity`). It is **display/authoring metadata only** — neither the battle simulation nor (as of the effect-based proficiency accrual, spike #1318) the XP path reads it: rarity reverts to display-only, since damage already reflects a skill's power and an explicit rarity weight would double-count. Like `IconPath` and the word of power, it is entity + contract only, not the lean battle model; existing rows backfill to `Common` (the migration default; the model carries no default so EF always writes the authored value on insert).

### Skill damage portions (weighted multi-type split)

A skill's direct hit carries a **weighted-portion collection** (`SkillDamagePortion { EDamageType Type, double Weight }`) rather than a single leaf `DamageType` (spike #1343, part A). It is an EF **child table keyed by `(SkillId, DamageType)`** — a skill carries at most one portion per leaf type — using the same child-table + admin child-saver pattern as a skill's `DamageMultipliers` (the shared upsert reconciler is `KeyedChangeSetProcessor`, keyed here by damage type). The collection lives on the **entity, contract, and the lean `Game.Core.Skills.Skill`** the battle reads (portions are a battle input, unlike rarity/word-of-power). Existing skills **backfill to a single `[{ Physical, 1.0 }]` portion** (pre-release, no grandfathering of the dropped single-type column). Weights are **raw and normalized at fire time** (`portionRaw = raw × weight ÷ Σweights`), so authoring one portion never forces rebalancing the others and an "even split" is just equal weights. A derived **`PrimaryDamageType`** (highest-weight portion, first-authored on a tie, `Physical` for an empty set) feeds the display surfaces — and, until the portion-aware battle pipeline (#1385) reads the split, the single-type direct-hit call — so the data is **inert** (battle behaviour unchanged: every skill is single-Physical-portion until multi-typed content is authored). The admin save enforces the portion invariant as anti-tamper (the editor's matching `warn` is advisory only): it **rejects a non-positive weight** and a change set that would leave a skill with **zero portions** — both break fire-time normalization (`Σweights == 0`, or a `weight ≤ 0` term). The frontend mirrors `PrimaryDamageType` in `$lib/battle/damage-types.ts` for parity.

### Item-granted skills (battle assembly)

An item carries an optional `GrantedSkillId` — a nullable FK to a skill it grants while equipped. It lives on the **entity + contract** and on the lean `Game.Core.Items.Item` (as an **id only**, keeping the item cache independent of the skill cache). Granted skills are **innate** — additive to and exempt from the selected-loadout cap. The battler's skill set is assembled in `BattleSnapshot.ToBattler` from the **already-captured `EquippedItems`** (no snapshot field added — granted skills are *derived*, like item attributes): the rule is **gather the granted ids from all active grant sources → de-duplicate by id (first wins) → order**, with player-selected skills first (their order) then the grants in `EEquipmentSlot` order (Helm→Accessory). The order/dedupe is the shared `BattleLoadout.OrderSkillIds` rule, mirrored tick-for-tick on the frontend for battle parity. Writing the gather as "all active sources" (not "iterate equipped items") makes a future set-bonus grant an additive source, not a rewrite. This is a deliberate player-favouring break in the player/enemy skill-count symmetry — enemies can't equip items (see [game-design.md](./game-design.md#skills)).

### Weapon type & the weapon-grants-a-signature invariant (authoring)

An item carries an optional `WeaponType` — an `EDamageType?` **constrained to a weapon leaf** (`Sword`/`Axe`/`Bow`/`Club`/`Dagger`/`Unarmed`), reusing `EDamageType` rather than a parallel enum (spike #1340 unified weapon type with weapon-leaf damage type). It lives on the **entity + contract** and the lean `Game.Core.Items.Item`. It is only meaningful on a `Weapon`-category item — the equipped weapon's type is what a weapon-typed skill matches against, resolving an empty slot as the virtual `Unarmed` fists (`weapon?.WeaponType ?? Unarmed`; the weapon-match gate itself is a later area). `DamageTypes.IsWeaponLeaf` / `WeaponLeaves` is the single source of truth for the classification, derived from the taxonomy table (mirrored on the frontend in `$lib/battle/damage-types.ts`).

The **no-stranding invariant** is enforced at admin save (`AdminItems.SaveItems`) as anti-tamper: a `Weapon` item must declare **both** a weapon-leaf `WeaponType` and a `GrantedSkillId` — every weapon brings a signature attack of its own type, which guarantees the player always fields ≥1 usable skill once the weapon-match gate dims mismatched selected skills. A non-weapon item may not carry a `WeaponType`. Intended tradeoff: no pure stat-stick weapons. The Workbench surfaces the same rule as a validation warning, but the backend is authoritative.

### Weapon-match gate + virtual-fists punch (battle assembly)

The fielded skill set is filtered by the equipped weapon at **battler assembly** (spike #1342), folded into the shared `BattleLoadout.OrderSkillIds` / `BattleSnapshot.GetBattleSkillIds` rule and mirrored tick-for-tick on the frontend (`battler.ts` `fillSkills` + `InventoryManager`). It is **authoritative on the backend** (the re-sim builds the battler from the snapshot) and **setup-time only — never on the deterministic tick loop**, so it adds no RNG-coupled parity surface; it still computes identically FE/BE over the same snapshot and is pinned by the parity suite. **No new snapshot field** — the equipped weapon type and its granted signature are derived from the already-resolved `EquippedItems`, exactly like `GrantedSkillId` resolution.

- **The gate.** The equipped weapon type is `weapon?.WeaponType ?? Unarmed` (the equipped `Weapon`-category item, else the virtual `Unarmed` "fists"). A skill whose type (`Skill.PrimaryDamageType`) is a **weapon leaf** (`DamageTypes.IsWeaponLeaf` — Sword/Axe/Bow/Club/Dagger/Unarmed) is fielded **iff** it matches the equipped weapon type (exact leaf-match); **weapon-agnostic** types (generic `Physical`, the elementals, the DoT leaves, caster types) are **never** filtered. Applied **uniformly to selected and granted** skills — a weapon's own signature matches by authoring, and the rule additionally covers the oddity of a non-weapon item that granted a weapon-typed skill (dormant unless its weapon is held).
- **Virtual-fists punch.** "Empty hands" is modelled as the virtual `Unarmed` fists whose signature is **punch** (`GameConstants.PunchSkillId`, a client-mirrored well-known seeded reference id, authored `Unarmed`-typed). The equipped weapon's granted signature is `weapon?.GrantedSkillId ?? PunchSkillId`, so punch is appended to the granted ids **only bare-handed** and the uniform gate fields it (Unarmed-typed, bare hands read Unarmed) — the no-stranding guarantee even when every selected skill is dimmed. A real `Unarmed` weapon (e.g. brass knuckles) **replaces** the fists and fields *its own* signature instead — no free bonus skill, symmetric with how a sword fields the sword's signature. The skill resolver the gate reads types through is **nullable** (`ISkills.TryGetSkill`): an id with no resolvable skill — an unauthored punch — yields a null type and is **dropped**, so the gate degrades gracefully rather than fielding a phantom.
- **Non-destructive.** The saved selected-skills loadout is untouched and `Player.TrySetSelectedSkills` is unchanged (still cap / no-dup / unlocked-only). Mismatched selected skills are simply **dormant** while the weapon mismatches and light back up when a matching weapon returns; the gate is contextual, not a save-time rule.

### Gear gated by proficiency (equip anti-cheat)

An item carries an optional `RequiredProficiencyId` + `RequiredProficiencyLevel` gate (a navigation-less FK like the zone `UnlockChallengeId`, paired with a level, like the zone `BossEnemyId`/`BossLevel`). It lives on the **entity + contract** and the lean `Game.Core.Items.Item`; the pure rule is `Item.MeetsProficiencyRequirement(levels)` (ungated → always met; gated → the player's level in that proficiency must be ≥ the required level, a missing entry counting as level 0). It is **enforced at equip time** — `Inventory.TryEquipItem` rejects a gated item the player hasn't qualified for, and `PlayerService.EquipItem` supplies the player's proficiency levels (read from the separate `PlayerProgress` aggregate, since proficiency levels live there, not on the battle player). This is **anti-cheat**: a tampered client can't bypass the frontend gate. Because proficiency level is monotonic the gate only matters at equip; **starter equipment** (persisted equipped at creation, not via `TryEquipItem`) bypasses it, as authoring is trusted. Admin save validates the gate references a non-retired proficiency at a level within its range. The frontend mirrors the rule (`meetsItemProficiencyRequirement` in `item-requirements.ts`) only to disable the affordance and surface the requirement; the backend is authoritative.

## Player-facing read projections

Player-facing reads also avoid exposing EF entities, but — unlike reference data — they reuse the existing **gameplay domain models** rather than dedicated contracts (those models are already lean enough), with the API DTOs projecting from them.

- **Statistics, challenge progress, and proficiency progress** (level + XP per proficiency) go through the single `IPlayerProgressRepository`, which also owns the progress write path, giving one source of truth for player-progress data. Each is a `(PlayerId, …)`-keyed row folded into the `PlayerProgress` aggregate with its own dirty set, persisted as one batched, absolute-value `ProgressUpdated` write-behind event whose handler upserts each kind idempotently (a re-applied event converges). A statistic's "no data yet" state is the **absence of its row**, never a magic `0` value — a stored value (0 included) is always a genuine recording. Min/max aggregation and the "at most" challenge path therefore key off row presence (`TryGetStatisticValue`), so a legitimate `0` minimum (e.g. an instant `FastestVictory`) records and counts correctly instead of being treated as empty. Challenge evaluation on battle completion is scoped to the challenges whose tracked statistic actually changed that battle — via a reverse index (statistic → challenges) bundled with the cached challenge snapshot — rather than re-scanning the whole authored catalog on every battle (a per-player hot path).
- **The skill loadout is sent as the full unlocked set** (each skill with its `Selected`/`Order`), mirroring how inventory is sent as the full unlocked item set; the client derives the ordered equipped loadout from it. The loadout cap is a generated client constant, not a per-player wire field.
- **The loadout is edited through one atomic socket command** that replaces the whole equipped set in order (handling select, deselect, and reorder in one path). Like the other player mutations it validates as **anti-cheat** — no duplicate ids, count within the cap, every id already unlocked — and rejects with no state mutation on any violation.

## HTTP vs WebSocket Communication

**The goal: once the session is live, every game-client read and player-state mutation goes over the socket — only the auth flow stays on HTTP.** This is what lets the player's state be held in-memory on the backend and updated in real time without re-synchronizing against Redis or the database on each request. Users connect to one instance at a time, and a player's socket commands are handled sequentially. The goal has been reached; the HTTP that remains is deliberately scoped to:

- the **auth flow** (login, account creation, refresh, logout, status, active-session, device-info) — it runs before a socket exists, and the active-session check must deliberately *not* open one;
- admin **persistence** endpoints; and
- the admin-only reads that have no socket equivalent yet (tags, user/role lookups).

**Player-state mutations belong on the socket because the socket serializes them with the battle loop.** Player writes are read-modify-write against the in-memory aggregate; the socket read loop processes one command per player at a time, so a write is naturally serialized with the idle battle-completion commands that mutate the same cached player. The same write over HTTP would run concurrently on a separate thread and could silently clobber a background battle save. Every player-mutating action is therefore a socket command and validates as anti-cheat (rejecting with no mutation) — e.g. a stat allocation only accepts the core attributes that actually have an allocation row, which blocks allocating into derived attributes.

**The per-player sequential guarantee is enforced by a per-socket command lock**, because server-initiated pushes (challenge-completion, socket-replaced) arrive over the Redis backplane and run *off* the read loop — so two paths could otherwise mutate the same cached player concurrently, the exact lost-update class the serialization exists to prevent. Each command also runs under a per-command timeout, and that cancellation token is plumbed down through the application services and the data tier (EF/Npgsql honour it natively; the Redis client is wrapped to honour it cooperatively) so a wedged command unwinds promptly and releases the lock rather than stalling every later command. A separate per-socket send lock guards WebSocket output, since overlapping sends on one socket are illegal.

## Single active connection (session takeover warning)

A player may only be connected to one instance at a time: the socket manager records the player's current socket id in Redis and, when a new socket replaces an existing one, signals the old socket to close. To let the client warn the user **before** that takeover happens, the authenticated `Login/ActiveSession` endpoint reports whether the player already has a live socket — deliberately over **HTTP, not a socket command**, because opening a socket to ask the question would itself trigger the replacement it is meant to warn about. Because the check is presence-based it covers "another tab" and "another device" uniformly.

The presence key carries a short TTL refreshed on every inbound socket message (the client heartbeats every 10s), so a live connection stays fresh while a connection that vanished without a clean close expires within the TTL. The key is claimed atomically (set-with-TTL in one operation) and rolled back if the post-claim setup fails, so a partial registration can't leave a ghost session that blocks the player.

## Graceful socket drain on shutdown

A stopping instance drains its live sockets cleanly rather than leaving them to be force-killed — the missing lifecycle piece for an app designed to run as multiple scalable instances (rolling deploys, scale-down). A singleton registry tracks every live socket and uses its shutdown hook to drain them in two phases: first stop the per-socket inactivity watchdogs (teardown now owns the close), then send each socket a normal-closure close frame and await the loops within a bounded timeout, aborting any still-blocked receive past the deadline so the loop unwinds rather than waiting out the host timeout. The client treats the normal closure as a cue to reconnect — which a rolling deploy routes to a healthy instance — so no client change was needed. The "draining has started" gate and the snapshot are flipped together under one lock, so a handshake that completes after the drain begins is closed on the spot with the same reason instead of slipping past the snapshot and lingering until the process is killed.

## Reference-data caching and versioning

The frontend caches each reference-data set in the browser and re-downloads only the sets that changed (see `frontend.md`). The `GetReferenceDataVersions` socket command supports this by returning a **content hash per set** in one round-trip. The hash is computed by hashing the same serialized models the client actually receives (SHA-256 over the camelCase JSON), so the version changes if and only if the client-visible data changes — there is no separate version counter to maintain or invalidate. The versioned set is exactly the set of reference-data socket commands (discovered by assembly scan), so a newly-added one is versioned automatically.

`GetReferenceDataVersions` is the **first** command the loading screen issues on every connect, so the hash is **memoized per set, keyed on the immutable cache snapshot it was computed from** — the serialize-and-hash runs once per cache swap rather than once per connect. Keying on the snapshot instance makes invalidation automatic: a build-then-swap publishes a new snapshot object (a memo miss), so an admin write can never serve a stale hash. Intrinsic (enum-derived) sets have no holder to swap, so they key on a process-stable sentinel and hash once for the process lifetime.

## Battle (setup, runtime & skill effects)

Setting up and running a battle is a domain decision, not an orchestration one: the application layer only snapshots the player and the chosen enemy loadout, sets the active battle, and delegates the game-logic choices to `Game.Core`. **Parity invariant:** battle logic runs on both the frontend and backend and the results must be identical — the backend replays the client's reported battle as anti-cheat — so these mechanics are mirrored on both sides and pinned by a parity matrix; they must never diverge. The tick-by-tick detail — encounter setup, snapshot reconstruction, zone-unlock anti-cheat, the RNG seed, and the skill-effect/DoT-HoT runtime — lives in [backend-battle.md](./backend-battle.md).

Two distinct constant classes back this: **`GameConstants`** holds values that must stay identical on both sides (tick size, loadout cap, …) and is `[ClientMirrored]` so codegen emits them to the TypeScript client, while **`ServerGameConstants`** holds server-only anti-cheat/reward clamps the client never evaluates (e.g. the exp-reward caps) and is deliberately not mirrored. Put a constant in the right one — a client-mirrored value referenced as a server-only one (or vice versa) is a recurring footgun.

### Damage types (taxonomy foundation)

Damage carries a discriminator (spike #1320). A skill/DoT instance has exactly one leaf `EDamageType` (physical, the four elements, the bleed/poison/burn DoT forms, and the **weapon leaves** below); per-type and per-category **amplification** (attacker) and **resistance** (defender) ride the normal attribute-modifier pipeline as `EAttribute` percentage values (decimal-percentage convention, base `0`), so any modifier source — proficiency, item, mod, skill-effect, class passive — can grant a typed buff with no new plumbing. The single piece of machinery is the lookup indirection in `Game.Core.Attributes.DamageTypes`: a static `Applies(type) → keys` map resolves a leaf type to the amp/resist **keys** (the leaf type itself plus cross-cutting `Elemental`/`DoT` categories) whose attributes apply to a hit, in a **fixed iteration order parity depends on**. That map and the key→attribute pairing **codegen to the client** (`$lib/battle/damage-types.ts`) like `StaticAttributeModifiers`, so the two simulators cannot drift. The **foundation is inert** — the amp/resist attributes are unread until the damage pipeline (later areas) consumes them, mirroring the #178 crit/dodge/block foundation.

**Weapon-type leaves (spike #1340).** Martial masteries (Swordsmanship, Axe mastery, …) are ordinary type-routed leaves — `Sword`/`Axe`/`Bow`/`Club`/`Dagger`/`Unarmed` — sitting under the `Physical` **category key** (`Applies()` = `[<weapon>, Physical]`), parallel to how the elements sit under `Elemental`. They are **statically per-skill typed** (a martial skill is authored with a fixed weapon type, not inherited from the equipped weapon) and purely **additive** on the shipped #1320 foundation (no migration beyond seeding the new amplification attributes). The one asymmetry with the elemental keys: a weapon key is **amplification-only** — there is no per-weapon resistance, so `DamageTypes.KeyInfo.Resistance` is nullable and a weapon hit resolves to `PhysicalResistance` via the shared `Physical` key (physical resistance stays the only physical mitigation lever; per-weapon resistance is a deferred opt-in). The enums grow **append-only**, so a weapon leaf no longer shares a numeric value with its key (the `EDamageType.Fire === EDamageTypeKey.Fire` coincidence holds only for the original #1320 leaves); a leaf type's own key is recovered as `Applies(type)[0]` (always first), which both simulators rely on. `Unarmed` is the no-weapon default. Offense proficiency routing reaches the weapon paths through the same `Applies → ActivityKeys.ForDamageKey` fold (a sword hit trains the weapon path **and** the `Physical`/Martial path).

## Caching, Pub/Sub & write-behind persistence

Most reference data is cached in-memory (see [Reference Data](#reference-data)), but **player data uses a write-behind cache where Redis is the source of truth**: a live player's aggregate is loaded once onto the connection-scoped session, mutated in-memory per socket command, and persisted to Postgres asynchronously through a Redis queue drained by a background service. Player progress (statistics + challenges) is a second such write-behind aggregate under its own key, and Redis pub/sub doubles as the WebSocket backplane and the carrier for cross-instance reference-data invalidations. The deep mechanics — the sliding TTLs, the domain-event → batched-publish path, dead-lettering and retry, drain serialization, and worker ownership — live in [backend-persistence.md](./backend-persistence.md).

On the `ICacheService` cache contract every value-accepting setter carries an expiry, and they fall into two camps. `Set` and `SetAndForget` take a **nullable** value where a **null value deletes the key** — the de-facto null-means-delete behaviour their generic `<T>` overloads rely on. `GetSet` (the atomic set-with-expiry) is **non-null**: writing a TTL implies writing a value, so a null would error its server-side Lua `SET` — it has no delete path to expose.

## Challenge-completion notifications (server push)

Completing a challenge unlocks its rewards as a side effect of battle-completion handling, decoupled from the socket command that triggered the battle — so the command's response can't carry the unlock, and it would otherwise surface only on the next page load (the client doesn't evaluate challenges itself; the server is authoritative). Instead the player aggregate raises a challenge-completed event and an API-layer handler pushes a server-initiated socket command to the player over the Redis backplane, so it reaches them on whichever instance holds their connection and covers **every** completion path (idle victory, won-abandon, even a loss that completes a counting challenge). Being server-initiated, the push carries a typed response but no request parameters, so codegen emits no client request type.

A server push that **faults** is escalated rather than silently dropped: the poisoned payload is dead-lettered (preserved on a holding queue for inspection, mirroring the player write-behind dead-letter path) and the affected socket is sent a one-way `ServerCommandFailed` notice carrying the failed command's name, so the client re-syncs the authoritative state the push would have updated (a failed `ChallengeCompleted` force-reloads challenge progress, which gates zone navigation) instead of silently diverging. Faults are **not retried**: the only server pushes have no transient failure mode (a poisoned payload can't be fixed by a retry) and re-executing across a released-lock window isn't generally safe. Faults are classified inside the command executor, which keeps a per-command **timeout** (`"Command timed out."`) distinct from a **lifetime/teardown cancellation** (a normal unwind, never an `"Internal Server Error"`); a server push reports its outcome so the pub/sub processor owns the dead-letter-plus-notify escalation.

## Proficiency XP accrual (battle-completion hook)

Proficiency XP is computed **server-side at battle completion, off the tick loop** — like statistics and challenge rewards, never inside the deterministic 40 ms simulation. This is the deliberate boundary that keeps it off the frontend/backend battle-parity surface: the simulation produces the `BattleStats` (the typed **output book** — `TypedDamageDealt` — and **incoming book** — `TypedDamageExposure`) and the player's power is the same `DefeatRewards` measure already computed server-side; the accrual reads those after the fact. Only the eventual proficiency *attribute bonus* is parity-sensitive (both sides compute it from player state at snapshot time), not the XP. The player-facing mechanic — the per-path power-normalized claim, two-book routing, and frontier walk — is owned by [game-design.md](./game-design.md#proficiency-xp--effect-based-power-normalized-accrual); this section is the server-side execution path only.

The accrual lives in one place — `ProficiencyRewardService` (mirroring `ChallengeRewardService`) — and **both** completion paths call it, so live and offline accrue identically: the live path runs it in `BattleStatisticsEventHandler` (reacting to `BattleCompletedEvent`, whose `PlayerPower` the victory path threads on), and the offline consolidation runs it per won battle in `BattleService`. It folds the two books onto each path's `ActivityKey` (`ActivityKeys.ForDamageKey` / `ForDamageKeyResist`) and has each path claim its XP via `ProficiencyXpCalculator` against its **frontier tier** (`IProficiencies.GetPath(id).Frontier`, read off the player's current proficiency levels). The `ActivityKey → paths` routing is a reverse index the proficiency cache builds, with **retired paths excluded** so a retired track is frozen for accrual. Because the frontier is re-resolved from progress each battle, the offline batch — replaying the same loadout over the away window — evolves identically to live. It writes through the same `PlayerProgress` aggregate / write-behind path as statistics and challenges (absolute values, so a re-applied event converges). The live path raises a `ProficiencyXpGainedEvent` that an API-layer notifier pushes to the client exactly like the challenge-completion push; the offline path suppresses the push and instead **folds each won battle's accrual across the away window** (`ProficiencyGainAccumulator` — XP summed per proficiency, level/residual XP final, milestones/skills/opened nodes unioned) onto the **welcome-back summary**, which carries the gains and opened nodes (reusing the live push's models so the gate renders offline and live the same way) and counts them in `HasProgress`.

### Milestone & open effects (derived openness, persisted only as skill grants)

A proficiency's **openness is derived, never stored**: a node is open iff it has accrued XP, has its previous in-path tier maxed, or has all its cross-path prerequisites maxed. There is no per-player "opened" flag and no migration — the same level + structure derivation the client uses (spike #982 decision 14), so XP routing needs no open-gate (a path opens the moment its activity-keyed effect trains it). The **only** persisted effect of the milestone/open layer is therefore the idempotent **skill grant** via `Player.UnlockSkill`, applied inside `ProficiencyRewardService` right where a tier crosses a milestone: a crossed milestone grants its `RewardSkillId`. Maxing a tier also **opens** its in-path successor (`Path.NextTier`) and any cross-path gateway whose prerequisites are now all maxed (`IProficiencies.DependentsOf`), but opening is **notification-only** — it grants no skill (the seed-skill machinery was removed in spike #1125; a freshly-opened tier's native skill is re-homed onto the predecessor tier's max-level milestone reward, or bootstrapped via synthesis for a cross-path/synthesized line). Because the milestone grant is idempotent and fires only on the level/max transition, the offline batch re-running the accrual per battle never double-grants. A character's starting roots are **emergent from its class kit** (spike #1126): the kit's skills open their paths on the first won battle through this same derived-openness rule, so there is no `StartsUnlocked` flag or root-seeding step at creation. The prerequisite graph is validated **acyclic** both at admin-authoring time (`AdminProficiencies.SetPrerequisites`) and as a build-time cache invariant (`ProficiencyPrerequisiteGraph`), since a cycle would soft-lock every node on it. The live push (`ProficiencyXpGainedEvent`) carries the granted skills and opened nodes alongside the XP; the offline path suppresses it.

## Attribute composition (battle-assembly parity)

The proficiency *attribute bonus* is one of three frontend↔backend **attribute-composition parity surfaces** assembled into the battler from player state at snapshot time. Each is composed into the modifier pipeline at a **fixed position** (float addition isn't associative, so order is part of the contract) and pinned **bit-exactly** by a mirrored parity test, since the backend replays the client's battle as anti-cheat:

- **Proficiency attribute bonus** — authored per-level/milestone payouts (`EAttributeModifierSource.Proficiency`), composed with the base set (allocations + gear) *before* the statics; **frozen into `BattleSnapshot`** at battle start.
- **Class locked base** (spike #1126 area D) — the level-scaled, non-reallocatable half of the hybrid attribute model (`BaseAmount + AmountPerLevel × level`, `EAttributeModifierSource.AttributeDistribution`), composed after gear and before the proficiency bonuses; the free pool is the reduced per-level stat-point grant.
- **Class signature passive** (spike #1126 area E) — the flat or attribute-scaled combat-identity bonus (`EAttributeModifierSource.Class`), composed **last**, after the statics, so a scaled passive reads fully-resolved attribute values.

The locked base and signature passive also feed the `DefeatRewards` power measurement. The composition-order contract, the `double`-arithmetic rule that keeps the decimal-authored values bit-identical to the frontend's IEEE-754 recompute, the live-frontend wiring, and the named parity tests live in [backend-attributes.md](./backend-attributes.md).

## Authentication

JWT bearer auth, not server sessions. Access tokens are signed (HMAC-SHA256), short-lived (15 min), and carry the user id + role claims; refresh tokens are opaque, long-lived (48h), single-use, and rotate on every refresh. Refresh tokens live server-side in Redis as a hash — chosen because auth already depends on Redis, so it adds no new durability assumption.

**The selected player is a token claim.** An account can own multiple characters, so which one a request acts as is carried as a `player_id` claim on the access token (and on the refresh-token record, so it survives a refresh) — set once a character is chosen, absent on a pre-selection token. Both the socket handshake and the session-rehydration path read the bound player from this claim rather than defaulting to the account's first player, so an evicted/uncached session re-binds to the *selected* character. Login no longer binds a player; see [Account/login orchestration](./backend-auth.md#accountlogin-orchestration).

The **token is the sole authorization source of truth** (roles are read off the validated principal, never the session cache; likewise the selected player), so a role change only takes effect on next login. A global fallback policy requires every endpoint to be authenticated unless explicitly marked anonymous. WebSockets authenticate via an `access_token` query-string parameter, since browsers can't set headers on the WebSocket handshake. The frontend owns token persistence, pre-emptive refresh, and the single-flight refresh path (see `frontend.md`).

**The validated token is likewise the sole authority for whether a caller is *authenticated* — never the gameplay session cache.** That cache (`Session_{userId}` → the in-flight `PlayerState`) is a volatile presentation convenience holding the in-flight battle state and the user→player binding; it can be evicted (Redis flush, sliding-TTL lapse) or simply never established on a given instance. A valid token with no cached session is therefore *authenticated-but-uncached*, not anonymous: the session is rehydrated in memory by binding the player named in the token's selected-player claim instead of reporting "not logged in" (a pre-selection token with no such claim is authenticated but has no player, surfaced as a graceful error). The claim is authoritative — a cached session bound to a different character than the claim names is re-bound to the claim's player. `SessionService.Authenticated` is gated only on the token-derived `UserId`; a cache hit/miss governs `HasPlayerSession`, never authentication. This rehydration is **in-memory only** — it re-binds the request's `PlayerState` but does **not** write the session cache, since that write would land on the concurrent HTTP thread and player-state writes belong on the socket (above). The cache is (re)populated only by login (which establishes the binding before any socket exists) and by the socket's write-behind path.

`SessionLoaderMiddleware` records only the token-derived `UserId` on every authenticated request; the `GetSession` read (and the rehydration on a miss) is deferred to the consumers that actually need player state — the socket handshake and the `Status`/`ActiveSession` auth endpoints — via `SessionInitializer`. So the rest of the authenticated HTTP surface (admin tooling, refresh, device-info) pays no per-request session-cache read.

**Passwords** use **PBKDF2-HMAC-SHA256** behind an `IPasswordHasher` abstraction (tunable work factor, per-hash salt, app-wide pepper), with constant-time verify and equivalent dummy work on an unknown username so timing never reveals whether an account exists. **Login is split around player selection** — `Login` verifies credentials and returns a pre-selection token plus the account's character summaries, then an authenticated `SelectPlayer` binds the chosen character and rotates the token to carry the `player_id` claim; an in-game `SwitchPlayer` changes the live character losslessly, and characters are created post-signup under a per-account cap. The anonymous auth endpoints are protected by a **per-IP rate limiter** and, layered on top, a **per-account exponential backoff** that is deliberately a bounded delay, **never a hard lockout**. The orchestration, hashing, and abuse-mitigation mechanics live in [backend-auth.md](./backend-auth.md).

## CORS allowed origins (deployment config)

The browser CORS policy's allowed origins are **configuration-bound, not hardcoded** — they are deployment-specific. The list supports multiple origins and is validated at startup (must be non-empty), so a misconfigured environment fails fast rather than silently rejecting every browser request. The local dev origin ships in the Development config.

## Unhandled exception handling

A centralized exception-handling middleware is the **single place** unhandled exceptions are logged and turned into a response. It converts any exception escaping the downstream pipeline into the project's consistent `ApiResponse` error envelope with a 500 status, so clients get the same `{ errorMessage }` shape they get for business failures rather than the host's default response. Internal details (the exception message) are surfaced **only in Development**; other environments get a generic message so nothing leaks. A client-disconnect cancellation is not treated as a server error, and an already-started response (e.g. an upgraded WebSocket) is left to unwind rather than corrupted. It runs **inside `RequestLoggingMiddleware`** so the 500 is set before the "Request Ended" event logs the status; `RequestLoggingMiddleware` therefore stays focused on structured start/end events and never logs the exception itself.

## Access Roles and Admin Authorization

Authenticated users are gated out of admin tooling unless they hold the `Admin` role. **Roles live in the signed auth token, not the session store** — the token is the source of truth for what a request is authorized to do, which keeps the check self-contained and avoids a second lookup; the trade-off is that a role change only takes effect on the user's next login. The admin authorization filter reads the role straight off the cryptographically validated principal, **never** off the gameplay session cache: that cache is a presentation concern decoupled from the token's validity, so keying off it would wrongly reject a valid admin whose session key happens to be absent (evicted, never established, or aged out).

## Admin tooling & user administration

Reference-data administration is modelled as a distinct **Content Authoring** bounded context (thin admin controllers → admin repositories that own persistence; the EF entity never surfaces in `Game.Api`), and user/role management is a separate **Identity / User Admin** context that shares the `User` aggregate with the auth flow. Both speak `Game.Abstractions.Contracts` types rather than entities, and every admin write reloads the in-memory reference caches. Users are archived (a soft delete that frees the username) or banned (blocked but name reserved). The API surface, the change-set/reconciler persistence patterns and their zero-based-id gotchas, the archive-vs-ban model, and the connection-tracking tables live in [backend-admin.md](./backend-admin.md).
