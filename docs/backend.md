# File Structure and Projects

The game is structured is similar to an onion architecture with multiple projects. The backend projects (all in folders of the same name) are:

- `Game.Abstractions`: This project contains shared abstractions and interfaces used across the backend, such as repository interfaces, infrastructure-service interfaces, and the read/write contracts (`Game.Abstractions.Contracts`). The EF entity models are **not** here — they live in `Game.Infrastructure` (see _Entity model location_ below).
- `Game.Api`: This project contains the ASP.NET Core Web API controllers, models, and related code for handling HTTP requests from the frontend, as well as WebSocket command handlers for real-time communication. It also includes a reflection-based code generator for automatically generating API client interfaces for the frontend based on the API controllers and WebSocket handlers.
- `Game.Api.Tests`: This project contains unit and integration tests for the `Game.Api` project, ensuring that all API endpoints and WebSocket handlers function correctly and that the battle logic is consistent with the frontend implementation.
- `Game.Application`: This project contains the application layer of the backend, including services for orchestrating game logic.
- `Game.Application.Tests`: This project contains unit and integration tests for the application layer, ensuring that all orchestration logic is correctly implemented and consistent with the frontend.
- `Game.Core`: This project contains the core domain logic of the game, including domain objects/services and the battle simulation logic. This is where the main game mechanics are implemented.
- `Game.Core.Tests`: This project contains unit tests for the core domain logic, ensuring that all game mechanics, including battle simulation, character progression, and item logic, are correctly implemented and consistent with the frontend.
- `Game.DataAccess`: This project contains the data access layer, including implementations of repositories from `Game.Abstractions` that interact with the database using Entity Framework and the cache and pub/sub implementations provided by `Game.Infrastructure`.
- `Game.Infrastructure`: This project contains the implementation of the cache and pub/sub interfaces defined in `Game.Abstractions` (currently using Redis), as well as the database context, the **EF entity models** (`Game.Infrastructure.Entities`), and migrations for EF Core.
- `Game.Infrastructure.Tests`: This project contains classical unit tests for the in-process logic in `Game.Infrastructure` that has no out-of-process dependency (currently the `BackgroundWorker` Start/Kill lifecycle). It is deliberately lightweight — it references only `Game.Infrastructure`, not `Game.TestInfrastructure` — because, per the testing guidelines below, anything coupled to Redis or the database is covered by integration tests instead of here.

# General Backend Guidelines

- Repositories should create and persist domain objects and are responsible for dispatching domain events before persisting changes. They should not contain any domain logic nor should they directly return entity models or database DTOs. Mappers should be used to convert between domain objects, entity models, and database DTOs and placed in the Mapping folder in the root of the `Game.DataAccess` project.
- All domain logic should be implemented in the `Game.Core` project, and the application layer should only be responsible for orchestrating calls to the repositories and domain objects/services to fulfill the use case. The API layer should only be responsible for handling HTTP/websocket requests, validating input, and returning responses, and should not contain any domain or orchestration logic.

## Testing Guidelines

Unit tests should be written according to the classical (Detroit) school of testing as much as possible. This project does not have any unmanaged dependencies, so you should be avoiding test doubles as much as possible in both unit and integration tests. If a service under test involves an out-of-process dependency such as the database or cache, it should only need to be tested through integration tests. Any classes that depend on out-of-process dependencies should NOT contain any logic worth unit testing. If you find that a service contains domain logic and depends on an out-of-process dependency, you should refactor the code to move said logic into an appropriate domain class that can be unit tested without the dependency, and then write integration tests only to validate the interaction with the out-of-process dependency.

When deciding whether to cover a branch, evaluate per-branch whether a worthwhile test exists rather than chasing the coverage number. The Redis primitives in `Game.Infrastructure` (`RedisMultiplexerFactory`, `RedisPubSubService`, `RedisQueue`) are the canonical example (#387, #399): their genuine logic is pinned by integration tests against the DI-resolved interface (`RedisQueueIntegrationTests`, `RedisPubSubServiceIntegrationTests` — typed queue round-trips, FIFO order, duplicate-handle-id rejection, and the `UnSubscribe` remove/unknown branches), while the residual uncovered branches are accepted as **measure-only**: the `RedisMultiplexerFactory` null-connection-string guards (defensive — config always supplies one) and the cache↔pub/sub multiplexer reuse-vs-connect branches (incidentally exercised at startup; provoking them deterministically would require disposing the process-wide shared multiplexer mid-suite, which is riskier than the branch is worth).

# Important Architectural Design Decisions

## Entity model location and isolation

The EF entity models are a persistence implementation detail of the data tier, **invisible to `Game.Api` and `Game.Application`**. This was completed in the entity-relocation capstone (#138, the last of the spike-#130 series; see `docs/spikes/130-hide-entity-models.md`).

- **Entities live in `Game.Infrastructure.Entities`,** next to `GameContext` — not in `Game.Abstractions`. The supporting `IZeroBasedIdentityEntity` marker and `NotLoadedException` moved with them (into `Game.Infrastructure`); the unused `IAuditableEntity` marker and the vestigial `PlayerState` entity (#142) were deleted as dead code.
- **Hiding mechanism — no `internal` needed.** `Game.DataAccess` references `Game.Infrastructure` with `PrivateAssets="Compile"`, so Infrastructure's compile-time types do **not** flow transitively to `Game.Api` (which references `Game.DataAccess`), and `Game.Application` never references `Game.Infrastructure` at all. The entities themselves stay `public` (EF and the test infrastructure need that) but are simply unreachable from the upper layers.
- **The data tier's own entity seam is internal.** `IEntityStore` (the low-level insert/update/delete tracker) is `internal` to `Game.DataAccess`, and the entity-returning reference lookups are exposed only on internal data-tier interfaces (`IItemEntityCache` etc. — see [Reference-data read contracts](#reference-data-read-contracts)). The public `Game.Abstractions` repo interfaces speak only domain models and contracts.
- **Architecture test (regression guard).** `Game.Api.Tests/Architecture/EntityIsolationTests` asserts that neither the `Game.Api` nor the `Game.Application` assembly references `Game.Infrastructure`. Because every entity type lives in `Game.Infrastructure`, "no reference to that assembly" is equivalent to "no dependency on the entity model", and it fails fast if the `PrivateAssets`/reference wiring ever regresses.

## Reference Data

Reference data in this app can be split into 2 categories: intrinsic and static. Intrinsic reference data is fundamentally encoded into the application and often has an associated enum or similar construct in the code (e.g. `EAttribute` or `EItemCategory`). These are persisted into the database for data integrity, but they do not need to be (and very rarely are) queried from the database. Static reference data, on the other hand, is not fundamentally encoded into the application but is typically cached in-memory as a `List` such that the Id of the entity is its index. IMPORTANT: The Id column for these tables is seeded to start at 0 to facilitate this indexing. This includes things like items, item modifications, skills, enemies, zones, and challenges. This data is stored in the database and queried on application startup to be cached in-memory for fast access. Since this data is static and does not change often, it is not a problem to cache it in-memory, and it allows for much faster access than querying the database every time it is needed.

**Caches are immutable snapshots loaded eagerly and replaced by build-then-swap reload (#356 → #357, #358).** A **singleton snapshot holder** per set (`ReferenceCacheHolder<TSnapshot>` and its six subclasses in `Game.DataAccess/Repositories/Caching`) owns the immutable cached snapshot; the scoped reference repos are thin readers over `holder.Current`. `Startup` awaits `IServiceProvider.InitializeReferenceCachesAsync()` (after migrations, before the host serves traffic) to load every holder up front, so a database problem at boot surfaces as a startup failure rather than on the first player request, and there is no lazy-fill path — reads of an unloaded holder throw rather than refilling inline. See [Reference-data cache reload (build-then-swap)](#reference-data-cache-reload-build-then-swap) for the reload mechanics.

> **The Id-as-index optimization requires contiguous ids — guaranteed by retirement, not convention.** Both tiers look these sets up *by array index* (`Items.GetItem`/`LookupItem` index the `OrderBy(Id)` cache directly; the frontend does `staticData.items[id]`). That is only correct while the data stays append-only, since the index equals the Id only when there are no gaps. A **hard delete** of a non-terminal record would open a permanent Id gap that silently mis-resolves every lookup above it (and orphan persisted player-state ids and battle snapshots that reference it). To make this structurally impossible, top-level reference records are **retired, not deleted** (#289 — see [Retiring reference data](#retiring-reference-data-content-lifecycle)).

### Retiring reference data (content lifecycle)

A top-level reference record (item, item mod, skill, enemy, zone, challenge) carries a nullable `RetiredAt` timestamp — the same flag-plus-audit shape as the User `ArchivedAt`/`BannedAt` soft-delete. Retiring takes a record *out of circulation* without erasing it; reinstating clears the timestamp. The guarantees:

- **The slot is permanent.** A retired record is never removed or renumbered, so the cache stays gap-free and every index lookup keeps resolving the right record — the contiguity invariant is enforced structurally rather than relying on "only delete the highest Id".
- **It stays resolvable by id.** `GetItem`/`LookupItem`/`GetEnemy`/… still return a retired record, so existing references — a player's equipped item or applied mods, a battle snapshot, an authored boss/reward/unlock-gate FK — keep working. Retirement is a catalogue/authoring concern only; it is **not** carried on the lean `Game.Core` battle models, so a retired item a player owns fights identically.
- **It is out of circulation.** The current concrete rule: a retired enemy is excluded from the random per-zone spawn tables (`EnemiesCacheHolder.BuildZoneEnemyTables`) so it no longer rolls as an idle encounter. (There is no gameplay path that picks randomly from *all* reference data — items/mods/skills are authored challenge rewards, enemies/bosses are authored per-zone — so the circulation surface is small.)
- **No hard delete of a reference record.** The admin "delete" affordance is replaced by Retire/Reinstate (an ordinary `RetiredAt` edit through `AddEdit*`). The six reference admin repos pass no `delete` handler to `ChangeSetProcessor`, so a stray top-level `Delete` change throws instead of corrupting data. Child collections (attributes, mod slots, spawn rows, …) are still added/edited/deleted normally.

`RetiredAt` is on the read contracts so the client and Workbench can show the state; it changes the reference-data version hash when toggled (an expected re-download). The Workbench's authoring pickers exclude retired records from *new* authoring while keeping an already-referenced retired value visible (#295 — see `frontend.md` → the Workbench section). Still deferred (#296): per-entity edge cases (idling in a retired zone, an in-progress challenge whose target/reward is retired, retiring a zone/challenge used as an unlock gate) and referenced-by warnings on retire.

**Tags deliberately keep the hard-delete lifecycle (#297).** Tags are *not* one of the six zero-based-id reference sets — they carry their own (non-zero-based) identity and are resolved by `.find`/Map on both tiers, so they never suffer the index mis-resolution bug retirement exists to prevent. Adding the `RetiredAt` lifecycle to them would be cost without that benefit (a label is not referenced by persisted player state or battle snapshots the way an item/skill is), so the admin "delete" stays a real hard delete. It is graceful for an **in-use** tag: the `ItemTags`/`ItemModTags` join FKs are `ON DELETE CASCADE`, so deleting a tag drops its join rows rather than throwing, and `AdminCacheReloadFilter` reloads the item/mod caches so the next read no longer carries the removed tag id (the change also flows to clients through the items/mods version hash). The Workbench's staged-delete UX surfaces the tag's item/mod usage counts (the Usage tab + list meta) before the admin commits the removal.

### Reference-data read contracts

The reference-data **read** path does not expose EF entities. The repository read methods return dedicated **read contracts** in `Game.Abstractions.Contracts` — a single published language shared by the game client's loading screen and the admin Workbench (which share the `Get*` socket commands). The entity→contract mapping lives in `Game.DataAccess` (`Mapping/*` for the cached in-memory lists; EF-translatable inline projections for the small `IAsyncEnumerable` reads), consistent with the data tier owning DTO mapping.

- **Reads return contracts:** the cached `All()` family (`IItems`/`ISkills`/`IEnemies`/`IZones`/`IItemMods`) and the small reference sets (`ITags`/`ITagCategories`) return `Game.Abstractions.Contracts` types. The internal cache stays as entities (the domain mappers need their full fidelity); `All()` projects that cache to contracts, preserving the zero-based Id-as-index ordering.
- **Gameplay reads are unchanged:** `GetItem`/`GetSkill`/`GetDomainEnemy`/`GetItemMod` still return lean `Game.Core` domain models. Cross-repo entity access within the data tier (e.g. `Enemies` building domain enemies from skill entities) goes through the internal `ISkillEntityCache`, never the public `ISkills` read contract.
- **Gameplay reads return shared, pre-materialized instances (#449).** `GetItem`/`GetItemMod`/`GetSkill` are on the per-battle hot path — invoked once per equipped item / applied mod / equipped skill during `PlayerMapper.ToCore` (cache miss) and `BattleSnapshot.ToBattler` (every battle validation). Rather than re-running the mapper to build a fresh domain graph on every call, the `Items`/`ItemMods`/`Skills` snapshots bundle the entity list **with** its pre-mapped `Game.Core` model list (the `ItemSnapshot`/`ItemModSnapshot`/`SkillSnapshot` records — the same build-and-publish-together shape as `EnemySnapshot`), and the read returns the shared instance by id. This is safe because the reference-data domain models (`Item`/`ItemMod`/`ItemModSlot`/`Skill`/`SkillEffect`) are **structurally immutable value objects** — every property is `init`-only and every collection is an `IReadOnlyList<>` (#547), so the read-only invariant is enforced by the compiler rather than only by review. A consumer therefore cannot reassign or grow the shared graph (e.g. set `ItemModSlot.ItemMod`, or `Add` to `Item.ModSlots`) and silently corrupt the cache for every player. Consumers already treated them read-only — the battle path composes modifiers into a separate `AttributeCollection`, applied mods live on the player's `UnlockedItemSlot` (never on the shared `Item.ModSlots`), and `BattleSkill` only reads its `Skill` — and the type now makes that the only thing the compiler permits. (`AttributeModifier` elements held inside those collections stay reference-mutable; they are a shared battle-engine primitive never mutated post-construction, so element-level immutability was left out of scope.) `Game.Core.Tests/Items/ReferenceDataImmutabilityTests` reflection-guards the invariant so a later edit can't quietly flip an `init` back to `set`. `IChallenges.All()` is the same idea for the whole list — it returns the immutable snapshot directly instead of copying it on every battle's challenge evaluation.
- **Admin writes keep entities inside the data tier:** the entity-returning lookups (`LookupItem`/`LookupItemMod`/`LookupSkill`/`GetEnemy`/`GetRandomEnemy`/`LookupZone`, and the `GetTagEntitiesFor*`/`GetTags` tag-graph methods) live on **internal data-tier interfaces** in `Game.DataAccess` (`IItemEntityCache`/`IItemModEntityCache`/`ISkillEntityCache`/`IEnemyEntityCache`/`IZoneEntityCache`/`ITagEntityQueries`), not on the public `Game.Abstractions` repo interfaces. Each reference repo implements both its public read interface and the internal entity-cache interface, and one shared scoped instance backs both. Their only consumers are the **Content Authoring admin repositories** in `Game.DataAccess` (see [Admin Tools API surface](#admin-tools-api-surface-content-authoring-context)). Neither `Game.Api` nor `Game.Application` reaches an entity through them: `BattleService` reads the `Zone` contract via `IZones.GetZone`, and `PlayerService` checks mod existence via `IItemMods.ValidateItemModId` (#137).
- **`IModel`** (the cross-boundary model marker) lives in `Game.Abstractions` so the contracts there satisfy the `IModel`-constrained `ApiResponse`/`Change` generics. The contracts keep the same property names/shape as the previous API DTOs, so the serialized wire shape — and therefore the reference-data version hash — is unchanged.

### Reference-data cache reload (build-then-swap)

Cache busting is an **eager build-then-swap** (stale-while-revalidate), not a null-and-lazily-refill — the direction settled in spike [#356](./spikes/356-reference-data-cache-reload.md) and implemented across #357 (eager startup load) and #358 (this design). Readers never observe an empty cache, never pay a refill query inline, and never see a torn snapshot.

- **A singleton holder owns each set's immutable snapshot.** `ReferenceCacheHolder<TSnapshot>` (`Game.DataAccess/Repositories/Caching`) holds the current snapshot and exposes `Current` (a lock-free read) and `ReloadAsync()`. The six subclasses each build their snapshot from a fresh `GameContext` resolved through `IServiceScopeFactory`, so a reload's DbContext lifetime is its own rather than borrowing whichever request scope is in flight (the awkward static-state-on-a-scoped-service seam the old design had is gone). The scoped reference repos are thin readers that project `holder.Current`.
- **The snapshot bundles derived structures so they swap atomically.** `EnemySnapshot` carries the enemy list **and** the per-zone spawn tables derived from it; building and publishing them together means a reader can never see a new enemy list against stale spawn tables. `ReloadAsync` builds the whole snapshot off to the side and publishes it with a single atomic reference assignment (`Volatile.Read`/`Write` — a generic field can't use the `volatile` keyword), so readers keep serving the previous snapshot until the swap and a **failed reload leaves the old snapshot in place**. A per-holder `SemaphoreSlim` serializes concurrent reloads; serializing (rather than coalescing onto an in-flight load) is what preserves read-your-writes — an awaited reload always starts a fresh query after the write it must reflect.
- **The admin filter awaits the reload, and must run after the commit.** `AdminCacheReloadFilter` resolves `IEnumerable<IReloadableReferenceCache>` (the holders, registered as that interface) and awaits `ReloadAsync()` on each after a successful admin write, so the Workbench's read-your-writes holds while players read the old snapshot with zero gap. Because the reload queries the database on a *fresh* context, it must run **after** the global `CommitFilter` has persisted the write; it therefore declares `Order = int.MinValue` so it is the outermost action filter and its post-action reload executes after the commit. Controllers apply it through the `[ReloadReferenceCaches]` attribute (a `ServiceFilterAttribute` subclass that bakes in that order), so a new admin controller can't silently default to order 0 and reload before the commit. A reload failure after a committed write propagates (the response is an error; the write persisted, so the admin can retry). The filter deliberately reloads **all** sets on every admin write (it does not know which entity changed); the sets are small and admin writes rare, so a Workbench save reloading each set several times is accepted (coalescing the *local* awaited reloads per request is a future optimization; the cross-instance broadcasts below already coalesce).
- **Cross-instance invalidation rides the Redis backplane (#359).** After a successful admin write (the write is committed once the filter's post-action code runs), the filter also broadcasts a reference-data-changed notification through the existing `IPubSubService` via `IReferenceDataChangeNotifier`, implemented by the `ReferenceCacheSynchronizer` hosted service (`Game.DataAccess`). Every instance subscribes at startup; a **foreign** notification signals a `CoalescingReferenceCacheReloader`, which debounces (default 500ms) and then runs one background reload sweep over all holders — so a Workbench save's burst of admin writes collapses into one (or few) sweeps per instance, on top of the per-holder reload gate. The payload is the publisher's process id, which the publishing instance uses to **skip its own message** (its caches were already reloaded synchronously in the filter). A failed background sweep is retried with exponential backoff (`ReferenceCacheReloadPolicy`) and never disturbs readers — the old snapshot keeps serving; once retries are exhausted the instance stays stale until the next notification, an accepted cost since pub/sub delivery is itself fire-and-forget best-effort (and a fresh boot always loads fresh data).

## Player-facing read projections

Player-facing reads do not expose EF entities to `Game.Api` either, but — unlike reference data — they reuse the existing **gameplay domain models** rather than dedicated read contracts (those domain models are already lean and lossless enough for these reads).

- **Statistics and challenge progress go through the single `IPlayerProgressRepository`,** which already owns the progress write path. Its `GetStatistics`/`GetChallenges` read methods return the `Game.Core.Progress` domain models (`PlayerStatistic`/`PlayerChallenge`), and the entity→domain mapping lives in `Game.DataAccess` (`Mapping/PlayerProgressMapper`, shared with `Load`). The API DTOs (`Game.Api/Models/Progress/*`) project from those domain models, so the read path references no entities. Consolidating onto `IPlayerProgressRepository` retired the duplicate entity-returning read repos (`IPlayerStatistics`/`IPlayerChallenges`), giving a single source of truth for player-progress data.
- **`Save` is self-sufficient — no implicit "load first" ordering.** Both `Load` and `Save` fetch the player's existing rows through a shared private helper that lazily loads and caches the tracked-entity snapshot for the lifetime of the scoped repository. The common battle-completion flow (load → mutate → save) reuses the cached snapshot, so no redundant query is issued; but a standalone `Save` simply fetches the snapshot on demand and upserts correctly rather than dereferencing an un-loaded field. This removed the prior temporal coupling between `Save` and a preceding `Load` (and its null-forgiving `!` usages) — #164.
- **Inventory and log-preference reads were already domain-sourced:** `PlayerData.FromPlayer` projects them from the `Player` aggregate. Their leftover entity-coupled mapping (`InventoryItem`'s entity constructor and `LogPreference.FromSource`) was dead and has been removed.
- **The skill loadout is sent as the full unlocked set, not just the equipped ids.** `PlayerData.UnlockedSkills` projects every `player.Skills` entry as an `UnlockedSkill { SkillId, Selected, Order }` — the direct parallel of `InventoryItem`'s `Equipped`/`EquipmentSlotId` (a skill's `Order` is null when unequipped, like an unequipped item's slot). Because `player.SelectedSkills` is already the deterministic `(Order, SkillId)` ordering (`PlayerMapper.ToCore`), an equipped skill's `Order` is just its index in that list. The client derives the ordered equipped loadout from this richer set, so there is no redundant equipped-id field. The loadout cap is **not** on the wire — it is a generated client constant (`MAX_SELECTED_SKILLS`, emitted from `Game.Core.GameConstants`; see [infrastructure.md](./infrastructure.md)), so the client renders "X / N" without hardcoding it and without a redundant per-player payload field (#306).
- **The loadout is edited through one atomic write command.** `SetSelectedSkills(orderedSkillIds)` — a socket command mirroring `SaveLogPreferences`, aligned with the WebSocket-everything direction — replaces the whole equipped set in order, handling select, deselect, and reorder through a single path. It delegates to `Player.TrySetSelectedSkills`, which enforces the loadout rules as **anti-cheat**: no duplicate ids, count ≤ `GameConstants.MaxSelectedSkills`, and every id already in the player's unlocked set; any violation rejects the change with no state mutation. On success it raises a single `SelectedSkillsChangedEvent` whose write-behind handler (`DataProviderSynchronizer.HandleSelectedSkillsChanged`) clears the player's `Selected`/`Order` flags and rebuilds them from the ordered list (delete-then-rebuild, idempotent on retry — the same shape as the mod-applied handler). This single-writer guarantee is why `PlayerSkill` can carry both `Selected` and `Order` without the two drifting.

## HTTP vs WebSocket Communication

The backend uses a mix of HTTP and WebSockets for communication with the frontend. Right now, HTTP is used for some operations and data fetching, while WebSockets are used for real-time communication such as battle updates. In the future, the goal is to move towards using WebSockets for all communication (other than login/account creation) between the frontend and backend, as this will also enable the Player state to be kept in-memory on the backend and updated in real-time without having to worry about synchronizing with the redis cache or db. Users will only be able to connect to one instance at a time for simplicity and any websocket commands will be handled sequentially for each player. However, HTTP will still be used for admin **persistence** endpoints (the `/api/AdminTools/*` writes), the few reference lookups that have no socket equivalent, and other operations not directly related to the core game.

**Player-state mutations belong on the socket — serializing them with the battle loop avoids lost updates.** Player writes are read-modify-write against the write-behind cache (load the cached `Player`, mutate, save it back); the socket read loop processes one command per player at a time (`SocketHandler.ReadLoop`), so a write on the socket is naturally serialized with the idle battle-completion commands (`DefeatEnemy`/`BattleLost`) that mutate the same cached player. An equivalent write over **HTTP** runs on a separate request thread *concurrently* with that loop, so a background battle save that loaded the player before the HTTP write but persisted after it silently clobbers the change — the client got a success response, yet a refresh shows the old value. This is why every player-mutating gameplay action is a socket command: `SetSelectedSkills`, `SaveLogPreferences`, `SetItemFavorite`, `UpdatePlayerStats` (the attribute allocation that #432 fixed), and the inventory writes `EquipItem`/`UnequipItem`/`ApplyMod`/`RemoveMod` (#463). With those moved, `PlayerController` is read-only — only `GET /api/Player` remains on HTTP (a read, which is safe to leave there).

**The per-player sequential guarantee is enforced by a per-socket command lock, because server-initiated commands run off the read loop (#478).** The read loop is not the only path into a player's commands: a server push (`ChallengeCompleted`, `SocketReplaced`) arrives over the Redis backplane and the pub/sub processor (`SocketManagerService.RegisterSocketCommandListener`) runs it on a thread-pool thread, concurrently with the read loop — both call `SocketHandler.ExecuteCommand` over the same cached `Player`. A `ChallengeCompleted` push mutating the player (`Player.CompleteChallenge`) could therefore interleave a read-modify-write with a `DefeatEnemy` in the read loop — the exact lost-update class the loop's serialization was meant to prevent. So `ExecuteCommand` takes a **per-socket `SemaphoreSlim(1,1)` command lock**, making the documented "handled sequentially for each player" guarantee real for *both* paths. Separately, a **per-socket send lock** (`SocketContext`) guards every `WebSocket` output (`SendData` and the close frame), since `WebSocket.SendAsync` forbids overlapping sends and the two paths (plus the ping/close paths) could otherwise interleave a multi-chunk frame or throw. The two locks are distinct because `ExecuteCommand` holds the command lock while calling `SendData`. (Both `SemaphoreSlim`s are intentionally left undisposed — like `ReferenceCacheHolder`'s gate — since they never materialize a wait handle and disposing risks an `ObjectDisposedException` against an in-flight pub/sub command during socket teardown.)

**A wedged command can't stall the queue forever — each runs under a per-command timeout (#483).** `ExecuteCommand` runs the command under a `CancellationTokenSource` budget (`DefaultCommandTimeout` = 25s — below the client's 30s per-request backstop in `api-socket.ts` and above StackExchange.Redis's 5s command timeout. Npgsql defaults to a 30s command timeout, so a genuinely slow DB call can be cut off a few seconds early; that's an accepted trade-off, since a >25s player command is already pathological and the client abandons it at 30s regardless). The token is threaded through `AbstractSocketCommand.ExecuteAsync`, so a cancellation-aware command unwinds itself, while one that ignores the token is bounded by a `Task.WaitAsync` on the same budget. On expiry the client immediately gets a `"Command timed out."` response so the read/pub-sub loop keeps draining — but the command lock is **not** released to the next command. It is handed to a continuation that releases it only once the abandoned task finally settles, so a still-in-flight read-modify-write of the shared cached `Player` can't race the next command (the lost-update class the lock exists to prevent). The abandoned command still commits its write (so it isn't lost), but `RunCommand` never sends — `ExecuteCommand` owns the single send — so the client never receives a second, late response for an id it already saw time out. The deliberate trade-off: a genuinely stuck dependency still queues later commands behind the lock until it clears (or the client reconnects, which takes over and closes the socket) — chosen over releasing the lock, which would restore the very race the per-socket lock was added for. The token is not yet plumbed below the command boundary (`ICacheService`/`IPubSubService` take none), so a hang inside a token-ignoring dependency relies on that dependency's own timeout to unwind; pushing cooperative cancellation down through the data-access layer is tracked as follow-up (#558).

As a step in that direction, the read-only reference data the loading screen needs (zones, enemies, items, skills, item mods, attributes, challenges, challenge types, statistic types) is exposed over the socket via `Get*` commands (`Game.Api/Sockets/Commands`) that mirror the corresponding `GET` endpoints. These share the `AbstractReferenceDataCommand<TModel>` base, which wraps each repository/intrinsic collection in an `ApiSocketResponse<IEnumerable<TModel>>`. Both the loading screen and the admin Workbench now load these sets over the socket (see [Reference-data caching and versioning](#reference-data-caching-and-versioning)), and the duplicate HTTP `GET` endpoints have been removed (#64). The admin reads can share the same cached socket commands because every admin write reloads these in-memory caches server-side (`AdminCacheReloadFilter`), so the next read serves the freshly written data — the mechanism the admin `Challenges` reads already relied on, now applied to every reference set (see [Reference-data cache reload](#reference-data-cache-reload-build-then-swap)). The only reference reads still served *exclusively* over HTTP are `Tags` and `Tags/TagCategories` (#318 removed the dead `ZoneEnemies`, `SlotsForItem`, `TagsForItem(Mod)`, `ItemCategories`, and `ItemModTypes` endpoints whose data had moved onto the socket contracts or codegen'd enums). (`StatisticTypes` is also exposed over HTTP, but it is dual-exposed — it has a socket twin too; see the next paragraph.)

Statistic-type metadata (`StatisticType { id, entityType, bossOnly, name }`) is intrinsic reference data computed from `EStatisticType` in `Core.Progress.StatisticType.GetAll()` (the DB seed exists only for integrity — there is no repository read). It is exposed both as `GET /api/Statistics/StatisticTypes` and the `GetStatisticTypes` socket command so the Statistics screen can resolve which entity kind each statistic breaks down by instead of hard-coding it. `StatisticType.GetEntityType` is the single source of truth for that mapping and must agree with the per-entity rows `PlayerProgress.RecordBattleCompleted` actually writes; `EntityType` is descriptive only, so changing it does not affect challenge evaluation (challenges resolve a statistic by its id + the challenge's target entity id, never by the declared `EntityType`). `BossOnly` is likewise a descriptive domain fact derived alongside `EntityType` (`GetBossOnly` — only `BossesDefeated`, which `RecordBattleCompleted` increments solely on dedicated-boss victories): it is the single source of truth for the "this statistic only counts bosses" rule, consumed by both the Statistics screen's display hint and the admin challenge editor, which restricts an enemy-scoped challenge's target picker to bosses when the tracked statistic is boss-only rather than special-casing `BossesDefeated`.

## Single active connection (session takeover warning)

A player may only be connected to one instance at a time: `SocketManagerService.RegisterSocket` records the player's current socket id in Redis (`PlayerSocket_{playerId}`) and, when a new socket replaces an existing one, emits `SocketReplaced` to the old socket to close it. To let the client warn the user **before** that takeover happens, `Login/ActiveSession` (an authenticated `GET`) reports whether the player already has a live socket via `SocketManagerService.HasActiveSocket` — a read of the same presence key, so the knowledge of the key's format stays in one place. The check is deliberately **HTTP, not a socket command**: opening a socket to ask the question would itself trigger the replacement it is meant to warn about. Because it is presence-based (rather than device- or tab-scoped), it covers both "another tab" and "another device" uniformly — any other live connection for the player.

- **The presence key is liveness-accurate via a heartbeat-refreshed TTL.** The key is written with a 30s TTL (`SocketPresenceTtl`) at register and refreshed on **every inbound socket message** — the client heartbeats every 10s (`api-socket.ts`), the same `_lastResponse` signal the 60s inactivity check already uses — so a live connection keeps its key fresh while a connection that vanished without a clean close (a tab/device closed without a WebSocket close frame, e.g. WebKit) lets the key expire within the TTL instead of lingering. The refresh is an `Expire` (extend-TTL) rather than a re-set, so it only ever prolongs the currently-registered key and never resurrects one a newer connection has taken over. Net effect: the warning fires immediately for a genuinely-open second tab/device, and the residual false-positive window — re-logging-in within ~30s of a non-graceful close and being warned about your own just-closed session — is bounded by the TTL (tunable; tolerates two missed heartbeats) and documented as acceptable for a best-effort, dismissible warning.
- **Possible future tightening:** a `BroadcastChannel`-based same-browser probe could give instant, zero-window detection of other tabs in the *same* browser (a crashed tab simply can't answer), layered on top of this server-side presence as polish — see the follow-up spike (#221).

## Graceful socket drain on shutdown

A stopping instance drains its live player sockets cleanly rather than leaving them to be force-killed when the process dies (#526) — the missing lifecycle piece for an app explicitly designed to run as multiple scalable instances. Without it, each per-socket read loop blocks on `ReceiveAsync` indefinitely, so on a rolling deploy or scale-down the host cannot signal those loops to stop or tell connected clients to reconnect elsewhere; the connections just hang until the OS/orchestrator kills the process, exactly the session churn the single-active-connection model is otherwise careful about.

- **`SocketConnectionRegistry` is the single long-lived owner of the connection set.** Because `SocketManagerService` is transient (one per handshake), the registry — a singleton registered as an `IHostedService` — tracks every live `SocketHandler` (register on `RegisterSocket`, unregister on teardown) and uses its `StopAsync` as the host's shutdown hook. This replaces the handlers' former untracked, fire-and-forget `Task.Run` loops: `Register` starts the loops and the handler exposes a `Completion` task so the drain can **await** them.
- **The drain is two-phase, carried by two cancellation tokens threaded into each handler's loops.** Outside of shutdown neither token is cancelled, so behaviour is unchanged. (1) A *stopping* token is cancelled first, stopping the per-socket inactivity watchdogs — their job is moot once the drain owns teardown, and leaving them running would just race the shutdown close. (2) The registry then sends every open socket a clean close frame (the dedicated `ESocketCloseReason.ServerShuttingDown`) and awaits the loops within a bounded `_drainTimeout` (5s, well under the host's default 30s shutdown timeout). If a client never completes the closing handshake within that window, a *drain-deadline* token is cancelled, aborting the still-blocked `ReceiveAsync` so the loop unwinds rather than waiting out the host timeout. The per-command timeout token (#483) is orthogonal — it bounds a single command; these bound the socket's whole lifetime.
- **The close frame is a normal closure, so the client reconnects rather than hangs.** `SocketContext.Close` already sends `WebSocketCloseStatus.NormalClosure`; the client's `handleClose` (`api-socket.ts`) settles in-flight requests with an error (callers retry) and re-opens the socket on its next 10s heartbeat — which a rolling deploy routes to a healthy instance. No client change was needed; `ServerShuttingDown` only enriches the close-frame *description*. The socket I/O methods (`SendData`/`ReadMessage`/`Close`) now take a `CancellationToken` (defaulting to none) so the loops can thread the drain token through to the underlying WebSocket calls, which previously passed `CancellationToken.None`.

## Reference-data caching and versioning

The frontend caches each reference-data set in the browser and re-downloads only the sets that changed (see `frontend.md`). The `GetReferenceDataVersions` socket command supports this by returning a content version per set in one round-trip.

- **The version is a content hash of the API models, computed on demand.** Each `AbstractReferenceDataCommand` exposes `ComputeVersion()` (via `IReferenceDataCommand`), which hashes the same serialized model collection the command returns (`ReferenceDataVersioning` — SHA-256 over the camelCase JSON, through the concrete model type so every property is included). Hashing what the client actually receives means the version changes if and only if the client-visible data changes, so no separate version counter has to be maintained or invalidated. It is recomputed per call — the command is hit roughly once per client boot — so there is nothing to invalidate when an admin edits reference data; memoizing it keyed off cache refresh is a possible future optimization if it ever shows up in profiling.
- **The versioned set is exactly the set of reference-data commands.** `GetReferenceDataVersions` is injected with `IEnumerable<IReferenceDataCommand>` (registered by `AddReferenceDataCommands`, which scans the assembly), so a newly-added `Get*` reference-data command is versioned automatically — there is no second list of datasets to keep in sync.

## Battle setup (enemy encounter)

Setting up a battle is a domain decision, not an orchestration one. `BattleService.StartBattle`/`StartBossBattle` (Application) only orchestrate — obtain the zone, snapshot the player and the chosen enemy loadout, set the active battle — and delegate the game-logic choices to `Game.Core`:

- **The `Game.Core.Zones.Zone` aggregate owns the per-zone encounter-level decisions.** It exposes the `[LevelMin, LevelMax]` idle range (and `RollEncounterLevel()`), and the dedicated boss's identity + fixed level (`BossEnemyId`/`BossLevel`/`HasBoss`). The data tier serves it via `IZones.GetDomainZone` (the gameplay/domain read, mapped by `ZoneMapper.ToCore`), alongside the existing `IZones.GetZone` contract read used by the reference-data catalogue. This replaced an unreferenced, mis-shaped `Zone` remnant (#149): the responsibility for "what level do you fight here, and who is the boss" now lives on one domain type instead of being threaded as loose ints.
- **`BattleFactory.CreateBattleEnemy(zone, resolver)` rolls a random idle encounter** within the zone's range and returns a battle-ready enemy with a capped, randomly-selected loadout (`Enemy.SelectBattleSkills()`, ≤4 skills). Enemy _selection_ (which enemy spawns) still lives in the data-access layer (the cached per-zone spawn table behind `IEnemies.GetRandomDomainEnemy`), so the factory takes a `Func<int, Enemy>` resolver rather than depending on a repository — keeping `Game.Core` free of data-access references.
- **`BattleFactory.CreateBossEnemy(zone, resolver)` builds the dedicated boss deterministically** for the "Challenge Boss" action: resolved at the zone's fixed `BossLevel` (no random roll) with its **full authored loadout** (`Enemy.SelectAllBattleSkills()`, no cap, no shuffle). The greater determinism is exactly what frontend/backend battle parity wants. The boss challenge is always available (no cooldown gate) and challenging does not change the player's current zone; `StartBossBattle` returns `null` when the zone has no boss authored, which the `ChallengeBoss` socket command surfaces as an error.
- **Zone-unlock is enforced at battle start (anti-cheat).** A zone carries an optional `Zone.UnlockChallengeId` (#190 — see the [game-design.md](./game-design.md#zone-progression--locking-zones-behind-challenges) zone-progression section). `BattleService` refuses to enter a locked zone: a `StartBattle` *transition* into one is ignored (the player keeps fighting their current zone) and a `StartBossBattle` against one returns `null`. The unlock rule lives on the domain `Zone.IsUnlocked(completedChallengeIds)`; the orchestration reads completion via `IPlayerProgressRepository.GetCompletedChallengeIds`. An ungated zone (the common case) pays no extra read, and a gated zone costs one indexed completion lookup incurred only on a real zone transition or a boss challenge — never per idle tick.
- **The chosen loadout is snapshotted rather than re-rolled.** The selected skill ids are stored on `PlayerState.ActiveEnemySkillIds` (symmetric with the player's `BattleSnapshot`) and sent to the client; validation (`EndBattleVictory`/`Loss`/`AbandonBattle` → `SimulateBattle`) reconstructs the enemy from that snapshot via `Enemy.SetBattleSkills(ids)`. Both sides therefore use the exact same loadout — required because battle logic runs on both the frontend and the backend.
- **The player side of the battle is captured and reconstructed in the domain.** `BattleSnapshot.FromPlayer` projects the player's battle-relevant state (level, stat allocations, equipped item + applied-mod ids, skill ids) into a minimal id-based snapshot at battle start, and `BattleSnapshot.ToBattler(resolveItem, resolveMod, resolveSkill)` rebuilds the player `Battler` from it at validation time — resolving the ids against the catalogs through caller-supplied resolvers (the same `BattleFactory` resolver pattern that keeps `Game.Core` free of data-access). Critically, reconstruction **reuses the same modifier-composition rules as the live `Player` aggregate** — `StatAllocation.ToModifier` and `Item.GetAttributeModifiers`, the building blocks of `Player.GetAllModifiers` — so a battle validated from a snapshot composes the identical attributes the player has live; this is the player half of the frontend/backend battle-parity guarantee, so the rules cannot be allowed to diverge. `Game.Application` (`BattleService`) only fetches the reference data and hands over the resolvers (#208).
- **A dedicated-boss battle is marked, not inferred.** `PlayerState.IsBossBattle` (+ `BattleZoneId`, the challenged zone) is captured at battle start and threaded through `BattleCompletedEvent`, so the resolve path knows a victory was a dedicated-boss clear rather than inferring it from `enemy.IsBoss` + `CurrentZoneId`. `PlayerProgress` keys the zone-clear off that marker, so only the challenge path clears a zone. The clear _semantics_ (#188) treat `ZonesCleared` as a count of distinct zones ever cleared — the per-zone entry is a binary flag and the global counter bumps only on a zone's first clear — while the per-boss/global `BossesDefeated` farm counter increments on every dedicated-boss victory (see the Zone Clears section in [game-design.md](./game-design.md)).
- **An abandon that resolved as a win is a full victory.** `AbandonBattle` (reached only when the player starts a new battle while one is active) re-simulates the in-progress fight capped at the elapsed wall-clock time. If the enemy died within that window the result is a genuine win, so it pays out through the same `RecordVictory` helper as `EndBattleVictory` — exp **and** the win/zone-clear/challenge credit — instead of booking the win while withholding the exp (#206). Both win paths share that one helper so they cannot drift. An abandon that did **not** resolve as a win is split by outcome: if the player died within the elapsed window it is a loss, otherwise (neither combatant died) it is an _incomplete abandon_, tracked as the distinct `BattlesAbandoned` statistic rather than a loss (#202 — see the [game-design.md](./game-design.md#battle-outcomes-battleswon--battleslost--battlesabandoned) battle-outcomes split).
- **The battle `seed` is reserved for the simulation's RNG, not for setup.** It is generated at battle start, persisted on `PlayerState.BattleSeed`, and sent to the client as the shared starting point for any randomness _inside_ the simulation (crit/dodge/etc., not yet implemented). Setup (level roll and skill selection) deliberately uses ambient randomness instead, so it never advances the seed's stream — otherwise the server (consuming draws during setup) and the client (which receives the loadout ready-made) would begin the simulation from divergent RNG states. (The boss path is already deterministic, so this only matters for the random idle encounter.) The RNG primitive that will consume the seed is already ported on both sides (`Game.Core/Battle/Mulberry32.cs` and `UI/src/lib/engine/mulberry32.ts`) — intentional forward-scaffolding for the seeded crit/dodge work (#178), not dead code. It has no live consumer yet, so a cross-implementation parity vector (`Mulberry32ParityTests` ↔ `mulberry32-parity.test.ts`) is the only thing guarding the two ports against silent drift; that vector must stay mirrored, exactly like the battle-simulation parity matrix.

## Battle runtime — skill effects (timed buffs/debuffs)

Skills can carry authored `SkillEffect`s that apply as **timed attribute modifiers** when the skill fires. Because battle logic runs on both the client and the server (the anti-cheat replay depends on them agreeing), every rule below executes identically on both sides — `Game.Core/Battle` (`BattleSkill`, `Battler`, `BattleContext`, `BattleSimulator`) and its frontend mirror (`skill.ts`, `battler.ts`, `battle-step.ts`) — and is pinned by the mirrored parity matrix (`BattleSimulatorParityTests` ↔ `battle-simulation-parity.test.ts`) plus mirrored `Battler`/`BattleSkill` unit tests.

- **Effects ride the real attribute graph.** Applying an effect adds a genuine `AttributeModifier` (source `SkillEffect`) to the target's live `AttributeCollection` and expiry removes it via `RemoveModifier` (#331), so derived attributes cascade for free — a temporary Strength buff raises MaxHealth exactly like a permanent one. This is the riskiest parity surface: both sides must compose attributes through their one shared path (backend `AttributeCollection`, frontend `computeAttributes`) **including mid-battle add/remove**. Float-identical results rely on modifier ordering matching — the backend `SortedLinkedList` inserts a new modifier at the end of its type-run (additives before multiplicatives), which is exactly where the frontend's append-to-`#modifiers` places it.
- **`BattleSkill.Update` fires effects after dealing damage** (pre-effect damage, then `BattleContext.ApplySkillEffect` routes `Self`→the active battler and `Opponent`→the target), so a self damage-buff never boosts its carrying hit, while an earlier loadout slot's effect does affect a later slot firing the same tick.
- **`Battler` owns the active-effect bookkeeping.** It tracks `{ sourceEffect, modifier, expiresAtMs }` per effect. The backend keys expiry to an **absolute per-battler simulated-time clock** (`_elapsedMs`, advanced one tick at a time), so `ActiveEffect` is **immutable** and per-tick expiry is a comparison rather than a countdown; the frontend mirror instead stores a relative `remainingMs` it decrements, because it also drives an interpolated render countdown for the effect chips. The two representations are functionally equivalent — they expire on the same tick — so this is an internal-representation difference that does not affect parity. `ApplyEffect` refreshes an already-active effect's duration (matched by the authored effect id) instead of stacking a second modifier — on the backend the refresh replaces the immutable instance with a new expiry; `AdvanceEffects(ms)` — called for both battlers at the **start of each tick** in `BattleSimulator.Simulate`, before any skill fires — advances the clock and removes the expired ones, so an effect influences exactly `DurationMs / MsPerTick` ticks (counting the application tick). `CurrentHealth` is left untouched when MaxHealth rises (no free heal) and clamped down when it drops, at both the apply and expire points.
- **No per-tick allocations.** The per-tick expiry pass early-returns when a battler has no active effects and otherwise iterates an index loop with no allocation (the same hot-path discipline as #286); the clock still advances on the early-return path so a later effect dates its expiry correctly. The active-effect list is created lazily on first apply.

The two per-second attributes (`DamageTakenPerSecond`, `HealthRegenPerSecond`) are consumed by an **end-of-tick DoT/HoT phase** (`BattleContext.ResolveDamageOverTime`, driven by `BattleSimulator` / `battleStep` after both battlers' skills resolve, reached only while both still live). The enemy resolves first — `Battler.ApplyDamageOverTime` (which **bypasses Defense**, unlike `TakeDamage`), a death check, then `Battler.ApplyHealOverTime` (capped at MaxHealth) — so an enemy DoT kill awards victory before the player's DoT applies (a same-tick mutual DoT kill favours the player). Statistics: DoT on the enemy adds to `PlayerDamageDealt` (not `HighestPlayerAttack`, not per-skill `SkillStats`); DoT on the player adds to `PlayerDamageTaken`; the player's post-cap healing accumulates into the new `BattleStats.PlayerDamageHealed`, wired through `PlayerProgress.RecordBattleCompleted` into the previously-unused `EStatisticType.DamageHealed`. This phase is parity-critical and mirrored on the frontend (`battler.ts` / `battle-step.ts`).

## Caching and Pub/Sub

As mentioned above, most of the reference data is cached in-memory on the backend for fast access. However, player data is cached in Redis and uses a write-behind caching strategy where the cache is the source of truth for player data, and changes are persisted to the database asynchronously. The application uses Redis pub/sub to trigger persistence of player data to the database whenever it changes, as a backplane for WebSocket communication, and to broadcast reference-data cache invalidations across API instances (#359 — see [Reference-data cache reload](#reference-data-cache-reload-build-then-swap)).

**The cached player aggregate carries a sliding idle TTL, not unbounded lifetime (#439).** `PlayerRepository` writes the `Player_{id}` key with a 48h expiry (`PlayerCacheTtl`) on every save and on the load-miss re-cache, and refreshes that TTL on every cache *hit* via a fire-and-forget `ExpireAndForget` (so the hot read path slides the expiration without an extra round-trip). Because an actively-playing player's socket commands rewrite the key on every save, sliding expiration keeps it alive for the whole session while a dormant player ages out of Redis instead of occupying memory forever — previously every player who ever logged in stayed cached forever via an expiry-less `SetAndForget`. The 48h budget mirrors the refresh-token lifetime: a player idle long enough for the key to lapse can no longer hold a live session anyway, so the only cost of expiry is one transparent DB reload. **Cache-miss fallback (the path that makes expiry safe):** `GetPlayer` already treats a missing key as a reload — it queries the player from the database (`GetPlayerFromDb`) and re-caches it — so an expired or evicted key is invisible to callers.

**Eviction interacts with write-behind durability, and the TTL is sized so it can't bite.** Because the cache is the source of truth, an event still draining from the `PlayerUpdateQueue` when its player's key disappears would let a subsequent `GetPlayer` fall through to a DB that lags until the queue drains, briefly serving pre-event state. The invariant is therefore *TTL ≫ max queue-drain time*: the sliding refresh keeps an active player's key alive, and even a dormant key's hours-scale TTL dwarfs the sub-second drain window, so a key never expires mid-drain. Redis's `maxmemory-policy` is the remaining lever under memory pressure — now that every player key (like the refresh-token and socket-presence keys) carries a TTL, a `volatile-*` policy lets Redis shed dormant player keys gracefully rather than failing writes under `noeviction` or evicting hot keys, but the deliberate TTL, not the eviction policy, is the primary control.

The write-behind path is wired through the domain-event pipeline. When the `Player` aggregate mutates it raises domain events; on save, `PlayerRepository` dispatches them in-process and `PlayerPersistencePublisher` (an `IDomainEventHandler` registered per event type in `AddDataAccess`) serializes each one into a `DomainEventEnvelope`. Rather than publishing per event, the publisher buffers each envelope into the scoped `PlayerUpdateBatch`, and `SavePlayer` flushes the whole batch onto the `PlayerUpdateQueue` Redis list as **one multi-value LPUSH** after the dispatch settles (`IPubSubService.PublishBatch` — a no-op on an empty batch, #559). So a save raising several player events (e.g. a stat allocation's `PlayerCoreUpdatedEvent` + `AttributeAllocationsChangedEvent`) costs a single queue round-trip instead of one per event; the queue contents are unchanged (an LPUSH of N values still adds N elements, drained one at a time). The `DataProviderSynchronizer` hosted service consumes that queue and applies each change to the database. The publisher is registered only for the player events that represent persistent state changes; events that are purely in-process reactions (e.g. `PlayerLeveledUpEvent`, whose persistent effect is already carried by `PlayerCoreUpdatedEvent`) are intentionally not registered, so they are never enqueued. This keeps the "which events get persisted" decision in the data-access layer alongside the `DataProviderSynchronizer` switch that consumes them, rather than encoding a persistence concern onto the domain events themselves.

**The source-of-truth cache write is deliberately fire-and-forget, because the in-memory aggregate — not the cache — is the read-modify-write base during a session.** A live player runs on a single WebSocket whose request-scoped `SessionService` lives for the whole connection (`SocketInterceptorMiddleware` blocks the `/socket` request on `WaitSocketClosed`). When the socket connects, `SocketInterceptorMiddleware` loads the `Player_{id}` key **once** up front — in a disposable scope, so no `GameContext` is held open for the connection — and stores the aggregate on the connection-lived `SessionService`, which exposes it synchronously as `SessionService.Player`. Every subsequent command on that socket mutates and re-saves the *same in-memory object* rather than re-reading Redis — each command runs in a fresh DI scope for its repositories, but the player it operates on comes from the connection-lived session. `SavePlayer` therefore writes the cache via `SetAndForget` while awaiting only the durable write-behind enqueue: a dropped cache write cannot poison the next command (it never reads the key) and self-heals on the next save, and Postgres still converges via the queue regardless. The one residual exposure is a dropped write *combined with* the in-memory aggregate being discarded (reconnect / socket-replace) before the next successful save — then a new connection's `GetPlayer` takes the stale-but-present cache *hit* instead of the converged DB row (a stale hit does not fall back to the DB, unlike the TTL-eviction *miss* above). This is a knowingly-accepted trade-off: guarding it would put an awaited Redis round-trip on the per-battle hot path to cover a rare race the in-memory model already shields the steady-state command stream from.

**Player progress (statistics + challenges) is a second cache-as-source-of-truth, write-behind aggregate (#550 — spike: [548-battle-end-io-write-behind.md](./spikes/548-battle-end-io-write-behind.md)).** It is cached under its own `Progress_{id}` key — kept separate from the `Player` blob (which `SavePlayer` re-serializes to cache in full on every save) because the stat set grows per-skill/per-enemy over time, so bundling it in would bloat every per-battle write — with the same sliding idle TTL. `PlayerProgressRepository` serves every read cache-first with a DB miss-reload, and **all** reads (the stats/challenges pages and the `GetCompletedChallengeIds` zone-unlock gate) go through the cache, so a just-completed unlock gates the zone correctly even while the DB still lags. `Save` writes the full snapshot to the cache and enqueues a single batched, **absolute-value** `ProgressUpdated` event onto the same `PlayerUpdateQueue`, carrying only the rows changed since load (`PlayerProgress` dirty-tracks its mutations; a battle touches ~10–20 rows, so one batched event beats granular per-stat events). `DataProviderSynchronizer` upserts them idempotently (absolute values, batched load-then-set, mirroring `HandleAttributeAllocationsChanged`). Unlike the `Player` aggregate, progress is published **directly by the repository** rather than through the domain-event dispatcher: it emits a single data-tier persistence event consumed only by the synchronizer, so routing it through the dispatcher would add ceremony for no gain.

Queue processing in `DataProviderSynchronizer` is resilient and never silently drops an event:

- **A malformed payload is dead-lettered, not retried.** A payload that fails to deserialize (or whose inner event payload is malformed) can never succeed, so it is logged as a warning and moved to the `PlayerUpdateDeadLetterQueue` Redis list for inspection instead of being discarded.
- **A valid event that fails on an unexpected error is retried with exponential backoff, then dead-lettered.** Transient failures (e.g. a database connection blip, deadlock, or timeout) often succeed on a second attempt, so each event is retried per `PlayerUpdateRetryPolicy` (default: 3 attempts, 200ms base backoff). The non-final attempts log a warning; once the retries are exhausted the event is logged as an error and pushed onto the dead-letter queue rather than dropped, so the change is preserved for later inspection/replay. Retrying is safe because the handlers are idempotent (existence checks, absolute `ExecuteUpdate`s, and delete-then-insert), so re-applying a partially-applied event converges to the same state.
- In every case the rest of the queued events are still processed.

The dead-letter queue is intentionally a passive holding list — there is no automatic redelivery consumer yet, and the queue read is still a destructive pop, so an event in flight when the process crashes is lost (a crash-safe reliable-queue / peek-then-acknowledge read remains future work). The retry-then-dead-letter path covers the common transient-error case and keeps a durable record of anything that exhausts its retries.

**Drains are serialized so same-player events apply in order (#578).** The pub/sub `BackgroundWorker` re-arms its wait independently of callback completion, so a wake arriving while a drain is in progress can dispatch a second `ProcessQueue` on another thread-pool thread; both then pop the shared queue via atomic `LPOP` and could apply two order-sensitive same-player events (an equip immediately followed by an unequip; the delete-then-rebuild loadout handler) out of order, persisting stale state until the cache self-heals on the next save. `ProcessQueue` therefore gates drains through a `SemaphoreSlim(1,1)` claimed with `WaitAsync(0)`: at most one drain runs at a time, and a wake that races the active drainer's exit is coalesced into a single follow-up pass (tracked by an interlocked flag set before claiming the gate and re-checked after releasing it) rather than dropped — so the whole queue still drains without ever running two passes concurrently and without stranding a late wake. (The other queue consumer, `SocketManagerService`'s per-socket command listener, is already serialized by the per-socket command lock, so this gate addresses the independent write-behind consumer specifically.)

**The dispatcher drains events raised during handling.** `DomainEventDispatcher.DispatchAsync(aggregate)` loops until the aggregate is quiescent, so events a handler raises on the aggregate while reacting to another event are dispatched in the **same** cycle rather than lingering until the next save. This matters for the challenge-reward path: `BattleStatisticsEventHandler` reacts to `BattleCompletedEvent` by unlocking rewards on the player (`Player.CompleteChallenge`, which raises the unlock events plus a `ChallengeCompletedEvent`); without draining, those secondary events would only flush on the *next* `SavePlayer`, delaying their persistence and the client notification below. A safety bound fails loudly if a handler raises events unboundedly.

**Handler dispatch is isolated, so one handler throwing never silently drops the rest of the batch (#412).** A batch is snapshotted and cleared off the aggregate *before* its handlers run (so a handler raising follow-up events doesn't re-process the batch). Previously, if a handler threw partway through, the remaining already-cleared events were never dispatched — their write-behind persistence / notifications were lost *before* reaching the resilient `DataProviderSynchronizer` safety net. The dispatcher now wraps each individual handler invocation: a thrown exception is captured and the remaining handlers, sibling events, and drained follow-up events are still dispatched. Once the aggregate is quiescent, any captured failures are rethrown together as an `AggregateException` (surfaced — and logged by `SocketHandler.ExecuteCommand` — rather than swallowed). This mirrors the write-behind consumer's "process the rest, never silently drop" philosophy. Note the chosen semantic is *no re-dispatch* (the cleared batch is not retried), deliberately avoiding double-applying non-idempotent in-process handlers such as `BattleStatisticsEventHandler` (`RecordBattleCompleted` increments cumulative stats); the residual gap — the specific event whose own handler threw is not itself retried — is tracked as follow-up work (a dispatch-layer retry/dead-letter mirroring the post-publish queue).

**Handlers for the same event type fire in registration order (#486).** The dispatcher keys each event type to an `ImmutableArray` of handler delegates rather than the previous `ConcurrentBag`, whose enumeration order was unspecified — so when more than one handler is registered for an event type, the order they run in is the order they were registered (one-time at startup in `AddDataAccess`/`Startup`), not nondeterministic across runs/instances. `ImmutableArray` keeps reads lock-free and iterates with a non-allocating struct enumerator on the dispatch hot path; registration is a copy-on-write `Add` under the existing `ConcurrentDictionary` keying.

## Challenge-completion notifications (server push)

Completing a challenge unlocks its rewards as a side effect of `BattleCompletedEvent` handling, decoupled from the socket command that triggered the battle — so the command's response cannot carry the unlock, and it would otherwise surface only on the next page load (the client does not evaluate challenges itself; the server is authoritative). Instead, `Player.CompleteChallenge` raises a `ChallengeCompletedEvent` (challenge id + any reward ids), and `ChallengeCompletedNotifier` (an API-layer `IDomainEventHandler`, registered in `Startup` because it depends on the socket infrastructure) pushes a `ChallengeCompleted` socket command to the player via `SocketManagerService.EmitSocketCommand`. Riding the Redis backplane like `SocketReplaced`, it reaches the player on whichever instance holds their connection and covers **every** completion path (idle victory, won-abandon, even a loss that completes a counting challenge), not just the explicit-victory response. The client applies it by unlocking the item/mod/skill locally so the reward is immediately usable. The push command is server-initiated only: it carries a typed response but no `Parameters` property, so codegen emits no client request type.

## Redis connection thread-pool floor

`RedisMultiplexerFactory` raises the process thread-pool minimum (to at least `MinThreadPoolThreads`, never lowering a host that already runs higher) before connecting any multiplexer. StackExchange.Redis completes async commands on the .NET thread pool, whose default minimum equals the CPU count; once busy threads exceed that minimum the CLR adds threads only ~1 per 500ms. On small, busy hosts (a CI runner hosting the API + Postgres + Redis, or multiple integration-test assemblies at once) that ramp-up stall could leave a reply unread in the socket past the 5s command timeout, surfacing as a spurious `RedisTimeoutException` (`WORKER: (Busy=N, Min=processorCount)`). Centralizing the bump in the single connection chokepoint covers production and every test process that touches Redis without scattering startup tweaks across entry points.

**The CI test process raises a higher, dedicated floor (#517).** The production floor (32) was sized for a single instance, but the `test-backend` job runs *every* test assembly concurrently under the `dotnet-coverage` profiler on a 2-core runner — a far heavier load on one process. Under it the 32-floor was still exceeded (failure diagnostics showed `Busy=35, Min=32, QueuedItems=36`), starving not just Redis replies but the Npgsql read and the test-only `RedisCleaner` multiplexer (which connects outside the factory). The starvation surfaced as an *intermittent* `RedisTimeoutException` / Npgsql `TimeoutException` in whichever integration test was running at the contention peak — not a logic bug in that test. The shared `IntegrationTestContainers` collection fixture (used by both integration assemblies) pre-warms a generous floor in its `InitializeAsync`, before any integration test touches the backing services. Production code is untouched.

## BackgroundWorker disposal and pub/sub worker ownership

`BackgroundWorker` (`Game.Infrastructure`) owns an `AutoResetEvent`, so it is `IDisposable`. `Kill()` and `Dispose()` are deliberately distinct: `Kill()` stops the worker from being scheduled again while keeping the instance queryable, whereas `Dispose()` also releases the OS wait handle (and implies `Kill()`). `Start()` publishes `IsRunning = true` **before** signaling the worker loop so the loop's trailing `IsRunning = false` always happens-after it — otherwise a fast (already-signaled) run could complete and reset the flag before the late `true` landed, wedging `IsRunning` stuck `true` until the next run. The flag is `volatile` so the transition is published across the threads that read it.

`RedisPubSubService` is **not** `IDisposable`. Its worker registry is `static` (process-wide) but the service is registered **transient**, so disposing one instance would tear down workers owned by every other instance. Each worker is therefore disposed at its own teardown point — when its subscription is removed (`UnSubscribe`) or when it is discarded because its id was already taken — which is the correct ownership boundary. (The single app-lifetime queue subscription created without an id, e.g. `DataProviderSynchronizer`, is intentionally not tracked or disposed; it lives for the process.)

## Admin Tools API surface (Content Authoring context)

The frontend admin tools were consolidated into a single entity-driven Workbench (see `docs/frontend.md`). That UI saves a whole record (identity + related collections) at once, but the backend keeps persisting each relationship through its own admin endpoint — the Workbench just orchestrates them.

Reference-data administration is modelled as a distinct **Content Authoring** bounded context (its own model over the shared database), implemented as namespaces within the existing projects rather than new projects. The EF entity is this context's persistence shape and is an implementation detail of `Game.DataAccess` — it never surfaces in `Game.Api`. (User administration is the separate Identity / User Admin context — see [User Administration](#user-administration-archive-vs-ban).)

- **Admin controllers are thin HTTP adapters.** `Game.Api/Controllers/Admin/*` accept an admin write contract, call an admin repository, and map the result to an `ApiResponse`. They reference no `Game.Abstractions.Entities` and no `IEntityStore`.
- **Admin write contracts live in `Game.Abstractions.Contracts.Admin`** (`Change<T>`, `EChangeType`, `AddEditAttributesData`, `SetTagsData`, and the `Set*Data` relationship DTOs). The whole-record `Change<T>` payloads reuse the reference-data read contracts in `Game.Abstractions.Contracts` (`Enemy`/`Item`/`Skill`/`Zone`/`Tag`/`ItemModSlot`/`Challenge`), so reading and authoring share one serialized shape. `Challenge` is the single read+write contract: the `GetChallenges` socket maps the gameplay domain `Challenge` (which `IChallenges` still serves for challenge evaluation) onto it, and the admin write path persists it.
- **Admin repositories own the persistence.** The `IAdmin*` interfaces (`Game.Abstractions/DataAccess/Admin`) are implemented in `Game.DataAccess/Repositories/Admin`. They encapsulate the change-set ordering (`ChangeSetProcessor`, also relocated here), the fresh-entity construction, and the `IEntityStore` calls; they reuse the cached reference repos for existence/diff lookups. A relationship setter returns `false` when its target record is missing, and the controller maps that to the user-facing "not found" message.
- **Cache reload stays a cross-cutting filter, discovered through `IReloadableReferenceCache`.** `AdminCacheReloadFilter` reloads the in-memory reference caches after every admin write (an awaited build-then-swap — see [Reference-data cache reload](#reference-data-cache-reload-build-then-swap)) so the next read serves the freshly written data. Each set's singleton snapshot holder implements `IReloadableReferenceCache` (`Game.Abstractions.DataAccess`) and is registered under that interface, so the filter (and the test bases, via `Game.TestInfrastructure`'s `ReferenceCacheReloader.ReloadAllAsync`) resolves `IEnumerable<IReloadableReferenceCache>` from DI instead of hand-maintaining a list of the caches — a newly-added cache is covered automatically (#312). The filter never touches entities, so it stayed at the presentation edge rather than moving into the repositories; controllers apply it through the `[ReloadReferenceCaches]` attribute, which bakes in `Order = int.MinValue` so the reload runs after `CommitFilter` persists the write and a new controller can't omit the ordering.

Two persistence gotchas the admin repositories respect (and any new admin write must too):

- **Updates must not be inferred from the key value.** `IEntityStore.Update` tracks the passed entity as `Modified` directly (`_context.Entry(entity).State = EntityState.Modified`) instead of going through `DbContext.Update`. `DbContext.Update` infers Add-vs-Edit from whether the key is "set", which misfires for our zero-based-identity tables (see [Reference Data](#reference-data)): the first row's `Id == 0` is the CLR/identity-column default, so EF would treat an edit of record 0 as a brand-new insert and silently duplicate it.
- **Don't update/delete cached entities directly.** Reference data is read with `AsNoTracking().Include(...)`, so a fetched child (e.g. a `SkillDamageMultiplier`) carries a loaded back-reference to its parent and siblings. Passing that instance to `Update`/`Delete` would drag the whole graph into the change tracker and re-insert it. Relationship endpoints therefore build fresh, navigation-free entities keyed only by their own ids (as `SetEnemySkills`/`SetZoneEnemies`/`SetSkillMultipliers` do) rather than mutating the cached objects.

## Authentication (JWT access + refresh tokens)

Authentication uses standard ASP.NET Core JWT bearer auth rather than the legacy cookie-backed pseudo-session token.

- **Short-lived signed access tokens + long-lived rotating refresh tokens.** `AccountService.Login` (called by `LoginController.Login`) verifies credentials and returns the issued tokens plus the player aggregate; the controller maps that to a `LoginResult` carrying an `AuthTokens` pair plus the player data. The **access token** is a JWT signed (HMAC-SHA256) behind the `IAccessTokenService` abstraction (`JwtTokenService`) using the configured `Jwt:SigningKey`, with a 15-minute lifetime (`AuthConstants.AccessTokenLifetime`); it carries the user id (`sub`) and `role` claims. The **refresh token** is an opaque, high-entropy random value with a 48-hour lifetime (`AuthConstants.RefreshTokenLifetime`). Clients send the access token as an `Authorization: Bearer` header; when it expires they call `Login/Refresh` to mint a new pair.
- **Standard claims pipeline.** `AddJwtBearer` validates the token (issuer/audience/signature/lifetime) and projects it onto `HttpContext.User`. A global `FallbackPolicy` requires every endpoint to be authenticated unless marked `[AllowAnonymous]` (login, account creation, refresh, logout). `SessionLoaderMiddleware` bridges the validated principal to the request-scoped `SessionService` (user id + roles + cached player state); inbound claim mapping is disabled so claim types stay verbatim (`sub`/`role`).
- **Refresh tokens are stored server-side in Redis, keyed by a SHA-256 hash of the value (`IRefreshTokenStore`/`RefreshTokenStore`).** Only the hash is persisted, the TTL enforces the lifetime, and consumption is an atomic get-and-delete so each refresh token is **single-use**: `Login/Refresh` rotates it (issues a brand-new pair and invalidates the old refresh token), and `Login/Logout` revokes it. Redis is the natural home because auth already depends on the Redis player session (a flushed cache logs users out regardless), so this adds no new durability assumption and avoids a DB migration.
- **WebSockets authenticate via an `access_token` query-string parameter.** Browsers cannot set an `Authorization` header on the WebSocket handshake, so the `/socket` endpoint reads the access token from the query string (wired through `JwtBearerEvents.OnMessageReceived`) — the canonical ASP.NET Core pattern for bearer auth over WebSockets.
- **The frontend consumes this scheme through its API client layer** — token persistence, pre-emptive refresh, the single-flight refresh path, and the socket query param all live in `UI/src/lib/api` (see `frontend.md` → Authentication).

### Password hashing (PBKDF2)

Passwords are hashed with **PBKDF2-HMAC-SHA256** behind the `IPasswordHasher` abstraction (`Game.Abstractions/Auth`), implemented by `Pbkdf2PasswordHasher` (`Game.Application/Auth`). This replaced a bespoke iterated-SHA-512 construction (a fast, GPU-friendly hash assembled by hand) with a purpose-built, deliberately-slow KDF.

- **Self-contained, tunable format.** A hash is stored as `$pbkdf2-sha256$<iterations>$<base64-salt>$<base64-key>`, so the work factor and a per-hash salt live alongside the key — the stored hash is fully self-contained. The iteration count is configured via `PasswordHashingOptions.Iterations` (default 600k, the OWASP floor for PBKDF2-HMAC-SHA256); tests lower it via `PasswordHashing:Iterations`.
- **Salt and pepper.** A fresh 128-bit random salt is generated for every hash and embedded in the output (so `Hash`/`Verify` take no salt parameter and there is no separate salt column). An application-wide pepper (`HashPepper` secret) is folded in via a standard HMAC pre-hash for defence-in-depth; it is injected through `PasswordHashingOptions` rather than process-global mutable static state, and validated on startup so a missing pepper still fails fast.
- **Transparent work-factor upgrade on login.** `Verify` returns `SuccessRehashNeeded` for a valid credential stored with an outdated work factor. `AccountService.Login` then re-hashes the verified plaintext (with a fresh salt) and persists it via `IUsers.UpdatePasswordHash` (a targeted `ExecuteUpdate`), so existing accounts upgrade on next login without a forced reset. Verification comparisons are constant-time (`CryptographicOperations.FixedTimeEquals`).

### Account/login orchestration lives in the application layer

Account creation and the login/refresh/logout flows are orchestrated by `AccountService` (`Game.Application/Services`), not the controller. `LoginController` is a thin HTTP adapter: it calls the service, maps the result to an `ApiResponse`, and wires the **request-scoped** `SessionService` (creating the session on login, clearing it on logout) — session identity is a presentation concern, so it stays in the API layer. `AccountService` owns the data orchestration: username availability, password hashing, delegating the new account's user-and-initial-player insert to the Identity context, credential verification, loading the player aggregate, and refresh-token rotation/revocation via `IRefreshTokenStore`. `Game.Application` references no EF entities (#137); it speaks only Identity contracts (`NewAccount`/`AccountCredentials`) and domain types.

- **The new-player defaults are a domain concern, built by `NewPlayerFactory` (`Game.Core/Players`).** "What does a new player look like?" — the starter skill set, the base attribute spread, and the default log preferences (plus the `StarterSkillCount`/`AttributeCount`/`StartingAttributeAmount`/`StartingZoneId` constants) — is a game rule, so it lives in `Game.Core` rather than the orchestration layer. `NewPlayerFactory.Create(name)` returns a `NewPlayer` blueprint (raw defaults, no identity or resolved reference-data objects). `AccountService` hashes the password into a `NewAccount` credential blueprint and hands both blueprints to `IUsers.CreateAccount` (the Identity context owns the `User` aggregate), which maps them to the linked user+player entity graph (`PlayerMapper.ToEntity`) and queues it in the unit of work. `AccountService` therefore neither encodes the defaults nor constructs entity graphs — it only orchestrates. Unlike `SavePlayer`'s write-behind path, account creation persists straight through the unit of work because a freshly created player has no cached state or domain events yet.
- **Access-token issuance is abstracted behind `IAccessTokenService` (`Game.Abstractions/Auth`).** The application layer must issue the access token as part of the login/refresh use case, but the concrete JWT implementation (`JwtTokenService`) sits at the presentation edge alongside the bearer-validation pipeline it mirrors. Depending on the abstraction keeps `Game.Application` free of the JWT libraries; the API composition root registers `JwtTokenService` as the implementation. Token-lifetime policy is shared via `AuthConstants` (`Game.Abstractions/Auth`) so the access-token signer and the refresh-token issuer reference one source.

## CORS allowed origins (deployment config)

The browser CORS policy's allowed origins are **configuration-bound, not hardcoded** (#585). The allowed frontend origins are deployment-specific, so `Startup` binds `CorsOptions` (an `AllowedOrigins` list) from the `Cors` configuration section and applies them via `UseCors(...).WithOrigins(...)`.

- **Multiple origins are supported** — `AllowedOrigins` is a list, so an environment can permit several frontend hosts.
- **Fail fast, not silently broken.** The options are registered with `Validate(... Count > 0).ValidateOnStart()`, so an environment with no configured origin fails at startup rather than silently rejecting every browser request. The local dev origin (`http://localhost:5174`) ships in `appsettings.Development.json` so local behaviour is unchanged, and the integration-test host supplies it in `GameServerFactory`.

## Access Roles and Admin Authorization

Authenticated users are gated out of the admin tooling unless they hold the `Admin` role.

- **Roles live in the auth token, not the session store.** `LoginController.Login` loads the user's roles and bakes their names into the access token's `role` claims (and into the refresh token's stored session data, so a rotated token keeps them), so the token is the source of truth for what a request is authorized to do. `SessionLoaderMiddleware` copies the validated principal's roles onto the request-scoped `SessionService` (which exposes `IsInRole`, used by `AdminRoleAuthorizationFilter`). This keeps the role check self-contained in the signed token and avoids a second lookup; the trade-off is that a role change only takes effect on the user's next login (acceptable for an admin grant, and revisitable if finer-grained revocation is ever needed).

## User Administration (Archive vs Ban)

`AdminUsersController` (under `Game.Api/Controllers/Admin`) exposes the admin user-management surface: search the roster (`GetUsers`), list the grantable roles (`GetRoles`), replace a user's roles (`SetUserRoles`), and the two lifecycle actions below. It shares the `/api/AdminTools/[action]` route prefix like the other admin controllers, but a few of its conventions differ on purpose:

- **Identity / User Admin is its own bounded context, and its repository reads speak contracts, not EF entities.** This context shares the `User` aggregate with the auth/login flow, so it sits alongside it rather than under Content Authoring. `IUsers`/`IRoles` return `Game.Abstractions.Contracts.Identity` types: the admin roster (`SearchUsers` → `AdminUser`, with nested `PlayerSummary`/`Role`; `GetRoles` → `Role`) is the published read language the Workbench consumes, and `GetUser` returns an `AccountCredentials` lookup for the login use case. The entity → contract projection lives in `Game.DataAccess` (`Users`/`Roles`), so the `User`/`Role` EF entities never surface in `Game.Api` or in the `AccountService` login path. `AccountCredentials` is deliberately **not** an `IModel`: it carries the password hash and is internal to the data tier + application layer, never serialized across the API boundary. Account creation is the matching **write** path: `IUsers.CreateAccount(NewAccount, NewPlayer)` takes a `NewAccount` credential blueprint (also a non-`IModel` Identity contract) plus the new-player blueprint and builds the linked user+player entity graph in the data tier, so `AccountService` constructs no entities (#137).
- **Archiving and banning are two distinct, non-destructive states, modeled as nullable timestamps on `User` (`ArchivedAt`, `BannedAt`).** Timestamps (rather than booleans) double as the flag _and_ an audit of when the action happened, and follow the idiomatic soft-delete shape.
  - **Archive = soft delete that frees the username.** An archived user is excluded from login (`IUsers.GetUser`) and from username-availability checks (`IUsers.CheckIfUsernameExists`, which gates account creation). Because the username check ignores archived rows, the username becomes available for a brand-new account — there is intentionally **no DB unique index** on `Username`, so an archived row and a new active row can share a username; the active one is the single non-archived match the queries resolve to. The admin roster, by contrast, does **not** hide archived users by default: `SearchUsers`/`CountUsers` take a tri-state `archived` filter (`null` = all users, `false` = active only, `true` = archived only), so an admin can still find and manage archived accounts.
  - **Ban = block while keeping the username reserved.** A banned user still counts toward username availability (so the name cannot be reused) and remains visible in the admin roster for later management. Surfacing the ban to the user at login time is intentionally out of scope for now.
- **Role changes take effect on next login.** As noted under [Access Roles and Admin Authorization](#access-roles-and-admin-authorization), roles live in the signed auth token, so a `SetUserRoles` change is reflected only after the affected user logs in again.

## User Connection Tracking

Connections are tracked with three tables rather than a full ledger, each keyed by the signal that actually distinguishes it:

- **`BrowserInfo` — the user-agent profile, deduplicated by `UserAgent`** (unique index). Holds only the server-observable signals carried on every request: the user-agent and the low-entropy client-hint headers (`Sec-CH-UA`/`-Mobile`/`-Platform`). It is genuinely shared across the many devices that report the same user-agent. Client hints are filled in only when missing, so a later request with fewer hints never clears earlier values.
- **`Device` — the device, deduplicated by `DeviceFingerprintHash`** (unique index). This is the most device-unique signal, so device-specific data lives here: the fingerprint, the capabilities (`deviceMemory`/`hardwareConcurrency`), and a FK to its `BrowserInfo` (the fingerprint incorporates the user-agent, so that mapping is stable). The fingerprint is computed client-side; the capabilities are reported by the frontend after login (they aren't on a regular request) via the authenticated `Login/DeviceInfo` endpoint.
- **`UserLogin` — a connection origin.** A surrogate identity key plus a unique index on `(UserId, IpAddress, DeviceId)`; `LastConnection` is updated in place for that combination rather than appended. (A unique index, not a composite PK, keeps the store-generated `Device` key out of the dependent's primary key — which EF cannot cleanly propagate when inserting both rows together.)

Keying `UserLogin` on the device means the per-request tracker needs the fingerprint on **every** request, but browsers don't send one on their own. So the frontend computes a deterministic fingerprint once (a SHA-256 of stable client signals, cached in `localStorage`) and the request layer attaches it as an `X-Device-Fingerprint` header on every authenticated request. This is the deliberate trade-off chosen over storing device data on a user-agent-keyed row (which would let two devices sharing a user-agent overwrite each other).

- **Recording happens in `LoginTrackingMiddleware`, after `SessionLoaderMiddleware`,** so it fires on authenticated **HTTP** requests (not per WebSocket command — those never re-enter the request pipeline, and a WebSocket handshake can't set the custom header anyway, so it's skipped). A request without the fingerprint header is skipped (no device to key on). Recording is **best-effort**: failures are logged and swallowed so tracking never affects the response. Because it runs outside the per-action `CommitFilter`, the tracking service commits its own unit of work. The trade-off is a small write per authenticated HTTP request; acceptable while most game traffic is on the socket, and throttleable later if HTTP volume grows.
