# Backend Battle Setup & Runtime

This doc covers the backend's battle **setup** (enemy/boss encounter construction and snapshotting) and **runtime** (skill effects and damage/heal-over-time). It is split from [backend.md](./backend.md) so the main doc stays an overview, with the tick-by-tick mechanics here next to the parity invariant that governs them.

**Parity invariant (why this detail matters).** Battle logic runs on **both** the frontend and the backend, and the backend replays the battle the client reports as anti-cheat, so the two simulators must agree tick-for-tick. Every rule below is mirrored on the frontend (`UI/src/lib/battle`) and pinned by the mirrored parity matrix (`BattleSimulatorParityTests` ↔ `battle-simulation-parity.test.ts`) plus mirrored unit tests; it must not be allowed to diverge. See [game-design.md](./game-design.md) for the player-facing mechanics.

## Battle setup (enemy encounter)

Setting up a battle is a domain decision, not an orchestration one. The application layer (`BattleService`) only orchestrates — obtain the zone, snapshot the player and chosen enemy loadout, set the active battle — and delegates the game-logic choices to `Game.Core`:

- **The `Zone` aggregate owns the per-zone encounter decisions** — the idle level range (and the roll within it) and the dedicated boss's identity + fixed level. The data tier serves a gameplay/domain `Zone` read alongside the contract read used by the reference catalogue, so "what level do you fight here, and who is the boss" lives on one domain type rather than being threaded as loose ints.
- **A random idle encounter** rolls a level within the zone's range and returns a battle-ready enemy with a capped, randomly-selected loadout (≤4 skills). Enemy _selection_ (which enemy spawns) stays in the data layer's cached per-zone spawn table; the factory takes a resolver function rather than a repository dependency, keeping `Game.Core` free of data-access references.
- **The dedicated boss is built deterministically** for the "Challenge Boss" action: resolved at the zone's fixed boss level (no random roll) with its full authored loadout (no cap, no shuffle). The greater determinism is exactly what battle parity wants. The boss challenge is always available and challenging it doesn't change the player's current zone; a zone with no boss authored surfaces an error.
- **Zone-unlock is enforced at battle start (anti-cheat).** A zone may carry an optional unlock-challenge gate (see the [zone-progression](./game-design.md#zone-progression--locking-zones-behind-challenges) section). Entering a locked zone is refused — an idle transition into one is ignored (the player keeps fighting their current zone) and a boss challenge against one is rejected. The rule lives on the domain `Zone`; the orchestration reads challenge completion. An ungated zone (the common case) pays no extra read, and a gated zone costs one indexed completion lookup only on a real transition or boss challenge — never per idle tick.
- **The chosen loadout is snapshotted, not re-rolled.** The selected skill ids are stored on the player state (symmetric with the player's own battle snapshot) and sent to the client; validation reconstructs the enemy from that snapshot, so both sides use the exact same loadout.
- **The player side is captured and reconstructed in the domain.** A minimal id-based snapshot (level, stat allocations, equipped item + applied-mod ids, skill ids) is projected at battle start and rebuilt into the player `Battler` at validation time, resolving ids against the catalogs through caller-supplied resolvers. Critically, reconstruction **reuses the same modifier-composition rules as the live `Player` aggregate**, so a battle validated from a snapshot composes the identical attributes the player has live — this is the player half of the parity guarantee and must not diverge. The application layer only fetches the reference data and hands over the resolvers.
- **A dedicated-boss battle is marked, not inferred.** A boss-battle flag (plus the challenged zone id) is captured at battle start and threaded through the completion event, so the resolve path knows a victory was a dedicated-boss clear rather than inferring it from the enemy + current zone. Only the challenge path clears a zone: `ZonesCleared` counts distinct zones ever cleared (a binary per-zone flag, the global counter bumping only on a zone's first clear), while the `BossesDefeated` farm counter increments on every dedicated-boss victory (see the Zone Clears section in game-design.md).
- **The active idle-loop mode is persisted for offline rewards.** A boolean `Player.AutoChallengeBoss` records what the idle loop is doing — `false` is idle-farming `CurrentZoneId`, `true` is auto-challenging **that same zone's** boss (the loop never targets a separate zone, so no boss zone is stored) — so the offline-rewards simulation (spike [#879](./spikes/879-offline-rewards.md)) can resume the correct loop at next login. The frontend syncs it from its live auto-fight state via the `SetAutoChallengeBoss` socket command (anti-cheat validated like a boss challenge: the current zone must be in circulation, unlocked, and have a boss), and it rides the existing core write-behind path. As a backstop mirroring the online auto-fight-off, a recorded dedicated-boss **loss or draw** resets the persisted mode to idle, so it tracks the player's present loop even if a frontend sync is missed.
- **An abandon that resolved as a win is a full victory.** An abandon (reached only when the player starts a new battle while one is active) re-simulates the in-progress fight bounded by how long the **client actually simulated** it (sent as `clientBattleMs` on the superseding `NewEnemy`/`ChallengeBoss`), **clamped to `[0, server-elapsed wall-clock]`** and then to **`DefaultMaxBattleMs`**. Bounding by the client's reported duration is what keeps the abandon accurate: a battle the client never fought (e.g. a server-prefetched next battle discarded by a zone/build change during the cooldown) reports `0`, so the replay runs no ticks and **records no outcome** rather than minting a phantom one from the post-cooldown wall-clock gap. The wall-clock cap still bounds it from above, so a won-abandon can't be claimed faster than real time allows (and the `DefaultMaxBattleMs` cap keeps a reported 2-minute stalemate a draw — a battle never runs past its cap on the client, so the replay must not either). If the enemy died within that window it pays out through the same victory helper as a normal win — exp **and** the win/zone-clear/challenge credit — so the two win paths can't drift. An abandon where the player died within the window is a loss; one where neither died is an _incomplete abandon_ — a draw, including a live timeout-draw — tracked as its own statistic rather than a loss (see the [battle-outcomes](./game-design.md#battle-outcomes-battleswon--battleslost--battlesabandoned) split). Whichever of the two it resolves to, the abandon applies the same post-battle cooldown `DefeatEnemy`/`BattleLost` would have (anchored to the moment of resolution) — otherwise a client that never sends those commands and instead only ever abandons via `NewEnemy`/`ChallengeBoss` could farm every kill's pacing cooldown away. The replacement battle `StartBattle`/`StartBossBattle` spawns in that same call is anchored to that cooldown's expiry rather than to `now`, so the cooldown just incurred actually paces the next fight (mirroring the prefetch bullet below) instead of being immediately bypassed. `StartBossBattle` anchors to *any* cooldown still in effect, not only one its own abandon just incurred (`#1920`): the always-available boss challenge has no cooldown *gate* on the request itself (unlike `NewEnemy`, gated on `IsOnCooldown` before it ever reaches `StartBattle`), so a normal (non-abandoned) victory/loss cooldown still running from the previous battle, or a challenge landing mid an already-running cooldown with nothing to abandon, must anchor the same way or the pacing cooldown is skipped entirely. `NewEnemy`/`ChallengeBoss`'s response carries that same cooldown alongside the fresh `EnemyInstance` (`#1881`/`#1884`) so the client waits it out before presenting the enemy/boss, rather than starting its battle clock ahead of the server's anchor. The still-in-progress hand-back (below) is unaffected — no outcome, no cooldown.
- **Victory-end validation and the post-battle cooldown use the server clock only — never a client-supplied timestamp.** A `DefeatEnemy` claim is accepted only when enough real server time has elapsed since battle start for the replay's duration to have completed (`now − BattleStartTime ≥ TotalMs − one tick` of slack for mid-tick start alignment). Network latency only ever *delays* a claim, so a legitimate win always satisfies this; being server-clock-only it is immune to client/server clock skew (an earlier client-timestamp check rejected legitimate wins whenever the two machines' clocks differed by more than a tick). The post-battle cooldown is anchored to the battle's **server-computed completion** (`BattleStartTime + TotalMs`), not to `now` — the gap between the two is the post-victory network latency, so subtracting it keeps the idle farm rate latency-independent without trusting the client clock. The client still reports its simulated duration as `clientTotalMs`, but **purely as a diagnostic** the server logs when it diverges from its own replay — it gates nothing. `BattleLost` shares the identical gate (`ClaimedBeforeBattleCouldFinish`, #1630): without it a tampered client could claim an instant loss and still have the full replayed statistics booked, farming statistic/challenge accrual at an accelerated rate.
- **The next idle battle is prefetched and bundled with the battle-end response.** To hide the `NewEnemy` round-trip latency under the post-battle cooldown, an idle victory (`DefeatEnemy`) and a boss loss (`BattleLost`) set up the *next* idle battle immediately (`PrepareNextIdleBattle`) and return it in their response, so the client begins it the instant the cooldown elapses with no extra request. Its `BattleStartTime` is anchored to the **scheduled cooldown expiry** (`EnemyCooldown`), not `now` — the cooldown is deterministic, so the next fight begins exactly when it elapses; anchoring there keeps the elapsed-time check above correct (latency only delays the claim past the scheduled completion) and the *following* cooldown correctly sized — anchoring to `now` would back-date the start by the whole cooldown and shrink (or zero) it. The bundled player snapshot is frozen at the previous battle's end, so the client uses the bundle **only while the zone and its battle-relevant state are unchanged** since then; a zone or build change during the cooldown falls back to a fresh `NewEnemy` whose snapshot matches what the client now simulates, preserving parity (and re-syncing the new zone). That fresh fetch supersedes the unfought prefetched battle, which the client reports as `clientBattleMs: 0` so the abandon records no phantom outcome (see the abandon bullet above). The boss-victory path is excluded — it paces itself via the Zone-Cleared overlay.
- **The battle `seed` is reserved for the simulation's RNG, not for setup.** It is generated at battle start from a **non-time entropy source** (a cryptographic RNG, not wall-clock ticks — a wall-clock seed is low-entropy and predictable between battles, unfit as the shared starting point for the parity-identical crit/dodge RNG), persisted on the player state, and sent to the client as the shared starting point for randomness _inside_ the simulation. Setup (level roll, skill selection) deliberately uses ambient randomness so it never advances the seed's stream — otherwise the server (drawing during setup) and the client (which receives the loadout ready-made) would begin the simulation from divergent RNG states. (The boss path is already deterministic, so this only matters for the random idle encounter.) The seed is threaded `PlayerState.BattleSeed → BattleService.SimulateBattle → BattleSimulator`, which constructs one `Mulberry32` and hands it to `BattleContext`; the frontend mirror seeds from `EnemyInstance.Seed`. Its first live consumer is the seeded crit/dodge runtime below — the `Mulberry32ParityTests` ↔ `mulberry32-parity.test.ts` vector still pins the two RNG ports against drift.

## Battle runtime — skill effects (timed buffs/debuffs)

Skills can carry authored effects that apply as **timed attribute modifiers** when the skill fires. Because battle logic runs on both sides, every rule below executes identically on `Game.Core/Battle` and its frontend mirror and is pinned by the mirrored parity matrix and unit tests.

- **Effects ride the real attribute graph.** Applying an effect adds a genuine attribute modifier to the target's live attribute collection and expiry removes it, so derived attributes cascade for free — a temporary Strength buff raises MaxHealth exactly like a permanent one. This is the riskiest parity surface: both sides must compose attributes through their one shared path, **including mid-battle add/remove**. Float-identical results rely on modifier ordering matching — a new modifier inserts at the end of its type-run (additives before multiplicatives) on both sides.
- **Effects fire after dealing damage** (self-effects route to the active battler, opponent-effects to the target), so a self damage-buff never boosts its carrying hit, while an earlier loadout slot's effect does affect a later slot firing the same tick.
- **Magnitude scales off the caster.** An effect's applied magnitude is `Amount + casterAttribute × ScalingAmount`, read from the **active (casting) battler** regardless of which battler it lands on — the same caster-scaling shape as a `DamageMultiplier` (`BattleContext.ApplySkillEffect` computes it and hands the resolved amount to `Battler.ApplyEffect`). A `ScalingAmount` of `0` is byte-identical to the unscaled authored amount, so this addition leaves existing effects unchanged. The resolved magnitude is frozen on the modifier at apply time; it does not re-scale as the caster's attributes change later.
- **The battler owns the active-effect bookkeeping.** Expiry is keyed to an absolute per-battler simulated-time clock advanced at the **start of each tick**, before any skill fires, so an effect influences exactly `DurationMs / MsPerTick` ticks (counting the application tick). Re-applying an active effect **stacks the magnitude** so magnitudes sum (additive amounts add, multiplicative factors compound), but all active applications **on the same attribute share one expiry** that **resets to the newest application's duration** (extending a longer-lived application or cutting a shorter one short); there are no independent per-portion expirations (#992). Applications on different attributes are independent, and **stacking is unbounded in magnitude but bounded in cost**: each application folds into the attribute's **single combined modifier per modifier type** rather than adding its own, so the modifier set and the active-effect list stay bounded by the affected-attribute count no matter how deep a buff re-applies — a persistently re-applied buff that never lapses keeps a battle O(ticks) instead of the O(ticks²) a per-application list produced. Because the fold changes the float grouping (`base + (a₁+a₂+…)` rather than `base + a₁ + a₂ + …`), both simulators must fold **identically** for the replay to stay bit-for-bit; this is parity-critical. `CurrentHealth` is left untouched when MaxHealth rises (no free heal) and clamped down when it drops. (The frontend keys expiry to the same absolute per-battler clock, deriving its render-facing `remainingMs` from it each tick rather than decrementing a countdown independently — #1705.)
- **No per-tick allocations.** The expiry pass early-returns when a battler has no active effects (still advancing the clock so a later effect dates its expiry correctly) and otherwise iterates without allocating; the per-attribute stack list is created lazily on first apply.

The per-second accumulators are consumed by an **end-of-tick DoT/HoT phase**, reached only while both battlers still live and after both their skills resolve. DoT is **typed** (spike [#1320](./spikes/1320-damage-types.md)): three accumulators — `BleedDamagePerSecond`, `PoisonDamagePerSecond`, `BurnDamagePerSecond` — replace the former single `DamageTakenPerSecond`, the type encoded by **which accumulator an effect targets** (no field on the effect). HoT (`HealthRegenPerSecond`) stays typeless. For each battler the typed DoT (which **bypasses the Toughness curve**, unlike a normal hit) then HoT (capped at MaxHealth) apply before its death check, so a heal-over-time can save a battler from an otherwise-lethal DoT tick (#1090). The enemy resolves first, so an enemy its own same-tick regen cannot save dies before the player's DoT applies (a same-tick mutual DoT kill favours the player). DoT on the enemy counts as player damage dealt; DoT on the player as damage taken; the player's post-cap healing accumulates into the damage-healed statistic. This phase is parity-critical and mirrored on the frontend.

The typed DoT phase loops the three types in the fixed `DamageTypes.DotAccumulators` order, applying each type's resistance through the same `applies(type) → keys` map as direct hits but **sampled live off the bearer** each tick (so a vulnerability debuff hurts existing DoTs immediately) — `Σ perSec_t × 40/1000 × (1 − Σ_{k ∈ applies(t)} bearer[kResistance])`. The caster's amplification is instead **frozen** into the accumulator at apply time (`BattleContext.ApplySkillEffect`), consistent with how the magnitude's caster scaling already freezes. DoT keeps **bypassing the Toughness curve**, so resistance is its only mitigation; each type's own tick's **health math is not floored** (negative resistance — vulnerability — and resistance above `1` — absorption — fall out for free), but the per-type offense booking is floored at `0` and capped at the health the tick actually removed (`Battler.HealthRemoved`, #2108) — an absorbed tick heals the bearer without booking negative typed damage. But the **aggregate** health change the phase realizes across all typed ticks **is capped at MaxHealth** when the summed total is a net heal — matching the reactive direct-hit absorption and the HoT cap (the game has no overheal/shield concept). A single typed DoT with zero resistance is byte-identical to the former single-accumulator outcome (the reduce-to-today identity), pinned by the parity matrix.

## Battle runtime — seeded crit / dodge / parry (player only)

The first live consumer of the seeded battle RNG (above). Only the **player** crits, dodges, and parries in this version; enemies do not, and the gating is purely on the acting battler — a clean seam for later enemy parity (note: enemy parry would add draws on player fires, unlike reflection whose determinism made enemy symmetry free). Like the rest of the runtime this executes identically on both sides (`BattleContext.DamageTarget` ↔ `battleStep`) and is pinned by the parity matrix; the seeded scenarios force each chance to `1`/`0` so the outcome is deterministic without depending on a draw, plus fractional-chance rows that pin the real draw positions.

- **Draw order is a pure function of the skill-fire sequence**, never of a roll outcome, so the per-tick draw count is fixed regardless of stats: each **player** skill fire takes **1 crit draw** (before its damage); then, if the enemy survives, each **enemy** skill fire takes **3 draws in fixed order — parry, dodge, and the parry counter's crit (#1457)** — all taken unconditionally (even when nothing can proc, and the counter-crit draw even when no parry procs or the defender has no counter skill). So the count is `playerFires×1 + enemyFires×3` (Block's former second draw was retired with the mechanic, spike #1330). A pure-buff (zero-damage) fire still draws. This decoupling from values and outcomes is the strongest parity guarantee, and is where #336's deferred skill-effect proc chance will later slot its own draw.
- **The math is player-only, gated on who acts.** *Crit:* `c < skill.CriticalChance × CriticalChanceMultiplier` ⇒ the raw damage is multiplied by `CriticalDamage` (read directly, like `CooldownRecovery`) **before** mitigation, so high crit damage punches through the Toughness curve. *Parry:* `p < ParryChance × ParryChanceMultiplier` ⇒ the hit deals `0` and the riposte fires (below); parry is checked **before** dodge and takes precedence, so dodge investment never starves the riposte. *Dodge:* `d < DodgeChance × DodgeChanceMultiplier` (dodge rework #1523) ⇒ the hit deals `0`. Chances are decimal fractions compared directly against a `[0,1)` draw.
- **The riposte is a first-class player fire through the shared pipeline (#1457).** On a parry, the defender strikes back with its **counter skill** — the equipped weapon's signature (`weapon?.GrantedSkillId ?? PunchSkillId`), resolved **once at battler assembly** like the weapon-match gate (`BattleSnapshot.ToBattler` ↔ the live `InventoryManager.counterSkillId`; an unresolvable id leaves no counter, so the parry negates without a riposte). The counter routes through the same player-hit resolution as a deliberate fire (`BattleContext.ResolvePlayerHit` ↔ `battle-step`'s `resolvePlayerHit`): amplification, the signature's own authored `CriticalChance` (its crit decision consuming the already-taken third draw), the execute multiplier, the overlay share claims, typed offense booking, and the enemy's reflection of the counter all apply. It is **damage-only** — no effects applied, no cooldown touched, no per-skill use recorded — and, like a dodged hit, a parried hit still applies the enemy skill's own effects (negation covers damage, not riders). A counter can kill; the existing after-turn death checks cover it (the same seam reflection kills use).
- **Riposte trains on the counter damage dealt.** The counter's net books into `BattleStats.PlayerCounterDamageDealt` — the `Parry` activity key's **direct** signal, like Retribution's reflected-damage tally rather than an overlay share claim — alongside the parry statistics (`AttacksParried`, `DamageParried`, the avoided post-mitigation damage mirroring `DamageDodged`).
- **Chance sourcing — dodge and crit both opt-in-multiplicative (dodge rework #1523; crit rework #1425, per-skill base #1453).** `DodgeChance` is base `0` and **authored-only** — granted by items/item-mods/skill-effects, never derived from a core attribute — the same committed-enabler template as `ParryChance`; `DodgeChanceMultiplier` only **scales** that authored chance and carries a base of `1` (like `CooldownRecovery`) plus an Agility derivation (spike #1426 — AGI the tempo & evasion amplifier; `ParryChanceMultiplier`/`CriticalChanceMultiplier` get the analogous Luck pair), so the opt-in holds because the multiplier scales a `0`-based enabler (`0 × mult = 0`) — an un-enabled build sits at exactly `0` dodge regardless of Agility. Crit's enabler is each `Skill`'s own authored `CriticalChance` (a raw base probability, defaulting to `0` — never crits unless authored); the `CriticalChanceMultiplier` attribute only scales that per-skill base with the same base-`1` + Luck-derivation shape. `CriticalDamage` keeps its Luck derivation and base — inert without a crit, so it pays off the moment an authored skill fires. The coefficients are a conservative strawman to be tuned during balancing, and `DodgeChance`/`CriticalChanceMultiplier` are also grantable by gear/mods/skill-effects through the normal modifier path. The frontend `SkillActivation` carries `crit`/`dodged` flags for the combat log (#800); the headless simulator and the backend ignore them, but the rolls and damage results stay identical across both sides.
- **Cadence is the fourth enabler × multiplier pair — composed at the charge site, not a draw (spike #1426).** `CooldownRecovery` was severed from the core attributes: it keeps its `1.0` base but no derivation, and the committed cadence channel is `CooldownBonus` (base `0`, authored-only) × `CooldownBonusMultiplier` (base `1`, Agility-derived, the Frequency path's target). The charge site (`Battler.GetCooldownMultiplier` ↔ `cooldownMultiplier`) reads the **effective rate** `CooldownRecovery + CooldownBonus × CooldownBonusMultiplier` each tick — a deterministic read (no RNG draw), so it adds no draw to the fixed count above, but it is **parity-critical** (it drives the fire cadence and thus every downstream tick) and must be composed identically on both sides. Like the proc multipliers, `0 × mult = 0` keeps a no-enabler build charging at exactly `CooldownRecovery` regardless of Agility.
- **Precision trains on the crit share claim ([#1448](./spikes/1398-utility-in-proficiency-system.md), reshaped by #1481).** The player-facing `CriticalDamageDealt` statistic stays the actual crit damage dealt (`totalNet`), but the Precision proficiency signal is a **separate** `CriticalBonusDealt` tally: per crit, the hit's **booked** damage (post-mitigation, capped at the health actually removed — #1482) × `φ(m − 1)`, where `φ(a) = a / (1 + a)` (`Battler.NormalizeInvestment`) and `m = CriticalDamage`. This is the **uniform overlay tally**: every overlay books `booked × φ(its own investment)` (#1481, superseding the earlier counterfactual normalized-marginal shape — see the spike's *Second refinement*). Strength-proportionality lives in `φ` (a `1.55×` and a `1.9×` crit train differently, which the full-hit tally didn't do), and because the booked basis sums to at most the enemy's health pool per won battle, per-battle claims are enemy-authored — stacking overlays pays out in kill speed, not tally inflation, and no tally runs a counterfactual curve evaluation. It is **backend-only**: the frontend simulator computes no battle stats and the statistic value ships from the server, so there is no parity mirror and the seeded health math above is untouched. Guarded like every other overlay (#1927): a `CriticalDamage` debuff at or below `1` (`m − 1 ≤ 0`) is a non-positive investment and books nothing, rather than a negative claim or, at `m = 0`, `φ`'s pole at `a = −1`.

- **Hex trains on the vulnerability share claim ([#1427](./spikes/1398-utility-in-proficiency-system.md), #1481).** A vulnerability enabler is a negative-resistance debuff on the opponent. To credit the *attacker's own debuff* — the work it did, not merely the net resistance state — each opponent-applied additive resistance debuff accrues its signed delta onto the target's effect stack (`ApplyEffect(tracksVulnerability)`), so `v = AppliedVulnerability(type)` is the resistance the attacker removed, independent of the target's base resistance **and** of any resistance the target buffs on itself (a self-buff never lowers `v`); riding the shared per-attribute stack expiry, `v` returns to `0` for free when the debuff lapses. Each direct hit and DoT tick then books its booked (health-capped) damage × `φ(v)` (`Battler.HexBonusForHit`) — accrued only on the player's offense. Backend-only like the crit bonus; the stack contribution and the `ApplyDamageOverTime` recorders never touch the health math, so they add no parity surface.

- **Momentum trains on the ramp share claim ([#1428](./spikes/1398-utility-in-proficiency-system.md), #1481).** A ramp enabler is a **stacking self-buff** to one of the attacker's typed amplification attributes (e.g. `FireAmplification`), authored as an ordinary skill effect — `ApplyEffect`'s existing stacking/refresh rules (magnitudes sum, expiry refreshes) already give a faster-casting build higher stacks for free, so no new engine mechanism is needed for the enabler itself. Each Self-targeted additive amplification effect accrues its magnitude onto the caster's own effect stack (`ApplyEffect(tracksMomentum)`), so `r = AppliedMomentum(type)` is exactly the ramp's own contribution to that type's amplification, isolated from any static (item/base) amplification the attacker already carries. Each direct hit of a ramped type books its booked damage × `φ(r)`. Backend-only, and scoped to direct hits: a DoT's frozen amplification already includes whatever ramp was active when the DoT was cast, but the tally itself does not extend to DoT ticks.

- **Sunder trains on the Toughness-debuff share claim ([#1429](./spikes/1398-utility-in-proficiency-system.md), #1481).** The anti-mitigation counterpart to Hex — an opponent-targeted additive `Toughness` debuff, tracked via `ApplyEffect(tracksSunder)` onto the target's effect stack so `s = AppliedSunder()` is the Toughness the attacker's debuff removed. Each direct hit books its booked damage × `φ(s / C)` — the investment made dimensionless by the mitigation curve's own characteristic magnitude (`C` = `GameConstants.ToughnessMitigationConstant`), the same constant the live curve divides by. Sunder pioneered the no-counterfactual shape (the nonlinear Toughness curve has no target-flat marginal); #1481 extended it to every overlay. **Direct-hit only**: DoT bypasses the Toughness curve entirely, so a Toughness debuff cannot affect it — unlike Hex's resistance debuff, there is no DoT counterpart. Backend-only like the other overlays.

- **Cull trains on the execute share claim, and is the one overlay whose enabler is real damage-calc logic ([#1430](./spikes/1398-utility-in-proficiency-system.md), #1481).** Unlike Hex/Sunder/Momentum — whose enablers ride the existing resistance/amplification/Toughness attributes and only train a backend-only signal — Cull's enabler, `ExecuteBonus` (authored-only, base `0`, like `DamageReflection`), scales the **real** damage dealt: on a player fire, before mitigation, the target's missing-health fraction (`(MaxHealth − CurrentHealth) / MaxHealth`, clamped `[0,1]` and sampled **once per fire** — like the crit draw, so an earlier portion's damage within the same multi-typed hit cannot change a later portion's bonus) scales `ExecuteBonus` into that fire's multiplier `1 + ExecuteBonus × missingHpFraction`, applied to the raw damage identically to `CriticalDamage` and composing multiplicatively with it. This makes Cull **parity-critical beyond its tally** — `BattleContext.DamageTarget` ↔ `battle-step`'s `dealPortionedDamage` must compute the accelerating multiplier identically, pinned by the `forcedExecute` parity scenario. The `CullBonusDealt` training signal is the uniform share claim: per portion, the booked damage × `φ(investment)` where `investment = ExecuteBonus × missingHpFraction`. A target at full health enables nothing; DoT has no counterpart (there is no per-tick "hit" to execute) — and because the basis is capped at the health actually removed (#1482), an execute one-shot claims only the sliver of health that existed, so overkill mints no Cull training.

- **Frequency trains on the cadence share claim — a pseudo-overlay ([#1426](./spikes/1426-attribute-identity-sourcing.md), #1527).** Unlike the true overlays above — whose enabler is a per-hit debuff/ramp/execute state — cadence's enabler is the **whole build's charge rate**: faster cycling *is* more hits, so it rides **every** landed player hit. Each player fire books its booked (health-capped) damage × `φ(investment)`, where `investment = effective cadence − 1` (`Battler.GetCooldownMultiplier − 1` = `CooldownRecovery + CooldownBonus × CooldownBonusMultiplier − 1`), **sampled once per fire** like the execute investment — a timed `CooldownBonus` effect therefore amplifies the claim only for the hits it enabled. An uncommitted build sits at exactly the base-`1` rate, so `investment = 0` and it books nothing; difficulty is self-supplied (the landed basis is bounded by the enemy's health pool, power sits in the accrual denominator). The `CadenceBonusDealt` signal is **backend-only** like the other overlay tallies — the effective-rate read it samples is already parity-pinned as the tick-loop charge rate (above, spike #1426), so the booking itself adds no new parity surface. Direct-hit only, including the parry riposte, which routes through the same player-fire pipeline.

## Battle runtime — deterministic damage reflection (spike #1330)

Replaced the old Block mechanic. On a **direct hit**, after the full mitigation pipeline below, the defender returns its `DamageReflection` share of the net damage to the attacker — `attacker.health −= net × defender.DamageReflection` — **bypassing the attacker's own mitigation** (otherwise it would double-dip to nothing). It is **deterministic** (no draw, so it never touches the RNG stream), applies in **both** directions wherever authored, and is **scoped to direct hits**: only a positive `net` reflects (a dodged or absorbed hit returns nothing), and DoT is never routed through it. `DamageReflection` is **authored-only** — base `0` with no `StaticAttributeModifiers` derivation, granted solely by gear/mod/proficiency/class — so an enemy reflects only where a designer authors it. Reflected damage can kill the attacker, so the simulator checks the attacker's death immediately after each side's turn (and the frontend gates the enemy's fire on both battlers being alive), enemy-death-first so a reflection kill wins the tie. Reflected damage is booked into the player's damage-dealt (player reflects onto the enemy) or damage-taken (enemy reflects onto the player) totals. Parity-critical and mirrored (`BattleContext.ReflectDamage`/`Battler.TakeReflectedDamage` ↔ `battle-step`'s `reflectDamage`/`Battler.takeReflectedDamage`).

## Battle runtime — damage typing (direct-hit amplification & resistance)

Every skill carries a leaf `DamageType` (spike [#1320](./spikes/1320-damage-types.md); eight leaf types — `Physical`, the four elementals, and the three DoT forms — with existing skills backfilled to `Physical`). Direct hits route their type through one shared lookup, the codegen'd `applies(type) → keys` map (`Game.Core.Attributes.DamageTypes` ↔ `$lib/battle/damage-types`), which resolves a hit to the attacker's amplification and the defender's resistance attributes — the leaf type itself plus any cross-cutting category (`Elemental`, `DoT`). This is parity-critical and mirrored: `BattleContext.DamageTarget`/`Battler` ↔ `battle-step`/`battler`, with the `applies()` iteration order fixed so the additive amp/resist sums fold bit-for-bit identically on both sides.

The direct-hit pipeline, attacker A → defender D, runs in this order (it both amplifies and resists in **both** player→enemy and enemy→player directions):

1. **Amplification** (attacker, additive): `dealt = raw × (1 + Σ_{k ∈ applies(type)} A[kAmplification])`.
2. **Crit** stays attacker-side and pre-mitigation (player-only, unchanged from the crit/dodge section above), so high crit damage still **punches through** the Toughness curve.
3. **Resistance** (defender, additive, **unclamped**): `mitigated = dealt × (1 − Σ_{k ∈ applies(type)} D[kResistance])`. A negative total (`res < 0`) amplifies the hit — **vulnerability**; a total above `1` (`res > 1`) drives `mitigated` negative — **absorption**, a net heal.
4. **Toughness curve last, and only on a positive hit** (spike [#1330](./spikes/1330-mitigation-rework.md), constant denominator #1487): when `mitigated > 0`, multiply by `1 − Toughness / (Toughness + C)` (a diminishing-returns reduction — EHP linear in `Toughness`, asymptoting below 100%, with `C` = `GameConstants.ToughnessMitigationConstant` the fixed half-point so a Toughness investment retains its mitigation % across progression). With Block's flat reduction now removed, this is the **only** mitigation step here and the stack is purely multiplicative — a positive hit can never be driven negative, so there is **no flat floor and no absorption-safe branch around one** (the branch the damage-types spike added vanished when the flat step it guarded was retired). When `mitigated ≤ 0` the absorbed heal passes through (mitigation can **neither heal nor deepen an absorption**) but is **capped at `MaxHealth`**, consistent with heal-over-time: the game has no overheal/shield concept, so the reactive absorption channel and the passive HoT channel behave alike. DoT still bypasses the Toughness curve entirely. The mitigation expression (`Battler.TakeDamage`/`ComputeNetDamage` ↔ `mitigateDamage`/`toughnessMitigatedDamage`) is parity-critical. A direct hit then feeds the deterministic reflection above.

**Weapon-leaf mechanics (spike #1340).** The martial weapon leaves (`Sword`/`Axe`/`Bow`/`Club`/`Dagger`/`Unarmed`, sitting under the `Physical` category key — design model in [game-design.md → Damage types](./game-design.md#damage-types)) ride this same pipeline with two mechanical wrinkles. A weapon key is **amplification-only**: there is no per-weapon resistance, so `DamageTypes.KeyInfo.Resistance` is nullable and a weapon hit resolves to `PhysicalResistance` via the shared `Physical` key (per-weapon resistance is a deferred opt-in). And the enums grow **append-only**, so a weapon leaf no longer shares a numeric value with its key (the `EDamageType.Fire === EDamageTypeKey.Fire` coincidence holds only for the original #1320 leaves); a leaf type's own key is recovered as `Applies(type)[0]` (always first), which **both simulators rely on**. Offense proficiency routing reaches the weapon paths through the same `Applies → ActivityKeys.ForDamageKey` fold (a sword hit trains the weapon path **and** the `Physical`/Martial path).

Amp/resist are **separate multiplicative budgets** (`× (1 + amp)` then `× (1 − res)`), never a single `(1 + amp − res)` — resistance must not cancel amplification 1:1. The amp/resist attributes default to `0` from every source, and `Toughness` defaults to `0`, so with no typed content and no Toughness both factors are an exact `1.0` and the curve is a no-op — the pipeline reduces to `raw` (the reduce-to-today identity, pinned by the parity matrix). Typed damage-over-time (the per-type accumulators) reuses the same `applies()` map in the end-of-tick DoT/HoT phase above.

### Multi-typed direct hits — the weighted portion split (spike [#1343](./spikes/1343-multi-typed-damage.md))

A skill's direct hit can carry **multiple leaf types with a split**: it owns a list of `SkillDamagePortion { Type, Weight }`, and **a multi-typed hit is the single-typed hit run once per portion and summed**. The one `raw` number (`BaseDamage + Σ multipliers`) is computed once and split `raw × weight ÷ Σweights` per portion; each portion runs the *unchanged* single-type pipeline above (amplify → crit → resist → Toughness) under its own type, and the per-portion nets are summed into `totalNet`. There is no new damage math — only a loop, kept off the RNG stream. This executes identically on `BattleContext.DamageTarget` ↔ `battleStep` (the `dealPortionedDamage` helper) and is pinned by the parity matrix.

- **Zero new RNG draws (the load-bearing parity invariant).** The per-fire draw count stays the fixed budget above (`playerFires×1 + enemyFires×3`), independent of portion count. A **single** crit decision per player fire multiplies *every* portion (`crit × Σ portions = Σ (crit × portion)`); a **single** parry/dodge decision zeroes the *whole* hit. A non-crit multiplies by an exact `1.0`, so it is byte-unchanged.
- **The weight total is folded in fixed portion order** (float addition is not associative), and the portions are iterated in that same fixed order — both are parity contracts, exactly like the `applies()` iteration order.
- **Per-portion `TakeDamage`, summed net, reflect once.** Each portion calls the unchanged `TakeDamage` sequentially, so vulnerability/absorption and the curve are per-portion for free. Reflection runs **once** on `totalNet` (it is linear and untyped, so once equals per-portion and gives a single clean reflection event). Each portion's absorption heal is capped at the `MaxHealth` room **at that point in the fixed order**, so the per-portion absorption cap is order-dependent (a deliberately fixed, parity-pinned behaviour, reachable only through authored absorption — resistance > 1 — on one portion of a multi-typed hit).
- **Per-hit stats are whole-hit; typed books are per-portion.** `HighestPlayerAttack`, `CriticalDamageDealt`, and the per-skill `RecordSkillUse` total use `totalNet` (one swing is one attack, deliberately uncapped — overkill is real feedback). In-battle `BattleStats` stay signed, but the Sum-aggregated lifetime damage statistics (`DamageDealt`/`DamageTaken`/`CriticalDamageDealt`) are floored at `0` at the recording seam so authored absorption can't decrement a lifetime total (#2143). `TypedDamageDealt` (per portion's net **capped at the health the portion actually removed** — a killing swing's overkill tail, and every later portion of it, books nothing, so the offense book's per-battle total is bounded by the enemy's health pool; the DoT recorder caps the same way across the fixed accumulator order — #1482) and `TypedDamageExposure` (per portion's pre-resist dealt) accumulate **per portion** under each portion's type — the cross-school proficiency signal, with no `ProficiencyXpCalculator`/accrual change. `TypedDamageResistanceMitigated` (`Battler.TypeResistanceMitigated`, #1454) accumulates alongside exposure — the resistance-only (Toughness-excluded) slice each portion's own type-resistance blocked — a backend-only side channel the resist-training split weights separately from the still-landed share; no accrual routing change beyond that weighting. The Precision signal `CriticalBonusDealt` (#1448/#1481) sums each portion's **booked** damage before applying `φ` once to the whole-hit investment, so a multi-typed crit still books one claim.
- **Reduce-to-single-portion identity.** With one portion at weight `w`, `raw × w ÷ w = raw`, the loop runs once, and the whole path is byte-for-byte the single-type hit — so the existing parity matrix is undisturbed and only multi-typed content changes outcomes. The display surfaces (floater/tooltip colour) read the derived `PrimaryDamageType` (highest-weight portion, lowest-numbered type on a tie). DoT stays per-effect (multiple effects for multiple DoT types), unchanged by this split.

## Offline rewards simulation

When a player returns after being away, `OfflineProgressService.SimulateOfflineProgress` (driven by the `GetOfflineProgress` socket command) replays the missed idle/boss battles and applies the rewards inline — the away window is bounded (≥5-minute floor, 10-hour cap) so the worst case sits well under the command timeout, and the command's cancellation token is threaded into the simulation loop. `OfflineProgressService` and the live `BattleService` share zone viability/relocation and boss-zone resolution through `ZoneResolutionService` (#1516), so the two loops can never resolve a zone differently. The player-facing mechanic is in [game-design.md](./game-design.md#offline-rewards); the path-specific points:

- **The simulation is parity-irrelevant.** The client never replays these battles — the backend is the sole authoritative simulator (`OfflineProgressSimulator`, pure `Game.Core`, reusing `BattleFactory`/`BattleSimulator`/`DefeatRewards`) — so there is no parity surface to mirror. Results stay consistent with online battles by reusing the shared simulator.
- **The loop mode + zone come from the durable aggregate, not the session.** `Player.AutoChallengeBoss` (+ the current zone) selects idle vs. boss; a retired/locked/bossless boss target falls back to idle, and a non-viable idle zone is lazily relocated the same way the live idle loop relocates.
- **A stale in-flight battle is settled first — or handed back still active (spike [#1073](./spikes/1073-idle-time-recovery.md), #1595).** A mid-battle disconnect leaves an active `PlayerState` battle; `BattleService.AbandonBattle`/`ResolveStaleBattle` resolve it exactly like an abandon before the away window simulates, so a concluded outcome (win/loss/genuine draw at the 2-minute cap) is credited once rather than re-abandoned by the first live `StartBattle` after the welcome-back gate. When the replay *doesn't* conclude and real elapsed time since `BattleStartTime` is still under that cap, the battle is genuinely still in progress rather than a stalemate — it is handed back unchanged (same enemy/seed/snapshot) with its elapsed offset instead of being booked as a phantom draw, and the away-window simulation is skipped entirely for that call (there is no settled outcome or leftover budget to spend, and `Player.LastActivity` is left untouched so the away clock keeps counting against the original disconnect). `NewEnemy`/`ChallengeBoss`/`GetOfflineProgress` all surface this the same way: the `EnemyInstance` they return carries a non-null `ElapsedOffsetMs`, plus the dedicated-boss flag (#1689) so the client resumes the correct loop (boss vs. idle) around the resumed fight. The client fast-forwards through the offset headlessly via `BattleEngine.replayToOffset` (#1597 — see [frontend.md](./frontend.md#battle-simulation--parity)). When the stale battle *is* settled with a credited outcome, `ResolveStaleBattle` also reports that battle's own duration (`StaleBattleResolution.SettledBattleMs`); `OfflineProgressService` deducts it (plus the post-battle cooldown) from the away budget the simulator below is about to spend, since that span already falls inside the same away window and would otherwise be credited twice (#1882).
- **Mid-window level growth (spike [#1073](./spikes/1073-idle-time-recovery.md) → #1253, #1601).** The simulator tracks cumulative exp and level in-loop (`OfflineSimulationParameters.StartingExp` + the snapshot's captured level), running the same clamp/threshold loop `Player.ApplyExp` does live via the shared `Game.Core.Players.ExpProgression.ApplyExp` helper — extracted so the curve can't drift between the two paths. A level crossed between battles (exp lands on victory, matching when it would live) re-derives the battler used to measure `DefeatRewards`, so the class locked base, the signature passive, and the reward power all grow across the away window exactly as they would live; the anti-grind treadmill decays with them instead of holding at its window-start value for up to 10 hours. `Player.GrantOfflineExp` remains the sole authority for actually applying the window's exp to the aggregate, post-loop — the simulator's own in-loop level/exp (`OfflineProgressResult.EndingLevel`/`EndingExp`) is pinned to agree with it exactly, since both run the identical shared loop. What still requires the player's hand — stat allocations, gear, loadout — stays frozen on the captured `BattleSnapshot` for the whole window; proficiency levels no longer do (next bullet, #1602).
- **Mid-window proficiency growth (#1602).** The proficiency-XP accrual math (frontier routing, XP claim/leveling, milestone/open triggers) is `Game.Core.Proficiencies.ProficiencyAccrual` — extracted from the application-layer `ProficiencyRewardService` (now a thin adapter over it for the live path) so the simulator can run the identical function **in-loop**, immediately after each victory, against its own working copy of the snapshot's captured `ProficiencyLevels` (deep-copied per run so growth never leaks onto the caller's frozen snapshot). A milestone crossed early in the window is therefore already baked into every later battle's attribute bonuses — unlike the pre-#1602 behavior, where the whole window fought at window-start proficiency levels because the accrual ran only after the simulation concluded. The rating battler measuring `DefeatRewards` (and through it every path's own `ratingDenominator`, spike #1526 Decision 5) is re-derived whenever a proficiency level-up actually changes an attribute modifier, exactly like a player level-up already does. Reward-skill grants are reported only (`ProficiencyXpResult.GrantedSkillIds`) — the pure accrual has no `Player`/`ISkills` dependency; `OfflineProgressService.ApplyOfflineRewards` folds each battle's already-computed result (`ProficiencyGainAccumulator`, unchanged) and applies the window's final per-proficiency level/XP plus skill grants once, through the same `ProficiencyRewardService.GrantRewardSkills` step the live path uses.
- **Reward application is consolidated and shares the live per-battle path.** Each simulated battle feeds the same `PlayerProgress.RecordBattleCompleted` the live `BattleCompletedEvent` handler uses; exp is granted per victory (batched into a single core-update via `Player.GrantOfflineExp` so a thousand-victory window doesn't flood the write-behind), and the affected challenges are evaluated **once** at the end through the shared `ChallengeRewardService` with the per-challenge client push suppressed (`Player.CompleteChallenge(..., notify: false)`) — the welcome-back summary is the notification.
- **The crediting loop's trailing edge carries forward instead of being dropped (#1596).** Each candidate battle is checked against the *remaining* budget before being credited: if its own duration fits, it is a real completed outcome (credited normally, consuming duration + cooldown); if the away-window boundary falls *inside* it instead, it never really concluded, so it is not credited as a win/loss/draw at all — crediting it regardless would double-count a phantom win and invert the resumed battle's elapsed offset. Two distinct outcomes ride on `OfflineProgressResult`: `RemainderMs` (the residual cooldown left in a *completed* battle's cooldown window, always ≤ the cooldown length) and `PendingBattle` (the exact enemy/seed of an *uncredited* boundary battle, with its true elapsed-so-far offset) — mutually exclusive, and both 0/null when nothing was simulated or the stalemate cutoff (a CPU-waste guard, not an overshoot) stopped the loop early. `OfflineProgressService` acts on whichever is set: a `PendingBattle` is handed back already active via `BattleService.HandBackPendingBattle` (`SetActiveBattle` backdated to its true offset) exactly like a still-in-progress stale-battle hand-back; a residual `RemainderMs` just sets `PlayerState`'s cooldown (the live `NewEnemy` cooldown gate naturally waits it out). The hand-back surfaces on the welcome-back summary's `ActiveBattle` the same way #1595's does, for the client's replay-to-offset (#1597).
- **CPU-waste guard.** The simulator takes an optional opening-batch stalemate cutoff: a run whose first batch is nothing but draws (an unwinnable-and-unloseable matchup that would otherwise simulate the whole budget of maximum-duration draws for nothing) stops early. Any win or loss in the opening batch disables it; the 10-hour cap bounds every run regardless.
