# Backend Battle Setup & Runtime

This doc covers the backend's battle **setup** (enemy/boss encounter construction and snapshotting) and **runtime** (skill effects and damage/heal-over-time). It is split from [backend.md](./backend.md) so the main doc stays an overview, with the tick-by-tick mechanics here next to the parity invariant that governs them.

**Parity invariant (why this detail matters).** Battle logic runs on **both** the frontend and the backend, and the backend replays the battle the client reports as anti-cheat, so the two simulators must agree tick-for-tick. Every rule below is mirrored on the frontend (`UI/src/lib/battle`) and pinned by the mirrored parity matrix (`BattleSimulatorParityTests` ↔ `battle-simulation-parity.test.ts`) plus mirrored unit tests; it must not be allowed to diverge. See [game-design.md](./game-design.md) for the player-facing mechanics.

## Battle setup (enemy encounter)

Setting up a battle is a domain decision, not an orchestration one. The application layer (`BattleService`) only orchestrates — obtain the zone, snapshot the player and chosen enemy loadout, set the active battle — and delegates the game-logic choices to `Game.Core`:

- **The `Zone` aggregate owns the per-zone encounter decisions** — the idle level range (and the roll within it) and the dedicated boss's identity + fixed level. The data tier serves a gameplay/domain `Zone` read alongside the contract read used by the reference catalogue, so "what level do you fight here, and who is the boss" lives on one domain type rather than being threaded as loose ints.
- **A random idle encounter** rolls a level within the zone's range and returns a battle-ready enemy with a capped, randomly-selected loadout (≤4 skills). Enemy _selection_ (which enemy spawns) stays in the data layer's cached per-zone spawn table; the factory takes a resolver function rather than a repository dependency, keeping `Game.Core` free of data-access references.
- **The dedicated boss is built deterministically** for the "Challenge Boss" action: resolved at the zone's fixed boss level (no random roll) with its full authored loadout (no cap, no shuffle). The greater determinism is exactly what battle parity wants. The boss challenge is always available and challenging it doesn't change the player's current zone; a zone with no boss authored surfaces an error.
- **Zone-unlock is enforced at battle start (anti-cheat).** A zone may carry an optional unlock-challenge gate (see the [zone-progression](./game-design.md#zone-progression--locking-zones-behind-challenges) section). Entering a locked zone is refused — an idle transition into one is ignored (the player keeps fighting their current zone) and a boss challenge against one is rejected. The rule lives on the domain `Zone`; the orchestration reads challenge completion. An ungated zone (the common case) pays no extra read, and a gated zone costs one indexed completion lookup only on a real transition or boss challenge — never per idle tick.
- **The chosen loadout is snapshotted, not re-rolled.** The selected skill ids are stored on the player state (symmetric with the player's own battle snapshot) and sent to the client; validation reconstructs the enemy from that snapshot, so both sides use the exact same loadout.
- **The player side is captured and reconstructed in the domain.** A minimal id-based snapshot (level, stat allocations, equipped item + applied-mod ids, skill ids) is projected at battle start and rebuilt into the player `Battler` at validation time, resolving ids against the catalogs through caller-supplied resolvers. Critically, reconstruction **reuses the same modifier-composition rules as the live `Player` aggregate**, so a battle validated from a snapshot composes the identical attributes the player has live — this is the player half of the parity guarantee and must not diverge. The application layer only fetches the reference data and hands over the resolvers.
- **A dedicated-boss battle is marked, not inferred.** A boss-battle flag (plus the challenged zone id) is captured at battle start and threaded through the completion event, so the resolve path knows a victory was a dedicated-boss clear rather than inferring it from the enemy + current zone. Only the challenge path clears a zone: `ZonesCleared` counts distinct zones ever cleared (a binary per-zone flag, the global counter bumping only on a zone's first clear), while the `BossesDefeated` farm counter increments on every dedicated-boss victory (see the Zone Clears section in game-design.md).
- **The active idle-loop mode is persisted for offline rewards.** A boolean `Player.AutoChallengeBoss` records what the idle loop is doing — `false` is idle-farming `CurrentZoneId`, `true` is auto-challenging **that same zone's** boss (the loop never targets a separate zone, so no boss zone is stored) — so the offline-rewards simulation (spike [#879](./spikes/879-offline-rewards.md)) can resume the correct loop at next login. The frontend syncs it from its live auto-fight state via the `SetAutoChallengeBoss` socket command (anti-cheat validated like a boss challenge: the current zone must be in circulation, unlocked, and have a boss), and it rides the existing core write-behind path. As a backstop mirroring the online auto-fight-off, a recorded dedicated-boss **loss or draw** resets the persisted mode to idle, so it tracks the player's present loop even if a frontend sync is missed.
- **An abandon that resolved as a win is a full victory.** An abandon (reached only when the player starts a new battle while one is active) re-simulates the in-progress fight bounded by how long the **client actually simulated** it (sent as `clientBattleMs` on the superseding `NewEnemy`/`ChallengeBoss`), **clamped to `[0, server-elapsed wall-clock]`** and then to **`DefaultMaxBattleMs`**. Bounding by the client's reported duration is what keeps the abandon accurate: a battle the client never fought (e.g. a server-prefetched next battle discarded by a zone/build change during the cooldown) reports `0`, so the replay runs no ticks and **records no outcome** rather than minting a phantom one from the post-cooldown wall-clock gap. The wall-clock cap still bounds it from above, so a won-abandon can't be claimed faster than real time allows (and the `DefaultMaxBattleMs` cap keeps a reported 2-minute stalemate a draw — a battle never runs past its cap on the client, so the replay must not either). If the enemy died within that window it pays out through the same victory helper as a normal win — exp **and** the win/zone-clear/challenge credit — so the two win paths can't drift. An abandon where the player died within the window is a loss; one where neither died is an _incomplete abandon_ — a draw, including a live timeout-draw — tracked as its own statistic rather than a loss (see the [battle-outcomes](./game-design.md#battle-outcomes-battleswon--battleslost--battlesabandoned) split).
- **Victory-end validation and the post-battle cooldown use the server clock only — never a client-supplied timestamp.** A `DefeatEnemy` claim is accepted only when enough real server time has elapsed since battle start for the replay's duration to have completed (`now − BattleStartTime ≥ TotalMs − one tick` of slack for mid-tick start alignment). Network latency only ever *delays* a claim, so a legitimate win always satisfies this; being server-clock-only it is immune to client/server clock skew (an earlier client-timestamp check rejected legitimate wins whenever the two machines' clocks differed by more than a tick). The post-battle cooldown is anchored to the battle's **server-computed completion** (`BattleStartTime + TotalMs`), not to `now` — the gap between the two is the post-victory network latency, so subtracting it keeps the idle farm rate latency-independent without trusting the client clock. The client still reports its simulated duration as `clientTotalMs`, but **purely as a diagnostic** the server logs when it diverges from its own replay — it gates nothing.
- **The next idle battle is prefetched and bundled with the battle-end response.** To hide the `NewEnemy` round-trip latency under the post-battle cooldown, an idle victory (`DefeatEnemy`) and a boss loss (`BattleLost`) set up the *next* idle battle immediately (`PrepareNextIdleBattle`) and return it in their response, so the client begins it the instant the cooldown elapses with no extra request. Its `BattleStartTime` is anchored to the **scheduled cooldown expiry** (`EnemyCooldown`), not `now` — the cooldown is deterministic, so the next fight begins exactly when it elapses; anchoring there keeps the elapsed-time check above correct (latency only delays the claim past the scheduled completion) and the *following* cooldown correctly sized — anchoring to `now` would back-date the start by the whole cooldown and shrink (or zero) it. The bundled player snapshot is frozen at the previous battle's end, so the client uses the bundle **only while the zone and its battle-relevant state are unchanged** since then; a zone or build change during the cooldown falls back to a fresh `NewEnemy` whose snapshot matches what the client now simulates, preserving parity (and re-syncing the new zone). That fresh fetch supersedes the unfought prefetched battle, which the client reports as `clientBattleMs: 0` so the abandon records no phantom outcome (see the abandon bullet above). The boss-victory path is excluded — it paces itself via the Zone-Cleared overlay.
- **The battle `seed` is reserved for the simulation's RNG, not for setup.** It is generated at battle start from a **non-time entropy source** (a cryptographic RNG, not wall-clock ticks — a wall-clock seed is low-entropy and predictable between battles, unfit as the shared starting point for the parity-identical crit/dodge RNG), persisted on the player state, and sent to the client as the shared starting point for randomness _inside_ the simulation. Setup (level roll, skill selection) deliberately uses ambient randomness so it never advances the seed's stream — otherwise the server (drawing during setup) and the client (which receives the loadout ready-made) would begin the simulation from divergent RNG states. (The boss path is already deterministic, so this only matters for the random idle encounter.) The seed is threaded `PlayerState.BattleSeed → BattleService.SimulateBattle → BattleSimulator`, which constructs one `Mulberry32` and hands it to `BattleContext`; the frontend mirror seeds from `EnemyInstance.Seed`. Its first live consumer is the seeded crit/dodge/block runtime below — the `Mulberry32ParityTests` ↔ `mulberry32-parity.test.ts` vector still pins the two RNG ports against drift.

## Battle runtime — skill effects (timed buffs/debuffs)

Skills can carry authored effects that apply as **timed attribute modifiers** when the skill fires. Because battle logic runs on both sides, every rule below executes identically on `Game.Core/Battle` and its frontend mirror and is pinned by the mirrored parity matrix and unit tests.

- **Effects ride the real attribute graph.** Applying an effect adds a genuine attribute modifier to the target's live attribute collection and expiry removes it, so derived attributes cascade for free — a temporary Strength buff raises MaxHealth exactly like a permanent one. This is the riskiest parity surface: both sides must compose attributes through their one shared path, **including mid-battle add/remove**. Float-identical results rely on modifier ordering matching — a new modifier inserts at the end of its type-run (additives before multiplicatives) on both sides.
- **Effects fire after dealing damage** (self-effects route to the active battler, opponent-effects to the target), so a self damage-buff never boosts its carrying hit, while an earlier loadout slot's effect does affect a later slot firing the same tick.
- **Magnitude scales off the caster.** An effect's applied magnitude is `Amount + casterAttribute × ScalingAmount`, read from the **active (casting) battler** regardless of which battler it lands on — the same caster-scaling shape as a `DamageMultiplier` (`BattleContext.ApplySkillEffect` computes it and hands the resolved amount to `Battler.ApplyEffect`). A `ScalingAmount` of `0` is byte-identical to the unscaled authored amount, so this addition leaves existing effects unchanged. The resolved magnitude is frozen on the modifier at apply time; it does not re-scale as the caster's attributes change later.
- **The battler owns the active-effect bookkeeping.** Expiry is keyed to an absolute per-battler simulated-time clock advanced at the **start of each tick**, before any skill fires, so an effect influences exactly `DurationMs / MsPerTick` ticks (counting the application tick). Re-applying an active effect **stacks the magnitude** so magnitudes sum (additive amounts add, multiplicative factors compound), but all active applications **on the same attribute share one expiry** that **resets to the newest application's duration** (extending a longer-lived application or cutting a shorter one short); there are no independent per-portion expirations (#992). Applications on different attributes are independent, and **stacking is unbounded in magnitude but bounded in cost**: each application folds into the attribute's **single combined modifier per modifier type** rather than adding its own, so the modifier set and the active-effect list stay bounded by the affected-attribute count no matter how deep a buff re-applies — a persistently re-applied buff that never lapses keeps a battle O(ticks) instead of the O(ticks²) a per-application list produced. Because the fold changes the float grouping (`base + (a₁+a₂+…)` rather than `base + a₁ + a₂ + …`), both simulators must fold **identically** for the replay to stay bit-for-bit; this is parity-critical. `CurrentHealth` is left untouched when MaxHealth rises (no free heal) and clamped down when it drops. (The frontend stores a decrementing `remainingMs` instead, to drive the render countdown — functionally equivalent, expiring on the same tick.)
- **No per-tick allocations.** The expiry pass early-returns when a battler has no active effects (still advancing the clock so a later effect dates its expiry correctly) and otherwise iterates without allocating; the per-attribute stack list is created lazily on first apply.

The per-second accumulators are consumed by an **end-of-tick DoT/HoT phase**, reached only while both battlers still live and after both their skills resolve. DoT is **typed** (spike [#1320](./spikes/1320-damage-types.md)): three accumulators — `BleedDamagePerSecond`, `PoisonDamagePerSecond`, `BurnDamagePerSecond` — replace the former single `DamageTakenPerSecond`, the type encoded by **which accumulator an effect targets** (no field on the effect). HoT (`HealthRegenPerSecond`) stays typeless. For each battler the typed DoT (which **bypasses the Toughness curve**, unlike a normal hit) then HoT (capped at MaxHealth) apply before its death check, so a heal-over-time can save a battler from an otherwise-lethal DoT tick (#1090). The enemy resolves first, so an enemy its own same-tick regen cannot save dies before the player's DoT applies (a same-tick mutual DoT kill favours the player). DoT on the enemy counts as player damage dealt; DoT on the player as damage taken; the player's post-cap healing accumulates into the damage-healed statistic. This phase is parity-critical and mirrored on the frontend.

The typed DoT phase loops the three types in the fixed `DamageTypes.DotAccumulators` order, applying each type's resistance through the same `applies(type) → keys` map as direct hits but **sampled live off the bearer** each tick (so a vulnerability debuff hurts existing DoTs immediately) — `Σ perSec_t × 40/1000 × (1 − Σ_{k ∈ applies(t)} bearer[kResistance])`. The caster's amplification is instead **frozen** into the accumulator at apply time (`BattleContext.ApplySkillEffect`), consistent with how the magnitude's caster scaling already freezes. DoT keeps **bypassing the Toughness curve**, so resistance is its only mitigation; it is **not floored** (negative resistance — vulnerability — and resistance above `1` — absorption — fall out for free, the latter healing uncapped, unlike the reactive direct-hit absorption which the HoT cap bounds). A single typed DoT with zero resistance is byte-identical to the former single-accumulator outcome (the reduce-to-today identity), pinned by the parity matrix.

## Battle runtime — seeded crit / dodge / block (player only)

The first live consumer of the seeded battle RNG (above). Only the **player** crits, dodges, and blocks in this version; enemies do not, and the gating is purely on the acting battler — a clean seam for later enemy parity. Like the rest of the runtime this executes identically on both sides (`BattleContext.DamageTarget` ↔ `battleStep`) and is pinned by the parity matrix; the seeded scenarios force each chance to `1`/`0` so the outcome is deterministic without depending on a draw.

- **Draw order is a pure function of the skill-fire sequence**, never of a roll outcome, so the per-tick draw count is fixed regardless of stats: each **player** skill fire takes **1 crit draw** (before its damage); then, if the enemy survives, each **enemy** skill fire takes **2 draws — dodge then block — unconditionally** (both always drawn, even when the hit is dodged). So the count is `playerFires×1 + enemyFires×2`. A pure-buff (zero-damage) fire still draws. This decoupling from values and outcomes is the strongest parity guarantee, and is where #336's deferred skill-effect proc chance will later slot its own draw.
- **The math is player-only, gated on who acts.** *Crit:* `c < CriticalChance` ⇒ the raw damage is multiplied by `CriticalDamage` (read directly, like `CooldownRecovery`) **before** mitigation, so high crit damage punches through the Toughness curve. *Dodge:* `d < DodgeChance` ⇒ the hit deals `0`. *Block:* `b < BlockChance` on a non-dodged hit ⇒ `TakeDamage` applies the Toughness curve and then subtracts the flat `BlockReduction`, floored at `0`. Chances are decimal fractions compared directly against a `[0,1)` draw.
- **Sourced from core attributes.** `StaticAttributeModifiers.All` feeds crit/dodge/block from raw allocations (Dexterity/Luck → crit, Agility → dodge, Endurance → block; `CriticalDamage`/`BlockReduction` carry a base), so the mechanic is live in real play; the coefficients are a conservative strawman to be tuned during balancing, and the values are also grantable by gear/mods/skill-effects through the normal modifier path. The frontend `SkillActivation` carries `crit`/`dodged`/`blocked` flags for the combat log (#800); the headless simulator and the backend ignore them, but the rolls and damage results stay identical across both sides.

## Battle runtime — damage typing (direct-hit amplification & resistance)

Every skill carries a leaf `DamageType` (spike [#1320](./spikes/1320-damage-types.md); eight leaf types — `Physical`, the four elementals, and the three DoT forms — with existing skills backfilled to `Physical`). Direct hits route their type through one shared lookup, the codegen'd `applies(type) → keys` map (`Game.Core.Attributes.DamageTypes` ↔ `$lib/battle/damage-types`), which resolves a hit to the attacker's amplification and the defender's resistance attributes — the leaf type itself plus any cross-cutting category (`Elemental`, `DoT`). This is parity-critical and mirrored: `BattleContext.DamageTarget`/`Battler` ↔ `battle-step`/`battler`, with the `applies()` iteration order fixed so the additive amp/resist sums fold bit-for-bit identically on both sides.

The direct-hit pipeline, attacker A → defender D, runs in this order (it both amplifies and resists in **both** player→enemy and enemy→player directions):

1. **Amplification** (attacker, additive): `dealt = raw × (1 + Σ_{k ∈ applies(type)} A[kAmplification])`.
2. **Crit** stays attacker-side and pre-mitigation (player-only, unchanged from the crit/dodge/block section above), so high crit damage still **punches through** the Toughness curve.
3. **Resistance** (defender, additive, **unclamped**): `mitigated = dealt × (1 − Σ_{k ∈ applies(type)} D[kResistance])`. A negative total (`res < 0`) amplifies the hit — **vulnerability**; a total above `1` (`res > 1`) drives `mitigated` negative — **absorption**, a net heal.
4. **Toughness curve + Block last, and only on a positive hit** (spike [#1330](./spikes/1330-mitigation-rework.md)): when `mitigated > 0`, multiply by `1 − Toughness / (Toughness + K·attackerLevel)` (a diminishing-returns reduction — EHP linear in `Toughness`, asymptoting below 100%, `K`·**attacker level** keeping the band stable as content scales), then subtract the flat `BlockReduction` on a block, floored at `0`. When `mitigated ≤ 0` the absorbed heal passes through (mitigation can **neither heal nor deepen an absorption**) but is **capped at `MaxHealth`**, consistent with heal-over-time: the game has no overheal/shield concept, so the reactive absorption channel and the passive HoT channel behave alike. DoT still bypasses the Toughness curve entirely. **The attacker's level is threaded into the mitigation call** (`Battler.TakeDamage`/`ComputeNetDamage` ↔ `mitigateDamage`/`toughnessMitigatedDamage`) — parity-critical.

Amp/resist are **separate multiplicative budgets** (`× (1 + amp)` then `× (1 − res)`), never a single `(1 + amp − res)` — resistance must not cancel amplification 1:1. The amp/resist attributes default to `0` from every source, and `Toughness` defaults to `0`, so with no typed content and no Toughness both factors are an exact `1.0` and the curve is a no-op — the pipeline reduces to `max(raw − Block, 0)` (the reduce-to-today identity, pinned by the parity matrix). Typed damage-over-time (the per-type accumulators) reuses the same `applies()` map in the end-of-tick DoT/HoT phase above.

## Offline rewards simulation

When a player returns after being away, `BattleService.SimulateOfflineProgress` (driven by the `GetOfflineProgress` socket command) replays the missed idle/boss battles and applies the rewards inline — the away window is bounded (≥5-minute floor, 10-hour cap) so the worst case sits well under the command timeout, and the command's cancellation token is threaded into the simulation loop. The player-facing mechanic is in [game-design.md](./game-design.md#offline-rewards); the path-specific points:

- **The simulation is parity-irrelevant.** The client never replays these battles — the backend is the sole authoritative simulator (`OfflineProgressSimulator`, pure `Game.Core`, reusing `BattleFactory`/`BattleSimulator`/`DefeatRewards`) — so there is no parity surface to mirror. Results stay consistent with online battles by reusing the shared simulator.
- **The loop mode + zone come from the durable aggregate, not the session.** `Player.AutoChallengeBoss` (+ the current zone) selects idle vs. boss; a retired/locked/bossless boss target falls back to idle, and a non-viable idle zone is lazily relocated the same way the live idle loop relocates.
- **A stale in-flight battle is settled first.** A mid-battle disconnect leaves an active `PlayerState` battle; the offline flow resolves it exactly like an abandon before simulating the away window, so its outcome is credited once rather than re-abandoned by the first live `StartBattle` after the welcome-back gate.
- **Reward application is consolidated and shares the live per-battle path.** Each simulated battle feeds the same `PlayerProgress.RecordBattleCompleted` the live `BattleCompletedEvent` handler uses; exp is granted per victory (batched into a single core-update via `Player.GrantOfflineExp` so a thousand-victory window doesn't flood the write-behind), and the affected challenges are evaluated **once** at the end through the shared `ChallengeRewardService` with the per-challenge client push suppressed (`Player.CompleteChallenge(..., notify: false)`) — the welcome-back summary is the notification.
- **CPU-waste guard.** The simulator takes an optional opening-batch stalemate cutoff: a run whose first batch is nothing but draws (an unwinnable-and-unloseable matchup that would otherwise simulate the whole budget of maximum-duration draws for nothing) stops early. Any win or loss in the opening batch disables it; the 10-hour cap bounds every run regardless.
