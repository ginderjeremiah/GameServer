# API Client: Token Refresh & Socket Lifecycle

The deep mechanics of the frontend API client layer (`$lib/api`). The architectural decisions — why all token handling lives in the client layer, and the socket-first communication goal — live in [frontend.md → Authentication](./frontend.md#authentication) and [backend.md → HTTP vs WebSocket Communication](./backend.md#http-vs-websocket-communication); this doc holds the lifecycle detail you need when working on the client layer itself. The server side of the socket lifecycle lives in [backend-sockets.md](./backend-sockets.md).

## Token refresh lifecycle (`auth.ts` / `token-store.ts`)

- **`token-store.ts`** is the single source of truth for credentials — it persists the `{ accessToken, refreshToken }` pair in local storage (surviving a refresh) and decodes the access token's `exp` for pre-emptive refresh.
- **`auth.ts`** owns the refresh lifecycle and **collapses concurrent callers onto a single in-flight refresh** — refresh tokens are single-use on the backend, so parallel refreshes would consume the token twice and log the user out. It uses raw `fetch` to stay out of the 401-retry path and avoid an import cycle. That collapsing is per-tab only, while the token pair lives in shared `localStorage` and two tabs are a supported state, so a failed refresh re-reads storage before clearing: if another tab already rotated the token out from under this one, the stored pair is adopted instead of wiping out a session that's still alive elsewhere. That guard only covers `refreshTokens`'s own clear — `handleAuthFailure` (called by the request/socket layers once a refresh is deemed dead) always clears unconditionally, so each caller re-reads storage itself immediately beforehand and skips calling it at all if a concurrent tab has since rotated in a fresh pair.
- **A refresh failure is only fatal when it is a definitive rejection.** `refreshTokens`/`ensureValidAccessToken` return a discriminated outcome: `rejected` means the backend affirmatively refused the presented refresh token (`Auth/Refresh` returns **400** for "Invalid or expired refresh token" — its only failure path; it never 401/403s) or no refresh token is stored at all. Everything else — a network throw, a non-400 non-2xx (a 5xx, the auth rate limiter's 429), or a malformed/unparseable 2xx body — is `retryable`: the stored pair may still be good, so tokens are left in place and no logout is forced. Callers route to `handleAuthFailure` only on `rejected`; on `retryable` the HTTP path surfaces the original 401 to the caller, and the socket path bails out of the connect and lets the keepalive ping's next `ensureSocket()` retry it.
- **`api-request.ts`** attaches the bearer header, refreshes pre-emptively, and on a 401 refreshes once and retries. The anonymous auth endpoints are exempt.
- **`api-socket.ts`** passes the access token as a WebSocket subprotocol (browsers can't set WebSocket handshake headers, and a query param would land in a downstream access log — see `backend-sockets.md`), refreshes pre-emptively before opening the handshake (mirroring the HTTP path, so a reconnect on an expired token doesn't eat a rejected handshake), and on an auth-rejected handshake refreshes and reconnects (bounded — see below). Because the open path is async, concurrent opens are collapsed onto a single in-flight connect.

## Socket request lifecycle

Each sent command is tracked in an in-flight map keyed by command id, pruned the moment its response resolves. The transport signals **every** failure — a server error, a dropped connection, or a per-request timeout — by **resolving** the promise with a response carrying an `error` field, never by rejecting; callers inspect `response.error` rather than wrapping in try/catch. When the socket closes, still-pending requests are settled this way instead of hanging, but in-flight commands are deliberately **not** blind-resent on reconnect (only the not-yet-sent queue is flushed) — commands like `DefeatEnemy` aren't idempotent and the lost response may well have been a success. The not-yet-sent queue is re-flushed only on a close that reconnects (the keepalive after a post-open drop, the auth-retry after a refresh); on a **terminal** close that won't reconnect (normal closure, auth-retry budget exhausted, no refresh token, or a failed refresh) the queued-but-unsent commands are settled with an error too, so a command queued during a connect that fails before it ever opens can't await a response forever.

A **per-request timeout** (30s) backstops the one case the close handler can't: the socket stays open but no reply arrives. It is armed only when a command is actually sent (not while it's queued waiting out a reconnect) and cleared on a real response. The cap is deliberately generous so legitimately-slow commands aren't spuriously failed.

## Socket connection lifecycle

The single module-singleton socket doubles as a background reconnect: a 10s keepalive ping (also used for latency) re-arms a dropped socket while idle. Because the singleton outlives any one connection, the lifecycle is torn down deliberately:

- **The keepalive is owned, not fire-and-forget** — its handle is cleared on a clean close, an unrecoverable auth failure, and an explicit `disconnect()`, and the ping self-terminates if it fires with no stored token, so it can't resurrect a socket for a logged-out user. `disconnect()` is called from the `SocketReplaced` teardown (which navigates client-side without a reload, so the ping would otherwise reconnect and fight the session that just took over).
- **`SocketReplaced` is handled app-wide, not scoped to the game engine.** It's registered once in the root layout (alongside `onSocketError`) rather than by `startGame`/`stopEngines`, so a takeover push is still acted on while parked on a non-`/game` route (e.g. `/admin`). The other three `startGame`-registered pushes (`ChallengeCompleted`, `ProficiencyXpGained`, `ServerCommandFailed`) stay scoped to the game engine's lifecycle — they only matter while its stores are live.
- **The keepalive also detects a half-open socket** — `readyState` stays `OPEN` even when the underlying connection is dead (common after laptop sleep/wake or a NAT drop), so each ping tracks whether the previous one's pong ever arrived. After `MAX_MISSED_PONGS` consecutive misses the socket is closed so the reconnect machinery above takes over, instead of every command silently eating its full per-request timeout. A command's own per-request timeout is also treated as half-open evidence when it coincides with an already-overdue pong (at least one miss already registered) — closing the socket right away rather than making commands in flight wait out the rest of the ping-only detection window. A timeout with no pong overdue is left alone, since a single legitimately slow command shouldn't close an otherwise healthy connection.
- **Auth-rejection retries are bounded** (5 in a row) so a flapping server can't drive an unbounded refresh/reconnect loop that burns single-use refresh tokens. The budget is spent only on an **actual token rotation** — a successful refresh that still failed to reconnect — never on a retryable refresh failure (network blip, transient server error), which consumes no refresh token and so must not count toward a bound that exists to stop token burn (#1992). The budget refills only after a reconnect stays open a stable interval, so a socket that opens then is immediately closed doesn't reset the count every cycle. Exhausting the budget is treated as terminal: the keepalive is torn down and the user is routed to re-auth, so the ping can't keep reconnecting (on the still-present access token) and bypass the bound.
