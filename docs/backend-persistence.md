# Backend Caching, Pub/Sub & Write-Behind Persistence

Player data is cached in Redis (the source of truth for persistence/rehydration; the live per-command working copy is the in-memory aggregate — see below) and persisted to Postgres asynchronously, while most reference data is cached in-memory (see [backend.md](./backend.md#caching-pubsub--write-behind-persistence)). This doc covers the write-behind mechanics for player data, plus the Redis connection thread-pool floor and background-worker ownership.

## Write-behind player cache

Player data uses a **write-behind cache where Redis is the source of truth** *for persistence and rehydration* — changes are persisted to the database asynchronously, and a cold read, reconnect, or cross-instance rehydration loads from Redis (the DB is only a lagging fallback on a miss). Redis is **not** the per-command working copy, though: during a live session the read-modify-write base is the **in-memory aggregate**, loaded from Redis once at connect and re-saved per command without ever being re-read. That two-layer split is the whole reason the cache write can be (and deliberately is) fire-and-forget — see [The write-behind path](#the-write-behind-path); a dropped cache write is invisible to the live session and self-heals on the next save, so it is *not* a lost-write bug. Redis pub/sub triggers the async persistence, doubles as the WebSocket backplane, and broadcasts reference-data cache invalidations across instances.

**The cached player is a lean serialization model, not the full aggregate.** The reference data the aggregate carries (item, item mod, and skill *definitions*) is **excluded** from the cached blob — only the player's own state plus the reference *ids* is serialized. On load the ids are re-resolved against the in-memory reference catalogs, so a cached player can never serve a stale reference snapshot behind an admin edit or a build-then-swap, and the per-save blob (the hot path) stays small by not re-serializing the reference graphs. The database read **projects straight into the same lean model** (no navigation graph is loaded), so the cache-hit and DB-miss paths share one rehydration that resolves references identically — including the loud fail-fast on an unresolvable owned reference (see [backend.md → Reference Data](./backend.md#reference-data)). The lean model mirrors the relational shape (flat applied mods, per-skill selected/order flags) so the projection is a trivial column copy, and its members are `required` so a projection that forgets a field is a compile error rather than a silently empty graph.

**The cached player aggregate carries a sliding idle TTL.** Every save (and every load-miss re-cache) writes the key with a fixed expiry, and every cache *hit* refreshes that TTL via a fire-and-forget expire. An actively-playing player's commands keep the key alive for the whole session, while a dormant player ages out of Redis instead of occupying memory forever. This is safe because a cache miss falls back to a DB reload, so an expired or evicted key is invisible to callers.

**The cached session carries the same sliding TTL, but as a correctness floor rather than a memory nicety.** The session holds the in-flight `PlayerState` (the active-battle snapshot + the user→player binding). The binding *is* reconstructable: a miss is rehydrated in-memory by the consumers that need player state (via `SessionInitializer`, binding the player named in the token's selected-player claim — `SessionLoaderMiddleware` itself only records the token-derived `UserId`), so a valid token is never mistaken for unauthenticated — authentication derives from the token, not this key. The active-battle snapshot, however, lives *only* here, so an eviction silently drops any in-flight battle. The TTL is therefore a floor against losing an active battle: a session should not be dropped while its refresh token is still valid. Both keys derive their budget from a single refresh-token-lifetime anchor — once a session is idle past it, the refresh token has also expired and the user must re-login anyway.

The TTL is sized so eviction can't bite mid-drain: because the cache is the source of truth, a key disappearing while its events are still draining would let a later read fall through to a lagging DB. The invariant is *TTL ≫ max queue-drain time*, which the hours-scale TTL dwarfs. Now that every player key carries a TTL, a `volatile-*` Redis eviction policy can shed dormant keys gracefully under memory pressure — but the deliberate TTL, not the eviction policy, is the primary control.

## The write-behind path

When the `Player` aggregate mutates it raises domain events; on save, the repository dispatches them in-process and a publisher serializes each into an envelope. Rather than publishing per event, the publisher **buffers the envelopes and flushes the whole batch onto the Redis queue as one multi-value push** after dispatch settles, so a save raising several events costs a single queue round-trip. A background service consumes that queue and applies each change to the database. The consume side mirrors the publish side: each enqueued event type has a discrete persistence handler, resolved per drain by a dispatcher keyed on the envelope's type name, so adding a persisted event is a new handler plus one registration rather than an edit to a central switch (an envelope whose type matches no handler is dead-lettered as a poison message). Only events representing persistent state changes are registered for publishing; purely in-process reactions are never enqueued — keeping the "which events persist" decision in the data-access layer rather than encoded onto the domain events. **The buffer is only cleared once the flush actually succeeds** (`PlayerUpdateBatch.FlushAsync`): a save whose events were already snapshotted and cleared off the aggregate by the dispatcher has nowhere else to re-raise them from, so if the publish itself throws (a transient Redis blip), draining first would silently discard them. The batch instead keeps them buffered for the *next* flush in the same scope — real for a command that saves the player more than once (e.g. a zone change followed by the battle result) — accepting an occasional harmless duplicate over a silent loss, consistent with the consume side's at-least-once/idempotent-handler contract below.

**A flush failure that outlives its scope forces the connection's in-memory player to reload.** The retry above only helps within one command's scope — the `PlayerUpdateBatch` is scoped per command (`SocketHandler` opens a fresh DI scope per command), so a flush that fails on a command's *last* save has nowhere left to retry from once that scope is disposed. Meanwhile the aggregate's mutation already happened before the flush ran and the connection holds that same in-memory `Player` for its whole lifetime (see above), so without intervention the stuck mutation would silently ride along and eventually reach Redis via some *later* command's unrelated save — with its own queued event permanently lost. `SavePlayer` therefore wraps a non-cancellation flush failure in `PlayerPersistenceFlushFailedException`, and `SocketHandler` reloads the connection's `Player` from the repository before its next command runs whenever that's thrown — discarding the failed command's in-memory mutations and converging back onto the last successfully-persisted state, consistent with the command having genuinely failed (#1632).

**The source-of-truth cache write is deliberately fire-and-forget**, because the in-memory aggregate — not the cache — is the read-modify-write base during a session. A live player runs on one WebSocket whose request-scoped session lives for the whole connection; the aggregate is loaded from Redis **once** up front and every later command mutates and re-saves that same in-memory object rather than re-reading Redis. So a dropped cache write can't poison the next command (it never reads the key) and self-heals on the next save, while Postgres still converges via the queue. The one residual exposure — a dropped write *combined with* the in-memory aggregate being discarded (reconnect / socket-replace) before the next save — is a knowingly-accepted trade-off, since guarding it would put an awaited Redis round-trip on the per-battle hot path.

**Player progress (statistics + challenges) is a second write-behind aggregate** under its own key — kept separate from the player blob (which is re-serialized in full on every save) because the stat set grows over time and would bloat every per-battle write. A deliberate consequence is that the battle-completion handler **reloads progress per battle** (a cache-first, typically warm read) rather than holding it in memory alongside the player for the connection's lifetime: co-locating it would either fold it back into the player blob (the bloat the split exists to avoid) or require a connection-scoped holder the application-layer handler can't reach (it sees only the player aggregate, not the presentation-layer session), so the per-battle load is accepted over inverting that dependency. All progress reads — including the zone-unlock gate — go through the cache with a DB miss-reload, so a just-completed unlock gates the zone correctly even while the DB lags. Its save enqueues an **absolute-value** event and *then* advances the cache (publish-before-cache, so a failed enqueue can't leave the cache holding a snapshot that was never queued and silently lose the write). Unlike the player aggregate, progress publishes directly from the repository rather than through the domain-event dispatcher: it emits one data-tier persistence event consumed only by the synchronizer, so routing it through the dispatcher would add ceremony for no gain. **A progress save raised within a player save shares that save's single batch flush** rather than issuing its own queue round-trip: the live battle-completion tick saves both the player aggregate and progress, and the progress save is reached *through* the player save's event dispatch (a `BattleCompleted` handler), so it buffers its event into the same scoped batch and defers its cache advance until the player save's flush enqueues it — collapsing the per-battle player and progress writes onto one round-trip while preserving publish-before-cache. A progress save *outside* a player save (e.g. the offline-rewards batch) flushes its own event, so it is never stranded.

## Queue processing resilience

Queue processing never silently drops an event, and the read itself is **at-least-once** rather than at-most-once:

- **The read reserves rather than destructively pops.** Each item is atomically moved onto a side processing list (Redis `LMOVE`) and only removed once it has been durably applied or dead-lettered (the *acknowledge*). A crash anywhere in between — a retry sleep, mid-apply, or after a failure but before the dead-letter write lands — leaves the item on the processing list rather than losing it. On startup the synchronizer **reclaims** any items a previous run left in flight (orphaned by a deploy, scale-down, or crash), moving them back onto the queue head in order before draining. At-least-once is safe because the handlers are idempotent (existence checks, absolute updates, delete-then-insert), so re-applying a reclaimed item converges. Idempotency holds even when two applies of the same event run **concurrently** across instances: the existence-check-then-insert is not atomic, so both can pass the check and insert, but the insert handlers absorb the resulting unique-constraint violation (backed by each table's unique key/index) as a benign no-op rather than surfacing it as a wasted retry. This rests on the index actually rejecting the duplicate, so any unique key over a **nullable** column must be declared `NULLS NOT DISTINCT` — Postgres otherwise treats nulls as distinct, and a global `PlayerStatistic` (null `EntityId`) would never collide, silently accumulating duplicate rows under the concurrent double-apply. **A no-op is the wrong resolution where the row's natural key is narrower than its payload, so those handlers settle last-writer-wins instead:** the `AppliedMod` PK `(player, item, slot)` excludes the mod id, so two *different* mods racing into one slot collide — absorbing that as a no-op would terminally drop the loser, so the mod handler resolves the violation with a server-side update to the latest mod. Equipment slots get a DB invariant of their own — a partial unique index on `(player, EquipmentSlotId) WHERE EquipmentSlotId IS NOT NULL` makes "one item per slot" a constraint rather than an app-code convention — and the equip handler vacates the destination slot server-side *before* placing the item (mirroring the mod handler's clear-then-write), retrying a bounded number of times on the rare cross-instance race. The processing list is a single shared key (not per-instance), so any instance's startup reclaim recovers another's orphans; cross-instance same-player ordering is already only best-effort (the drain gate below serializes within an instance, not across the fleet), so the reclaim introduces no new reordering hazard — at worst a still-live item is applied twice. **Best-effort ordering is a real hazard the order-sensitive handlers must tolerate, not just the reclaim's double-apply.** The player and progress aggregates share the queue but are *separate enqueues*, so they are ordered independently, not jointly. Within the player aggregate, a child-row handler whose target presupposes an earlier insert — equip/select/favorite presuppose the item/skill unlock — must therefore **insert the missing row if the unlock is reordered behind it** rather than mutating only existing rows (which silently drops the change until a later edit self-heals the DB). The exception is unequip, where "missing row" already *is* the desired end state (unequipped = no equipped row), so its no-op converges without an insert.
- **A malformed payload is dead-lettered, not retried** — it can never succeed, so it is logged and moved to a dead-letter list for inspection.
- **A valid event that fails on an unexpected error is retried with exponential backoff, then dead-lettered** — transient failures (a DB connection blip, deadlock, timeout) often succeed on a second attempt.
- In every case the rest of the queued events still process.

**The dead-letter queue's depth is surfaced, not silent, and an admin surface can inspect and replay it.** Nothing drains it automatically, so a warning is logged whenever its depth grows past the last reported value (and on the first drain after startup, catching a standing backlog), so accumulating poison messages are visible to alerting rather than piling up unseen. On top of that, an admin-only Ops surface (under the shared `/api/AdminTools/*` route) offers a non-destructive **peek** that classifies each entry (malformed, unknown-type, or replayable — derived from the same parse the synchronizer does, and the owning player id pulled from the payload) and a guarded **replay** that moves selected — or all — entries back onto the player update queue and wakes the synchronizer. Replay re-enqueues onto the destination *before* removing from the dead-letter queue, so a crash can only duplicate an entry (absorbed by the idempotent handlers), never lose it — the reason it can't use a destructive pop — and it only re-enqueues entries genuinely on the queue (a snapshot-validated selection), so it can't inject an arbitrary message.

**The socket command dead-letter queue (below) shares this same peek/replay surface and contracts** (`Game.Api/Services/Admin/SocketCommandDeadLetters`, reusing `DeadLetterEntry`/`DeadLetterInspection`/`DeadLetterReplayResult`), but its replay target is fundamentally different: there is no second Redis list to re-enqueue onto. A dead-lettered command instead carries the **player id** it was addressed to (`SocketCommandDeadLetterEnvelope`, not the bare command — the specific socket it originally failed on may no longer exist by the time an operator replays it), and replay redelivers it via `SocketManagerService.EmitSocketCommand(command, playerId)`, which resolves whatever socket is **currently** live for that player. An entry whose player has no live socket right now is left queued rather than silently dropped — there is no destination to reserve onto first, so redelivery only proceeds once it can actually happen, and an operator can retry later once the player reconnects.

**Drains are serialized so same-player events apply in order.** The pub/sub worker can dispatch a second drain on another thread while one is in progress; two drains popping the shared queue could apply order-sensitive same-player events (an equip immediately followed by an unequip) out of order. Drains are therefore gated so at most one runs at a time, and a wake that races the active drainer's exit is coalesced into a single follow-up pass rather than dropped.

**The queue is drained once on startup, not only on a wake.** Pub/sub wakes are at-most-once and fire-and-forget, so an item enqueued while no subscriber was connected (across a restart, or a wake dropped during a brief outage) would otherwise wait for the *next* publish. The synchronizer subscribes first, reclaims any orphaned in-flight items, then runs one drain over the queue, so anything already enqueued is applied immediately; subscribing before draining means an item enqueued *during* the startup drain still gets a wake, and the drain gate serializes the two (the reserve/acknowledge read is idempotent, so they can't double-apply).

**Shutdown drains to a clean boundary, mirroring the graceful socket drain.** On stop the synchronizer signals its stopping token and awaits the in-flight drain (startup or a wake) within a bounded timeout, so an in-progress apply finishes rather than being cut off mid-`SaveChanges`. The drain loop checks the token between items, so a stop ends after the current item is acknowledged; anything still queued (or reserved but not yet acknowledged) is reclaimed and re-applied on the next startup, so the bound — sized under the host's shutdown timeout — loses nothing. The same token is threaded through the startup reclaim/drain **and into the reserve/reclaim queue ops themselves** (the queue honours it cooperatively the same way the cache client does — see [backend.md](./backend.md#http-vs-websocket-communication)), so a stop requested mid-boot unwinds the startup work, and a *wedged* Redis round-trip mid-reclaim/reserve unwinds promptly rather than only at the next item boundary. Once an item is reserved it is processed and acknowledged without the token, so the in-flight apply always finishes cleanly.

**The per-event drain handlers deliberately take no cancellation token.** The `IPlayerUpdateHandler.HandleAsync` signature, the dispatcher, and the handlers' `SaveChangesAsync` calls thread none — a considered choice, not an oversight. Honouring the *shutdown* token inside a handler would let a stop mid-save cancel a good write, which the retry path would then dead-letter rather than complete — the opposite of the clean-finish guarantee above. A separate *per-operation* drain timeout was weighed and rejected as redundant: a wedged save is already bounded by Npgsql's command timeout and unwound by the retry-then-dead-letter path, so threading a token through the handler interface would widen the surface for no behavioural gain.

## Domain-event dispatch

**The dispatcher drains events raised during handling.** It loops until the aggregate is quiescent, so events a handler raises while reacting to another are dispatched in the **same** cycle rather than lingering until the next save — which matters for the challenge-reward path, where reacting to a battle-completed event unlocks rewards and raises the unlock + completion events that drive the client push. A safety bound fails loudly if a handler raises events unboundedly.

**Handler dispatch is isolated, so one handler throwing never silently drops the rest of the batch.** A batch is snapshotted and cleared off the aggregate *before* its handlers run (so a handler raising follow-ups doesn't re-process it). Each handler invocation is wrapped: a thrown exception is captured and the remaining handlers, sibling events, and drained follow-ups still dispatch; once the aggregate is quiescent, captured failures are rethrown together (surfaced and logged, not swallowed). The chosen semantic is *no re-dispatch* — deliberately avoiding double-applying non-idempotent in-process handlers (e.g. one that increments cumulative stats) — with the residual gap (the specific event whose own handler threw is not itself retried) tracked as follow-up work.

**Handlers for the same event type fire in registration order** (a deterministic ordered collection, set once at startup), so when more than one handler is registered the order is reproducible across runs and instances rather than nondeterministic.

**The generic logging handler never serializes an event.** It logs only the event type name, never the event object — several events carry whole aggregates (the player, the enemy), so serializing them would pull player identity, stats, and inventory into the logs. An event opts into richer logging by exposing a deliberately-curated list of **safe scalar ids only** (player id, event-specific ids, counts, flags), attached as a structured logging scope. The default is therefore safe-by-default: a new event leaks nothing until someone deliberately projects reviewed scalars.

## Redis connection thread-pool floor

The Redis multiplexer factory raises the process thread-pool minimum before connecting any multiplexer (never lowering a host that already runs higher). StackExchange.Redis completes async commands on the .NET thread pool, whose default minimum equals the CPU count; once busy threads exceed that minimum the CLR adds threads only slowly (~1 per 500ms). On small, busy hosts that ramp-up stall can leave a reply unread past the 5s command timeout, surfacing as a spurious timeout exception. Centralizing the bump in the single connection chokepoint covers production and every test process that touches Redis without scattering startup tweaks across entry points.

The CI test process raises a higher, dedicated floor: it runs every test assembly concurrently under the coverage profiler on a 2-core runner — a far heavier single-process load that exceeded the production floor and starved Redis replies, the Npgsql read, and the test cleanup multiplexer alike. The shared integration-test fixture pre-warms a generous floor before any test touches the backing services. Production code is untouched.

## BackgroundWorker disposal and pub/sub worker ownership

`BackgroundWorker` owns an OS wait handle, so it is `IDisposable`. `Kill()` and `Dispose()` are deliberately distinct: `Kill()` stops it from being scheduled again while keeping the instance queryable, whereas `Dispose()` also releases the wait handle (and implies `Kill()`). `Start()` publishes `IsRunning = true` **before** signaling the loop so the loop's trailing `false` always happens-after it — otherwise a fast (already-signaled) run could reset the flag before the late `true` landed, wedging it stuck `true`. The flag is `volatile` so the transition is visible across the threads that read it.

`RedisPubSubService` is **not** `IDisposable`: its worker registry is process-wide `static` but the service is registered transient, so disposing one instance would tear down workers owned by every other instance. Each worker is instead disposed at its own ownership boundary — when its subscription is removed, or when it is discarded because its id was already taken. The worker-backed `Subscribe` overloads therefore **require an id**; the write-behind consumer subscribes under its instance id and tears its worker down in `StopAsync` (there is no untracked app-lifetime subscription).
