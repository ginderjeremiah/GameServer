# WebSocket Connection & Command Lifecycle

The deep mechanics of the socket layer. The architectural goal — why every in-game read and player-state mutation goes over the socket — lives in [backend.md → HTTP vs WebSocket Communication](./backend.md#http-vs-websocket-communication); this doc holds the lifecycle detail you need when working on the socket layer itself.

## Handshake auth via WebSocket subprotocol, not a query string

Browsers can't set an `Authorization` header on the WebSocket handshake, so the access token has to travel some other way. It rides as the sole requested WebSocket subprotocol (`Sec-WebSocket-Protocol`) rather than an `access_token` query-string parameter: a query string ends up verbatim in the request line of any downstream access log (the production edge proxy's default nginx format logs `$request`, which includes the full URL), while a subprotocol is a handshake header no default access log format records. `JwtBearerEvents.OnMessageReceived` (`Startup.cs`) reads the token from `HttpContext.WebSockets.WebSocketRequestedProtocols` when the request has none via the normal bearer path; `SocketInterceptorMiddleware` then echoes that same value back via `AcceptWebSocketAsync(subProtocol)` — required because a browser fails the handshake outright if it offered a subprotocol list and the server's response selects none.

## Per-socket command serialization & timeouts

**The per-player sequential guarantee is enforced by a per-socket command lock**, because server-initiated pushes (challenge-completion, socket-replaced) arrive over the Redis backplane and run *off* the read loop — so two paths could otherwise mutate the same cached player concurrently, the exact lost-update class the serialization exists to prevent. Each command also runs under a per-command timeout, and that cancellation token is plumbed down through the application services and the data tier (EF/Npgsql honour it natively; the Redis client is wrapped to honour it cooperatively) so a wedged command unwinds promptly and releases the lock rather than stalling every later command. A separate per-socket send lock guards WebSocket output, since overlapping sends on one socket are illegal.

## Single active connection (session takeover warning)

A player may only be connected to one instance at a time: the socket manager records the player's current socket id in Redis and, when a new socket replaces an existing one, signals the old socket to close. To let the client warn the user **before** that takeover happens, the authenticated `Auth/ActiveSession` endpoint reports whether the player already has a live socket — deliberately over **HTTP, not a socket command**, because opening a socket to ask the question would itself trigger the replacement it is meant to warn about. Because the check is presence-based it covers "another tab" and "another device" uniformly.

The presence key carries a TTL refreshed on every inbound socket message (the client heartbeats every 10s), so a live connection stays fresh while a connection that vanished without a clean close eventually expires. The TTL is pinned to *at least* the inactivity watchdog's own worst-case bound (`DefaultInactivityTimeout` + `DefaultInactivityPollInterval`) rather than a shorter, UI-responsive value — a shorter TTL would let a backgrounded-but-still-open socket (browsers throttle hidden-tab timers well past the 10s heartbeat interval, see spike #922) lose its presence claim to a switch-credit or account-level takeover claim (below) while the watchdog still considers it live, reintroducing the exact lost-update race those claims exist to prevent. The trade-off is that `HasActiveSocket` takes longer to report an uncleanly-vanished connection as gone. The key is claimed atomically (set-with-TTL in one operation) and rolled back if the post-claim setup fails, so a partial registration can't leave a ghost session that blocks the player.

**Enforced per account, not just per character.** The decided multiple-characters model (spike #922) is one live character at a time *per account*, but presence above is keyed by character (`playerId`), which by itself only stops the same character from having two live sockets. `RegisterSocket` additionally claims an account-level key (`AccountSocket_{userId}`) naming the account's current live character, using the same atomic claim shape. If a *different* character on the same account still holds that slot — a second tab/device that selected another character without tearing its first socket down — its socket is signalled to close exactly like a same-character takeover. This closes a gap where two live characters on one account would otherwise both write their in-flight battle state into the same account-keyed session cache (`Session_{userId}`, deliberately kept account-keyed — see [spike #922](./spikes/922-multiple-players-per-account.md)), last-writer-wins, silently discarding whichever wrote first (#1817). The key's stored value carries the claiming *socket's* id alongside the player id (not the player id alone), so a same-character reconnect's old-socket teardown — a compare-and-delete keyed on what it claimed — can never match, and therefore never delete, a newer socket's still-live claim (#2235).

The same key also backs the in-game character-switch credit (`docs/backend-auth.md` → "Switching characters in-game is lossless"): before crediting a departed character's idle progress over HTTP, `SocketManagerService.TryClaimForSwitchCredit` atomically reserves the key (a compare-and-set against "unset") instead of a plain presence read, so a socket registering concurrently for that same player either finds the key already taken (the credit wins, the registration proceeds and simply overwrites the claim once it's stale or released) or a fresh `RegisterSocket` call defers behind an active claim (bounded by the claim's own short TTL) rather than starting its battle loop while the credit's read-modify-write is still in flight (#2041).

Each claim carries a unique id (`switch-credit:{guid}`), not a shared literal — a registration that has been deferring behind one claim can tell it apart from a newer claim that replaced it (the original released, a fresh switch-away credit landed) and restarts its wait against the new claim's own full TTL, rather than a fixed deadline anchored to when the registration itself started waiting. Without that, a registration that burned most of its wait window behind claim #1 could CAS over a seconds-old claim #2 once its own (unrelated) deadline lapsed, reintroducing the #2041 race the claim exists to prevent (#2374).

## Graceful socket drain on shutdown

A stopping instance drains its live sockets cleanly rather than leaving them to be force-killed — the missing lifecycle piece for an app designed to run as multiple scalable instances (rolling deploys, scale-down). A singleton registry tracks every live socket and uses its shutdown hook to drain them in two phases: first stop the per-socket inactivity watchdogs (teardown now owns the close), then send each socket a normal-closure close frame and await the loops within a bounded timeout, aborting any still-blocked receive past the deadline so the loop unwinds rather than waiting out the host timeout. The client treats the normal closure as a cue to reconnect — which a rolling deploy routes to a healthy instance — so no client change was needed. The "draining has started" gate and the snapshot are flipped together under one lock, so a handshake that completes after the drain begins is closed on the spot with the same reason instead of slipping past the snapshot and lingering until the process is killed.

## Self-bounded `Close`/`SendData` (wedged-peer protection)

**Both `SocketContext.Close` and `SendData` self-bound and fall back to `WebSocket.Abort()`.** A client that stops reading (a TCP zero-window) can wedge a chunked send mid-frame while holding the per-socket send lock; those chunk sends deliberately run under `CancellationToken.None` once a frame has begun (to avoid interleaving a half-sent frame), so a caller cancelling its *wait* for the lock never unblocks the send itself. Both methods therefore apply their own short internal timeout on top of whatever token they're given — covering the watchdog's terminal close, which passes no token at all, and the ordinary command-response send, which is awaited *after* the per-command timeout budget while still holding the per-socket command lock — and abort the connection outright rather than waiting on a wedge that can otherwise only be broken by the peer. This is what guarantees the inactivity watchdog, the shutdown drain, and an ordinary response send all complete rather than hanging (and pinning their lock, handler, and in-memory `Player`) on a wedged peer that keeps sending inbound traffic without ever reading its responses.
