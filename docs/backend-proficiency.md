# Backend Proficiency Accrual (execution paths)

The server-side execution paths for proficiency XP, split from [backend.md](./backend.md#proficiency-xp-accrual-battle-completion-hook) so the main doc stays an overview. The player-facing mechanic — the per-path power-normalized claim, two-book routing, and frontier walk — is owned by [game-design.md](./game-design.md#proficiency-xp--effect-based-power-normalized-accrual).

Proficiency XP is computed **server-side at battle completion, off the tick loop** — like statistics and challenge rewards, never inside the deterministic 40 ms simulation. This is the deliberate boundary that keeps it off the frontend/backend battle-parity surface: the simulation produces the `BattleStats` (the typed **output book** — `TypedDamageDealt` — and **incoming book** — `TypedDamageExposure`) and the player's and enemy's `CombatRating`s are the same measure `DefeatRewards` computes server-side; the accrual reads those after the fact. Only the eventual proficiency *attribute bonus* is parity-sensitive (both sides compose it from player state at snapshot time), not the XP.

## The two accrual paths (live & offline)

The accrual **math** — frontier routing, the XP claim via `ProficiencyXpCalculator`, leveling, and the milestone/open triggers — is `Game.Core.Proficiencies.ProficiencyAccrual` (#1602): a pure domain function taking a small resolver-func bundle (`ProficiencyCatalog`) plus read/write funcs for a proficiency's current `(level, xp)`, with no `IProficiencies`/`ISkills`/`PlayerProgress`/`Player` dependency of its own. **Both** completion paths run it, so live and offline accrue identically, but they call it differently:

- The **live** path goes through `ProficiencyRewardService` (mirroring `ChallengeRewardService`) — a thin application-layer adapter that bundles `IProficiencies` into the `ProficiencyCatalog`, backs the read/write funcs with the `PlayerProgress` aggregate, grants the milestone reward skill(s) `ProficiencyAccrual` only *reports* (`ProficiencyXpResult.GrantedSkillIds`, via idempotent `Player.UnlockSkill`), and raises the client push. `BattleStatisticsEventHandler` calls it per victory, reacting to `BattleCompletedEvent` (whose `PlayerRating`/`EnemyRating` the victory path threads on).
- The **offline** simulator (`OfflineProgressSimulator`) calls `ProficiencyAccrual` directly, **in-loop** after every victory, against its own working copy of the snapshot's captured proficiency levels — so a milestone crossed early in the away window is already baked into the attribute bonuses of every later simulated battle, not just recorded after the whole window has already played out. `OfflineProgressService.ApplyOfflineRewards` then folds each battle's already-computed result (`ProficiencyGainAccumulator`, unchanged) and applies the window's final state — the per-proficiency level/XP and the granted skills — through the same aggregate/skill-grant step `ProficiencyRewardService` exposes (`GrantRewardSkills`), once, at the end.

The `ActivityKey → paths` routing is a reverse index the proficiency cache builds, with **retired paths excluded** (a retired path is simply absent from the catalog `ProficiencyAccrual` is handed) so a retired track is frozen for accrual. The live path raises a `ProficiencyXpGainedEvent` that an API-layer notifier pushes to the client exactly like the challenge-completion push; the offline path suppresses the push and instead **folds each won battle's accrual across the away window** (`ProficiencyGainAccumulator` — XP summed per proficiency, level/residual XP final, milestones/skills/opened nodes unioned) onto the **welcome-back summary**, which carries the gains and opened nodes (reusing the live push's models so the gate renders offline and live the same way) and counts them in `HasProgress`.

## Milestone & open effects (derived openness, persisted only as skill grants)

A proficiency's **openness is derived, never stored**: a node is open iff it has accrued XP, has its previous in-path tier maxed, or has all its cross-path prerequisites maxed. There is no per-player "opened" flag and no migration — the same level + structure derivation the client uses (spike #982 decision 14), so XP routing needs no open-gate (a path opens the moment its activity-keyed effect trains it). The **only** persisted effect of the milestone/open layer is therefore the idempotent **skill grant** via `Player.UnlockSkill`, applied inside `ProficiencyRewardService` right where a tier crosses a milestone: a crossed milestone grants its `RewardSkillId`. Maxing a tier also **opens** its in-path successor (`Path.NextTier`) and any cross-path gateway whose prerequisites are now all maxed (`IProficiencies.DependentsOf`), but opening is **notification-only** — it grants no skill (the seed-skill machinery was removed in spike #1125; a freshly-opened tier's native skill is re-homed onto the predecessor tier's max-level milestone reward, or bootstrapped via synthesis for a cross-path/synthesized line). Because the milestone grant is idempotent and fires only on the level/max transition, the offline batch re-running the accrual per battle never double-grants. A character's starting roots are **emergent from its class kit** (spike #1126): the kit's skills open their paths on the first won battle through this same derived-openness rule, so there is no `StartsUnlocked` flag or root-seeding step at creation. The prerequisite graph is validated **acyclic** both at admin-authoring time (`AdminProficiencies.SetPrerequisites`) and as a build-time cache invariant (`ProficiencyPrerequisiteGraph`), since a cycle would soft-lock every node on it. The live push (`ProficiencyXpGainedEvent`) carries the granted skills and opened nodes alongside the XP; the offline path suppresses it.
