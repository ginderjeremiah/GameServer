# Backend Attribute Composition (battle-assembly parity)

This doc covers how a battler's attributes are composed from player state at battle-snapshot time — the modifier-pipeline order and the three frontend↔backend parity surfaces the proficiency and class systems add. It is split from [backend.md](./backend.md) so the main doc stays an overview, with the composition-order detail here next to the parity invariant that governs it.

**Parity invariant (why this detail matters).** Each surface below is composed into the battler from player state at snapshot time on **both** the frontend and the backend, and the backend replays the client's battle as anti-cheat, so the two must produce the **same attributes down to the last bit**. The composition order is part of the contract — float addition isn't associative, so a modifier appended in a different position diverges in the last bits and the no-tolerance replay flags a legitimate player. Every rule is mirrored under `UI/src/lib/battle` and pinned bit-exactly by the named parity tests; it must never diverge. See [game-design.md](./game-design.md) for the player-facing mechanics and [backend-battle.md](./backend-battle.md) for the tick-by-tick runtime that consumes these attributes.

## Proficiency attribute bonuses (frozen in the battle snapshot)

A proficiency's per-level/milestone payouts pay out as **attribute bonuses** through the same modifier pipeline as stat allocations and gear: each authored `ProficiencyModifier` becomes an `AttributeModifier` with the new `EAttributeModifierSource.Proficiency` source, and a player at level *L* contributes the cumulative bonus of every authored payout level at or below *L* (`Proficiency.ModifiersForLevel`). This is the **one frontend↔backend parity surface** the proficiency system adds — both sides compose the bonus from player state at snapshot time — so the rule is mirrored in `UI/src/lib/battle/proficiency-modifiers.ts` and pinned by `ProficiencyBonusParityTests` / `proficiency-bonus-parity.test.ts`. **The composition order is part of the contract:** the proficiency modifiers must sit with the base set (stat allocations + gear) *before* the static engine modifiers (`StaticAttributeModifiers`), so the additive accumulation order — and therefore the result down to the last bit — is identical on both sides. The backend gets this for free (`BattleSnapshot.GetModifiers` concatenates the proficiency modifiers onto the base set, and `AttributeCollection` appends the statics last); the frontend `Battler` feeds them through `BattleAttributes.setData` for the same reason rather than appending them after the statics. The parity scenarios assert **bit-exactly** (a tolerance would hide a last-bit ordering divergence) and include a derived attribute that carries a static additive base (MaxHealth) to pin that order.

The bonuses are **frozen into `BattleSnapshot` at battle start** (`ProficiencyLevels`), exactly like the captured stat allocations and gear — a level gained while idling takes effect on the next battle, and an offline window fights at a stationary power even as its victories accrue XP. Because proficiency progress lives on the separate `PlayerProgress` aggregate (not the battle aggregate), `BattleService` reads the levels through the lean `IPlayerProgressRepository.GetProficiencies` accessor when it builds the snapshot and threads a proficiency resolver into `BattleSnapshot.ToBattler`/`GetModifiers` (so the same bonus also flows into the `DefeatRewards` power measurement — a proficiency-boosted player is measured at its true power). Until proficiencies are authored and opened, every player's captured set is empty, so the bonus is a no-op and the anti-cheat replay is unchanged. **The live frontend wiring** (reading the player's proficiency levels + the reference per-level modifiers into the live player battler at assembly) ships with the client proficiency data delivery (#1119) — the `playerProficiencies` store composes the bonuses via the shared `proficiency-modifiers.ts` (#1117) and the battle engine feeds them through the same per-spawn change-detection as equipment/attributes; #1117 ships the shared computation and the parity vectors.

## Class locked base & the hybrid attribute model (spike #1126 area D)

A character's attributes are a **hybrid**: a level-scaled, non-reallocatable **class locked base** plus a reduced **free pool** of manually-allocated stat points. The locked base is the class's `AttributeDistributions` resolved at `BaseAmount + AmountPerLevel × level` — the same shape (and `EAttributeModifierSource.AttributeDistribution` source) an enemy uses — derived purely from `(class, level)` and **never stored**. It assembles into the battler in `BattleSnapshot.GetModifiers` (the captured `ClassId` resolved through a class resolver threaded by `BattleService`), composed **after gear and before the proficiency bonuses, ahead of the static engine modifiers** — a fixed position, since float addition isn't associative. This is a **second frontend↔backend parity surface**: the rule is mirrored in `UI/src/lib/battle/class-modifiers.ts` and pinned bit-exactly by `ClassLockedBaseParityTests` / `class-locked-base-parity.test.ts` (with a derived-attribute scenario to pin the order, and a fractional-per-level scenario to pin the arithmetic). **`AttributeDistribution.GetDistributionModifier` evaluates `BaseAmount + AmountPerLevel × level` in `double`** (casting each authored `decimal` operand first) rather than in decimal-then-cast, so it is bit-identical to the frontend's IEEE-754 recompute for *any* authored value — a fractional per-level term would otherwise diverge in the last bits between decimal and double, and the no-tolerance replay would flag a legitimate player. This applies to the enemy distribution (also client-recomputed) as much as the class locked base. The free pool is the per-level stat-point grant (`GameConstants.StatPointsPerLevel`), **reduced** because the locked base now supplies the bulk of attribute growth; `NewPlayerFactory` seeds the free-pool allocations **empty** (the locked base provides the starting spread — seeding it too would double-count). Offline, the locked base is frozen at the snapshot's window-start level like every other captured input (the window is stationary).

Unlike the proficiency bonus (a no-op until proficiency content existed), **every character has a class with a locked base, so the live frontend battler must compose it too or the anti-cheat replay diverges** — the wiring ships here, not deferred. The player's class distributions ride the `PlayerData` payload (`LockedBaseDistribution`, resolved from the class catalogue in the login projection), and `PlayerManager` memoises `classLockedBaseModifiers(distribution, level)` into the live battler (recomputed on level-up, since the base is level-scaled) in the same `additionalModifiers` slot as the proficiency bonuses, before the statics. The distribution is delivered with the player (not via the `GetClasses` reference set) because a character's class is fixed and its fingerprint is the only class data the battler needs; the class **reference** delivery + the create-character picker remain #1225's scope. The attributes screen rendering locked-base-vs-free-pool distinctly is also #1225 (it needs no further data).

## Class signature passive (spike #1126 area E)

A class's **signature passive** is the durable combat-identity bonus it feeds through the same attribute pipeline — flat (`Amount`) or **attribute-scaled** (`Amount + ScalingAmount × value(ScalingAttribute)`, mirroring how a skill effect scales off its caster). It resolves to a single `AttributeModifier` (`ClassSignaturePassive.GetModifier`) tagged with the new `EAttributeModifierSource.Class` source, and is composed into the battler **last** — after the free pool, locked base, proficiency bonuses, **and the static engine modifiers** — so an attribute-scaled passive reads its scaling attribute's *fully-resolved* value (the snapshot state a V1 passive sees, so it never depends on itself) and lands in the same per-attribute apply order on both sides. The backend adds it in `BattleSnapshot.ToBattler` after building the `AttributeCollection`; the frontend battle engine adds it after building the battler (a data-less #811 re-arm keeps it, so it's composed only on a full rebuild). This is a **third frontend↔backend parity surface**: the rule is mirrored in `UI/src/lib/battle/class-modifiers.ts` (`classSignaturePassiveModifier`) and pinned bit-exactly by `ClassSignaturePassiveParityTests` / `class-signature-passive-parity.test.ts` (flat, core-scaled, derived-scaled, self-scaling, multiplicative, and a fractional-scaling case to pin the `double` arithmetic). Like the locked base, the passive rides the `PlayerData` payload (`SignaturePassive`, resolved from the class in the login projection) so the live battler can compose it. It is also **included in the `DefeatRewards` power measurement** — via `BattleSnapshot.GetModifiersWithSignaturePassive`, which resolves the passive against a transient collection and appends it to the snapshot's modifier list — so a flat (or core-additive-scaled) signature passive counts toward power exactly like the class locked base, rather than a class-identity bonus being silently dropped from the reward heuristic (`SumCoreAttributes` still naturally omits the derived-attribute / multiplicative cases, as it does for every source). The read-only attribute-breakdown inspector also surfaces the `Class` source (and the locked base / proficiency bonuses), assembling them in the same order the battler does so its totals match the simulation.
