# Backend Admin Tooling & User Administration

This doc covers the admin/identity surface: the Content Authoring admin API and its persistence patterns, user administration (archive vs ban), and user-connection tracking. It is split from [backend.md](./backend.md) so the main doc — which a gameplay-focused newcomer reads first — stays a get-up-to-speed overview. The admin-authorization gate (the `Admin` role) and the high-level auth/JWT decisions these build on remain in [backend.md](./backend.md) (_Authentication_ and _Access Roles and Admin Authorization_); the auth mechanics live in [backend-auth.md](./backend-auth.md).

## Admin Tools API surface (Content Authoring context)

The frontend admin tools are a single entity-driven Workbench (see `docs/frontend.md`) that saves a whole record (identity + related collections) at once — but the backend keeps persisting each relationship through its own admin endpoint; the Workbench just orchestrates them.

Reference-data administration is modelled as a distinct **Content Authoring** bounded context (its own model over the shared database, as namespaces within the existing projects rather than new projects). The EF entity is this context's persistence shape and is an implementation detail of `Game.DataAccess` — it never surfaces in `Game.Api`. (User administration is a separate Identity / User Admin context — see [User Administration](#user-administration-archive-vs-ban).)

- **Admin controllers are thin HTTP adapters** — accept an admin write contract, call an admin repository, and hand back its result. They reference no entities. **Every admin write — top-level save or relationship setter — reports through one `AdminSaveResult`** (success, or a user-facing failure message the data tier produces, e.g. `"Enemy not found."`); it implicitly converts to an `ApiResponse`, so each endpoint returns the result directly and the "not found" copy and the success/error mapping live in one place instead of being re-copied per endpoint.
- **Admin write contracts live in `Game.Abstractions.Contracts.Admin`** and reuse the reference-data read contracts for the whole-record payloads, so reading and authoring share one serialized shape.
- **Admin repositories own the persistence** (`Game.DataAccess/Repositories/Admin`): the change-set ordering, fresh-entity construction, and the entity-store calls, reusing the cached reference repos for existence/diff lookups. A relationship setter (or an identity-level edit) targeting a missing *owner* returns a not-found rejection the controller surfaces — and identity-level saves validate the whole change set up front so they reject *before* staging anything, since the commit filter would otherwise persist the rest of an accepted batch. That identity-level **Edit-existence** rejection lives in the change-set processor itself (each save just supplies the record-set's existence predicate — a cache lookup or the zero-based-id bounds check), so a new identity-level write path can't silently forget it; a record-set-specific FK reference an *Add* carries (e.g. a zone's boss/unlock) is still validated by that save before delegating. The two kinds of related-collection write share a helper apiece: **explicit change sets** (the client sends per-item Add/Edit/Delete) go through a change-set processor, while **full desired-set** writes (the client sends the whole intended collection) go through a reconciler that diffs against the existing rows to derive the delete/update/insert. Both build fresh, navigation-free entities so a cached `Include(...)` graph is never dragged into the change tracker. The attribute-keyed change-set setters (item/item-mod attributes, skill damage multipliers) share one such helper, and the item/item-mod tag setters route through the reconciler. A top-level Delete against a *retire-only* reference set (enemies, items, item mods, skills, zones, challenges — these are retired, not deleted) is unsupported input, so the change-set processor rejects it as a graceful business failure (the same `AdminSaveResult` → 400 path as a not-found edit) rather than throwing an opaque 500. A change set that names the same key more than once is likewise rejected up front (it would otherwise double-track the entity into an opaque EF 500) — the same standard the reconciler holds; the guard covers the value-tracked Edit/Delete changes (an Add's id is a store-generated sentinel, so two Adds never collide), while the attribute-keyed processor dedups the whole batch since an attribute key is meaningful for an Add too.
- **Child-collection edits/deletes of a missing *member* are guarded no-ops, by contract** (distinct from the identity-level edit-of-missing rejection above). An explicit-change-set setter guards its Edit/Delete by the owner's current membership, so a change naming a child the owner doesn't have is reconciled away — neither applied nor rejected. This is deliberate: unlike a blind identity-level edit (whose 0-row `UPDATE` throws), the guard can't crash, and the change set is a delta the client computed against a baseline it just read (the Workbench re-reads after every save), so an edit/delete of an already-absent child settles idempotently to the same end state. The full desired-set reconciler reaches the same end state by construction (it only ever deletes what's absent from the desired set and inserts what's new).
- **Cache reload stays a cross-cutting filter, discovered through an interface.** Every admin write reloads the in-memory reference caches (an awaited build-then-swap — see [Reference-data cache reload](./backend.md#reference-data-cache-reload-build-then-swap)) so the next read serves the freshly written data. Each cache holder is registered under a common reloadable-cache interface, so the filter resolves them all from DI and a newly-added cache is covered automatically. The filter touches no entities, so it stays at the presentation edge; controllers apply it through an attribute that bakes in an ordering so the reload runs *after* the commit filter persists the write.

Two persistence gotchas the admin repositories respect (and any new admin write must too):

- **Updates must not be inferred from the key value.** The entity store tracks the passed entity as `Modified` directly rather than going through `DbContext.Update`, which infers Add-vs-Edit from whether the key is "set" — that misfires for the zero-based-identity tables (see [Reference Data](./backend.md#reference-data)), where the first row's `Id == 0` is the CLR/identity-column default, so EF would treat an edit of record 0 as a brand-new insert and silently duplicate it. `GameContext.SaveChangesAsync` carries a second, belt-and-suspenders guard — a zero-based-identity fixup that forces the literal `0` back onto any key/FK EF marked temporary by reading `0` as an unset store-generated value. Its FK branch is live (a non-nullable `int` FK referencing record 0 of a store-generated principal); its PK branch (a non-`Added` record whose `Id == 0`) is, in EF Core 10, an unreachable no-op — a `Modified`/`Deleted`/`Unchanged` entry's key cannot be temporary, EF rejects that state outright — but is retained deliberately as defence against older/future EF semantics or an unforeseen attach path that would otherwise UPDATE the wrong row.
- **Don't update/delete cached entities directly.** Reference data is read with `AsNoTracking().Include(...)`, so a fetched child carries a loaded back-reference to its parent and siblings; passing that instance to update/delete would drag the whole graph into the change tracker and re-insert it. Relationship endpoints therefore build fresh, navigation-free entities keyed only by their own ids.

## Content-lifecycle retirement guards

Reference records are retired, not deleted (see [backend.md → Retiring reference data](./backend.md#retiring-reference-data-content-lifecycle)). Most retirements are safe, but two would author *live* content into a permanently broken state the runtime won't paper over, so the admin save rejects them up front (reported via `AdminSaveResult.Failure`; the Workbench shows the same rule as a validation warning, but the backend is authoritative):

- **No spawnable enemies in a live zone.** `AdminEnemies.SaveEnemies` rejects retiring an enemy that is the **last active spawn of a live (non-retired) zone**, so a live zone is never authored into the no-spawnable-enemies state — the workflow is to retire the *zone* first, then its enemies. It scopes to *live* zones deliberately; the runtime relocation (a player in a now-unviable zone is moved to the nearest viable one) is the defense-in-depth backstop for the cases it doesn't cover (a player standing in, reconnecting into, or persisted against an already-retired zone).
- **No retiring a path that gates a live proficiency gateway.** `AdminPaths.SavePaths` rejects newly retiring a path when one of its proficiency tiers is named as a prerequisite by a **live (non-retired-path) gateway** — the frozen track never accrues, so that prerequisite can never max and the live gateway could never open (a permanent soft-lock). The guard fires only on the live → retired transition (reinstating only ever *unblocks*, so it is unguarded); the workflow is to retire the dependent gateway's path first, or remove the prerequisite. This complements the runtime gateway-open suppression (a retired path is dropped from the gated side of the prerequisite index), which handles the inverse case but cannot rescue a *live* gateway gated by a now-retired path.

## User Administration (Archive vs Ban)

The admin user-management surface (search the roster, list grantable roles, replace a user's roles, plus the two lifecycle actions below) shares the admin route prefix, but a few conventions differ on purpose:

- **Identity / User Admin is its own bounded context whose repository reads speak contracts, not entities.** It shares the `User` aggregate with the auth/login flow, so it sits alongside it rather than under Content Authoring. The reads return `Game.Abstractions.Contracts.Identity` types (the published read language the Workbench consumes), with the entity→contract projection in `Game.DataAccess`, so the `User`/`Role` entities never surface in `Game.Api` or the login path. The credential-lookup contract is deliberately **not** an `IModel` — it carries the password hash and stays internal to the data + application tiers, never serialized across the API boundary. Account creation is the matching write path: it takes credential + new-player blueprints and builds the linked user+player entity graph in the data tier, so the application layer constructs no entities.
- **Archiving and banning are two distinct, non-destructive states, modeled as nullable timestamps on `User`** (`ArchivedAt`, `BannedAt`). Timestamps double as the flag *and* an audit of when it happened, following the idiomatic soft-delete shape.
  - **Archive = soft delete that frees the username.** An archived user is excluded from login and from username-availability checks, so the username becomes available for a brand-new account. Uniqueness is enforced by a **partial unique index** on `Username` filtered to `WHERE ArchivedAt IS NULL` (see [backend-auth.md → Account/login orchestration](./backend-auth.md#accountlogin-orchestration)): the filter excludes archived rows, so an archived row and a new active row can share a username, while two *active* rows cannot — and a concurrent double-create can't slip past the up-front availability check. The admin roster does *not* hide archived users: its search takes a tri-state `archived` filter (all / active-only / archived-only) so an admin can still find and manage them.
  - **Ban = block while keeping the username reserved.** A banned user still counts toward username availability (so the name can't be reused) and stays visible in the roster. The ban is **enforced at login**: a banned account is rejected even with correct credentials, but only *after* the password verifies, so an anonymous probe can't enumerate ban status — only the account owner learns they are banned.
- **Role changes take effect on next login**, since roles live in the signed auth token (see [Access Roles and Admin Authorization](./backend.md#access-roles-and-admin-authorization)).
- **Privileged actions self-protect against admin lockout.** The controller supplies the acting admin's id (from the validated token, via the session) to the data tier, which rejects an admin banning/archiving their own account, stripping their own Admin role, or removing the Admin role from the last remaining admin (an admin with no other *usable* admin holding the role). A **usable admin** is one that is neither archived nor banned — i.e. one that can still log in to recover the instance; both lockout guards (role removal and archive/ban) share this single definition of the surviving-admin pool. The decision is a pure rule in `Game.Core` (`AdminLockoutPolicy`, unit-tested); the data tier only gathers the facts (the target's roles, whether another usable admin remains) and applies it. The last-admin check-and-act is **atomic**: a role removal that drops Admin locks the Admin role row (`FOR UPDATE`), re-reads the surviving-admin fact under that lock, and commits its mutation in-tier (rather than deferring to the request unit of work, the same exception [account creation](./backend-auth.md#accountlogin-orchestration) makes), so two concurrent demotions of different admins can't both observe another admin remaining and leave the system admin-less.

## User Connection Tracking

Connections are tracked with three tables rather than a full ledger, each keyed by the signal that actually distinguishes it:

- **`BrowserInfo` — the user-agent profile, deduplicated by user-agent** (unique index). Holds only the server-observable signals on every request (the user-agent and the low-entropy client-hint headers), and is genuinely shared across the many devices that report the same user-agent. Client hints fill in only when missing, so a later request with fewer hints never clears earlier values.
- **`Device` — the device, deduplicated by fingerprint hash** (unique index — the most device-unique signal). Holds device-specific data: the fingerprint, the reported capabilities, and an FK to its `BrowserInfo`. The fingerprint is computed client-side; the capabilities are reported by the frontend after login via an authenticated endpoint.
- **`UserLogin` — a connection origin**: a surrogate key plus a unique index on (user, IP, device); the last-connection timestamp is updated in place for that combination rather than appended. (A unique index, not a composite PK, keeps the store-generated device key out of the dependent's primary key, which EF can't cleanly propagate when inserting both rows together.)

Keying `UserLogin` on the device means the per-request tracker needs the fingerprint on **every** request, but browsers don't send one. So the frontend computes a deterministic fingerprint once (a hash of stable client signals, cached in `localStorage`) and attaches it as a header on every authenticated request — the deliberate trade-off chosen over storing device data on a user-agent-keyed row (where two devices sharing a user-agent would overwrite each other).

Recording happens in middleware after the session loads, so it fires on authenticated **HTTP** requests only — not per WebSocket command (those never re-enter the request pipeline, and a handshake can't set the custom header anyway). A request without the fingerprint header is skipped. Recording is **best-effort** — failures are logged and swallowed so tracking never affects the response. The data tier owns the commit (it runs outside the per-action commit filter) and the get-or-create rows are conflict-tolerant: two concurrent connections from the same new device both read "absent" and insert, so the dedup unique indexes turn the lost insert into a unique violation the repository catches, then rebuilds against the now-committed rows and saves in place (the same catch-and-reload as account creation). The write is deduped per (user, IP, device) by a 5-minute in-memory memo (populated only after a successful save), so steady-state traffic costs a cache lookup rather than a DB round-trip per request.
