# Backend Admin Tooling & User Administration

This document covers the admin/identity surface: the Content Authoring admin API and its persistence patterns, user administration (archive vs ban), and user-connection tracking. It is split out from [backend.md](./backend.md) so the main doc — which a gameplay-focused newcomer reads first — stays a get-up-to-speed overview. The admin-authorization gate (the `Admin` role) and the auth/JWT pipeline these build on remain in [backend.md](./backend.md) (_Authentication_ and _Access Roles and Admin Authorization_).

## Admin Tools API surface (Content Authoring context)

The frontend admin tools were consolidated into a single entity-driven Workbench (see `docs/frontend.md`). That UI saves a whole record (identity + related collections) at once, but the backend keeps persisting each relationship through its own admin endpoint — the Workbench just orchestrates them.

Reference-data administration is modelled as a distinct **Content Authoring** bounded context (its own model over the shared database), implemented as namespaces within the existing projects rather than new projects. The EF entity is this context's persistence shape and is an implementation detail of `Game.DataAccess` — it never surfaces in `Game.Api`. (User administration is the separate Identity / User Admin context — see [User Administration](#user-administration-archive-vs-ban).)

- **Admin controllers are thin HTTP adapters.** `Game.Api/Controllers/Admin/*` accept an admin write contract, call an admin repository, and map the result to an `ApiResponse`. They reference no `Game.Abstractions.Entities` and no `IEntityStore`.
- **Admin write contracts live in `Game.Abstractions.Contracts.Admin`** (`Change<T>`, `EChangeType`, `AddEditAttributesData`, `SetTagsData`, and the `Set*Data` relationship DTOs). The whole-record `Change<T>` payloads reuse the reference-data read contracts in `Game.Abstractions.Contracts` (`Enemy`/`Item`/`Skill`/`Zone`/`Tag`/`ItemModSlot`/`Challenge`), so reading and authoring share one serialized shape. `Challenge` is the single read+write contract: the `GetChallenges` socket maps the gameplay domain `Challenge` (which `IChallenges` still serves for challenge evaluation) onto it, and the admin write path persists it.
- **Admin repositories own the persistence.** The `IAdmin*` interfaces (`Game.Abstractions/DataAccess/Admin`) are implemented in `Game.DataAccess/Repositories/Admin`. They encapsulate the change-set ordering (`ChangeSetProcessor`, also relocated here), the fresh-entity construction, and the `IEntityStore` calls; they reuse the cached reference repos for existence/diff lookups. A relationship setter returns `false` when its target record is missing, and the controller maps that to the user-facing "not found" message. **An identity-level `Save*` edit of a missing record is the same kind of not-found rejection, not a silent no-op:** `SaveItems`/`SaveItemMods` validate the whole change set up front (any `Edit` whose id isn't resolvable) and apply nothing — rejecting before staging, because the `CommitFilter` would otherwise persist the rest of an accepted batch. (The other identity-level saves don't yet front-validate edits — they let EF throw on the 0-row update; see #646.) The two kinds of related-collection write share a helper apiece, so each setter is just a declarative mapping to entity operations: the **explicit change set** ones (the client sends per-item Add/Edit/Delete — `Set*Attributes`, `SaveModSlots`, skill multipliers/effects) go through `ChangeSetProcessor`, while the **full desired-set** ones (the client sends the whole intended collection — `SetAttributeDistributions`/`SetSkills`/`SetSpawns`/`SetEnemies`) go through `ChildCollectionReconciler`, which diffs against the existing rows with `HashSet` lookups to derive the delete/update/insert (a pure join row like `EnemySkill` simply omits the update). Both build fresh, navigation-free entities for every write so a cached `Include(...)` graph is never dragged into the change tracker.
- **Cache reload stays a cross-cutting filter, discovered through `IReloadableReferenceCache`.** `AdminCacheReloadFilter` reloads the in-memory reference caches after every admin write (an awaited build-then-swap — see [Reference-data cache reload](./backend.md#reference-data-cache-reload-build-then-swap)) so the next read serves the freshly written data. Each set's singleton snapshot holder implements `IReloadableReferenceCache` (`Game.Abstractions.DataAccess`) and is registered under that interface, so the filter (and the test bases, via `Game.TestInfrastructure`'s `ReferenceCacheReloader.ReloadAllAsync`) resolves `IEnumerable<IReloadableReferenceCache>` from DI instead of hand-maintaining a list of the caches — a newly-added cache is covered automatically (#312). The filter never touches entities, so it stayed at the presentation edge rather than moving into the repositories; controllers apply it through the `[ReloadReferenceCaches]` attribute, which bakes in `Order = int.MinValue` so the reload runs after `CommitFilter` persists the write and a new controller can't omit the ordering.

Two persistence gotchas the admin repositories respect (and any new admin write must too):

- **Updates must not be inferred from the key value.** `IEntityStore.Update` tracks the passed entity as `Modified` directly (`_context.Entry(entity).State = EntityState.Modified`) instead of going through `DbContext.Update`. `DbContext.Update` infers Add-vs-Edit from whether the key is "set", which misfires for our zero-based-identity tables (see [Reference Data](./backend.md#reference-data)): the first row's `Id == 0` is the CLR/identity-column default, so EF would treat an edit of record 0 as a brand-new insert and silently duplicate it.
- **Don't update/delete cached entities directly.** Reference data is read with `AsNoTracking().Include(...)`, so a fetched child (e.g. a `SkillDamageMultiplier`) carries a loaded back-reference to its parent and siblings. Passing that instance to `Update`/`Delete` would drag the whole graph into the change tracker and re-insert it. Relationship endpoints therefore build fresh, navigation-free entities keyed only by their own ids (as `SetEnemySkills`/`SetZoneEnemies`/`SetSkillMultipliers` do) rather than mutating the cached objects.

## User Administration (Archive vs Ban)

`AdminUsersController` (under `Game.Api/Controllers/Admin`) exposes the admin user-management surface: search the roster (`GetUsers`), list the grantable roles (`GetRoles`), replace a user's roles (`SetUserRoles`), and the two lifecycle actions below. It shares the `/api/AdminTools/[action]` route prefix like the other admin controllers, but a few of its conventions differ on purpose:

- **Identity / User Admin is its own bounded context, and its repository reads speak contracts, not EF entities.** This context shares the `User` aggregate with the auth/login flow, so it sits alongside it rather than under Content Authoring. `IUsers`/`IRoles` return `Game.Abstractions.Contracts.Identity` types: the admin roster (`SearchUsers` → `AdminUser`, with nested `PlayerSummary`/`Role`; `GetRoles` → `Role`) is the published read language the Workbench consumes, and `GetUser` returns an `AccountCredentials` lookup for the login use case. The entity → contract projection lives in `Game.DataAccess` (`Users`/`Roles`), so the `User`/`Role` EF entities never surface in `Game.Api` or in the `AccountService` login path. `AccountCredentials` is deliberately **not** an `IModel`: it carries the password hash and is internal to the data tier + application layer, never serialized across the API boundary. Account creation is the matching **write** path: `IUsers.CreateAccount(NewAccount, NewPlayer)` takes a `NewAccount` credential blueprint (also a non-`IModel` Identity contract) plus the new-player blueprint and builds the linked user+player entity graph in the data tier, so `AccountService` constructs no entities (#137).
- **Archiving and banning are two distinct, non-destructive states, modeled as nullable timestamps on `User` (`ArchivedAt`, `BannedAt`).** Timestamps (rather than booleans) double as the flag _and_ an audit of when the action happened, and follow the idiomatic soft-delete shape.
  - **Archive = soft delete that frees the username.** An archived user is excluded from login (`IUsers.GetUser`) and from username-availability checks (`IUsers.CheckIfUsernameExists`, which gates account creation). Because the username check ignores archived rows, the username becomes available for a brand-new account — there is intentionally **no DB unique index** on `Username`, so an archived row and a new active row can share a username; the active one is the single non-archived match the queries resolve to. The admin roster, by contrast, does **not** hide archived users by default: `SearchUsers`/`CountUsers` take a tri-state `archived` filter (`null` = all users, `false` = active only, `true` = archived only), so an admin can still find and manage archived accounts.
  - **Ban = block while keeping the username reserved.** A banned user still counts toward username availability (so the name cannot be reused) and remains visible in the admin roster for later management. Surfacing the ban to the user at login time is intentionally out of scope for now.
- **Role changes take effect on next login.** As noted under [Access Roles and Admin Authorization](./backend.md#access-roles-and-admin-authorization), roles live in the signed auth token, so a `SetUserRoles` change is reflected only after the affected user logs in again.

## User Connection Tracking

Connections are tracked with three tables rather than a full ledger, each keyed by the signal that actually distinguishes it:

- **`BrowserInfo` — the user-agent profile, deduplicated by `UserAgent`** (unique index). Holds only the server-observable signals carried on every request: the user-agent and the low-entropy client-hint headers (`Sec-CH-UA`/`-Mobile`/`-Platform`). It is genuinely shared across the many devices that report the same user-agent. Client hints are filled in only when missing, so a later request with fewer hints never clears earlier values.
- **`Device` — the device, deduplicated by `DeviceFingerprintHash`** (unique index). This is the most device-unique signal, so device-specific data lives here: the fingerprint, the capabilities (`deviceMemory`/`hardwareConcurrency`), and a FK to its `BrowserInfo` (the fingerprint incorporates the user-agent, so that mapping is stable). The fingerprint is computed client-side; the capabilities are reported by the frontend after login (they aren't on a regular request) via the authenticated `Login/DeviceInfo` endpoint.
- **`UserLogin` — a connection origin.** A surrogate identity key plus a unique index on `(UserId, IpAddress, DeviceId)`; `LastConnection` is updated in place for that combination rather than appended. (A unique index, not a composite PK, keeps the store-generated `Device` key out of the dependent's primary key — which EF cannot cleanly propagate when inserting both rows together.)

Keying `UserLogin` on the device means the per-request tracker needs the fingerprint on **every** request, but browsers don't send one on their own. So the frontend computes a deterministic fingerprint once (a SHA-256 of stable client signals, cached in `localStorage`) and the request layer attaches it as an `X-Device-Fingerprint` header on every authenticated request. This is the deliberate trade-off chosen over storing device data on a user-agent-keyed row (which would let two devices sharing a user-agent overwrite each other).

- **Recording happens in `LoginTrackingMiddleware`, after `SessionLoaderMiddleware`,** so it fires on authenticated **HTTP** requests (not per WebSocket command — those never re-enter the request pipeline, and a WebSocket handshake can't set the custom header anyway, so it's skipped). A request without the fingerprint header is skipped (no device to key on). Recording is **best-effort**: failures are logged and swallowed so tracking never affects the response. Because it runs outside the per-action `CommitFilter`, the tracking service commits its own unit of work. The trade-off is a small write per authenticated HTTP request; acceptable while most game traffic is on the socket, and throttleable later if HTTP volume grows.
